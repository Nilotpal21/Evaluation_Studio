# Billing, Quotas & Operations

Monitor your workspace's resource consumption, manage your plan, and operate the Agent Platform 2.0 at scale. This page covers billing and usage tracking, platform limits, customer-managed encryption keys, analytics, self-hosted deployment, and upgrade procedures.

## Billing & Usage

**Required role:** Owner or Admin to manage billing; Operator to view usage

### Viewing your current plan

Go to **Settings > Account > Billing**. The top of the page shows your current plan, billing cycle, and renewal date.

### Credit balance

If your workspace uses a credit-based billing model, the billing page displays:

- **Total allocated credits** -- Credits available for the current billing period
- **Credits consumed** -- How many credits have been used
- **Remaining credits** -- Available balance
- **Credit burn rate** -- Average daily consumption for the period

Credits are consumed by LLM API calls, session activity, and feature usage. The credit cost varies by model tier and operation type.

#### Low credit alerts

Set up alerts to be notified when credits are running low:

1. Go to **Settings > Account > Billing**.
2. In the **Alerts** section, click **Add alert**.
3. Configure the alert:
   - **Type** -- Select "Credit low"
   - **Threshold** -- The credit percentage at which to trigger (for example, 20% remaining)
   - **Channel** -- Email or webhook
   - **Target** -- The email address or webhook URL to notify
4. Click **Save**.

### LLM usage analytics

The billing page includes detailed analytics on LLM consumption:

**Summary metrics** displayed at the top:

- **Total sessions** -- Number of agent sessions in the selected period
- **Total tokens** -- Input and output tokens consumed across all models
- **Total estimated cost** -- Calculated cost based on model pricing
- **Active members** -- Team members who triggered LLM calls

**Breakdowns available:**

- **Daily usage trends** -- Time-series chart of token consumption. Use it to identify usage spikes, spot trends, and verify that usage drops after decommissioning agents.
- **By provider** -- Pie chart showing token distribution across LLM providers.
- **By model** -- Input tokens, output tokens, average latency, error rate, and estimated cost per model.
- **By agent** -- Identifies agents that could benefit from a lower-cost tier, have unexpectedly high token usage, or are inactive.

### Usage thresholds and alerts

Configure alerts to prevent cost overruns:

- **Usage threshold alerts** -- Notify when a percentage of the monthly token budget is reached (e.g., 80%).
- **Health degraded alerts** -- Notify when LLM provider connections experience degraded performance.
- **Feature limit alerts** -- Notify when approaching plan limits for sessions, members, or projects.

Configure alerts at **Settings > Account > Billing > Alerts**.

### Billing periods

Usage is tracked in billing periods that align with your subscription cycle:

- **Monthly billing** -- Usage resets on your billing anniversary date.
- **Annual billing** -- Monthly usage is tracked within the annual period for budgeting, but limits are enforced monthly.

Each billing period records total sessions, messages, tokens, tool calls, estimated cost, peak concurrent sessions, and invoice status.

### Plan upgrades

To upgrade your plan:

1. Go to **Settings > Account > Billing**.
2. Click **Upgrade plan** or **Compare plans**.
3. Review the available tiers and their features.
4. Select the plan you want.
5. Confirm the upgrade.

Plan upgrades take effect immediately. The billing difference is prorated for the remainder of the current billing period.

> **Note:** Plan downgrades take effect at the end of the current billing period. Features exclusive to your current plan remain accessible until the period ends.

### Cost optimization tips

- **Use the right tier for the job.** Route simple classification tasks to Fast-tier models instead of Powerful-tier models. The cost difference can be 10x or more.
- **Set daily token budgets.** Even if your monthly budget is generous, a daily limit prevents a single runaway agent from consuming a month's worth of tokens in a day.
- **Review agent-level usage monthly.** Identify agents with high per-session token counts and consider whether their prompts can be optimized or their context windows reduced.
- **Enable conversation sliding windows.** For long-running agents, sliding windows limit the conversation history sent to the LLM, reducing input token costs.
- **Archive inactive projects.** Inactive projects with deployed agents may still consume resources from scheduled tasks or webhook-triggered sessions.

## KMS (Bring Your Own Key)

The Key Management Service (KMS) feature lets you control the encryption keys used to protect your workspace's sensitive data. Instead of relying on platform-managed keys, you provide your own key material from a supported cloud provider.

**Required role:** Owner

> **Plan requirement:** KMS is available on Enterprise plans.

### Why bring your own key

By default, the Agent Platform 2.0 encrypts all sensitive data using platform-managed encryption keys. Bring Your Own Key (BYOK) gives you:

- **Key custody** -- You own and control the master encryption key. You can revoke it at any time.
- **Compliance** -- Meet regulatory requirements that mandate customer-managed encryption keys (SOC 2, HIPAA, FedRAMP).
- **Key rotation control** -- Rotate keys on your schedule, according to your organization's cryptographic policies.
- **Audit trail** -- Your cloud provider logs every key usage event, giving you an independent audit trail.

### Encryption architecture

The platform uses an envelope encryption model:

1. **Key Encryption Key (KEK)** -- Your master key, stored in your cloud provider's key management service. The platform never has access to the raw key material.
2. **Data Encryption Keys (DEKs)** -- Short-lived keys generated by the platform to encrypt individual data items. Each DEK is wrapped (encrypted) by your KEK before storage.
3. **Encryption at rest** -- When the platform encrypts a secret, it generates a DEK, encrypts the data, wraps the DEK with your KEK, and stores both. Decryption reverses the process.

DEKs are scoped to a specific project and environment, with configurable rotation intervals.

### Supported KMS providers

| Provider             | Key type                | Region support      |
| -------------------- | ----------------------- | ------------------- |
| **AWS KMS**          | Symmetric CMK (AES-256) | All AWS regions     |
| **Azure Key Vault**  | RSA or AES keys         | All Azure regions   |
| **Google Cloud KMS** | Symmetric AES-256 keys  | All GCP regions     |
| **External KMS**     | Any REST-compatible KMS | Self-hosted or SaaS |

### Setting up KMS

**Step 1: Create a key in your provider**

Create a symmetric encryption key in your cloud provider's KMS:

- **AWS KMS:** Create a Customer Managed Key (CMK). Note the Key ID and region. Create an IAM policy granting `kms:Encrypt`, `kms:Decrypt`, `kms:GenerateDataKey`, and `kms:DescribeKey` permissions.
- **Azure Key Vault:** Create a Key Vault and generate a key. Note the Key Vault URL and key name. Grant the Agent Platform 2.0 service principal the "Key Vault Crypto User" role.
- **Google Cloud KMS:** Create a key ring and key. Note the resource name. Grant the Agent Platform 2.0 service account the `cloudkms.cryptoKeyEncrypterDecrypter` role.

**Step 2: Configure KMS in the Agent Platform 2.0**

1. Go to **Settings > Team > KMS**.
2. Click **Configure KMS**.
3. Select your provider.
4. Enter the key details:
   - **Key ID** -- The identifier for your key (ARN for AWS, URL for Azure, resource name for GCP)
   - **Region** -- The region where the key is hosted
   - **Authentication** -- Service account credentials or role-based access configuration
5. Click **Validate connection**.
6. If validation succeeds, click **Enable KMS**.

**Step 3: Migration**

When you enable KMS for the first time, the platform re-encrypts existing secrets and credentials using your key. Existing DEKs are re-wrapped with your new KEK. New encryption operations use your key immediately. The migration status is visible on the KMS settings page.

### Key rotation

Rotate your encryption key periodically to limit the impact of key compromise.

**Automatic rotation:** If your cloud provider supports automatic key rotation (AWS KMS and GCP KMS do), enable it in your provider's console. The platform automatically detects the new key version.

**Manual rotation:**

1. Create a new key version in your cloud provider.
2. Go to **Settings > Team > KMS**.
3. Click **Rotate key**.
4. Enter the new key version details.
5. Click **Initiate rotation**.

The platform starts using the new key version for all new encryption operations, re-wraps existing DEKs in the background, and marks the old key version as "decrypt only" once complete.

> **Important:** Do not destroy the old key version in your cloud provider until the Agent Platform 2.0 shows all data has been re-encrypted with the new version. Destroying a key version while data still depends on it results in permanent data loss.

### KMS health monitoring

The platform periodically verifies connectivity to your KMS provider. If the health check fails, the platform follows the configured failure policy:

- **Fail-closed** -- All encryption and decryption operations fail. Agents cannot access secrets. Safest option but disrupts service.
- **Graceful degradation** -- The platform continues operating with cached DEKs until the KMS connection is restored.

Configure the failure policy at **Settings > Team > KMS > Failure policy**.

### Disabling KMS

To revert to platform-managed encryption:

1. Go to **Settings > Team > KMS**.
2. Click **Disable KMS**.
3. Confirm the action.

The platform re-encrypts all data using platform-managed keys. Your cloud KMS key is no longer accessed after migration completes.

## Advanced Analytics

Advanced analytics provide deep visibility into agent performance, user sessions, and execution traces.

**Required role:** Owner, Admin, or Operator to view analytics; Member with project access for project-scoped views

> **Plan requirement:** Advanced analytics are available on Professional and Enterprise plans.

### Agent performance dashboard

The agent performance dashboard shows aggregate metrics:

| Metric                        | Description                                              |
| ----------------------------- | -------------------------------------------------------- |
| **Total sessions**            | Number of agent sessions in the selected time range      |
| **Avg. session duration**     | Mean time from session start to last message             |
| **Resolution rate**           | Percentage of sessions that reached a successful outcome |
| **Avg. messages per session** | Mean number of user and agent messages per session       |
| **Total tokens consumed**     | Input + output tokens across all models                  |
| **Total estimated cost**      | Calculated LLM cost for the period                       |
| **Error rate**                | Percentage of sessions that encountered errors           |

Filter by date range, project, agent, or model. Charts include session volume over time, token usage, cost trends, top agents by session count and cost, and error breakdown.

### Session explorer

Browse and search individual sessions to understand user interactions and agent behavior.

1. Go to **Settings > Analytics > Sessions**.
2. Search by session ID, user message content, or agent name. Filter by status, date range, or agent.
3. Click a session to see the conversation transcript, tool calls, model usage, state transitions, and metadata.

### Trace viewer

The trace viewer provides the deepest level of detail, showing the execution trace of a single interaction.

Each trace records a **span tree** -- a hierarchical view of all operations:

- LLM calls (prompt, response, tokens, latency)
- Tool executions (name, input, output, duration)
- Step transitions (for agents with a FLOW section)
- Guardrail evaluations (category, score, decision)
- Knowledge base queries (query text, retrieved chunks, relevance scores)

**Trace search syntax:**

| Operator     | Example                  | Description              |
| ------------ | ------------------------ | ------------------------ |
| `agent:`     | `agent:customer-support` | Filter by agent name     |
| `status:`    | `status:error`           | Filter by outcome        |
| `model:`     | `model:claude-4-sonnet`  | Filter by model used     |
| `duration:>` | `duration:>2s`           | Filter by total duration |
| `tokens:>`   | `tokens:>5000`           | Filter by token count    |

### Custom dimensions

Configure custom dimensions that are automatically extracted from session data:

1. Open the project in Studio.
2. Go to **Project settings > Advanced**.
3. In the **Trace dimensions** section, add session data keys (for example, `customer_tier`, `region`, `channel`).
4. Click **Save**.

Custom dimensions appear in analytics filters, letting you segment metrics by business-relevant attributes.

### Exporting analytics data

- **Session export:** Apply filters in the session explorer, click **Export**, and select CSV or JSON.
- **API access:** Use the Agent Platform 2.0 API with an API key that has the `analytics:read` scope for programmatic access.

### Configuration reference

Configuration is organized into schemas validated at startup. Invalid configuration causes the service to fail fast with a descriptive error. Key configuration categories:

- **Server:** `server.port`, `server.host`, `server.apiUrl`, `server.frontendUrl`
- **Database:** `database.url` (MongoDB connection string), `database.poolSize`
- **Redis:** `redis.url`, `redis.enabled`, TLS settings, cluster mode
- **JWT/Auth:** `jwt.secret` (32+ characters required), `jwt.expiresIn`, Google OAuth credentials
- **LLM:** Default provider, API keys per provider, `llm.defaultModel`, `llm.fastModel`, fallback chain, timeout, cache
- **Encryption:** `encryption.key` (256-bit, required for credential storage), `encryption.algorithm`
- **Security:** PII detection/redaction, rate limiting, IP allowlists
- **Observability:** Trace sampling, metrics, logging level, OTLP endpoint
- **Voice:** Twilio, Deepgram, ElevenLabs, LiveKit, Jambonz credentials
- **Sandbox:** Code execution timeout, Python/JavaScript pod URLs

### Database setup

**MongoDB:** Deploy MongoDB 7.0+ as a replica set (required for change streams and transactions). The platform creates indexes and collections automatically on first startup.

**Redis:** Deploy Redis 7+ with persistence enabled (AOF recommended). For production, use Redis Sentinel or Redis Cluster for high availability. Enable TLS for production deployments.

**ClickHouse (optional):** Deploy for analytics if you need usage dashboards, conversation analytics, or custom metrics.

### Security checklist

Complete before exposing the platform to production traffic:

- TLS on all public endpoints
- 256-bit encryption key configured
- Strong JWT signing secret (32+ characters)
- MongoDB and Redis authentication enabled
- PII detection and redaction enabled
- Rate limiting enabled with appropriate limits
- Admin routes restricted to known IP ranges
- API key rotation configured (default 90 days)
- Outbound domain restriction for tool calls
- Audit logging enabled and exported to SIEM
- Secrets stored in a vault (not in environment variables)

## Upgrade & Migration

Plan and execute platform upgrades with minimal downtime.

### Compatibility matrix

| Platform version | MongoDB | Redis | Node.js | Kubernetes | Breaking changes |
| ---------------- | ------- | ----- | ------- | ---------- | ---------------- |
| 1.x              | 7.0+    | 7.0+  | 20 LTS  | 1.28+      | --               |
| 0.9.x            | 6.0+    | 6.2+  | 20 LTS  | 1.27+      | --               |
| 0.8.x            | 6.0+    | 6.2+  | 18 LTS  | 1.26+      | Config schema v2 |

> **Upgrade path:** Always upgrade one major version at a time. Skipping versions is not supported.

### Pre-upgrade checklist

- Read release notes for every version between your current version and the target.
- Back up MongoDB (`mongodump` or filesystem snapshot) and Redis (`BGSAVE`).
- Verify sufficient disk space (2x current database size recommended).
- Confirm infrastructure meets the target version's requirements.
- Test the upgrade in a staging environment first.
- Schedule a maintenance window if the upgrade includes database migrations.
- Notify affected users of planned downtime.

### Upgrade procedures

**Docker Compose:**

```bash
docker compose down
docker compose pull
docker compose up -d
# Verify health
curl http://localhost:3112/health
curl http://localhost:5173/api/health
```

**Kubernetes (Helm):**

```bash
helm repo update
helm diff upgrade abl-platform abl-platform/abl-platform \
  -f values.yaml --version <target-version>
helm upgrade abl-platform abl-platform/abl-platform \
  -f values.yaml --version <target-version> --wait --timeout 10m
kubectl rollout status deployment/abl-runtime -n agent-platform
```

For zero-downtime rolling upgrades, configure `maxSurge: 1` and `maxUnavailable: 0`. New pods pass readiness checks before old pods terminate. WebSocket connections drain gracefully.

### Database migrations

The platform applies most schema changes automatically on startup (indexes, collection schemas, TTL indexes). For upgrades requiring manual intervention, the release notes include exact commands:

```bash
# Dry run
npx abl-migrate --target <version> --dry-run

# Apply
npx abl-migrate --target <version> --apply

# Verify
npx abl-migrate --target <version> --verify
```

### Breaking changes log

**Version 1.0:**

- Config schema v3: validated by Zod schemas at startup. Invalid configuration causes fail-fast.
- MongoDB required: SQLite/Prisma support removed.
- Redis required for production: session state and distributed locks require Redis.

**Version 0.9:**

- Encryption key must be 256-bit hex string.
- JWT secret must be at least 32 characters.

**Version 0.8:**

- Config schema v2: flat env vars moved to structured schemas. `env-mapping` utility provides backward compatibility.
- Rate limit configuration moved to `security.rateLimiting`.

### Configuration diff tool

Compare configuration between versions:

```bash
npx abl-config-diff --from <current-version> --to <target-version>
```

Outputs added, changed, and removed options plus validation warnings.

### Rollback procedures

**Docker Compose:** Stop services, restore previous image tags, restore database backups if needed, restart.

**Kubernetes:** `helm rollback abl-platform <previous-revision>`, restore database backup if needed.

### Post-upgrade verification

After upgrading, verify:

1. Health checks pass for all services.
2. Existing agents execute correctly (run a test conversation).
3. Channel connections are active and receiving messages.
4. Knowledge bases are searchable.
5. Traces and analytics are recording new data.
6. Scheduled jobs (session cleanup, retention) are running.

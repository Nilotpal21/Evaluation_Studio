# AI Model Configuration

Agent Platform 2.0 is provider-neutral. Your agents can use models from multiple LLM providers, and you can switch between them without changing your agent definitions. This page covers how providers work, how to configure credentials and models, the model resolution chain, fallback chains, and cost management.

**Required role:** Owner or Admin

## How LLM Providers Work

### Provider architecture

Model configuration in the Agent Platform 2.0 follows a layered approach:

1. **Workspace-level models** -- You register AI models with your workspace, providing credentials and setting default parameters. These models become available to all projects in the workspace.
2. **Model tiers** -- Each registered model is assigned to a tier (Fast, Balanced, Powerful, or Voice). Agents reference tiers, not specific models, which lets you swap underlying models without changing agent definitions.
3. **Project-level overrides** -- Individual projects can override which tier maps to which operation (for example, using a Powerful-tier model for extraction tasks that default to Balanced).

### Model resolution chain

When the Runtime executes an agent, it resolves the model through a 5-level priority cascade, stopping at the first match:

| Priority | Source                  | Description                                                   |
| -------- | ----------------------- | ------------------------------------------------------------- |
| 0        | **Deployment override** | Model pinned to a specific deployment (e.g., an A/B test)     |
| 1        | **Agent DSL**           | Model or `operation_models` declared in the `.agent.abl` file |
| 2        | **Agent DB config**     | Per-agent model override in the agent settings UI             |
| 3        | **Project DB config**   | Project-level `ModelConfig` with tier-to-model mapping        |
| 4        | **Tenant model**        | Workspace-level default model for the resolved tier           |

If no model is resolved at any level, the request fails with a clear error. The platform never falls back to a hard-coded default.

### Fallback chains

Configure fallback models to handle provider outages or rate limits. When the primary model fails (timeout, rate limit, or provider error), the Runtime automatically retries with each fallback in order.

How fallback works:

- Fallback models are tried in order. The first successful response is used.
- Each fallback attempt uses its own credentials (you might fall back from Anthropic to OpenAI, which requires different API keys).
- The platform tracks which model ultimately served the request, so you can see fallback usage in analytics.
- Fallback is transparent to the agent -- it receives a response regardless of which model served it.

To configure fallback models:

1. Go to **Settings > AI Configuration > Models**.
2. Select a registered model.
3. In the **Fallback** section, add one or more fallback models in priority order.
4. Click **Save**.

You can also declare fallback models directly in your agent definition:

```abl
AGENT CustomerSupport
  DESCRIPTION "Handles customer inquiries"
  MODEL "anthropic/claude-sonnet-4"
  FALLBACK_MODELS ["openai/gpt-4o", "google/gemini-2.5-flash"]
```

### Cost and token tracking

Every LLM call tracks token usage and estimated cost. This data flows into the platform's analytics pipeline, where you can see it broken down by agent, project, model, and time period.

| Metric             | Description                                                                        |
| ------------------ | ---------------------------------------------------------------------------------- |
| **Input tokens**   | Tokens sent to the model (system prompt + conversation history + tool definitions) |
| **Output tokens**  | Tokens generated by the model (response text + tool calls)                         |
| **Estimated cost** | Calculated from token counts and the model's known pricing                         |
| **Model used**     | Which model served the request (important when fallbacks are in play)              |
| **Latency**        | Time from request to first token (time-to-first-token) and total request duration  |

The platform maintains a catalog of known model pricing. For custom or self-hosted models where pricing is not known, token counts are still tracked -- you can apply your own cost formulas.

### Context window management

Large conversations accumulate tokens. The Runtime automatically manages the context window to stay within model limits and control costs:

- **Tool result compression** -- Large tool results are compressed before being added to the conversation.
- **Prior turn truncation** -- Tool results from previous conversation turns are replaced with short placeholders.
- **Conversation compaction** -- When the conversation history grows beyond a threshold, older messages are summarized to reduce token usage while preserving important context.

## Supported Providers

| Provider             | Auth type                     | Models                                             | Key strengths                                                  |
| -------------------- | ----------------------------- | -------------------------------------------------- | -------------------------------------------------------------- |
| **Anthropic**        | API key                       | Claude 4 Opus, Claude 4 Sonnet, Claude 3.5 Haiku   | Strong reasoning, long context (200K tokens), tool use, vision |
| **OpenAI**           | API key                       | GPT-4.1, GPT-4o, GPT-4o mini, o3, o4-mini          | Broad capabilities, vision, function calling                   |
| **Azure OpenAI**     | API key + endpoint            | GPT-4, GPT-4o (Azure deployments)                  | Enterprise compliance, regional deployment                     |
| **Google**           | API key                       | Gemini 2.5 Pro, Gemini 2.5 Flash, Gemini 2.0 Flash | Very large context windows (1M+ tokens), fast inference        |
| **Amazon Bedrock**   | IAM credentials               | Claude, Titan, Bedrock-hosted models               | AWS integration, VPC-private access                            |
| **Custom / LiteLLM** | API key, Bearer, or OAuth 2.0 | Any OpenAI-compatible endpoint                     | Unified interface to 100+ model providers                      |

Each provider requires its own credentials (API keys or cloud-native authentication). You configure credentials at the workspace level, and they are encrypted at rest.

## Add Provider Credentials

Before registering models, add your provider's API credentials:

1. Go to **Settings > AI Configuration > LLM Providers**.
2. Click **Add credential**.
3. Select the provider from the dropdown.
4. Enter a display name (for example, "Production Anthropic Key").
5. Enter the API key and, for Azure or custom providers, the endpoint URL.
6. Select the credential scope:
   - **Workspace** -- Available to all projects and members in the workspace.
   - **Personal** -- Available only to you, useful for experimentation.
7. Click **Validate and save**.

The platform validates credentials by making a test API call. If validation fails:

- Confirm the API key is correct and active.
- Confirm the endpoint URL is correct (Azure, custom, and LiteLLM).
- Confirm your network allows outbound HTTPS to the provider.

> **Security:** API keys are encrypted at rest using the workspace encryption key and never displayed in plaintext after initial entry. See [Workspace Configuration](./workspace-configuration) for security details.

### Credential lifecycle

Each credential tracks:

- **Active/Inactive status** -- Deactivate during key rotation without deleting.
- **Last used** -- Timestamp of the most recent API call.
- **Last validated** -- Timestamp of the most recent successful validation.

## Model Tiers & Defaults

Model tiers decouple agent definitions from specific model choices. Agents reference tiers, and the platform resolves the tier to a specific model at runtime.

| Tier         | Intended use                                     | Typical models                                  |
| ------------ | ------------------------------------------------ | ----------------------------------------------- |
| **Fast**     | Classification, routing, quick responses         | Claude 3.5 Haiku, GPT-4o mini, Gemini 2.0 Flash |
| **Balanced** | General-purpose tasks balancing quality and cost | Claude 4 Sonnet, GPT-4o, Gemini 2.5 Flash       |
| **Powerful** | Complex reasoning, analysis, content generation  | Claude 4 Opus, o3, Gemini 2.5 Pro               |
| **Voice**    | Real-time voice interaction                      | GPT-4o Realtime, Gemini 2.0 Flash (Live)        |

### Setting workspace defaults

1. Go to **Settings > AI Configuration > LLM Providers**.
2. In the **Default models** section, select a model for each tier dropdown.
3. Click **Save defaults**.

Defaults apply to all new projects. Existing projects retain their assignments unless updated.

### Choosing the right model

Selecting a model is a trade-off between three dimensions: quality, speed, and cost.

- **Start with a balanced model.** Most agents perform well with a balanced-tier model. Only move to a more powerful (or faster) model if you identify a specific need.
- **Use fast models for internal operations.** Entity extraction, field validation, and summarization often do not need the most powerful model.
- **Use powerful models for complex reasoning.** If your agent needs to make nuanced decisions or synthesize information from multiple sources, a powerful model produces better results.
- **Match the model to the context window.** If your agent handles long documents, choose a model with a large context window. Gemini models offer 1M+ token contexts; Claude models offer 200K tokens.
- **Consider latency for real-time interactions.** Voice agents and real-time chat need fast time-to-first-token.
- **Test with real conversations.** The best way to choose a model is to test your agent with representative conversations and compare quality, speed, and cost.

## Register Custom Models

To make a specific model available in your workspace:

1. Go to **Settings > AI Configuration > Models**.
2. Click **Add model**.
3. Select the provider and model from the catalog (or enter a custom model ID for LiteLLM/custom endpoints).
4. Assign the model to a tier (**Fast**, **Balanced**, **Powerful**, or **Voice**).
5. Associate a credential from the ones you registered.
6. Set default hyperparameters:

| Parameter             | Default | Description                                                           |
| --------------------- | ------- | --------------------------------------------------------------------- |
| **Temperature**       | 0.7     | Controls randomness. Lower values produce more deterministic outputs. |
| **Max tokens**        | 4096    | Maximum number of tokens the model can generate in a response.        |
| **Top P**             | 1.0     | Nucleus sampling threshold.                                           |
| **Frequency penalty** | 0.0     | Reduces repetition of token sequences.                                |
| **Presence penalty**  | 0.0     | Encourages the model to discuss new topics.                           |
| **Timeout (ms)**      | 30000   | Request timeout in milliseconds.                                      |

7. Mark capabilities:

| Capability         | Description                                       |
| ------------------ | ------------------------------------------------- |
| **Text**           | Standard text generation (all models)             |
| **Tools**          | Function/tool calling support                     |
| **Streaming**      | Server-sent events streaming for real-time output |
| **Vision**         | Image input processing                            |
| **Realtime voice** | WebSocket-based real-time voice interaction       |

The platform uses these flags to validate agent configurations. If an agent requires tool calling but the assigned model does not support it, a validation warning is shown.

8. Click **Save**.

You can adjust hyperparameters per model registration. Individual agents can also override hyperparameters at the agent level.

## Project-Level Overrides

Projects inherit the workspace's model configuration but can customize how tiers map to operations.

### Operation tier overrides

1. Go to **Project > Settings > LLM Configuration**.
2. For each operation, select the tier override from the dropdown.
3. Click **Save**.

| Operation        | Default tier | Description                             |
| ---------------- | ------------ | --------------------------------------- |
| `extraction`     | Fast         | Parsing structured data from user input |
| `validation`     | Fast         | Input validation and format checking    |
| `tool_selection` | Fast         | Choosing which tool to invoke           |
| `response_gen`   | Balanced     | Generating user-facing responses        |
| `summarization`  | Balanced     | Summarizing conversations or documents  |
| `reasoning`      | Powerful     | Multi-step reasoning and analysis       |
| `coordination`   | Powerful     | Supervisor routing and orchestration    |
| `realtime_voice` | Voice        | Real-time voice interactions            |

These overrides only affect the current project. Other projects continue using the workspace defaults.

### Allowing project credentials

By default, projects use workspace-level LLM credentials. To allow individual projects to register their own credentials (for example, to use a dedicated API key with separate billing):

1. Go to **Settings > AI Configuration > LLM Providers**.
2. Toggle **Allow project-level credentials** to enabled.

When enabled, project Admins can add provider credentials scoped to their project.

## Provider-Specific Configuration

### Anthropic

- **Base URL:** `https://api.anthropic.com` (default, configurable for proxy setups)
- **API version:** `2023-06-01` (default)
- **Extended thinking:** Enable per-project for models that support it

### OpenAI

- **Standard and Azure endpoints** are treated as separate providers.
- For Azure, you need the deployment URL, API key, and API version.
- Reasoning models (o3, o4-mini) support a `reasoningEffort` parameter (`low`, `medium`, `high`).

### Google Gemini

- Long-context models (Gemini 2.5 Pro: up to 1M tokens) are available for document-heavy workflows.
- Voice-capable models (Gemini 2.0 Flash) support the **Voice** tier for real-time interactions.

### Amazon Bedrock

- Provide IAM access key and secret key (or configure an IAM role for the runtime).
- Model IDs use the Bedrock format (for example, `anthropic.claude-v2`).

### LiteLLM / custom endpoints

- Set the proxy URL in workspace settings.
- Model IDs use the LiteLLM format (for example, `openai/gpt-4o` or `anthropic/claude-sonnet-4`).
- The platform routes requests through the LiteLLM proxy, which handles provider normalization.

## LLM Policies

Workspace-level LLM policies govern usage limits and provider restrictions:

| Setting                      | Description                                                                   |
| ---------------------------- | ----------------------------------------------------------------------------- |
| **Allowed providers**        | Restrict which LLM providers can be used in this workspace                    |
| **Model allowlist**          | Restrict specific models that can be used                                     |
| **Monthly token budget**     | Maximum tokens across all models for the billing month                        |
| **Daily token budget**       | Maximum tokens per day (prevents runaway costs)                               |
| **Max requests per minute**  | Rate limit on LLM API calls from this workspace                               |
| **Token budget per project** | Set monthly token limits per project or per user                              |
| **Platform demo mode**       | Enable demo mode with platform-provided credentials (for evaluation purposes) |

To configure these settings:

1. Go to **Settings > AI Configuration > LLM Providers**.
2. Click **LLM Policy**.
3. Adjust the settings as needed.
4. Click **Save**.

> **Tip:** Set daily token budgets conservatively during initial rollout. You can increase them as you understand your agents' actual consumption patterns. Monitor usage in [Billing, Quotas & Operations](./billing-quotas-and-operations).

## Troubleshooting

### Model validation fails

- Confirm the API key is valid by testing it directly with the provider's API.
- For Azure deployments, verify the endpoint URL includes the deployment name.
- Check that your workspace's network (or proxy configuration) allows outbound HTTPS connections to the provider.

### Agent cannot find a model for the requested tier

- Verify that at least one active model is assigned to the tier the agent is requesting.
- Check that the model's credential is active and not expired.

### Token budget exceeded

- Review current consumption in **Settings > Account > Billing**.
- Increase the daily or monthly budget in the LLM Policy settings.
- Consider switching high-volume operations to a Fast-tier model to reduce token usage.

COORDINATION

Multi-agent patterns. Three mechanisms: HANDOFF (transfer control), DELEGATE (sub-task with result), ESCALATE (route to human). Supports local and remote agents via A2A protocol.

How Coordination Works
COORDINATION defines how agents collaborate. HANDOFF transfers the conversation to another agent (with optional return). DELEGATE runs a sub-agent to completion and returns the result. ESCALATE routes to a human agent when AI cannot or should not continue. All three support local and remote (A2A) agents.

HANDOFF — Transfer Control
Basic HANDOFF Transfer with context and return › 
RETURN_HANDLERS:
  await_next_request:
    RESPOND: "Billing issue resolved. Anything else?"
    CONTINUE: true

HANDOFF:
  - TO: Billing_Agent
    WHEN: "user asks about billing or payment"
    PRIORITY: 1
    CONTEXT:
      pass: ["customer_id", "policy_number"]
      summary: "Customer needs billing help for policy {{policy_number}}"
      history: auto    # auto | none | summary_only | full
      # For bounded raw history use:
      # history:
      #   mode: last_n
      #   count: 10
      memory_grants:
        - path: "user.preferred_language"
          access: read
    RETURN: true
    ON_RETURN:
      handler: await_next_request
Creates a new thread for Billing_Agent. Pushes current agent onto the handoff stack. The child agent receives __return_to_parent__ as an available tool. When called, control pops back to Agent A and the named return handler runs.
One-Way Transfer RETURN: false — no comeback › 
  - TO: Claims_Specialist
    WHEN: "complex claim requiring specialist"
    PRIORITY: 2
    CONTEXT:
      pass: ["claim_type", "damage_amount", "policy_number"]
      summary: "High-value claim needs specialist review"
      history: full
    RETURN: false              # one-way transfer
One-way transfer. The parent agent's thread is closed. Claims_Specialist takes full ownership of the conversation. No __return_to_parent__ tool is provided.
Remote Agent (A2A) Cross-service handoff via A2A protocol › 
  # Remote agent (A2A protocol)
  - TO: External_Verification
    WHEN: "need third-party verification"
    REMOTE:
      location: remote
      endpoint: "https://partner-api.example.com/a2a"
      protocol: a2a
      auth:
        type: bearer
      timeout: "60s"
    CONTEXT:
      pass: ["customer_id"]
    RETURN: true
    ASYNC: true                # non-blocking
Remote handoffs use the A2A (Agent-to-Agent) protocol. The runtime sends a task to the remote endpoint and can poll for completion. ASYNC: true makes it non-blocking — the parent continues and checks back later.
Use ASYNC: true for remote agents that take more than a few seconds. The runtime will poll or receive a webhook when the remote agent finishes.
RETURN_HANDLERS + MAP Structured post-return behavior › 
RETURN_HANDLERS:
  payment_complete:
    RESPOND: "Payment confirmed! Receipt sent to your email."
    CONTINUE: true

HANDOFF:
  - TO: Payment_Agent
    WHEN: "user wants to make a payment"
    CONTEXT:
      pass: ["customer_id", "amount_due"]
      history: auto
    RETURN: true
    ON_RETURN:
      handler: payment_complete
      map:
        status: payment_status
        total: amount_paid
The child return result is mapped into parent session fields first, then the named return handler decides what the supervisor says or clears next.

WHEN string Yes Natural language or CEL condition to trigger
PRIORITY number No Evaluation order when multiple match (lower = first)
CONTEXT.pass string[] No Variables to pass to child agent
CONTEXT.summary string No Summary message for child agent (supports templates)
CONTEXT.history string | object No auto | none | summary_only | full | { mode: last_n, count }
CONTEXT.memory_grants object[] No Explicit durable memory grants to share with child
RETURN boolean No Whether child can return to parent (default: true)
ON_RETURN string | object No Built-in action, named handler, or structured handler + map object
RETURN_HANDLERS object No Reusable post-return handlers referenced by ON_RETURN
REMOTE.location enum No local | remote
REMOTE.endpoint string No A2A endpoint URL
REMOTE.protocol enum No a2a (Agent-to-Agent protocol)
REMOTE.auth object No Authentication config (type: bearer | api_key)
REMOTE.timeout string No Timeout for remote agent response
ASYNC boolean No Non-blocking handoff (default: false)
ASYNC_TIMEOUT string No Max wait for async completion
History Modes
Mode What Child Receives Best For
auto Use the authored summary when available; otherwise fall back to a bounded recent transcript Default safe handoffs that should avoid full transcript transfer
none No conversation history — clean slate Independent tasks, privacy-sensitive transfers
summary_only Authored handoff summary only, with no raw messages Strict no-transcript transfers when the authored summary is sufficient
full Complete conversation history Complex cases where every detail matters
{ mode: last_n, count } Bounded raw history using a typed block (`mode: last_n`, `count: 5`) Recent context without full history overhead
The runtime creates a new thread for the target agent and pushes the current agent onto the handoffStack. If RETURN: true, the child agent gets a __return_to_parent__ tool. Calling it pops the stack and resumes the parent. Cycle detection prevents A → B → A infinite loops.
AB003 — RETURN: true without ON_RETURN emits a warning (user gets no feedback on return). Empty CONTEXT.pass emits info-level diagnostic. Compiler validates that the target agent exists in the project at compile time.
DELEGATE — Sub-Task Execution
Basic DELEGATE Run sub-agent, get structured result › 
DELEGATE:
  - AGENT: Policy_Lookup_Agent
    WHEN: "need to verify policy details"
    PURPOSE: "Look up and validate customer policy"
    INPUT:
      policyId: "{{policy_number}}"
      customerName: "{{customer_name}}"
    RETURNS:
      policyData: "result.policy"
      isValid: "result.valid"
    USE_RESULT: "Use the policy data to verify the claim"
    TIMEOUT: "30s"
    ON_FAILURE:
      type: respond
      message: "Policy lookup is unavailable. Let me try another way."
The sub-agent runs synchronously to completion. Its result is mapped via the RETURNS block into the parent's variable space. The parent's LLM receives USE_RESULT as guidance on how to interpret the data.
Remote DELEGATE Cross-service delegation via A2A › 
  # Remote delegate
  - AGENT: Credit_Check_Service
    WHEN: "need credit check"
    REMOTE:
      location: remote
      endpoint: "https://credit-api.internal/a2a"
      protocol: a2a
    INPUT: { customerId: "{{customer_id}}" }
    RETURNS: { creditScore: "result.score" }
    TIMEOUT: "45s"
    ON_FAILURE:
      type: retry
      count: 2
Remote delegates send the INPUT payload to the A2A endpoint and wait for the result. The TIMEOUT applies to the entire round-trip. On failure, the retry strategy attempts up to count times before falling back.
ON_FAILURE Strategies 4 failure handling modes › 
# Strategy 1: respond — tell user and stop
ON_FAILURE:
  type: respond
  message: "Service is unavailable right now."

# Strategy 2: continue — proceed without the result
ON_FAILURE:
  type: continue              # parent proceeds, result is null

# Strategy 3: escalate — route to human
ON_FAILURE:
  type: escalate
  reason: "Sub-agent failed, human review needed"

# Strategy 4: retry — try again N times
ON_FAILURE:
  type: retry
  count: 3                    # max retries before falling back
Type Behavior Best For
respond Send message to user, stop delegation User-facing failures, graceful degradation
continue Proceed without result (null) Optional enrichment, non-critical lookups
escalate Trigger ESCALATE to human Critical failures that need human intervention
retry Retry up to count times Transient network errors, remote service flakiness
All DELEGATE Fields
Field Type Required Purpose
AGENT string Yes Sub-agent name (must exist in project)
WHEN string Yes Natural language or CEL condition to trigger
PURPOSE string No Description of what the sub-agent should accomplish
INPUT object No Variables to pass as input (supports templates)
RETURNS object No Map sub-agent result to parent variables
USE_RESULT string No LLM guidance on how to use the result
TIMEOUT string No Max execution time (e.g. "30s", "2m")
ON_FAILURE object No Failure strategy: respond | continue | escalate | retry
REMOTE object No Remote agent config (location, endpoint, protocol, auth)
DELEGATE runs the sub-agent synchronously within the parent's reasoning loop. The result appears as a tool result to the parent LLM. Max nesting depth is 10 (A delegates to B delegates to C... up to 10 levels). Each level gets its own thread.
AB005 — Missing TIMEOUT emits an info diagnostic (sub-agent could run indefinitely). Missing ON_FAILURE also emits info (unhandled failures will propagate as errors to parent).
ESCALATE — Human Handoff
Full ESCALATE Configuration Triggers, context, routing, resolution › 
ESCALATE:
  triggers:
    - WHEN: "damage_amount > 50000"
      reason: "High-value claim requires human review"
      priority: high
      tags: ["high-value", "claims"]
    - WHEN: "user explicitly requests human agent"
      reason: "Customer requested human assistance"
      priority: medium
    - WHEN: "turn_count > 30 AND NOT is_complete"
      reason: "Extended conversation without resolution"
      priority: low

  contextForHuman:
    - name: "claim_summary"
      template: "Policy: {{policy_number}}, Damage: ${{damage_amount}}, Type: {{claim_type}}"
      include: ["policy_number", "damage_amount", "claim_type", "incident_date"]
    - name: "conversation_context"
      include: ["customer_name", "customer_sentiment"]

  routing:
    queue: "claims_team"
    skills: ["insurance", "high-value"]

  onHumanComplete:
    - condition: "resolution == 'approved'"
      action: "Notify customer their claim has been approved."
    - condition: "resolution == 'denied'"
      action: "Explain denial reason to customer: {{denial_reason}}"
    - condition: "resolution == 'need_info'"
      action: "Ask customer for additional information: {{info_needed}}"
Trigger Conditions When escalation fires ›

Triggers are evaluated after each LLM turn. The first matching trigger fires the escalation. Conditions can be CEL expressions (for data checks) or natural language (for LLM-evaluated intent).

Trigger Type Example Evaluation
CEL expression "damage_amount > 50000" Evaluated by CEL engine against session variables
Natural language "user explicitly requests human agent" Evaluated by LLM against conversation context
Turn count "turn_count > 30 AND NOT is_complete" CEL with built-in turn_count variable
Sentiment "customer_sentiment == 'frustrated'" Requires sentiment extraction in GATHER
Priority Levels Queue ordering for human agents ›
Priority SLA Hint Use Case
critical Immediate Safety issues, legal compliance, system failures
high Minutes High-value transactions, angry customers, complex claims
medium Hours Customer requests human, moderate complexity
low Best effort Long conversations without resolution, optional review
Priority affects queue ordering in the contact center integration. Higher priority items are routed first. Tags and skills enable skill-based routing to specialized human agents.
Context Package What the human agent receives › 
contextForHuman:
  - name: "claim_summary"
    template: "Policy: {{policy_number}}, Damage: ${{damage_amount}}, Type: {{claim_type}}"
    include: ["policy_number", "damage_amount", "claim_type", "incident_date"]
  - name: "conversation_context"
    include: ["customer_name", "customer_sentiment"]
The context package is assembled from session variables and the template. The human agent sees a structured summary plus the raw variable values listed in include. Sensitive fields marked with sensitive: true in GATHER are masked in the context package.
Resolution Handling onHumanComplete — what happens after › 
onHumanComplete:
  - condition: "resolution == 'approved'"
    action: "Notify customer their claim has been approved."
  - condition: "resolution == 'denied'"
    action: "Explain denial reason to customer: {{denial_reason}}"
  - condition: "resolution == 'need_info'"
    action: "Ask customer for additional information: {{info_needed}}"
When the human agent resolves the escalation, the onHumanComplete conditions are evaluated. The matching action becomes the AI agent's next response. Variables like denial_reason come from the human agent's resolution form.
Always define at least a default onHumanComplete action. If the human resolves with an unexpected status, the agent needs a fallback behavior.
AB002 — ESCALATE triggers without reason emit a warning. Missing contextForHuman emits info (human agent gets no context). Missing onHumanComplete means the conversation ends after human resolution with no AI follow-up.
Comparison — When to Use Which
Aspect HANDOFF DELEGATE ESCALATE
Control Transfers to another agent Sub-agent runs, returns result Routes to human
Conversation User talks to new agent User unaware (transparent) User talks to human
Return Optional (RETURN: true/false) Always returns result Optional (onHumanComplete)
Threading New thread, push to stack Sub-thread, result to parent Paused, resumed on resolution
Blocking Full transfer (or async) Synchronous (blocks parent) Suspends agent thread
Remote A2A with ASYNC option A2A with TIMEOUT Contact center integration
Max depth Stack-based (cycle detection) 10 levels nesting N/A (human endpoint)
Use case Different domain/specialty Lookup, validation, enrichment Beyond AI capability

Use HANDOFF when...

The conversation needs a different specialist. The user's intent has shifted to another domain (billing, claims, tech support). The new agent takes over the conversation.

Use DELEGATE when...

You need data or a sub-task completed. Policy lookup, credit checks, address validation. The user does not interact with the sub-agent — it runs behind the scenes.

Use ESCALATE when...

AI should not continue. High-value decisions, frustrated customers, explicit human requests, regulatory requirements, or extended unresolved conversations.

Multi-Agent Patterns
Supervisor + Specialists Central router with domain agents › 
# Supervisor_Agent — routes to specialists
RETURN_HANDLERS:
  billing_follow_up:
    RESPOND: "Billing handled. What else can I help with?"
    CONTINUE: true
  claims_follow_up:
    RESPOND: "Claims issue resolved. Anything else?"
    CONTINUE: true
  policy_follow_up:
    RESPOND: "Policy question handled. How else can I help?"
    CONTINUE: true

HANDOFF:
  - TO: Billing_Agent
    WHEN: "billing, payment, invoice questions"
    RETURN: true
    ON_RETURN:
      handler: billing_follow_up

  - TO: Claims_Agent
    WHEN: "claims, damage, incidents"
    RETURN: true
    ON_RETURN:
      handler: claims_follow_up

  - TO: Policy_Agent
    WHEN: "policy details, coverage, renewal"
    RETURN: true
    ON_RETURN:
      handler: policy_follow_up
The Supervisor pattern is the most common multi-agent architecture. The supervisor is lightweight (minimal TOOLS/GATHER) and focuses on intent classification and routing. Each specialist is a full agent with its own GATHER, TOOLS, and CONSTRAINTS.

Delegation Chain Agent enriches data via multiple sub-agents › 
# Claims_Agent delegates to multiple sub-agents for data
DELEGATE:
  - AGENT: Policy_Lookup_Agent
    WHEN: "need policy details"
    INPUT: { policyId: "{{policy_number}}" }
    RETURNS: { policyData: "result.policy" }
    TIMEOUT: "15s"

  - AGENT: Fraud_Check_Agent
    WHEN: "claim filed, need fraud screening"
    INPUT: { claimData: "{{claim_details}}" }
    RETURNS: { fraudScore: "result.score", flagged: "result.flagged" }
    TIMEOUT: "20s"
    ON_FAILURE: { type: continue }

  - AGENT: Damage_Assessment_Agent
    WHEN: "need damage estimate"
    INPUT: { photos: "{{damage_photos}}", description: "{{incident_description}}" }
    RETURNS: { estimate: "result.amount" }
    TIMEOUT: "30s"
Delegation chains let a single agent orchestrate multiple sub-tasks. Each delegate runs independently. Use ON_FAILURE: continue for non-critical enrichment so the parent can proceed even if one sub-agent fails.
Escalation with Fallback Delegate fails, escalate to human › 
# Try automation first, escalate if it fails
DELEGATE:
  - AGENT: Auto_Refund_Agent
    WHEN: "customer requests refund"
    INPUT: { orderId: "{{order_id}}", amount: "{{refund_amount}}" }
    RETURNS: { refunded: "result.success" }
    TIMEOUT: "15s"
    ON_FAILURE:
      type: escalate
      reason: "Automated refund failed, needs manual processing"

ESCALATE:
  triggers:
    - WHEN: "refund_amount > 500"
      reason: "High-value refund requires approval"
      priority: high
  contextForHuman:
    - name: "refund_details"
      template: "Order: {{order_id}}, Amount: ${{refund_amount}}"
      include: ["order_id", "refund_amount", "customer_name"]
This pattern combines DELEGATE and ESCALATE. The agent tries automated resolution first. If the sub-agent fails (timeout, error) or if the amount exceeds a threshold, it escalates to a human. This is the recommended pattern for high-stakes operations.
Anti-Patterns

Circular Handoffs

Agent A hands off to B, which hands off back to A. The runtime detects cycles and throws an error, but the conversation is already disrupted. Design clear handoff hierarchies.

DELEGATE without TIMEOUT

A sub-agent without a timeout can hang indefinitely, blocking the parent. Always set TIMEOUT — 30s is a good default for local agents, 60s for remote.

ESCALATE without Context

Escalating to a human with no contextForHuman. The human agent has to ask the customer to repeat everything. Always include a summary and relevant variables.

HANDOFF with full History

Using history: full for every handoff wastes tokens and may expose sensitive data. Use auto by default, opt into strict summary_only only when an authored summary is enough, and use full only when every detail matters.

Deep Delegation Nesting

A delegates to B delegates to C delegates to D... Deep nesting creates latency and debugging nightmares. Keep delegation chains to 2-3 levels max.

No ON_FAILURE Strategy

A DELEGATE without ON_FAILURE propagates errors to the parent's LLM as raw error text. Define explicit failure handling for a graceful user experience.

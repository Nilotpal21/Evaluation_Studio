How Memory Works
MEMORY defines what the agent remembers within a session and across sessions. SESSION holds ephemeral variables for the current conversation. PERSISTENT stores facts in a cross-session FactStore. REMEMBER auto-triggers writes to persistent memory. RECALL loads relevant memory at key moments.

Session Memory

Ephemeral variables that live for one conversation. Reset per-session or per-step. Used for tracking state, counters, collected data. Stored in session.data.values.

Persistent Memory

Facts that survive across sessions. Scoped to user or project. Loaded on session start, written on REMEMBER triggers. Stored in FactStore keyed by (tenantId, userId, projectId).

Mode Behavior

per_step Reset to initial value when flow advances to a new step. Useful for step-local counters.

Scopes

user scope

Keyed by (tenantId, userId). Each user has their own facts. "User prefers Spanish" is user-scoped. Loaded via factStore.getMany(userId, paths).

project scope

Keyed by (tenantId, projectId). Shared across all users in the project. "Business hours are 9-5" is project-scoped. Loaded via projectFactStore.getMany(projectId, paths).

Batch-loaded on session start via getMany(). User-scoped facts use the FactStore keyed by (tenantId, userId). Project-scoped facts use a separate projectFactStore keyed by (tenantId, projectId). Values are injected into session context and available as {{user.preferred_language}} in templates.
Use access: read for project-level config that agents should not modify. Use readwrite for user preferences the agent can update via REMEMBER triggers.
REMEMBER Event-driven writes to persistent memory › 
MEMORY:
  REMEMBER:
    - WHEN: claim_type != null
      STORE:
        value: "{{claim_type}}"
        target: "user.last_claim_type"
      TTL: "30d"
    - WHEN: user_sentiment == "frustrated"
      STORE:
        value: "true"
        target: "user.had_frustrating_experience"
      TTL: "90d"
Field Type Purpose
WHEN string (CEL) Condition expression evaluated after state changes
STORE.value string (template) Value to persist — supports {{var}} templates
STORE.target string Persistent path to write to (must match a PERSISTENT entry)
TTL string (duration) Auto-expiry: 30d, 24h, 90d, 1y
Evaluation Triggers
state change tool call complete entity extraction session variable SET flow step transition
REMEMBER rules are evaluated after state changes, tool calls, and entity extractions. When the WHEN condition matches, the STORE writes to the FactStore in a fire-and-forget pattern (non-blocking). TTL sets an expiry on the fact — expired facts are automatically pruned and return null on read.
Always set a TTL. Facts without TTL persist forever and accumulate unbounded storage. Use 30d for transient preferences, 90d for behavioral patterns, 1y for stable user attributes.
RECALL Event-driven memory loading + prompt injection › 
MEMORY:
  RECALL:
    - ON: session:start
      INSTRUCTION: "Check if user has previous claims and greet accordingly."
    - ON: tool:fetch_policy:after
      INSTRUCTION: "Note the policy details for personalized context."
      ACTION:
        type: inject_context
        paths: ["user.claim_history", "user.preferred_language"]
Field Type Purpose
ON string (event) Event that triggers the recall
INSTRUCTION string Natural language guidance injected into system prompt
ACTION.type enum inject_context | load_memory | prompt_llm
ACTION.paths string[] Persistent paths to load (for inject_context / load_memory)
Event Types
Event Fires When
session:start New session begins, after persistent memory is loaded
session:end Session is closing (cleanup, final persist)
tool:<name>:before Just before a specific tool executes
tool:<name>:after After a specific tool returns its result
step:<name>:enter Flow enters a named step
step:<name>:exit Flow leaves a named step
intent:<name> A specific intent is detected
Action Types

inject_context

Loads persistent paths and injects their values into the current LLM context. The INSTRUCTION tells the LLM how to use this context. Most common action.

load_memory

Loads persistent paths into session variables without injecting into the prompt. Useful for data the agent needs for logic but not for conversation.

prompt_llm

Triggers an additional LLM call with the recalled context and instruction. Used for complex reasoning about recalled data before continuing the conversation.

(no action)

When only INSTRUCTION is provided without ACTION, the instruction text is injected into the system prompt's "Recalled Memory Instructions" section. No data is loaded.

RECALL instructions are injected as _recallPrompts into the system prompt's "Recalled Memory Instructions" section. When ACTION includes paths, those persistent values are fetched from the FactStore and made available as context. Multiple RECALL entries can fire on the same event.
The /recall slash command provides event autocomplete (session:start, tool:*, step:*) and a path picker that lists all declared PERSISTENT paths.
Built-in Context Variables
These variables are always available in templates, conditions, and expressions. They are automatically populated by the runtime and do not need to be declared in SESSION.
Variable Type Description
channel string Current channel: web, voice, sms, whatsapp, api

locale string Full locale: en-US, de-DE, ja-JP

Use built-in variables in REMEMBER conditions: WHEN: turn_count > 5 AND intent == "escalate". Use in RECALL events: ON: session:start fires once when turn_count == 0.
Memory vs Gather vs Constraints

GATHER

Collects information from the user explicitly. Typed fields with prompts, validation, and retry logic. The user provides the data.

MEMORY

Stores computed and derived values. Session state, persistent facts, and auto-triggered writes. The system manages the data.

CONSTRAINTS

Checks values as business rules. Enforces boundaries on what the agent can do. Does not store or collect — only validates.

When to Use Which

Need user input? GATHER. Need to track state? SESSION. Need cross-session facts? PERSISTENT + REMEMBER. Need to enforce rules? CONSTRAINTS.

Common Patterns
Session State Machine Track conversation flow state › 
MEMORY:
  SESSION:
    - claim_status: string = "new"
      # transitions: new → collecting → reviewing → submitted → closed
    - validation_attempts: number = 0
      reset: per_step
    - escalation_reason: string

# In flow steps, use SET to advance state:
# SET claim_status = "collecting"
# SET validation_attempts = validation_attempts + 1
Use session vars as a lightweight state machine. Track status, retry counts, and intermediate results. Reset per-step counters automatically clean up when the flow advances.
User Preference Learning Remember and apply user preferences › 
MEMORY:
  PERSISTENT:
    - user.preferred_language
      scope: user
      type: string
    - user.communication_style
      scope: user
      type: string
      # formal | casual | technical

  REMEMBER:
    - WHEN: language != user.preferred_language AND turn_count > 3
      STORE:
        value: "{{language}}"
        target: "user.preferred_language"
      TTL: "1y"

  RECALL:
    - ON: session:start
      INSTRUCTION: "Greet the user in their preferred language and match their communication style."
      ACTION:
        type: inject_context
        paths: ["user.preferred_language", "user.communication_style"]
On each session, the agent recalls the user's preferences and adapts. If the user switches language mid-conversation and stays with it, REMEMBER updates the preference for next time.
Conversation Continuity Resume where the user left off › 
MEMORY:
  PERSISTENT:
    - user.last_claim_id
      scope: user
      type: string
    - user.last_interaction_topic
      scope: user
      type: string
    - user.open_issues
      scope: user
      type: array

  REMEMBER:
    - WHEN: result.claim_id != null
      STORE:
        value: "{{result.claim_id}}"
        target: "user.last_claim_id"
      TTL: "90d"

  RECALL:
    - ON: session:start
      INSTRUCTION: "If user has an open claim, ask if they want to continue with it. Reference the claim ID."
      ACTION:
        type: inject_context
        paths: ["user.last_claim_id", "user.open_issues"]
Continuity creates a natural experience: "Welcome back! Last time we were working on claim CLM-12345. Would you like to continue?" The user feels recognized without repeating themselves.
Anti-Patterns

Persistent paths without scope

Omitting scope defaults to user, but makes intent unclear. Always be explicit: scope: user or scope: project. Reviewers and future maintainers need to know the isolation boundary.

REMEMBER without TTL

Facts without TTL persist forever in the FactStore. Over time this creates unbounded storage growth. Always set a TTL — even 1y for "permanent" facts forces periodic review.

Too many session variables

Declaring 20+ session vars makes the agent's state opaque and hard to debug. Keep session vars under 10. Use PERSISTENT for anything that needs to survive the session, and GATHER for user-provided data.

RECALL without INSTRUCTION

Loading paths via inject_context without an INSTRUCTION means the LLM gets raw data with no guidance on how to use it. Always pair ACTION with a clear INSTRUCTION.

Validation Rules
Rule Level Source
REMEMBER target not in PERSISTENT Error Analyzer — STORE.target must reference a declared PERSISTENT path
RECALL paths not in PERSISTENT Error Analyzer — ACTION.paths must reference declared PERSISTENT paths
REMEMBER without TTL Warning Language service — unbounded storage growth risk
Unused PERSISTENT path Info Analyzer — declared but never referenced by REMEMBER or RECALL
Invalid reset mode Error Parser — must be per_session | per_step | never
Duplicate session var name Error Parser — session variable names must be unique
Invalid RECALL event Error Parser — must match pattern: session:* | tool:*:before/after | step:*:enter/exit | intent:*

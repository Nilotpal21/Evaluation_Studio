FLOW

Step-by-step conversation control. Determines execution mode: without FLOW → reasoning, with FLOW → scripted or mixed mode.

The Big Decision — Execution Mode
The presence or absence of FLOW: determines how the entire agent executes. This is the single most important architectural decision in an agent definition.

Reasoning Mode (no FLOW)

The LLM drives the conversation freely. It decides when to call tools, what to ask the user, and when the task is complete. Best for open-ended tasks: research, analysis, creative work, general Q&A.

Flow Mode (with FLOW)

Steps execute in a defined sequence. Each step gathers data, calls tools, or responds — then transitions to the next step. Best for structured processes: bookings, forms, onboarding, claims.

Mixed Mode (FLOW + REASONING: true steps)

Combine scripted flow with LLM-powered steps. Some steps are deterministic (gather, respond), others let the LLM reason freely. Best of both worlds for complex workflows that need flexibility at certain points.

How It's Detected

Runtime checks !!agentIR.flow. That's it. No MODE declaration needed — the deprecated MODE: keyword produces a parser error if used. 

# Reasoning mode — no FLOW section
AGENT: Research_Assistant
GOAL: "Help users research topics"
TOOLS:
  web_search(query: string) -> object
    description: "Search the web"
# No FLOW: → LLM drives everything

# ─────────────────────────────────────

# Flow mode — structured steps
AGENT: Hotel_Booking
GOAL: "Book hotels for users"
FLOW:
  steps: [welcome, collect, search, confirm]
  # Each step defines REASONING: true/false
Flow Structure
FLOW:
  # Step order — defines the sequence
  steps:
    - welcome
    - collect_trip_info
    - search_hotels
    - select_hotel
    - collect_guest_info
    - confirm

  # Optional: explicit entry point (defaults to first step)
  entry_point: welcome

  # Global digressions — available in ALL steps
  global_digressions:
    - INTENT: "cancel"
      RESPOND: "Cancelled. Start over?"
      GOTO: welcome
    - INTENT: "help"
      RESPOND: "I can help with hotel bookings..."
      RESUME: true

  # Step definitions follow...
  welcome:
    REASONING: false
    ...
Field Type Required Purpose
steps string[] ● Ordered list of step names. Defines traversal order.
entry_point string ○ First step to execute. Defaults to steps[0].
global_digressions Digression[] ○ Intent handlers available in every step.
[step_name] FlowStep ● Step definitions — one for each name in steps[].
Every name in steps[] must have a matching step definition. Orphaned definitions (not in steps[]) are ignored. COMPLETE is always a valid THEN target.

Two Step Types
Every step in FLOW must declare REASONING: true or REASONING: false. This is not optional — the parser errors without it.
REASONING: false Scripted / Deterministic Step ›

No LLM involvement. Step executes a fixed sequence: GATHER → SET → CALL → RESPOND → THEN. Fast, predictable, zero LLM cost. 

collect_trip_info:
  REASONING: false

  # Collect user input
  GATHER:
    - destination: required
      prompt: "Where would you like to stay?"
    - checkin_date: required
      type: date
      prompt: "Check-in date?"
    - checkout_date: required
      type: date
      prompt: "Check-out date?"
    - num_guests
      type: number
      default: 2

  # Show collected data
  PRESENT: |
    Destination: {{destination}}
    Dates: {{checkin_date}} to {{checkout_date}}
    Guests: {{num_guests}}

  # Allow user to fix values
  CORRECTIONS: true

  # Exit condition
  COMPLETE_WHEN: destination AND checkin_date AND checkout_date

  # Handle off-topic
  DIGRESSIONS:
    - INTENT: "weather"
      CALL: get_weather(destination)
      RESPOND: "Weather: {{forecast}}, {{temp}}°F"
      RESUME: true

  # Handle in-step corrections
  SUB_INTENTS:
    - INTENT: "change destination"
      CLEAR: [destination]
      RESPOND: "What's the new destination?"

  THEN: search_hotels

REASONING: true LLM-Powered Step ›

Delegates to the full ReasoningExecutor for this step. The LLM gets a step-specific goal, scoped tools, and runs its agentic loop until EXIT_WHEN is met or MAX_TURNS is reached. 

research_options:
  REASONING: true

  # Step-specific goal (required for reasoning steps)
  GOAL: "Find the best hotel options matching the user's criteria"

  # Limit which tools the LLM can use in this step
  AVAILABLE_TOOLS:
    - search_hotels
    - check_availability
    - get_reviews

  # Exit when condition is met
  EXIT_WHEN: hotel_selected == true

  # Safety limits
  MAX_TURNS: 8

  # Step-specific constraints
  STEP_CONSTRAINTS:
    - "Only recommend hotels within budget"

  THEN: confirm_booking
Reasoning Step Fields
Field Type Req Purpose
REASONING true ● Marks as LLM-powered step
GOAL string ● What the LLM should achieve in this step
AVAILABLE_TOOLS string[] ○ Subset of agent tools available. Omit = all tools.
EXIT_WHEN string ○ CEL condition to exit the reasoning loop
MAX_TURNS number ○ Max LLM iterations (default: 10)
STEP_CONSTRAINTS string[] ○ Additional instructions injected into system prompt
THEN string ○ Next step after reasoning completes
ReasoningExecutor runs the full agentic loop: system prompt → LLM call → tool execution → repeat. The step's GOAL replaces the agent's GOAL in the prompt for this step. AVAILABLE_TOOLS filters the tool list.
REASONING: true without GOAL → parser error. REASONING: false with GOAL → parser warning (ignored). REASONING: false with AVAILABLE_TOOLS → parser warning.
Branching & Transitions
THEN Unconditional transition › 
welcome:
  REASONING: false
  RESPOND: "Welcome!"
  THEN: collect_info       # always goes here

# Special targets:
  THEN: COMPLETE           # marks session complete
THEN target must exist in steps[] or be COMPLETE. Invalid targets → analyzer error AB007.
ON_SUCCESS / ON_FAIL Branch on CALL result › 
create_booking:
  REASONING: false
  CALL: create_booking(hotel_id, guest_name, email)

  ON_SUCCESS:
    RESPOND: "Booking confirmed! ID: {{booking_id}}"
    SET: { booking_confirmed: "true" }
    THEN: COMPLETE

  ON_FAILURE:
    RESPOND: "Booking failed: {{error_message}}"
    THEN: select_hotel

  # ON_SUCCESS/ON_FAILURE can also have sub-branches:
  ON_SUCCESS:
    branches:
      - IF: result.status == "confirmed"
        RESPOND: "Confirmed!"
        THEN: COMPLETE
      - IF: result.status == "pending"
        RESPOND: "Pending approval..."
        THEN: wait_for_approval
ON_RESULT Conditional branches on CALL result values › 
check_eligibility:
  REASONING: false
  CALL: check_policy_status(policy_number)

  ON_RESULT:
    - IF: result.status == "active"
      RESPOND: "Policy is active. Proceeding..."
      THEN: collect_claim_details

    - IF: result.status == "expired"
      RESPOND: "Your policy has expired."
      THEN: COMPLETE

    - # ELSE (no IF condition)
      RESPOND: "Unknown status: {{result.status}}"
      THEN: error_handler
ON_INPUT Branch on user message content › 
main_menu:
  REASONING: false
  RESPOND: "What would you like to do? 1) File claim 2) Check status 3) Get help"

  ON_INPUT:
    - IF: input contains "file" OR input contains "1"
      THEN: file_claim

    - IF: input contains "status" OR input contains "2"
      THEN: check_status

    - IF: input contains "help" OR input contains "3"
      THEN: help

    - # ELSE
      RESPOND: "Please choose 1, 2, or 3."
      THEN: main_menu
GATHER in Flow Steps
collect_guest_info:
  REASONING: false

  GATHER:
    - guest_name: required
      prompt: "Full name (as on ID):"
    - guest_email: required
      type: email
      prompt: "Email for confirmation:"
      validation: "is_email(value)"
      retryPrompt: "That doesn't look like a valid email."
    - guest_phone
      type: phone
      prompt: "Phone number (optional):"
      infer: true
      inferConfidence: 0.9
    - special_requests
      type: string
      required: false
      activation: progressive     # only ask if conversation allows
      dependsOn: ["guest_name"]   # only after name is collected

  strategy: hybrid                  # llm | pattern | hybrid

  CORRECTIONS: true
  COMPLETE_WHEN: guest_name AND guest_email
  THEN: review_booking
Flow GATHER extracts entities from each user message, validates them, then prompts for the next missing required field. strategy: hybrid uses regex for structured types (email/phone/date) and LLM for free-text. infer tries to extract values the user didn't explicitly provide.
Gather Feature Purpose
strategy llm (LLM extracts all) | pattern (regex only) | hybrid (regex + LLM)
CORRECTIONS: true User can say "change my email" to re-collect a field
COMPLETE_WHEN CEL expression — step advances when true
PRESENT Template shown after each extraction cycle to confirm values
activation: progressive Field only asked if conversation naturally reaches it
dependsOn Field only asked after listed dependencies are collected
Digressions

Handle off-topic user messages without losing the current step's state. Can be global (all steps) or step-specific. 

# Global — available in every step
global_digressions:
  - INTENT: "cancel"
    KEYWORDS: ["cancel", "stop", "quit", "never mind"]
    RESPOND: "Cancelled. Would you like to start over?"
    GOTO: welcome              # jump to a step

  - INTENT: "help"
    RESPOND: "I can help you book hotels, check availability..."
    RESUME: true               # return to current step after

# Step-specific — only in this step
collect_info:
  REASONING: false
  DIGRESSIONS:
    - INTENT: "weather"
      CALL: get_weather(destination)
      RESPOND: "Weather in {{destination}}: {{forecast}}"
      RESUME: true

    - INTENT: "price_check"
      CONDITION: destination != null   # only if destination known
      DELEGATE: Price_Check_Agent
      RESUME: true
Field Type Purpose
INTENT string Intent name to match
KEYWORDS string[] Keyword triggers (matched against user message)
CONDITION string CEL guard — only handle if true
RESPOND string Reply to user
CALL string Execute a tool
DELEGATE string Delegate to another agent
GOTO string Jump to a different step (abandons current)
RESUME boolean true = return to current step after handling
CLEAR string[] Clear variables before resuming
Use RESUME: true for informational digressions (weather, help). Use GOTO for navigational ones (cancel, restart). Digressions with GOTO abandon the current step's gathered data.
Sub-Intents

Handle in-step corrections and modifications. Unlike digressions, sub-intents always resume within the current step. 

select_hotel:
  REASONING: false
  GATHER:
    - hotel_selection: required

  SUB_INTENTS:
    - INTENT: "filter by price"
      SET: { sort_by: "price" }
      RESPOND: "Sorted by price. Here are the options..."

    - INTENT: "sort by rating"
      SET: { sort_by: "rating" }
      RESPOND: "Sorted by rating."

    - INTENT: "change dates"
      CLEAR: [checkin_date, checkout_date]
      RESPOND: "What are your new dates?"

    - INTENT: "more details"
      CALL: get_hotel_details(hotel_selection)
      RESPOND: "{{hotel_details}}"
Field Type Purpose
INTENT string Intent name to match
RESPOND string Reply to user
SET Record Update session variables
CLEAR string[] Clear variables (triggers re-collection)
CALL string Execute a tool
RESUME boolean Always true (implicit) — stays in current step

  REASONING: false
  RESPOND: "Choose your room type:"

  ACTIONS:
    elements:

Data Transform
display_results:
  REASONING: false

  # Transform search results before display
  TRANSFORM:
    source: "search_result.hotels"
    itemVar: "hotel"              # AS hotel
    target: "filtered_hotels"     # INTO filtered_hotels
    filter: "hotel.price <= budget"
    map:
      name: "hotel.name"
      price: "hotel.price"
      rating: "hotel.rating"
    sortBy:
      field: "price"
      order: asc
    limit: 5

  RESPOND: "Top {{filtered_hotels.length}} hotels within budget:"
SET & CLEAR — Variable Management
process_selection:
  REASONING: false

  # SET — assign values (supports expressions)
  SET:
    - variable: "total_price"
      expression: "room_price * num_nights"
    - variable: "booking_status"
      expression: "'pending'"
    - variable: "user.name"            # dot notation for nested
      expression: "guest_name"

  # CLEAR — remove variables
  CLEAR: ["temp_search_results", "_internal_cache"]

  THEN: confirm
All FLOW Validation Rules
Rule Level Message
Missing REASONING Error Step 'X' must declare REASONING: true or REASONING: false
REASONING:true no GOAL Error Step 'X' has REASONING: true but no GOAL
REASONING:false + GOAL Warning GOAL has no effect on deterministic steps
REASONING:false + AVAILABLE_TOOLS Warning Tools have no effect on deterministic steps. Use CALL.
Invalid THEN target Error Step 'X' references non-existent step 'Y' (analyzer AB007)
No path to COMPLETE Warning Flow has no path reaching COMPLETE (analyzer AB004)
Deprecated MODE Error MODE is no longer supported. Use REASONING per-step.
Common Flow Patterns
Linear Flow A → B → C → COMPLETE › 
FLOW:
  steps: [greet, collect, process, done]
  greet: { REASONING: false, RESPOND: "Hi!", THEN: collect }
  collect: { REASONING: false, GATHER: [...], THEN: process }
  process: { REASONING: false, CALL: submit(...), THEN: done }
  done: { REASONING: false, RESPOND: "Done!", THEN: COMPLETE }
Loop-Back on Failure Try → Fail → Retry › 
book:
  REASONING: false
  CALL: create_booking(...)
  ON_SUCCESS:
    THEN: COMPLETE
  ON_FAILURE:
    RESPOND: "Failed. Try different options?"
    THEN: search    # loop back
Mixed Mode — Scripted + Reasoning Structure with flexibility › 
welcome:
  REASONING: false
  RESPOND: "Welcome!"
  THEN: gather_requirements

gather_requirements:
  REASONING: false             # structured collection
  GATHER: [...]
  THEN: research

research:
  REASONING: true              # LLM-powered research!
  GOAL: "Find best options"
  AVAILABLE_TOOLS: [search, compare]
  EXIT_WHEN: options_ready
  THEN: confirm

confirm:
  REASONING: false             # back to scripted
  CALL: submit(...)
  THEN: COMPLETE
Mixed mode is powerful: use scripted steps for data collection and confirmation, reasoning steps for complex decision-making. The LLM only runs where you need it.
Conditional Routing (Menu) User choice → different paths › 
main_menu:
  REASONING: false
  RESPOND: "Choose: 1) File claim 2) Check status 3) Help"
  ON_INPUT:
    - IF: input contains "1"
      THEN: file_claim
    - IF: input contains "2"
      THEN: check_status
    - RESPOND: "Please choose 1, 2, or 3."
      THEN: main_menu    # loop

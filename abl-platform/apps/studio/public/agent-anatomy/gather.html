How Gather Works
GATHER defines what information the agent needs from the user. The runtime extracts values from user messages using LLM or pattern matching, validates them, and prompts for missing required fields. GATHER appears both at the agent top-level (reasoning mode) and inside FLOW steps.

Top-Level GATHER (Reasoning Mode)

All entities extracted by LLM before the first reasoning loop. Values stored in session. Used by constraints, tools, and completion conditions. LLM asks for missing fields naturally during conversation.

Flow Step GATHER (Flow Mode)

Entities extracted per-step. Step blocks until all required fields are collected (COMPLETE_WHEN). Supports CORRECTIONS, PRESENT template, and explicit prompts per field.

Basic Structure
GATHER:
  customer_name:
    prompt: "What is your full name?"
    type: string
    required: true

  email:
    prompt: "Your email address?"
    type: email
    required: true
    validate: "is_email(value)"
    retryPrompt: "That doesn't look like a valid email."

  budget:
    prompt: "What's your budget?"
    type: number
    required: false
    default: 1000
    semantics:
      unit: "USD"
      format: "currency"
8 Field Types
string Free-text input › 
customer_name:
  prompt: "What is your full name?"
  type: string
  required: true

# With validation pattern
policy_number:
  prompt: "Policy number?"
  type: string
  required: true
  validate: "matches(value, '^POL-[0-9]{6}$')"
  retryPrompt: "Format: POL-123456"
  maxRetries: 3

# With extraction pattern (regex)
order_id:
  prompt: "Order number?"
  type: string
  extractionPattern: "ORD-([0-9]+)"
  extractionGroup: 1
Extracted by LLM from natural language. If extractionPattern is set, regex runs first before LLM extraction.
number Numeric values with optional units › 
damage_amount:
  prompt: "Estimated damage amount?"
  type: number
  required: true
  semantics:
    unit: "USD"
    format: "currency"
    convertTo: "USD"         # auto-convert other currencies

num_guests:
  prompt: "How many guests?"
  type: number
  default: 2
  validate: "value >= 1 AND value <= 20"
LLM extracts numeric values from text like "about fifty thousand dollars" → 50000. Semantics enable unit recognition and conversion.
boolean Yes/no values › 
is_urgent:
  prompt: "Is this an urgent claim?"
  type: boolean
  required: false
  default: false
Extracted from "yes", "no", "yeah", "nope", "definitely", "not really", etc.
date Dates and date ranges › 
incident_date:
  prompt: "When did the incident occur?"
  type: date
  required: true
  semantics:
    format: "YYYY-MM-DD"
    locale: "en-US"

# Date range collection
travel_dates:
  prompt: "What are your travel dates?"
  type: date
  range: true               # collects {low: date, high: date}
Handles natural language: "last Tuesday", "March 15th", "2 weeks ago", "next Friday". Uses JS date libs + LLM for ambiguous cases. range: true collects start/end as {low, high}.
email Email addresses with built-in validation › 
contact_email:
  prompt: "Email for confirmation?"
  type: email
  required: true
Auto-validates email format using regex. Extracts from text: "you can reach me at john@example.com" → "john@example.com". No extra validate needed.
phone Phone numbers with locale awareness › 
contact_phone:
  prompt: "Phone number?"
  type: phone
  required: false
  semantics:
    locale: "en-US"
Extracts phone numbers from text. Validates format based on locale. Handles: "call me at 555-123-4567", "+1 (555) 123-4567".
enum Selection from predefined values › 
claim_type:
  prompt: "What type of claim? (auto, home, health, life)"
  type: enum
  required: true

# With lookup table for validation
state:
  prompt: "Which state?"
  type: string
  semantics:
    lookup: "valid_states"      # references LOOKUP_TABLES
LLM maps user input to valid values: "car accident" → "auto". When semantics.lookup is set, validates against LOOKUP_TABLES (with optional fuzzy matching). Invalid values trigger retryPrompt.
array Collect multiple values › 
damaged_items:
  prompt: "What items were damaged?"
  type: array
  list: true
  required: true

# Preference collection
brand_preferences:
  prompt: "Any brand preferences?"
  type: string
  preferences: true           # collects accept/desire/avoid/refuse signals
list: true collects an array of values. preferences: true collects structured preference signals: {accept: [], desire: [], avoid: [], refuse: []}. Preferences are stored in persistent memory via REMEMBER triggers.

Validation — 3 Modes
REGEX Pattern matching — fast, zero cost › 
policy_number:
  prompt: "Policy number?"
  type: string
  validate: "matches(value, '^POL-[0-9]{6}$')"
  validationProcess: REGEX     # default
  retryPrompt: "Format must be POL-123456. Try again."
  maxRetries: 3

age:
  type: number
  validate: "value >= 18 AND value <= 120"
  retryPrompt: "Age must be between 18 and 120."
Use CEL functions: matches(value, regex), abl.length(value), range checks with AND/OR. The value variable holds the extracted field value.
CODE Programmatic validation — custom logic › 
credit_card:
  prompt: "Card number for verification?"
  type: string
  validate: "luhn_check(value)"
  validationProcess: CODE
  retryPrompt: "Invalid card number. Please check and try again."
CODE validation runs a programmatic check (Luhn algorithm, checksum, etc.). More powerful than regex but requires the validation function to be registered.
LLM Natural language validation — nuanced checks › 
incident_description:
  prompt: "Describe what happened."
  type: string
  validate: "Description must be specific enough to process a claim. It should include what happened, when, and where."
  validationProcess: LLM
  retryPrompt: "Could you provide more detail? I need what happened, when, and where."
  maxRetries: 2
LLM evaluates the natural language validation rule against the value. Expensive (LLM call per validation) but handles nuanced quality checks regex can't.
Use sparingly. Reserve for subjective quality checks like "is this description detailed enough?" Prefer REGEX for format validation.
Field Inference
destination:
  prompt: "Where would you like to go?"
  type: string
  required: true
  infer: true                  # try to extract from context
  inferConfidence: 0.8          # min confidence (0-1)
  inferConfirm: true            # ask user to confirm

# User said earlier: "I'm planning a trip to Paris next week"
# → LLM infers: destination = "Paris" (confidence: 0.95)
# → Since 0.95 > 0.8 threshold, agent asks: "I see you mentioned Paris. Is that correct?"
# → If inferConfirm: false, it would accept silently
Field Default Behavior
infer: true false LLM tries to extract this field from prior conversation context
inferConfidence 0.8 Min confidence to accept inferred value. Below threshold → ask user.
inferConfirm: true true Even above threshold, ask user "Is X correct?" before accepting
inferConfirm: false — Accept silently if above confidence threshold
Inference runs after entity extraction. The LLM reviews conversation history and tries to fill unasked fields. Useful for multi-entity messages: "Book a hotel in Paris for 2 guests, checking in March 15" fills destination + num_guests + checkin_date in one pass.
Sensitivity & PII Handling
ssn:
  prompt: "Last 4 digits of your SSN for verification?"
  type: string
  required: true
  sensitive: true
  sensitiveDisplay: mask
  maskConfig:
    showFirst: 0
    showLast: 4
    char: "*"
  transient: true               # auto-delete after session

credit_card:
  prompt: "Card number?"
  type: string
  sensitive: true
  sensitiveDisplay: redact      # completely hide in logs/UI
  transient: true
Option Display Example
redact [REDACTED] SSN: [REDACTED]
mask Partial show SSN: ****1234
replace Custom text SSN: [On file]
Sensitive fields are excluded from trace events, logs, and LLM context after collection. transient: true auto-deletes the value when the session ends — it's never persisted.
Semantics — Understanding Meaning
budget:
  prompt: "What's your budget?"
  type: number
  semantics:
    unit: "USD"                # expected unit
    format: "currency"          # formatting hint
    convertTo: "USD"            # auto-convert from other currencies

departure_time:
  prompt: "When do you want to leave?"
  type: date
  semantics:
    format: "YYYY-MM-DD HH:mm"
    locale: "en-US"

address:
  prompt: "What's the address?"
  type: string
  semantics:
    components: ["street", "city", "state", "zip"]

product_name:
  prompt: "Which product?"
  type: string
  semantics:
    lookup: "product_catalog"  # validate against LOOKUP_TABLES
Semantic Type Purpose
format string Expected format (date format, currency, etc.)
unit string Expected unit (USD, km, kg, etc.)
convertTo string Auto-convert to this unit
locale string Locale for date/number parsing (en-US, de-DE)
components string[] Structured sub-parts (address components)
lookup string Validate against a LOOKUP_TABLE
koreEntityType string Map to Kore.ai entity type (platform integration)
Activation Modes — When to Ask
4 Activation Modes Control when fields are prompted › 
# Always asked
name:
  activation: required          # default — always collect

# Asked if conversation allows
special_requests:
  activation: optional          # only if user brings it up

# Asked progressively (later in conversation)
detailed_preferences:
  activation: progressive       # asked after required fields done

# Conditional — only when expression is true
manager_email:
  activation:
    when: "damage_amount > 50000"  # only for high-value claims

# Field dependencies
room_type:
  dependsOn: ["hotel_selection"]   # only after hotel is chosen
Mode Behavior
required Always prompted. Step/agent blocks until collected.
optional Only collected if user mentions it. Never explicitly asked.
progressive Asked after all required fields are done, if conversation is going well.
{when: expr} Only activated when the CEL expression evaluates to true.
Prompt Modes
# Ask mode — explicitly prompt the user
destination:
  promptMode: ask              # default — agent asks user
  prompt: "Where would you like to go?"

# Extract-only — silently extract, never ask
user_sentiment:
  promptMode: extract_only     # extract from context, don't ask
  type: string
  infer: true

ask (default)

Agent explicitly asks the user for this field using the prompt text. Standard collection behavior.

extract_only

Agent never asks — only extracts if the user mentions it naturally. Great for metadata, sentiment, context signals that shouldn't be prompted.

GATHER in Flow Steps
collect_trip_info:
  REASONING: false

  GATHER:
    - destination: required
      prompt: "Where would you like to stay?"
    - checkin_date: required
      type: date
      prompt: "Check-in date?"
    - checkout_date: required
      type: date
      prompt: "Check-out date?"
    - num_guests
      type: number
      default: 2
      prompt: "How many guests?"

  strategy: hybrid                # llm | pattern | hybrid

  PRESENT: |
    Here's what I have:
    Destination: {{destination}}
    Check-in: {{checkin_date}}
    Check-out: {{checkout_date}}
    Guests: {{num_guests}}

  CORRECTIONS: true            # user can say "change my dates"
  COMPLETE_WHEN: destination AND checkin_date AND checkout_date
  THEN: search_hotels
Strategy Behavior Best For
llm LLM extracts all entities Free-text, ambiguous inputs
pattern Regex-only extraction Structured data (email, phone, dates)
hybrid Regex first, then LLM for remainder Mix of structured + free-text
Flow GATHER blocks the step until COMPLETE_WHEN is satisfied. Each user message triggers extraction → validation → lookup → store. PRESENT shows collected values after each cycle. CORRECTIONS lets user say "change my email" to re-collect.
Validation Rules
Rule Level Source
Unreferenced gather field Info Analyzer AB006 — field defined but never used in flow/constraints/complete
Invalid type Error Parser — type must be string|number|boolean|date|email|phone|enum|array
Missing prompt Warning Language service — field without prompt (how will agent ask?)
Common Patterns
Progressive Form Required first, optional later › 
GATHER:
  name: { type: string, required: true, activation: required }
  email: { type: email, required: true, activation: required }
  phone: { type: phone, required: false, activation: progressive }
  preferences: { type: string, required: false, activation: progressive }
Conditional Fields Only ask based on prior answers › 
GATHER:
  claim_type: { type: enum, required: true }
  vehicle_make:
    type: string
    activation: { when: "claim_type == 'auto'" }
  property_address:
    type: string
    activation: { when: "claim_type == 'home'" }
Sensitive Data Collection PII with masking and auto-delete › 
GATHER:
  ssn_last4:
    prompt: "Last 4 digits of SSN?"
    type: string
    sensitive: true
    sensitiveDisplay: mask
    maskConfig: { showFirst: 0, showLast: 4, char: "*" }
    transient: true
    validate: "matches(value, '^[0-9]{4}$')"
Anti-Patterns

Too many required fields

More than 5-6 required fields feels like an interrogation. Use activation: progressive and infer: true to reduce explicit asking.

No retryPrompt

When validation fails without a retryPrompt, the user sees a generic "invalid input" message. Always provide helpful retry guidance.

LLM validation for formats

Using validationProcess: LLM to validate email format. Use REGEX — it's free and instant. Reserve LLM for subjective quality checks.

Missing sensitive flag

Collecting SSN, credit cards, or personal data without sensitive: true. This data ends up in logs, traces, and LLM context.

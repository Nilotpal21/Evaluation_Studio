NLU — Natural Language Understanding
NLU defines how the agent understands user input: languages, intents with patterns, typed entities, categories, and optional embedding-based semantic matching. NLU runs before the LLM reasoning loop to classify intent and extract structured data.

Full NLU Configuration Languages, intents, entities, embeddings › 
NLU:
  languages: ["en", "es"]
  defaultLanguage: "en"
  intents:
    - name: "file_claim"
      patterns: ["file a claim", "submit claim", "new claim", "report damage"]
      examples: ["I need to file a claim for my car accident"]
      entities: ["claim_type"]
    - name: "check_status"
      patterns: ["claim status", "where is my claim", "check on claim"]
  entities:
    - name: "claim_type"
      type: enum
      values: ["auto", "home", "health", "life"]
      synonyms:
        auto: ["car", "vehicle", "automobile"]
        home: ["house", "property", "dwelling"]
    - name: "policy_id"
      type: pattern
      pattern: "POL-[0-9]{6}"
  categories:
    - name: "urgency"
      values: ["low", "medium", "high", "critical"]
  embeddings:
    enabled: true
    provider: "openai"
    model: "text-embedding-3-small"
    threshold: 0.75
    cacheTtl: 3600
Intents Pattern-based intent classification ›
Field Type Purpose
name string Intent identifier, referenced in FLOW transitions and conditions
patterns string[] Short phrases that trigger this intent (keyword matching)
examples string[] Full example utterances for embedding-based matching
entities string[] Entities to extract when this intent is detected
Patterns use keyword/phrase matching (fast, zero cost). Examples use embedding similarity when embeddings.enabled: true (slower, more accurate). Both can be combined — patterns run first, embeddings as fallback.
Entities Typed entity extraction ›
Field Type Purpose
name string Entity identifier, maps to GATHER fields
type enum | pattern enum: predefined values with synonyms. pattern: regex extraction.
values string[] Valid values for enum type
synonyms object Map of value → alternative phrases (enum only)
pattern string Regex for pattern type extraction 
# Enum entity — maps synonyms to canonical values
# "car accident" → claim_type: "auto"
# "house fire" → claim_type: "home"

# Pattern entity — regex extraction
# "My policy is POL-123456" → policy_id: "POL-123456"
Enum entities map user language to canonical values via synonyms. Pattern entities use regex for structured data like IDs, codes, and phone numbers. Entities feed into GATHER fields with the same name.
Categories & Embeddings Classification and semantic matching › 
categories:
  - name: "urgency"
    values: ["low", "medium", "high", "critical"]
  - name: "topic"
    values: ["billing", "technical", "account", "general"]

embeddings:
  enabled: true
  provider: "openai"            # embedding provider
  model: "text-embedding-3-small" # embedding model
  threshold: 0.75              # min similarity score (0-1)
  cacheTtl: 3600              # cache embeddings for 1 hour
Categories classify each message into predefined buckets (urgency, topic, etc.). Used for routing and analytics. Embeddings enable semantic intent matching — more robust than keyword patterns but slower and costlier. Cache reduces repeated embedding calls.
BEHAVIOR_PROFILES — Contextual Overrides
Behavior profiles override agent behavior based on runtime conditions (channel, time, user segment). They modify instructions, tools, voice settings, response limits, gather behavior, and flow steps. Multiple profiles can activate simultaneously — higher PRIORITY wins on conflicts.
Voice Channel Profile Full example with 6 override sections › 
BEHAVIOR_PROFILE: Voice_Channel
PRIORITY: 10
WHEN: channel == "voice"

instructions: "Keep responses short. Use simple language."

tools:
  hide: ["generate_pdf", "show_chart"]
  add:
    - speak_slowly(text: string) -> object
      description: "Speak text slowly for clarity"
voice:
  instructions: "Speak clearly at a moderate pace"
response:
  max_response_length: 100
  fallback_format: "text"
gather:
  validation_style: "lenient"
  confirmation: "always"
flow:
  skip: ["show_images", "display_chart"]
  overrides:
    welcome:
      respond: "Hi there! How can I help you today?"
      voice:
        ssml: "<speak>Hi there! <break time='300ms'/> How can I help?</speak>"
6 Override Sections What profiles can change ›
Section Purpose Example
instructions Additional instructions merged into system prompt "Keep responses under 50 words for voice."
tools Hide tools, add channel-specific tools Hide PDF generation on voice, add TTS tools
voice Voice-specific settings (SSML, pace, tone) "Speak clearly at a moderate pace"
response Response length limits, format fallback max_response_length: 100 for voice
gather Validation style, confirmation behavior Lenient validation on voice (no spelling)
flow Skip steps, override step responses Skip image display steps on voice channel
Multiple Profiles Priority-based conflict resolution › 
BEHAVIOR_PROFILE: Voice_Channel
PRIORITY: 10
WHEN: channel == "voice"
instructions: "Keep responses short."

BEHAVIOR_PROFILE: After_Hours
PRIORITY: 20
WHEN: time.hour < 9 OR time.hour > 17
instructions: "Inform user that live agents are unavailable."
tools:
  hide: ["escalate_to_human"]

# If voice call at 8pm:
# Both profiles activate.
# After_Hours (priority 20) wins on conflicts.
# Non-conflicting settings merge (both instructions apply).
When multiple profiles activate, non-conflicting settings merge. On conflicts (same field), higher PRIORITY wins. Instructions from all active profiles are concatenated. Tool hide/add lists are merged (union).
LOOKUP_TABLES — External Validation Data
Lookup tables provide reference data for entity validation and gather field checking. Three sources: inline (hardcoded values), collection (MongoDB), and API (external HTTP). Referenced from GATHER fields via semantics.lookup and from NLU entities.
3 Lookup Sources Inline, Collection, API › 
LOOKUP_TABLES:
  valid_states:
    source: inline
    values: ["CA", "NY", "TX", "FL", "WA"]
    caseSensitive: false
    fuzzyMatch: true
    fuzzyThreshold: 0.85

  product_catalog:
    source: collection
    tableName: "products"
    field: "name"
    timeoutMs: 5000

  currency_codes:
    source: api
    endpoint: "https://api.exchangerate.host/list"
    field: "currencies"
    timeoutMs: 3000
Source Details Configuration per source type ›
Source Fields Best For
inline values (string array) Small, static lists (states, categories, statuses). No external dependency.
collection tableName, field, timeoutMs Dynamic data from MongoDB (products, customers). Refreshed per query.
api endpoint, field, timeoutMs External reference data (currencies, zip codes). Cached per session.
Option Type Default Purpose
caseSensitive boolean false Whether matching is case-sensitive
fuzzyMatch boolean false Enable fuzzy string matching (Levenshtein distance)
fuzzyThreshold number 0.85 Minimum similarity score for fuzzy match (0–1)
timeoutMs number 5000 Timeout for collection/API lookups
Inline tables are loaded at agent compile time. Collection tables query MongoDB at runtime. API tables fetch on first use and cache for the session. Fuzzy matching handles typos: "Caifornia" matches "California" at threshold 0.85.
Referencing Lookup Tables From GATHER and NLU › 
# In GATHER — validate field against lookup
GATHER:
  state:
    prompt: "Which state?"
    type: string
    semantics:
      lookup: "valid_states"     # references LOOKUP_TABLES.valid_states

  product:
    prompt: "Which product?"
    type: string
    semantics:
      lookup: "product_catalog"  # references LOOKUP_TABLES.product_catalog

# In NLU — entity values from lookup
NLU:
  entities:
    - name: "currency"
      type: enum
      valuesFrom: "currency_codes"  # dynamic values from API lookup
Use semantics.lookup in GATHER for validation. Use valuesFrom in NLU entities for dynamic enum values. Both reference the same LOOKUP_TABLES by name.
MULTI_INTENT — Handling Multiple User Intents
Users often express multiple intents in a single message: "Check my claim status and also update my address." MULTI_INTENT controls how the agent handles these compound requests — process them sequentially, in parallel, or ask for clarification.
Configuration Strategy, limits, thresholds › 
MULTI_INTENT:
  strategy: sequential
  max_intents: 3
  confidence_threshold: 0.6
  queue_max_age_ms: 600000
  enabled: true
Field Type Default Purpose
enabled boolean true Enable multi-intent detection and handling
strategy enum sequential How to process detected intents (see strategies below)
max_intents number 3 Maximum intents to detect per message
confidence_threshold number 0.6 Minimum confidence to accept a detected intent (0–1)
queue_max_age_ms number 600000 Max age for queued intents before they expire (10 minutes)
5 Strategies sequential, parallel, primary_queue, disambiguate, auto ›
Strategy Behavior Best For
sequential Process intents one at a time, in order detected. Complete first before starting second. Form-based flows where each intent needs full data collection.
parallel Process all intents simultaneously. Merge results into a single response. Simple lookups: "Check balance and show recent transactions."
primary_queue Process the highest-confidence intent immediately. Queue others for later. Complex agents where one intent dominates but user also mentions side requests.
disambiguate Ask the user which intent to handle first before proceeding. Ambiguous inputs where intent priority matters: "Cancel or modify my order?"
auto Runtime picks the best strategy based on intent complexity and agent type. General-purpose agents. The runtime uses heuristics to choose. 
# User: "Check my claim status and update my address"

# sequential → handle check_status first, then update_address
# parallel → run both lookups simultaneously
# primary_queue → handle highest-confidence intent, queue the other
# disambiguate → "Would you like to check status or update address first?"
# auto → runtime decides (probably sequential for these intents)
The runtime detects multiple intents using NLU (patterns + embeddings). Intents below confidence_threshold are discarded. Remaining intents are processed according to the strategy. Queued intents expire after queue_max_age_ms.
Runtime Behavior
NLU runs as a pre-pass before the LLM reasoning loop. Intent classification and entity extraction results are injected into the LLM context, helping it understand user intent without re-analyzing from scratch.
Behavior profiles are evaluated at the start of each turn. The runtime checks all WHEN conditions against the current session context (channel, time, user attributes). Active profiles are merged by priority and applied to the agent configuration for that turn.
Lookup tables with source: collection query MongoDB at runtime with the configured timeout. API lookups are cached per session. If a lookup times out, the field validation falls back to LLM-based validation (if available) or accepts the value with a warning.
Multi-intent detection runs after NLU intent classification. If multiple intents are detected above confidence_threshold, the runtime applies the configured strategy. Queued intents persist in session state and are presented to the user when the current intent completes.
Common Patterns
Multilingual Agent NLU with language detection › 
NLU:
  languages: ["en", "es", "fr"]
  defaultLanguage: "en"
  intents:
    - name: "greeting"
      patterns: ["hello", "hola", "bonjour", "hi", "hey"]
    - name: "file_claim"
      patterns: ["file claim", "presentar reclamacion", "deposer reclamation"]
Include patterns in all supported languages. The NLU pipeline detects the language first, then matches against patterns in that language. The agent responds in the detected language.
Channel-Adaptive Agent Behavior profiles per channel › 
BEHAVIOR_PROFILE: Voice_Channel
PRIORITY: 10
WHEN: channel == "voice"
response: { max_response_length: 100 }
tools: { hide: ["show_chart", "generate_pdf"] }

BEHAVIOR_PROFILE: Slack_Channel
PRIORITY: 10
WHEN: channel == "slack"
response: { fallback_format: "slack" }

BEHAVIOR_PROFILE: Web_Chat
PRIORITY: 10
WHEN: channel == "web"
response: { fallback_format: "markdown" }
Define one behavior profile per channel. The WHEN condition matches the active channel. Non-applicable tools are hidden, response formats adapt, and gather validation adjusts for the input modality.
Dynamic Validation with Lookups External data for field validation › 
LOOKUP_TABLES:
  valid_zip_codes:
    source: api
    endpoint: "https://api.zippopotam.us/us"
    field: "post code"
    timeoutMs: 3000

GATHER:
  zip_code:
    prompt: "What's your zip code?"
    type: string
    validate: "matches(value, '^[0-9]{5}$')"
    semantics:
      lookup: "valid_zip_codes"
    retryPrompt: "Please enter a valid 5-digit US zip code."
Combine regex validation (format check) with lookup validation (existence check). The regex runs first (free, instant), then the lookup confirms the value exists in the external source.
Anti-Patterns

Too many NLU intents

Defining 50+ intents with overlapping patterns. Intent classification becomes unreliable. Group related intents and let the LLM handle sub-classification.

Behavior profiles without PRIORITY

Multiple profiles with the same default priority. Conflict resolution becomes unpredictable. Always set explicit PRIORITY values.

API lookups without timeout

External API lookups with no timeoutMs. A slow API blocks the entire conversation. Always set timeouts and handle failures gracefully.

Parallel multi-intent on complex flows

Using strategy: parallel when intents require GATHER data collection. Parallel collection of overlapping fields causes confusion. Use sequential for complex flows.

Validation Rules
Rule Level Source
Duplicate intent name Error Parser — intent names must be unique
Entity referenced but not defined Error Analyzer — intent references entity not in NLU.entities
Overlapping patterns Warning Language service — two intents share identical patterns
Lookup table unreferenced Info Analyzer — table defined but not used in GATHER or NLU
Missing PRIORITY on profile Warning Language service — behavior profile without explicit priority
API lookup without timeout Warning Analyzer — external lookup with no timeoutMs

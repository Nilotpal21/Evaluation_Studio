Full Agent DSL Example
AGENT: Insurance_Claim_Agent
GOAL: "Help users file insurance claims"

PERSONA: |
  Professional, empathetic claims assistant.

LIMITATIONS:
  - "Cannot approve claims over $50,000"

EXECUTION:
  model: "claude-sonnet-4-20250514"
  temperature: 0.3

TOOLS:
  fetch_policy(policyId: string) -> object
    description: "Retrieve policy details"
  submit_claim(data: object) -> {claimId: string}
    description: "Submit insurance claim"

GATHER:
  policy_number:
    prompt: "Policy number?"
    type: string
    required: true
    validate: "matches(value, '^POL-[0-9]{6}$')"
  damage_amount:
    prompt: "Estimated damage?"
    type: number
    required: true

MEMORY:
  SESSION:
    - claim_status: string = "pending"
  PERSISTENT:
    - user.claim_history

CONSTRAINTS:
  pre_submit:
    - REQUIRE: damage_amount < 50000
      ON_FAIL: "Needs manager approval."

GUARDRAILS:
  pii_protection:
    kind: input
    check: not_matches_pattern(input, "\\b\\d{3}-\\d{2}-\\d{4}\\b")
    action: redact

HANDOFF:
  - TO: Billing_Agent
    WHEN: "billing questions"
    CONTEXT:
      pass: ["policy_number"]

ON_START:
  RESPOND: "Hello! I'm your claims assistant."

COMPLETE:
  - WHEN: claim_submitted == true
    RESPOND: "Claim submitted! Ref: {{claim_id}}"

ON_ERROR:
  - type: tool_timeout
    retry: 2
    respond: "Retrying..."

Field Maps To
role → goal.description
persona → persona.description
expertise → appended to persona
limitations → limitations[]
If IDENTITY is used, gray out standalone GOAL / PERSONA / LIMITATIONS — they conflict.
LIMITATIONS:optionalPrompt-level boundaries for what the agent should avoid, decline, or explain› 
LIMITATIONS:
  - "Cannot process payments directly"
  - "Cannot access medical records"
  - "Must escalate claims over $50,000"
Injected as {{limitations}} in system prompt. Shapes the LLM's scope and refusal behavior, but is not deterministic runtime enforcement.

Runtime: Sets the default language for NLU, date parsing, and response formatting. Injected as {{language}} context variable.

FLOW:determines modeStep-by-step conversation — without → reasoning, with → scripted/mixed› 
FLOW:
  steps: [welcome, collect_info, search, confirm]

  welcome:
    REASONING: false
    RESPOND: "Welcome! How can I help?"
    THEN: collect_info

  collect_info:
    REASONING: false
    GATHER:
      - destination: required
      - checkin_date: required
        type: date
    COMPLETE_WHEN: destination AND checkin_date
    THEN: search

  search:
    REASONING: true              # LLM-powered
    GOAL: "Find best matching hotels"
    AVAILABLE_TOOLS: [search_hotels]
    EXIT_WHEN: hotel_selected == true
    MAX_TURNS: 5
    THEN: confirm

  confirm:
    REASONING: false
    CALL: create_booking(hotel_id, guest_name)
    ON_SUCCESS:
      RESPOND: "Booked! ID: {{booking_id}}"
      THEN: COMPLETE
Step Compatibility
Field REASONING: true REASONING: false
GOAL required ignored
AVAILABLE_TOOLS limits tools ignored (use CALL)
GATHER — collect input
CALL — execute tool
RESPOND — send message
THEN next step next step
DIGRESSIONS — off-topic handling
Every step must declare REASONING. REASONING:true requires GOAL. Invalid THEN targets → error.
FlowStepExecutor traverses. Reasoning steps → ReasoningExecutor. Max 100 iterations. COMPLETE always valid.
Safety → Guardrails Deep Dive
GUARDRAILS:optionalContent safety — 6 kinds, 3 tiers, 7 actions› 
GUARDRAILS:
  # Tier 1 — Pattern (zero cost)
  ssn_detection:
    kind: input
    check: not_matches_pattern(input, "\\b\\d{3}-\\d{2}-\\d{4}\\b")
    action: redact
    message: "SSN redacted for security."

  # Tier 2 — Model-based (~100ms)
  toxicity:
    kind: output
    provider: "openai-moderation"
    category: "hate"
    threshold: 0.7
    action: block

  # Tier 3 — LLM-based (~1-3s)
  compliance:
    kind: output
    llm_check: "Does the response make unauthorized commitments?"
    action: reask
    max_reasks: 2

Memory → Memory Deep Dive
MEMORY:optional4 sub-systems — SESSION, PERSISTENT, REMEMBER, RECALL› 
MEMORY:
  SESSION:
    - claim_status: string = "pending"
    - attempts: number = 0

  PERSISTENT:
    - user.preferred_language
      scope: user
      access: readwrite

  REMEMBER:
    - WHEN: claim_type != null
      STORE: { value: "{{claim_type}}", target: "user.last_claim_type" }
      TTL: "30d"

  RECALL:
    - ON: session:start
      INSTRUCTION: "Check if user has previous claims."
System Scope Lifetime
SESSION Per-session Session duration. Types + defaults + per-step reset.
PERSISTENT Per-user / project Permanent. FactStore. Batch-loaded at start.
REMEMBER Cross-session TTL-based. Event-driven writes.
RECALL Per-session On-demand. Injected as prompt instructions.
Multi-Agent Coordination → Coordination Deep Dive
HANDOFF:optionalTransfer conversation to another agent› 
RETURN_HANDLERS:
  billing_follow_up:
    RESPOND: "Billing resolved. Anything else?"
    CONTINUE: true

HANDOFF:
  - TO: Billing_Agent
    WHEN: "user asks about billing"
    PRIORITY: 1
    CONTEXT:
      pass: ["customer_id", "policy_number"]
      summary: "Customer needs billing help"
      history: auto
    RETURN: true
    ON_RETURN:
      handler: billing_follow_up
Creates thread, pushes to handoffStack. RETURN:true → child gets __return_to_parent__. Prevents cycles. Supports A2A.
DELEGATE:optionalRun sub-agent to completion, get result› 
DELEGATE:
  - AGENT: Policy_Lookup_Agent
    WHEN: "need to verify policy"
    INPUT: { policyId: "{{policy_number}}" }
    RETURNS: { policyData: "result.policy" }
    TIMEOUT: "30s"
    ON_FAILURE: { type: respond, message: "Lookup failed." }
ESCALATE:optionalTransfer to human agent with priority› 
ESCALATE:

Lifecycle → Lifecycle Deep Dive
ON_START:optionalWelcome message, initial setup — runs once› 
ON_START:
  SET: { session_type: "claims" }
  CALL: load_customer_profile
  RESPOND: "Hello! I'm your claims assistant."
Runs once (idempotent). Sequence: SET → CALL → RESPOND → DELEGATE. In flow mode, also executes first step.
COMPLETE:optionalCompletion conditions — runtime-evaluated, not LLM› 
COMPLETE:
  - WHEN: claim_submitted == true
    RESPOND: "Claim submitted! Ref: {{claim_id}}"
    STORE: "user.last_claim_id"
CompletionDetector evaluates after each turn. No LLM tool. In flow: THEN: COMPLETE triggers same.
ON_ERROR:optionalError handlers with retry and backoff› 
ON_ERROR:
  - type: tool_timeout
    retry: 2
    retryDelay: 1000
    retryBackoff: exponential
    respond: "Service slow, retrying..."
  - type: tool_error
    respond: "Something went wrong."
    then: escalate

How Lifecycle Works
LIFECYCLE constructs control what happens when an agent session begins, ends, encounters errors, and at each turn boundary. ON_START initializes the session, COMPLETE defines when the agent is done, ON_ERROR handles failures with retry logic, and HOOKS inject behavior at four lifecycle points.

Reasoning Mode

ON_START fires once before the first reasoning loop. COMPLETE conditions are evaluated after each turn. HOOKS wrap every turn in the reasoning loop.

Flow Mode

ON_START fires once, then also executes the first flow step. COMPLETE is triggered by THEN: COMPLETE in flow steps or by condition evaluation. HOOKS wrap each step execution.

ON_START — Session Initialization
Basic ON_START SET + CALL + RESPOND › 
ON_START:
  SET:
    session_type: "claims"
    initialized: "true"
  CALL: load_customer_profile
  RESPOND: "Hello! I'm your claims assistant. How can I help today?"
Runs once per session (idempotent). Execution order is deterministic: SET → CALL → RESPOND → DELEGATE. SET before CALL ensures variables are available for the tool.
With Voice SSML + voice instructions › 
ON_START:
  RESPOND: "Welcome to claims assistance."
  VOICE:
    ssml: "<speak>Welcome to <emphasis>claims</emphasis> assistance.</speak>"
    instructions: "Speak warmly and professionally"
VOICE is optional. When present, the voice channel uses SSML/instructions instead of the plain RESPOND text. Text channel still gets RESPOND.
With Rich Content Action buttons on start › 
ON_START:
  RESPOND: "Welcome! Choose an option:"
  ACTIONS:
    elements:
      - id: "file_claim"
        type: button
        label: "File a Claim"
      - id: "check_status"
        type: button
        label: "Check Status"
ACTIONS renders interactive UI elements (buttons, cards, carousels) alongside the RESPOND text. The user's button selection is sent as the first message.
With Delegation Delegate to setup agent › 
ON_START:
  DELEGATE: Context_Loader_Agent
DELEGATE hands off to another agent during start. The delegated agent runs to completion, then control returns. Useful for complex initialization sequences that are themselves agent-based.
ON_START Fields
Field Type Purpose
SET Record<string, string> Initialize session variables. Runs first.
CALL string Tool name to invoke during initialization.
RESPOND string Welcome message sent to the user.
VOICE VoiceConfig Voice channel config: ssml, instructions.
RICH_CONTENT RichContentAST Structured rich content (cards, images).
ACTIONS ActionSetAST Interactive elements (buttons, forms).
DELEGATE string Agent name to delegate initialization to.
In flow mode, ON_START also triggers the first flow step after completing its own actions. The sequence is: ON_START actions → first flow step entry.
COMPLETE — Session Termination
Condition-Based Completion Runtime-evaluated, NOT LLM-driven › 
COMPLETE:
  # Simple condition
  - WHEN: claim_submitted == true
    RESPOND: "Your claim has been submitted! Reference: {{claim_id}}"
    STORE: "user.last_claim_id"

  # Compound condition
  - WHEN: all_fields_collected AND order_confirmed == true
    RESPOND: "All done! Your order is being processed."

  # With voice goodbye
  - WHEN: user.browsing_complete == true
    RESPOND: "Thank you for using our service!"
    VOICE:
      instructions: "Say goodbye warmly"
COMPLETE conditions are evaluated by the runtime against session values after each turn. There is no __complete__ LLM tool. The LLM does not decide when to complete — the conditions do.
Flow Mode Completion THEN: COMPLETE in flow steps › 
# In flow mode, reaching THEN: COMPLETE triggers the same chain
confirm_order:
  RESPOND: "Order confirmed. Thank you!"
  THEN: COMPLETE

# The COMPLETE conditions above ALSO apply in flow mode
# Either path triggers completion
In flow mode, THEN: COMPLETE triggers the same completion chain as condition-based COMPLETE. Both set session.isComplete = true and execute tryThreadReturn() to pop back to the parent agent.
COMPLETE Fields
Field Type Purpose
WHEN CEL expression Condition evaluated against session values after each turn.
RESPOND string Farewell message. Supports {{variable}} interpolation.
VOICE VoiceConfig Voice channel config for the goodbye message.
RICH_CONTENT RichContentAST Structured rich content in the final message.
ACTIONS ActionSetAST Interactive elements (e.g., feedback buttons).
STORE string Persistent path to write (e.g., user.last_claim_id).
CompletionDetector evaluates all WHEN conditions against session values after each turn. First matching condition wins. Sets session.isComplete = true, executes the RESPOND/STORE actions, then calls tryThreadReturn() to pop back to the parent agent (if delegated).
AB004 — Flow agents should have a path to COMPLETE. Reasoning agents should have COMPLETE conditions. An agent without COMPLETE conditions may run indefinitely.
ON_ERROR — Error Handling
tool_timeout Retry with backoff strategies › 
ON_ERROR:
  - type: tool_timeout
    retry: 3
    retryDelay: 1000
    retryBackoff: exponential    # fixed | exponential | linear
    retryMaxDelay: 10000
    respond: "Service is slow, retrying..."
Backoff Delay Pattern Example (base 1000ms)
fixed Same delay every time 1000, 1000, 1000
exponential Doubles each retry 1000, 2000, 4000
linear Adds base each retry 1000, 2000, 3000
executeWithRetry() handles the retry loop with backoff. retryMaxDelay caps the maximum wait between retries regardless of backoff strategy.
tool_error Subtypes, fallback, escalation › 
ON_ERROR:
  # Tool error with subtype filtering
  - type: tool_error
    subtypes: ["connection_error", "auth_error"]
    respond: "There was a technical issue."
    then: escalate

  # With rich response for user choice
  - type: tool_error
    respond: "Service issue. Try again?"
    ACTIONS:
      elements:
        - id: "retry"
          type: button
          label: "Try Again"
        - id: "skip"
          type: button
          label: "Skip This Step"
subtypes narrows the handler to specific error categories. Without subtypes, the handler catches all errors of that type. then: escalate triggers escalation (handoff to human agent or parent agent).
invalid_input User input errors › 
ON_ERROR:
  - type: invalid_input
    respond: "I didn't understand that. Could you rephrase?"
Fires when the runtime cannot parse or validate user input. Different from GATHER validation failures — this handles messages that are completely unparseable or out of scope.
default + backtrackTo Generic fallback with recovery › 
ON_ERROR:
  # Generic fallback
  - type: default
    respond: "Something went wrong. Let me try a different approach."
    backtrackTo: previous_step    # go back to a safe step
backtrackTo rewinds the flow to a named step, discarding intermediate state. Useful for recovery from errors deep in a flow. The named step re-executes from scratch.
default catches any error type not matched by a more specific handler. Always define a default handler as the last entry.
ON_ERROR Fields
Field Type Purpose
type enum tool_timeout | tool_error | invalid_input | default | custom
subtypes string[] Narrow to specific error subcategories.
respond string Message shown to the user on error.
VOICE VoiceConfig Voice channel config for the error message.
RICH_CONTENT RichContentAST Structured rich content in the error message.
ACTIONS ActionSetAST Interactive elements (retry buttons, etc.).
retry number Number of retry attempts before giving up.
retryDelay number (ms) Initial delay between retries.
retryBackoff enum fixed | exponential | linear
retryMaxDelay number (ms) Maximum delay cap regardless of backoff.

error-handler-router.ts matches errors against handlers in priority order: step-level ON_ERROR (FlowStep.onError) → agent-level ON_ERROR → built-in default. Step-level handlers override agent-level for that specific step.
HOOKS — Turn-Level Lifecycle Injection
4 Hook Points before_agent, after_agent, before_turn, after_turn › 
HOOKS:
  before_agent:
    call: initialize_session_token
    set:
      session_start_time: "{{abl.now()}}"

  after_agent:
    call: cleanup_session

  before_turn:
    call: refresh_auth_token
    set:
      turn_start: "{{abl.now()}}"

  after_turn:
    set:
      last_activity: "{{abl.now()}}"
      turn_count_custom: "{{turn_count + 1}}"
    call: log_turn_analytics
Hook When Runs
before_agent After ON_START, before first message processing Once per session
after_agent After COMPLETE fires, before session cleanup Once per session
before_turn Before every message is processed Every turn
after_turn After every message is processed Every turn
Each hook can use: call (invoke a tool), set (update session variables), respond (send a message), VOICE, RICH_CONTENT, ACTIONS. Hook actions execute in the order: set → call → respond.
Hook Actions Reference What each hook can do ›
Action Type Purpose
call string Tool to invoke. Receives current session context.
set Record<string, string> Variables to set/update. Supports {{expression}} interpolation.
respond string Message to send to the user.
VOICE VoiceConfig Voice channel config for the hook message.
RICH_CONTENT RichContentAST Structured rich content.
ACTIONS ActionSetAST Interactive elements.
Use before_turn for auth token refresh, rate limiting, or input preprocessing. Use after_turn for analytics logging, state snapshots, and activity tracking.
Complete Lifecycle Order
1. Session created
2. ON_START fires — SET → CALL → RESPOND → DELEGATE (once)
3. before_agent hook — initialize session token, set start time (once)
4. For each message:
   a. before_turn hook — refresh auth, set turn_start
   b. Process message — reasoning loop or flow step execution
   c. after_turn hook — log analytics, update last_activity
   d. Check COMPLETE conditions — evaluate WHEN expressions
5. If error: ON_ERROR handler — step-level → agent-level → default
6. If complete: COMPLETE handler — RESPOND + STORE + tryThreadReturn()
7. after_agent hook — cleanup session, final logging (once)
8. Session ends
The message loop (steps 4a-4d) repeats for every user message until a COMPLETE condition matches or an unrecoverable error occurs. Errors during the loop are caught by ON_ERROR and may retry, backtrack, or escalate without terminating the session.
Common Patterns
Welcome with Profile Load Initialize context before greeting › 
ON_START:
  SET:
    channel: "{{abl.channel()}}"
    locale: "{{abl.locale()}}"
  CALL: load_customer_profile
  RESPOND: "Welcome back, {{customer_name}}! How can I help?"
  VOICE:
    instructions: "Greet by name, sound warm and familiar"
CALL executes before RESPOND, so the tool result (customer profile) is available for template interpolation in the welcome message.
Graceful Error Recovery Layered error handlers with escalation › 
ON_ERROR:
  # Retry timeouts
  - type: tool_timeout
    retry: 2
    retryBackoff: exponential
    retryDelay: 1000
    respond: "One moment, retrying..."

  # Auth errors get escalated immediately
  - type: tool_error
    subtypes: ["auth_error"]
    respond: "I need to connect you with a specialist."
    then: escalate

  # Other tool errors: backtrack and try again
  - type: tool_error
    respond: "Let me try a different approach."
    backtrackTo: initial_step

  # Catch-all
  - type: default
    respond: "I'm having trouble. Let me transfer you to support."
    then: escalate
Order matters. More specific handlers (subtypes) should come before generic ones. The router matches top-to-bottom and uses the first match.
Session Analytics Track timing and turn counts with hooks › 
HOOKS:
  before_agent:
    set:
      session_start: "{{abl.now()}}"
      total_turns: "0"

  before_turn:
    set:
      turn_start: "{{abl.now()}}"

  after_turn:
    set:
      total_turns: "{{total_turns + 1}}"
      last_turn_duration: "{{abl.now() - turn_start}}"
    call: log_turn_analytics

  after_agent:
    set:
      session_duration: "{{abl.now() - session_start}}"
    call: save_session_summary
Anti-Patterns

Heavy Processing in ON_START

Calling slow APIs or running complex computations in ON_START delays the welcome message. The user sees nothing while initialization runs. Keep ON_START fast — defer heavy work to the first turn or a background tool.

No COMPLETE Conditions

An agent without COMPLETE conditions never terminates on its own. The session runs indefinitely until timeout or user abandonment. Always define at least one COMPLETE condition.

Retrying Indefinitely

Setting retry: 99 without retryMaxDelay can lock the agent in a retry loop for minutes. Cap retries at 3-5 and always define a fallback for when retries are exhausted.

Hooks with Side Effects

Calling external APIs in before_turn that can fail — if the hook fails, the entire turn fails. Keep hooks lightweight. Use tools with their own error handling for risky operations.

Validation Rules
Rule Level Source
Missing COMPLETE conditions Warning Analyzer AB004 — agent may run indefinitely
Flow without path to COMPLETE Warning Analyzer AB004 — flow steps never reach THEN: COMPLETE
Invalid error type Error Parser — type must be tool_timeout|tool_error|invalid_input|default|custom
Unknown hook point Error Parser — must be before_agent|after_agent|before_turn|after_turn
ON_START references undefined tool Error Analyzer — CALL tool name not in TOOLS
COMPLETE WHEN with invalid CEL Error Language service — CEL expression parse failure
Retry without retryDelay Info Language service — retry set but no delay configured

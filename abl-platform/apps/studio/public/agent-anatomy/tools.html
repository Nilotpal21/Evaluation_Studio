How Tools Work
DSL defines tool
→
Compiler → IR
→
Runtime → JSON Schema
→
LLM sees as tool_use
→
LLM calls tool
→
ToolExecutor runs binding
→
Result → LLM
Tools are converted to JSON Schema via ablTypeToJsonSchema() and sent as LLM tool_use definitions. When the LLM returns a tool call, the runtime executes the appropriate binding (HTTP fetch, MCP call, sandbox eval, etc.) and feeds the result back. Multiple tool calls execute in parallel. 
# Minimal tool — type is inferred, binding is project-level
TOOLS:
  fetch_policy(policyId: string) -> object
    description: "Retrieve insurance policy details"
Tool Structure — All Fields
Field Type Required Default Purpose
name string ● — Unique identifier. Used in CALL, AVAILABLE_TOOLS, and LLM tool_use.
description string ● — Tells the LLM when and how to use this tool. Critical for accuracy.
parameters ToolParam[] ○ [] Input parameters with name, type, required, default, validate, description.
returns ToolReturn ○ {type:"object"} Return type: string, number, object, array. Can have nested fields.
type ToolType ○ inferred http | mcp | lambda | sandbox | async_webhook
hints ToolHints ○ {} cacheable, latency, side_effects, requires_auth, timeout
storeResult boolean ○ true Save result in session data for later reference.
confirmation object ○ — require: always | never | when_side_effects. immutableParams[].
piiAccess enum ○ — tools | user | logs | llm — declares PII exposure level.
contextAccess object ○ — {read: string[], write: string[]} — session variable permissions.
onResult object ○ — {set: Record<string, string>} — auto-set variables on success.
onError object ○ — {set: Record<string, string>} — auto-set variables on error.

5 Binding Types — Deep Dive
HTTP REST API integration › 
TOOLS:
  fetch_policy(policyId: string) -> object
    description: "Retrieve insurance policy by ID"
    type: http
    http:
      method: GET
      endpoint: "https://api.insurance.com/v2/policies/{{policyId}}"
      auth: bearer
      timeout: 15000
      retry: 2
      retryDelay: 500
      headers:
        X-Request-Source: "agent"
        Accept: "application/json"
      rateLimit: 100          # requests per minute
      circuitBreaker:
        threshold: 5          # failures before open
        resetMs: 30000       # 30s before retry
HTTP Methods
GETPOSTPUTPATCHDELETE
Auth Types
Type Config Fields Use Case
none — Public APIs
api_key headerName Simple key auth (X-API-Key)
bearer — JWT / Bearer token
oauth2_client tokenUrl, clientId, clientSecret, scopes Server-to-server OAuth
oauth2_user tokenUrl, clientId, scopes User-delegated OAuth
saml provider SAML SSO
custom customHeaders Custom auth headers
All HTTP Binding Fields
Field Type Default Purpose
endpoint string — URL with {{param}} interpolation
method string GET HTTP method
auth AuthType none Authentication strategy
authConfig object — Auth-specific configuration
timeout number 30000 Request timeout (ms)
retry number 0 Retry attempts on failure
retryDelay number 1000 Delay between retries (ms)
headers Record {} Custom request headers
queryParams Record {} URL query parameters
bodyTemplate string — Custom body template
rateLimit number — Max requests per minute
circuitBreaker object — threshold + resetMs
HTTP tools execute via fetch() with timeout, retry, and circuit breaker. URL params interpolated from tool call arguments. Body is JSON by default. Results > 4KB are compressed.
Compiler validates: method must be GET/POST/PUT/PATCH/DELETE. Endpoint must be a string.
MCP Model Context Protocol server › 
TOOLS:
  search_knowledge(query: string, limit: number) -> object
    description: "Search the knowledge base via MCP"
    type: mcp
    mcp:
      server: "knowledge-base-server"
      tool: "search"         # defaults to tool name if omitted
Field Type Required Purpose
server string ● MCP server name (must be configured in project)
tool string ○ Tool name on server. Defaults to the ABL tool name.
MCP tools route to the configured MCP server. Parameters are forwarded as-is. Server must be registered in project MCP configuration.
MCP tools are the simplest binding — just point to a server. The server handles auth, execution, and response formatting.
Lambda Serverless function invocation › 
TOOLS:
  generate_pdf(claimId: string, format: string) -> {url: string}
    description: "Generate claim PDF document"
    type: lambda
    lambda:
      function: "claim-pdf-generator"    # logical name or ARN
      runtime: "nodejs20"              # nodejs20 | python3.12
      timeout: 60000
Field Type Required Purpose
function string ● Function name or ARN
runtime string ○ nodejs20 | python3.12
timeout number ○ Execution timeout (ms)
Use Lambda for compute-heavy tasks (PDF generation, image processing) that need more resources than sandbox provides.
Sandbox Inline code execution › 
TOOLS:
  calculate_premium(age: number, coverage: number, tier: string) -> number
    description: "Calculate insurance premium based on age and coverage"
    type: sandbox
    sandbox:
      runtime: javascript        # javascript | python
      timeout: 5000              # max execution time (ms)
      memoryMb: 128              # memory limit
      code: |
        const rates = { basic: 0.02, standard: 0.03, premium: 0.05 };
        const rate = rates[tier] || rates.standard;
        const ageFactor = age > 50 ? 1.5 : age > 30 ? 1.2 : 1.0;
        return Math.round(coverage * rate * ageFactor * 100) / 100;
Field Type Required Purpose
runtime string ● javascript | python
code string ○ Inline source code. Params available as variables.
timeout number ○ Max execution time (ms)
memoryMb number ○ Memory limit (MB)
Code runs in an isolated sandbox. Tool parameters are injected as local variables. Return value becomes the tool result.
Compiler validates: runtime must be javascript or python.
Great for calculations, data transforms, format conversions — anything that doesn't need external network access.
Async Webhook Long-running operations with callback › 
TOOLS:
  request_manager_approval(claimId: string, amount: number) -> {approved: boolean, comments: string}
    description: "Request manager approval for a claim (may take hours)"
    type: async_webhook
    async_webhook:
      endpoint: "https://approvals.internal/api/submit"
      method: POST
      headers:
        X-Source: "agent-platform"
      callbackUrlField: "callbackUrl"   # where to inject callback URL
      timeoutSeconds: 86400           # 24 hours max wait
Field Type Required Purpose
endpoint string ● URL to POST the request to
method string ○ POST | PUT | PATCH
headers Record ○ Custom headers
callbackUrlField string ○ Field name for the callback URL (default: "callbackUrl")
timeoutSeconds number ○ Max wait time (default: 3600 = 1hr)
Runtime sends the request with a callback URL injected. Session suspends until the external system calls back with the result. Session can be resumed by other messages in the meantime.
Use for human-in-the-loop approvals, external processing that takes minutes/hours, or any operation where you can't wait synchronously.
Tool Hints
TOOLS:
  fetch_policy(policyId: string) -> object
    description: "Retrieve policy details"
    hints:
      cacheable: true           # result can be cached
      latency: fast             # fast | medium | slow
      side_effects: false       # does NOT modify state
      requires_auth: true       # needs authentication
      timeout: 10000            # tool-specific timeout override
Hint Type Purpose
cacheable boolean If true, runtime may cache identical calls within a session
latency fast | medium | slow Helps runtime optimize parallel vs sequential execution
side_effects boolean If true, tool modifies external state (used by confirmation logic)
requires_auth boolean Whether tool needs authenticated session
timeout number Override the global tool_timeout for this specific tool
Tool Confirmation
TOOLS:
  submit_claim(data: object) -> {claimId: string}
    description: "Submit insurance claim — irreversible"
    confirmation:
      require: always            # always | never | when_side_effects
      immutableParams:
        - "policyId"             # these params can't be changed after confirmation

always

User must confirm every invocation. Use for irreversible actions like payments, submissions, deletions.

never

No confirmation needed. Use for read-only lookups and safe operations.

when_side_effects

Confirm only if hints.side_effects is true. Smart default for mixed tools.

immutableParams

Listed params are locked after first confirmation — prevents the LLM from silently changing critical values between retries.

Context Access & PII
TOOLS:
  update_profile(name: string, email: string) -> object
    description: "Update customer profile"
    contextAccess:
      read: ["customer_id", "session_type"]
      write: ["profile_updated"]
    piiAccess: user              # tools | user | logs | llm
    onResult:
      set:
        profile_updated: "true"
    onError:
      set:
        last_error: "profile_update_failed"
PII Level Meaning
tools PII only in tool params — not in logs or LLM context
user PII visible to user and tools but not logged
logs PII appears in logs (for debugging) but not in LLM context
llm PII fully visible to LLM — use with caution
Tool Imports
TOOLS:
  IMPORT: "project://common_tools.tool.abl"
  IMPORT: "project://payment_tools.tool.abl"
    only: ["process_payment", "refund"]   # import specific tools

  # Agent-specific tools alongside imports
  custom_calculator(x: number) -> number
    description: "Agent-specific calculation"
    type: sandbox
    sandbox:
      runtime: javascript
      code: |
        return x * 1.1;
Tool imports let you share tool definitions across agents. Define once in a .tool.abl file, import in many agents. Agent-specific tools can coexist with imports.
Validation Rules
Rule Level Message
Missing name Error Tool must have a name
Duplicate name Error Tool name "X" is already defined
Invalid tool_type Error Unknown tool type "X" — must be http | mcp | lambda | sandbox | async_webhook
Invalid HTTP method Error HTTP method must be GET | POST | PUT | PATCH | DELETE
Invalid sandbox runtime Error Sandbox runtime must be javascript | python
Impl props in agent DSL Warning endpoint, method, auth, code, runtime belong in project tool definitions
The compiler (tool-schema-validator.ts) runs all these checks. The parser also warns when implementation properties (endpoint, method, etc.) appear directly in agent DSL tools — these should typically live in project-level tool definitions.
Common Patterns
Read-only lookup
get_weather(city: string) -> {temp: number, forecast: string}
  description: "Get current weather"
  hints:
    cacheable: true
    latency: fast
    side_effects: false
Write with confirmation
delete_account(userId: string) -> {deleted: boolean}
  description: "Permanently delete user account"
  confirmation:
    require: always
  hints:
    side_effects: true
    latency: slow
Search with parallel queries
lookup_tool(queries: object[]) -> {results: array}
  description: "Search domain data across namespaces in parallel"
  parameters:
    queries:
      type: object[]
      required: true
      items:
        query: { type: string, required: true }
        namespace: { type: string, required: true }
        filters: { type: object }
Auto-set variables on result
create_order(items: array) -> {orderId: string}
  description: "Create purchase order"
  storeResult: true
  onResult:
    set:
      order_placed: "true"
      last_order_id: "{{result.orderId}}"
  onError:
    set:
      order_error: "true"
Anti-Patterns

Missing description

Tools without description confuse the LLM. It won't know when to use it. Always write a clear, actionable description.

Too many tools

More than 15-20 tools dilutes LLM attention. Use AVAILABLE_TOOLS in flow steps to scope. Or split into sub-agents.

Vague param names

data: object is useless. Be specific: claimDetails: {incident_date: string, description: string}

No error handling

Tools that can fail should have ON_ERROR handlers. Without them, the LLM gets a raw error and may hallucinate a response.

How Execution Works
EXECUTION configures the LLM engine: which model to use, how creative or deterministic it should be, timeout limits, thinking mode for complex reasoning, and per-operation model overrides. It's the performance and cost tuning knob for every agent.

All 15 Fields
Model Selection Primary, fallback, per-operation ›
Field Type Default Purpose
model string "claude-sonnet-4-20250514" Primary LLM model for all operations
fallback_model string none Used when primary model fails or times out
operation_models object none Per-operation model overrides (extraction, classification, generation) 
# Model resolution priority:
# 1. operation_models.{operation} — if the current operation matches
# 2. model — primary model
# 3. fallback_model — on primary failure

operation_models:
  extraction: "gpt-4o-mini"       # entity extraction — fast, cheap
  classification: "gpt-4o-mini"  # intent classification — fast, cheap
  generation: "claude-sonnet-4-20250514"  # response generation — quality
ModelResolutionService resolves the model per session. It checks operation type first, then falls back to primary, then fallback. This lets you use cheap models for extraction while keeping quality models for generation.
Generation Parameters Temperature, tokens ›
Field Type Default Purpose
temperature number 0.3 LLM creativity (0 = deterministic, 1 = creative). Low for factual tasks, higher for creative.
max_tokens number 4096 Maximum tokens in a single LLM response. Limits cost and response length.
For customer service agents, keep temperature at 0.1–0.3. For creative writing or brainstorming agents, use 0.7–0.9. Never use 1.0 in production — outputs become unreliable.
Timeouts 5 timeout types ›
Field Type Default Purpose
tool_timeout number (ms) 30000 Max time for a single tool execution. Prevents runaway API calls.
llm_timeout number (ms) 60000 Max time for a single LLM call. Prevents hanging on slow models.
session_idle_timeout number (ms) 300000 Session expires after this idle time (5 minutes default).
voice_latency_target number (ms) 500 Target latency for voice channel responses. Triggers optimizations. 
# Timeout cascade:
# tool_timeout (30s) — individual tool call
# llm_timeout (60s) — single LLM inference
# session_idle_timeout (300s) — session cleanup
# voice_latency_target (500ms) — optimization hint, not a hard limit
When tool_timeout fires, the tool returns an error and the LLM retries or uses a fallback. When llm_timeout fires, the runtime switches to fallback_model if available, otherwise returns error_default message.
Iteration Limits Reasoning and flow loop guards ›
Field Type Default Purpose
max_reasoning_iterations number 10 Max LLM reasoning loops per user message. Prevents infinite tool-calling chains.
max_flow_iterations number 100 Max flow step transitions per session. Prevents infinite flow loops.
If max_reasoning_iterations is hit, the agent stops reasoning and returns whatever it has. If max_flow_iterations is hit, the flow terminates with an error. Both indicate design issues — check for circular logic.
Thinking Mode Extended reasoning (Claude) ›
Field Type Default Purpose
enable_thinking boolean false Enable Claude's extended thinking mode for complex reasoning.
thinking_budget number 2048 Max tokens allocated to internal reasoning (not shown to user). 
EXECUTION:
  enable_thinking: true
  thinking_budget: 2048

# When enabled, Claude uses internal reasoning tokens
# to "think through" complex decisions before responding.
# Thinking tokens are not shown to the user.
# Costs more (thinking tokens are billed) but improves
# quality on multi-step reasoning, math, and analysis.
Thinking mode activates Claude's extended thinking. The model uses up to thinking_budget tokens for internal chain-of-thought reasoning before generating the visible response. These tokens are billed but not shown to the user.
Enable thinking for agents that do complex analysis, multi-step calculations, or policy evaluation. Disable for simple FAQ or routing agents — it adds latency and cost with no benefit.
Compaction Auto-summarization on long conversations ›
Field Type Default Purpose
compaction_threshold number 0.8 Trigger compaction when context usage reaches this ratio (0–1). 
EXECUTION:
  compaction_threshold: 0.8

# When conversation context reaches 80% of the model's
# context window, the runtime auto-summarizes older messages.
# Summary replaces old messages, freeing token space.
# Gathered values and key decisions are preserved.
Compaction triggers auto-summarization when context approaches the model's limit. The runtime summarizes older conversation turns into a compact summary, preserving gathered field values and key decisions. This prevents context overflow on long conversations.
Inline Gather Entity extraction alongside tools ›
Field Type Default Purpose
inline_gather boolean true Include _extract_entities tool alongside domain tools in reasoning mode. 
EXECUTION:
  inline_gather: true

# When true, the runtime injects an _extract_entities tool
# into the LLM's tool list during reasoning mode.
# The LLM can call it alongside domain tools to extract
# gather fields from the user's message in the same turn.
# When false, entity extraction runs as a separate pre-pass.
With inline_gather: true, the LLM sees _extract_entities as a callable tool during reasoning. This lets it extract gather fields and call domain tools in a single reasoning turn, reducing round-trips.
Voice Latency Optimization
EXECUTION:
  voice_latency_target: 500   # 500ms target for voice
  model: "claude-sonnet-4-20250514"
  fallback_model: "gpt-4o-mini"  # faster fallback for voice

# When voice_latency_target is set:
# 1. Runtime prefers streaming responses
# 2. Shorter max_tokens for faster first-token
# 3. May switch to fallback_model if primary is too slow
# 4. Compaction runs more aggressively (smaller context)
The voice_latency_target is an optimization hint, not a hard timeout. The runtime uses it to tune streaming behavior, context size, and model selection for voice channels. Lower targets may trigger automatic model downgrades to meet latency.
Common Patterns
Fast Responses Low latency, low cost › 
EXECUTION:
  model: "gpt-4o-mini"
  temperature: 0.1
  max_tokens: 1024
  max_reasoning_iterations: 3
  enable_thinking: false
  voice_latency_target: 300
Best for: FAQ bots, simple routing, quick lookups. Low temperature for consistent answers. Small model for speed. Limited iterations to prevent over-thinking.
Deep Reasoning High quality, complex analysis ›

  model: "claude-sonnet-4-20250514"
  temperature: 0.5
  max_tokens: 8192
  max_reasoning_iterations: 15
  enable_thinking: true
  thinking_budget: 4096
  fallback_model: "gpt-4o"

Best for: Claims processing, financial analysis, medical triage. Thinking mode for step-by-step reasoning. Higher token limits for detailed responses. More iterations for multi-tool workflows.

Cost-Optimized Cheap models for bulk operations › 
EXECUTION:
  model: "gpt-4o-mini"
  temperature: 0.1
  max_tokens: 2048
  enable_thinking: false
  operation_models:
    extraction: "gpt-4o-mini"
    classification: "gpt-4o-mini"
    generation: "gpt-4o-mini"
Best for: High-volume, low-complexity tasks. Same cheap model for all operations. No thinking overhead. Useful for triage, tagging, and simple data collection.
Anti-Patterns

Thinking on simple agents

Enabling enable_thinking on FAQ or routing agents. Adds 1–3 seconds of latency and extra cost for no quality gain.

No fallback model

Running without fallback_model. When the primary model has an outage, every request fails instead of degrading gracefully.

High temperature for data collection

Setting temperature > 0.5 for agents that collect structured data. The LLM hallucinates field values and validation becomes unreliable.

Unlimited iterations

Setting max_reasoning_iterations: 100. A confused LLM will loop calling tools endlessly, burning tokens and time.

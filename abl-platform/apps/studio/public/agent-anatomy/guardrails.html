GUARDRAILS

Content safety checks on input and output. Three evaluation tiers from zero-cost regex to full LLM evaluation. Seven action types from blocking to auto-fix.

How Guardrails Work
Guardrails are content filters — they evaluate user input and agent output against rules. Unlike CONSTRAINTS (business rules on data), guardrails are about content safety: preventing prompt injection, PII leaks, toxic output, unauthorized commitments.

Input guardrails fire before the LLM call — a blocked input never reaches the model. Output guardrails fire twice: once as streaming (async, fire-and-forget for fast feedback) and once as a final batch check after the reasoning loop completes. The pipeline evaluates Tier 1 → 2 → 3 in order. A terminal action (block, escalate) stops the pipeline.
Guardrail Structure — All Fields
GUARDRAILS:
  guardrail_name:
    kind: input              # when to check
    check: "CEL expression"   # Tier 1: local pattern
    provider: "model-name"    # Tier 2: ML classifier
    llm_check: "question"     # Tier 3: LLM evaluation
    action: block             # what to do on violation
    message: "User message"   # message shown to user
    priority: 10              # lower = higher priority
Field Type Required Purpose
name string ● Unique guardrail identifier
kind enum ● When to evaluate: input | output | both | tool_input | tool_output | handoff
check string ○ Tier 1: CEL expression (zero cost, <1ms)
provider string ○ Tier 2: ML classifier name (e.g., "openai-moderation")
category string ○ Tier 2: Classification category (e.g., "hate", "violence")
threshold number ○ Tier 2: Confidence threshold (0.0–1.0)
llm_check string ○ Tier 3: Natural language question for LLM to evaluate
action enum ● block | warn | redact | escalate | fix | reask | filter
message string ○ Message shown to user (or replacement text for redact)
priority number ○ Evaluation order. Lower = evaluated first. Default: 100
severity_actions Record ○ Graduated actions by severity level (safe/low/medium/high/critical)
fix_strategy string ○ For action:fix — truncate | strip_html | redact_pii | normalize | custom
fix_expression string ○ CEL expression for custom fix strategy
max_reasks number ○ For action:reask — max regeneration attempts. Default: 2
filter_min_length number ○ For action:filter — minimum remaining length
streaming boolean ○ Evaluate during streaming (output guardrails only)
streaming_interval string ○ token | sentence | chunk_size
6 Kinds — When to Check
Kind Evaluates When Use Case
input User message Before LLM call Prompt injection, PII detection, content filtering
output Agent response After LLM call (streaming + batch) Toxicity, credential leak, compliance
both User + agent Both directions Universal content rules
tool_input Tool arguments Before tool execution Prevent malicious tool parameters
tool_output Tool results After tool execution Filter sensitive data from tool responses
handoff Context bundle Before handoff to another agent Prevent PII leaking across agent boundaries
3 Evaluation Tiers

Tier 1 — Pattern

Cost: Zero
Speed: <1ms
Trigger: check: field
Method: CEL expression with abl.* functions
Best for: Regex patterns, keyword matching, length checks

Tier 2 — Model

Cost: API call
Speed: ~100ms
Trigger: provider: field
Method: External ML classifier
Best for: Toxicity, sentiment, content classification

Tier 3 — LLM

Cost: LLM call
Speed: 1–3s
Trigger: llm_check: field
Method: Natural language evaluation
Best for: Nuanced checks, compliance, commitment detection

Tier 1 — Pattern Matching (CEL) Zero cost, <1ms › 
GUARDRAILS:
  # SSN detection
  ssn_detection:
    kind: input
    check: not_matches_pattern(input, "\\b\\d{3}-\\d{2}-\\d{4}\\b")
    action: redact
    message: "[SSN REDACTED]"

  # Credit card detection
  cc_detection:
    kind: input
    check: not_matches_pattern(input, "\\b\\d{4}[- ]?\\d{4}[- ]?\\d{4}[- ]?\\d{4}\\b")
    action: redact
    message: "[CARD REDACTED]"

  # Prompt injection
  injection_guard:
    kind: input
    check: |
      not_matches_pattern(input.lower(),
        "ignore (previous|prior|all|above) (instructions|rules|prompts)")
      AND not_matches_pattern(input.lower(),
        "(act as|pretend to be|you are now) (admin|root|system)")
    action: block
    message: "I cannot process that request."

  # Message length limit
  length_guard:
    kind: input
    check: abl.length(input) < 5000
    action: block
    message: "Message too long. Please keep it under 5000 characters."

  # Output credential leak prevention
  credential_leak:
    kind: output
    check: not_matches_pattern(output, "(Bearer |api[_-]?key|secret|token)\\s*[:=]\\s*[A-Za-z0-9]{20,}")
    action: redact
    message: "[CREDENTIALS REDACTED]"
Available CEL Functions
Category Functions
Pattern matches_pattern(text, regex), not_matches_pattern(text, regex)
String abl.upper, abl.lower, abl.trim, abl.substring, abl.replace, abl.split
Length abl.length(text)
Type abl.is_string, abl.is_number, abl.is_array
Utility abl.coalesce, abl.now
Tier 1 guardrails run locally with zero LLM cost. Use them for all pattern-based checks: PII, injection, length, keyword blocking. They're evaluated first and can short-circuit expensive Tier 2/3 checks.
Tier 2 — Model-Based Classification ~100ms, API call ›

  # OpenAI moderation API
  hate_speech:
    kind: output
    provider: "openai-moderation"
    category: "hate"
    threshold: 0.7
    action: block
    message: "Response blocked for safety."

  violence_check:
    kind: output
    provider: "openai-moderation"
    category: "violence"
    threshold: 0.8
    action: warn

  # Graduated severity actions
  toxicity_graduated:
    kind: output
    provider: "openai-moderation"
    category: "harassment"
    severity_actions:
      low: warn
      medium: reask
      high: block
      critical: escalate
Field Purpose
provider ML classifier name (registered in project guardrail config)
category What to check: hate, violence, self-harm, sexual, harassment
threshold Confidence threshold 0.0–1.0. Higher = stricter.
severity_actions Different actions at different severity levels
Tier 2 evaluators send content to an external ML API. The provider must be configured in the project's guardrail provider registry (provider-registry.ts).
Tier 3 — LLM Evaluation 1–3s, LLM call › 
GUARDRAILS:
  # Compliance check
  unauthorized_commitments:
    kind: output
    llm_check: "Does this response make any unauthorized promises, guarantees, or financial commitments that the agent is not authorized to make?"
    action: reask
    max_reasks: 2

  # Tone check
  professional_tone:
    kind: output
    llm_check: "Is this response professional, empathetic, and appropriate for a customer service context? Flag if condescending, dismissive, or rude."
    action: reask
    max_reasks: 1

  # Factual grounding
  hallucination_check:
    kind: output
    llm_check: "Does this response contain claims that are not supported by the tool results or conversation context? Flag unsupported facts."
    action: reask
Tier 3 sends the content to an LLM with the llm_check as a question. The LLM returns a yes/no verdict with reasoning. Most expensive but handles nuanced, context-dependent checks.
Use sparingly — each Tier 3 check adds 1–3s latency and costs an LLM call. Reserve for compliance, hallucination, and tone checks that regex/ML can't handle.
7 Action Types
block Terminal — reject content entirely › 
injection_guard:
  kind: input
  check: not_matches_pattern(input.lower(), "ignore previous instructions")
  action: block
  message: "I cannot process that request."

Behavior: Stops pipeline. Input blocks prevent LLM call. Output blocks suppress the response. The message is sent to the user instead.

warn Non-terminal — log and continue › 
suspicious_pattern:
  kind: input
  check: not_matches_pattern(input, "\\b(hack|exploit|bypass)\\b")
  action: warn
  message: "Suspicious content detected"

Behavior: Logs the violation to trace store. Content continues through pipeline. Warning injected into system prompt on next turn.

redact Non-terminal — replace sensitive content › 
ssn_redact:
  kind: input
  check: not_matches_pattern(input, "\\b\\d{3}-\\d{2}-\\d{4}\\b")
  action: redact
  message: "[SSN REDACTED]"

Behavior: Matched content is replaced with the message text. The redacted version continues through the pipeline. Original never reaches the LLM (for input) or the user (for output).

escalate Terminal — route to human agent › 
severe_violation:
  kind: input
  check: not_matches_pattern(input, "threat|harm|illegal")
  action: escalate
  message: "This conversation requires human review."

Behavior: Stops pipeline. Sets session.isEscalated = true. Routes to human agent with full context. Use for severe safety violations.

fix Non-terminal — auto-repair content › 
html_strip:
  kind: output
  check: not_matches_pattern(output, "<script|<iframe|javascript:")
  action: fix
  fix_strategy: strip_html

length_trim:
  kind: output
  check: abl.length(output) < 2000
  action: fix
  fix_strategy: truncate

pii_auto_redact:
  kind: output
  check: not_matches_pattern(output, "\\b\\d{3}-\\d{2}-\\d{4}\\b")
  action: fix
  fix_strategy: redact_pii
Fix Strategies
truncatestrip_htmlredact_piinormalizecustom (CEL)

Behavior: Automatically repairs the content using the specified strategy. The fixed version continues through the pipeline.

reask Conditional — ask LLM to regenerate (output only) › 
compliance_check:
  kind: output
  llm_check: "Does this make unauthorized commitments?"
  action: reask
  max_reasks: 2       # max regeneration attempts

Behavior: Asks the LLM to regenerate the response with the guardrail violation feedback. Retries up to max_reasks times. Falls back to block if all retries fail. Only works on output guardrails.

filter Non-terminal — remove violating portions › 
profanity_filter:
  kind: output
  check: not_matches_pattern(output, profanity_regex)
  action: filter
  filter_min_length: 10   # don't filter if result would be too short

Behavior: Removes the matching portions while preserving the rest. If the remaining content is shorter than filter_min_length, falls back to block.

Action Comparison
Action Terminal? Input Output Modifies Content? Use Case
block Yes ✓ ✓ No — replaces Severe violations, injection
warn No ✓ ✓ No Logging, soft boundaries
redact No ✓ ✓ Yes — replaces matched PII, credentials
escalate Yes ✓ ✓ No — routes to human Severe safety issues
fix No ✗ ✓ Yes — repairs HTML stripping, truncation
reask Conditional ✗ ✓ Yes — regenerates Compliance, tone
filter No ✗ ✓ Yes — removes portions Profanity, sensitive topics
5 Built-in Guardrail Templates

Available from packages/compiler/src/platform/guardrails/builtin-templates.ts. All are Tier 1 (zero cost).

detect_instruction_override Input · Block › 
detect_instruction_override:
  kind: input
  check: |
    not_matches_pattern(input.lower(),
      "ignore (previous|prior|all|above) (instructions|rules|prompts)")
  action: block
  message: "I cannot process that request."

Catches: "ignore previous instructions", "disregard all prior rules", "forget your system prompt"

Streaming Guardrails
realtime_toxicity:
  kind: output
  provider: "openai-moderation"
  category: "hate"
  threshold: 0.7
  action: block
  streaming: true
  streaming_interval: sentence   # token | sentence | chunk_size
Interval Behavior Latency
token Evaluate every token (highest cost) Per-token delay
sentence Evaluate at sentence boundaries Minimal delay
chunk_size Evaluate at chunk boundaries Configurable
Streaming guardrails run async alongside the LLM response stream. If a violation is detected mid-stream, the response is truncated and the user sees the guardrail message. The final batch check after streaming catches anything the streaming check missed.
Graduated Severity Actions
content_safety:
  kind: output
  provider: "openai-moderation"
  category: "harassment"
  severity_actions:
    safe: warn           # just log it
    low: warn            # log + inject warning
    medium: reask         # ask LLM to regenerate
    high: block           # block the response
    critical: escalate    # route to human
safelowmediumhighcritical
Graduated actions let you handle different violation severities differently. A mildly inappropriate response gets a reask; a severely harmful one gets escalated. Much better UX than binary block/allow.
Guardrails vs Constraints — When to Use Which
Aspect Guardrails Constraints
Purpose Content safety (what is said) Business rules (what is allowed)
Evaluates Message text content Session data / variables
Example "Block SSN in user messages" "Require policy to be active"
Timing Before/after LLM call Before processing + between steps
Method Regex, ML, LLM evaluation CEL expressions on session values
Actions block, warn, redact, fix, reask, filter, escalate respond, block, escalate, handoff, collect_field, goto_step, retry_step
Use guardrails for content-level checks (injection, PII, toxicity, compliance). Use constraints for business logic checks (eligibility, limits, required data).

GUARDRAILS:
  # Tier 1 — fast pattern checks first
  pii_guard:
    kind: both
    check: not_matches_pattern(text, SSN_OR_CC_REGEX)
    action: redact
    priority: 1

  # Tier 2 — ML classification
  toxicity:
    kind: output
    provider: "openai-moderation"
    category: "hate"
    threshold: 0.6
    action: block
    priority: 50

  # Tier 3 — LLM for nuanced checks (last, most expensive)
  compliance:
    kind: output
    llm_check: "Does this make unauthorized commitments?"
    action: reask
    max_reasks: 2
    priority: 100
Anti-Patterns

All Tier 3

Using llm_check for everything. Each adds 1–3s latency. Use Tier 1 for patterns, Tier 2 for classification, Tier 3 only for nuanced checks.

No priority ordering

Without priorities, cheap Tier 1 checks might run after expensive Tier 3. Always set priority: lower number = runs first.

Reask on input

action: reask only works on output. On input it silently does nothing. Use block or redact for input.

Overly strict regex

SSN regex catching phone numbers. Credit card regex catching tracking numbers. Test your patterns against real data.

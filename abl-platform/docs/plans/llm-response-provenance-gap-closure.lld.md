## 1. Scope

This plan closes the remaining provenance propagation gaps after the Web SDK consumer contract was proven in isolation. The implementation now covers:

1. live runtime websocket delivery
2. HTTP/internal chat live response and persistence
3. persistence, session detail, and history readback
4. Studio/Web SDK live hydration plus trace-only recovery
5. async resume, pending replay, and cross-pod websocket delivery
6. async channel webhooks/direct-send metadata
7. direct-channel outcome shaping and A2A structured-response parity
8. omnichannel transcript backfill
9. one black-box runtime SDK E2E
10. final live websocket/resume canonical metadata forwarding
11. publish-safe client typing parity for Web SDK and Studio
12. active session-detail provenance preservation before persisted history takes over
13. Studio replay-side provenance enrichment for existing metadata-less assistant messages
14. web debug proactive greeting parity for structured-only assistant payloads
15. paginated history/backfill parity with active runtime tails
16. in-memory `contentEnvelope` parity for active detail and websocket resume snapshots
17. voice assistant-history metadata parity
18. resume snapshot metadata parity from runtime to Studio
19. KoreVG realtime/S2S transcript provenance for assistant turns
20. active-session cursor pagination fallback for long persisted histories

## 2. Canonical Contract

```ts
type ResponseProvenanceKind = 'scripted' | 'llm' | 'mixed';

interface ResponseProvenance {
  schemaVersion: 1;
  kind: ResponseProvenanceKind;
  disclaimerRequired: boolean;
  usedLlmInternally: boolean;
}

interface ResponseMessageMetadata {
  isLlmGenerated: boolean;
  responseProvenance: ResponseProvenance;
}
```

### Semantics

1. `isLlmGenerated` means customer-visible response content was generated by an LLM.
2. Internal-only LLM work does not flip the flag by itself.
3. `kind: 'mixed'` means customer-visible LLM output happened in the same turn as internal-only LLM work.
4. Simulated fallback traces are ignored.
5. Non-LLM fallback outcomes for async/webhook channels stay `scripted`.

## LLD outlines (items 1, 3, 4)

### Item 1 (CP4) — Platform-mock amnesty

**Root cause:** route handler tests like `apps/studio/src/app/api/**/__tests__/*.test.ts`
import real route handlers, then `vi.mock('@agent-platform/arch-ai')`,
`vi.mock('@agent-platform/design-tokens')`, etc. to shortcut through real
service logic. Result: the test file's imports fail silently in production
wiring, tests pass anyway.

**Approach: 2-phase strangler.**

1. **Audit phase (CP4 commit 1):** write `docs/plans/2026-04-16-platform-mock-cleanup.md`.
   For each `vi.mock(<internal>)`:
   - Classify: (a) extract pure function, (b) dependency-inject the service,
     (c) refactor the test to use real service + test harness.
   - Record expected file churn per violation.
2. **Cleanup phase (CP4 commits 2–3):** batches of 10–15 files.
   - Never delete a test; always refactor it to use real services.
   - For each internal dependency, extract the pure logic into a testable
     module (pattern: `provider-cache.ts` extracted from `session-llm-client.ts`).
   - The `.claude/hooks/platform-mock-lint.sh` hook already blocks new violations;
     Phase 2 in a follow-up run will tighten to zero tolerance.

**Exit criteria:** platform-mock violation count (grep of `vi.mock.*@agent-platform\|vi.mock.*@abl`)
drops from 77 to ≤30 in this run; the rest have written cleanup plans.

### Item 3 (CP3) — Stream stop/abort

**Root cause:** `apps/studio/src/hooks/useArchChat.ts` uses `fetch('/api/arch-ai/message', {...})`
with no `AbortController`. Once a message is in flight, the UI cannot cancel it.
The backend still streams to completion and writes journal entries. User has no
way to recover from a wrong prompt mid-stream except waiting it out (BUILD phase
can take 5+ minutes).

**Approach:**

1. Add `abortControllerRef: React.MutableRefObject<AbortController | null>` to
   `useArchChat` hook state.
2. When a send starts, allocate a fresh `AbortController`, pass `signal` into
   `fetch`.
3. Expose `stop()` function from the hook: calls `abortControllerRef.current?.abort()`,
   transitions streaming state to `aborted`, emits a local journal entry
   `{ type: 'analysis', summary: 'User stopped stream' }` (client-side marker; the
   backend's journal remains canonical).
4. UI: send button becomes a stop button while `streamingState === 'streaming'`.
   Use the existing disabled-input logic from CP2 #2 to gate the stop button (can
   only stop while streaming).
5. Backend side: the `AbortController.abort()` closes the fetch, which causes
   `response.body.getReader().read()` to reject. On the server, Next.js route
   handler sees the connection close and the coordinator should (already does
   via SSE contract §Reconnection) handle partial state. No backend changes
   required for v1 — the next user message implicitly resumes.
6. Edge case: if the user aborts mid-widget (e.g. during `ask_user`), the
   `pendingInteraction` already persisted in session metadata remains. Next
   mount will re-render the widget correctly via the resume path (validated by
   CP2 #4 session-recovery work).

**Exit criteria:**

- Stop button visible during streaming; click stops the request within 100ms.
- `AbortController.abort()` is called with explicit reason `'user_stopped'` so
  downstream telemetry can distinguish abort from error.
- E2E test: send a BUILD-phase message, click stop after 2 seconds, assert
  `streamingState === 'idle'` and no `done` SSE event fired client-side.
- No console error spam on abort (filter `AbortError` in the fetch catch).

### Item 4 (CP5) — `spec-document` tests

**Root cause:** `packages/arch-ai/src/spec-document/` has zero test files per
the review. It's the schema source of truth for Interview phase and the
persistence layer for `update_specification` tool calls. Silent corruption here
breaks the whole Arch flow.

**Approach: extract-then-test, not mock-then-test.**

1. Read the `spec-document/` module and inventory exported functions.
2. For each function:
   - If it's pure (`input → output`, no side effects) — write direct unit tests.
   - If it touches MongoDB — extract the pure transformation into a sibling
     module, make the DB-facing function a thin wrapper. Test the pure module.
   - If it uses a transaction — test the transaction path with a real in-memory
     MongoDB (not a mock). Use `@shelf/jest-mongodb` or the existing test DB
     pattern from elsewhere in `packages/arch-ai/`.
3. Minimum coverage target for CP5: 80% statements on the pure modules, 60% on
   transaction paths. Zero `vi.mock` of anything under `@agent-platform/*`.
4. Commit 1: schema + pure operations (expected to be ~40% of module).
5. Commit 2: transaction/persistence paths including error handling (the
   "logged but not re-thrown" P2 bug becomes a regression test here).

**Exit criteria:**

- `spec-document/` module has ≥ 15 test cases.
- Zero platform mocks.
- `spec-document` transaction errors fail loudly with a typed error.
- Coverage report in `docs/arch/14-test-coverage.md` updated.

locally in docs/plans/2026-04-16-arch-v03-7-to-9-plan.md`

No `--no-verify`. No `--amend` after hook failure. No Co-Authored-By lines.

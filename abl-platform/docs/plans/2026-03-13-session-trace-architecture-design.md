# Session & Trace Architecture Simplification

**Date**: 2026-03-13
**Status**: Design complete — ready for implementation planning

---

## Problem Statement

The current session and trace architecture has accumulated complexity that causes real bugs:

1. **Dual session IDs**: Every session has a MongoDB `_id` (UUIDv7) and a runtime UUID (`crypto.randomUUID()`). The mapping is stored in `Session.runtimeSessionId`. Every trace lookup requires resolving one to the other, with fallback logic scattered across endpoints.

2. **Three-layer trace storage**: Trace events live in an in-memory TraceStore (per-pod, ephemeral), get batched to ClickHouse via EventStore (5-second buffer), and are queried through a waterfall of in-memory → ClickHouse fallbacks. Events in the buffer window are lost on pod restart.

3. **Zombie sessions**: WebSocket `web_debug` sessions use `detach` on disconnect, keeping the session in the RuntimeExecutor's memory Map. The DB marks the session as ended, but the executor still holds it. The traces endpoint's ownership check then produces false 404s because the stale executor entry has mismatched metadata.

4. **No graceful EventStore flush**: On shutdown, the ClickHouse BufferedWriter's pending events are discarded because `eventStore.close()` is never called before `closeClickHouseClient()`.

### Observed Bug (2026-03-13)

Session `019ce2c4-6b92-...` had 4 messages and 11,730 tokens but only 1 trace event (`session_created`) in ClickHouse. Root cause chain:

- Session detached (not ended) in executor on WS disconnect
- Traces endpoint found the stale executor entry → ownership check failed → 404
- Other trace events were lost because the runtime was restarted without flushing the EventStore buffer

## Design Decisions

### Q&A Log

**Q1: What's the target deployment topology?**
A: **Multi-pod with any-pod routing (Scenario C)**. Any message can land on any pod. Sessions must be resumable anywhere. This is the hardest case and the right one to design for.

**Q2: What's the source of truth for session state?**
A: Redis should NOT store session state — Redis memory is expensive and session state can be large (IR blobs, conversation history, gather data). Redis stays thin and bounded.

**Q3: Tolerance for data loss on pod crash?**
A: **Milestone-based persistence (Option D)**. Accept that in-flight message processing is lost on pod crash. The user's last message is in MongoDB; the LLM response can be regenerated by replaying. Session structural state (current agent, flow step, gather progress) is persisted to MongoDB on milestones (message completion, handoff, session end) asynchronously.

**Q4: What about hundreds of pods × thousands of sessions hitting MongoDB?**
A: Milestone-based writes are per-session, but with consistent-hash load balancing the writes are spread across pods naturally. Each pod writes its own sessions. Not a write storm because milestones are spaced by user message intervals (seconds to minutes).

**Q5: How do sessions route to the same pod?**
A: **Consistent hash on session ID at the load balancer** (Istio `DestinationRule` with consistent hashing, or Nginx `hash $http_x_session_id consistent`). This gives pod affinity without application-level routing. WebSocket connections are naturally sticky. HTTP API requests are consistently routed. When pods scale up/down, only ~1/N sessions remap → rehydration.

**Q6: Should WAL be enabled by default for trace events?**
A: **Yes**. The WAL infrastructure already exists (`packages/eventstore/src/resilience/`). Enable `EVENTSTORE_RESILIENCE_ENABLED=true` by default. Cost is ~0.1ms per event for a disk append. Prevents trace event loss on crash.

**Q7: What about the dual session ID problem?**
A: **Single session ID**. Use the runtime UUID (`crypto.randomUUID()`) as the MongoDB `_id`. Pass it explicitly when creating the DB session record. The `runtimeSessionId` column becomes redundant. All dual-lookup fallback code (`findSessionByRuntimeId`) can be removed.

**Q8: Should active sessions hot-swap IR when a new agent version is deployed?**
A: **No — session IR is pinned for lifetime by deployment.** After investigating the codebase, this is already how it works:

- `AgentVersion.irContent` stores every compiled IR version permanently in MongoDB
- `Deployment.agentVersionManifest` maps agent names → pinned version numbers
- `RuntimeSession.versionInfo` is set at session creation and never changes
- On rehydration (pod crash/switch), the serialized session snapshot includes the pinned IR — it does NOT re-resolve from the latest deployment
- New deployments only affect **new sessions**. Existing sessions keep their pinned versions until they end.

This is simpler and correct. No Redis Pub/Sub needed for IR invalidation. The only Pub/Sub channels remain: KMS cache invalidation (existing) and any future cross-pod notifications.

## Architecture Design

### Storage Roles

| Component                       | Role                                                                                               | Hot path?                                                     |
| ------------------------------- | -------------------------------------------------------------------------------------------------- | ------------------------------------------------------------- |
| **In-memory** (RuntimeExecutor) | Working session state — full RuntimeSession with IR, history, handoff stack                        | Yes — zero latency                                            |
| **MongoDB**                     | Durable store — sessions, messages, agents, milestone snapshots                                    | Write: async on milestones. Read: on rehydration              |
| **ClickHouse**                  | Trace events — WAL-backed buffered writes                                                          | Write: per event (via WAL + buffer). Read: historical queries |
| **Redis**                       | Distributed locks, rate limiting, quota tracking, pub/sub (version invalidation, KMS invalidation) | Bounded, thin                                                 |

### What Goes Away

- Redis SessionService as session state store (L1/L2 cache layer)
- Dual session IDs and all dual-lookup fallback code
- `detach` vs `end` zombie session problem (simplified lifecycle)
- Stale executor ownership checks in traces endpoint
- TraceStore as a separate query authority (ClickHouse becomes the single trace source for queries)

### Session Lifecycle

```
CREATE (any pod)
  → RuntimeExecutor.createSessionFromResolved()
  → session.id = crypto.randomUUID()    ← this IS the MongoDB _id
  → MongoDB: create Session record with _id = session.id
  → TraceStore: register session
  → EventStore: emit session_created → WAL → ClickHouse

MESSAGE (consistent-hash routes to same pod)
  → RuntimeExecutor processes message in-memory
  → TraceEmitter: emit events → WAL → ClickHouse buffer
  → MongoDB: persist messages (async queue)
  → MongoDB: persist session milestone snapshot (async, on message completion)

POD SWITCH (consistent hash changes on scale event, or pod crash)
  → New pod receives message for unknown session
  → SessionService: restore session from Redis snapshot (includes pinned IR)
  → If Redis miss: MongoDB read session + messages, recompile pinned IR version
  → Rebuild RuntimeSession in-memory
  → Continue processing — same IR version, same state

AGENT VERSION DEPLOYED
  → New deployment record created in MongoDB
  → Existing sessions: unaffected (IR pinned by versionInfo from creation)
  → New sessions: resolve against new deployment → get new IR versions

SESSION END
  → MongoDB: final session record (status, totals, timestamps)
  → RuntimeExecutor: remove from memory
  → EventStore: emit session_ended → WAL → ClickHouse
```

### Trace Pipeline (Simplified)

```
Event generated
  → TraceEmitter.emit()
    → WAL append (0.1ms, crash-safe)
    → In-memory TraceStore (for live WebSocket streaming to Studio)
    → EventStore buffer → ClickHouse (5-second batch)

Query path:
  → Live session on this pod? → TraceStore (in-memory, instant)
  → Historical / other pod?   → ClickHouse (single source of truth)

No more waterfall fallback logic. No more "try memory, then try ClickHouse,
then try alternate session ID."
```

### Single Session ID

```
Before (dual ID):
  MongoDB _id:  019ce2c4-6b92-76e8-8dd9-4e6aa714c217  (UUIDv7)
  Runtime UUID: 6caba522-43d3-4871-9e3f-62c3a22ed371  (crypto.randomUUID)
  Linked by:    Session.runtimeSessionId column

After (single ID):
  Everywhere:   6caba522-43d3-4871-9e3f-62c3a22ed371
  MongoDB _id = RuntimeSession.id = TraceStore key = ClickHouse session_id
```

Migration: The `runtimeSessionId` column stays for backward compatibility with existing data. New sessions use the unified ID. Old sessions continue to work via the existing fallback code until they age out.

### IR Version Pinning (Already Implemented)

Sessions are pinned to a deployment at creation time. The IR never changes during a session's lifetime.

```
Persistence chain:
  AgentVersion.irContent         → compiled IR blob per version (permanent in MongoDB)
  Deployment.agentVersionManifest → { "agent_a": "1.2.3", "agent_b": "1.0.0" }
  RuntimeSession.versionInfo     → { deploymentId, environment, versions: { ... } }
  SessionService snapshot        → serialized IR included for cross-pod rehydration

Rehydration path:
  1. SessionService.loadSession(sessionId) → HydratedSession with agentIR + compilationOutput
  2. Rebuild RuntimeSession with pinned IR (no re-resolution from latest deployment)
  3. If Redis miss: fall back to MongoDB → load AgentVersion by pinned version numbers
```

## Immediate Fixes (Implemented 2026-03-13)

### Fix #1: Graceful Shutdown Flush

**File**: `apps/runtime/src/server.ts`

Added `eventStore.store.close()` and `eventStore.emitter.close()` to the shutdown sequence, before `closeClickHouseClient()`. This flushes the BufferedWriter's pending events to ClickHouse on SIGTERM/SIGINT.

### Fix #2: Traces Endpoint Ownership Check

**File**: `apps/runtime/src/routes/sessions.ts`

When `findSessionById(sessionId, tenantId)` succeeds (tenant-scoped query), skip the RuntimeExecutor ownership check. The DB lookup already verified tenant isolation. The executor check was producing false 404s for detached sessions with stale metadata.

## Open Questions

1. ~~**IR hot-swap state compatibility**~~ — **Resolved**: Sessions pin IR for lifetime. No hot-swap, no compatibility issue.

2. **Rehydration performance**: SessionService snapshot in Redis includes serialized IR — so the typical rehydration is a Redis GET (~1ms) not a MongoDB read + recompile. MongoDB fallback (~25-45ms) only happens if Redis entry expired. Is the Redis TTL for session snapshots long enough?

3. ~~**Migration strategy**~~ — **Resolved**: Platform is dev-only. Clean cut: wipe all sessions (MongoDB `sessions` + `messages` collections, Redis session keys, optionally ClickHouse `platform_events`). Deploy single-ID code. Remove all `findSessionByRuntimeId` fallback code and `runtimeSessionId` column. No backward compatibility needed.

4. ~~**TraceStore role post-simplification**~~ — **Resolved**: TraceStore stays as a **live-streaming layer only** (not a query source). It's backed by `RedisTraceStore` (Redis Streams + Pub/Sub) which already handles cross-pod streaming. Studio live debugging subscribes via WS → RedisTraceStore. Historical/REST queries go to ClickHouse only. No waterfall fallback logic needed.

5. ~~**Consistent hash rebalancing**~~ — **Resolved**: 0.5% rehydration on scale events is acceptable. One-time ~25-45ms hit per affected session. Load balancer is **Nginx** with `hash $http_x_session_id consistent` for pod affinity. WebSocket connections are naturally sticky.

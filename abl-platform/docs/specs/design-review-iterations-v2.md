# Crawl Together — 7-Iteration Design Review (v2)

Review of the revised architecture: Search-AI self-contained, CrawlIntelligenceService,
intent + sample URL, 5-phase execution, PageHandlers, Crawl Worker.

Each iteration: (1) question the assumption, (2) find the problem, (3) counter with
prior learnings, (4) net assessment.

---

## Iteration 1: Can CrawlIntelligenceService Replace an Agent Reasoning Loop?

### The Assumption

The revised design replaces the ABL Runtime's reasoning loop (LLM → tool call →
result → LLM → repeat) with a `CrawlIntelligenceService` that makes 8-12 targeted
LLM calls. The service "knows" the workflow — it doesn't need the LLM to decide
what to do next. Each phase is a method with deterministic orchestration and
specific LLM prompts.

### The Problem

An agent reasoning loop is ADAPTIVE. The LLM decides what to do based on what it
sees. A service with predefined phases is RIGID. What happens when the site doesn't
fit the 5-phase model?

1. **No sitemap, no obvious URL pattern.** Phase 0 returns nothing useful. Phase 1
   opens the sample URL but there's no list of target URLs to filter. The service
   can't proceed to Phase 2 because it doesn't know what pages exist. An agent
   would explore — click links, follow navigation, discover pages dynamically. The
   service has no exploration mode.

2. **Sample URL doesn't represent the target content.** User says "crawl all
   product specs" but the sample URL is a listing page, not a product page. The
   service builds a handler for listing pages, then the worker tries to run it on
   product pages. Mismatch.

3. **Multi-step authentication.** The site requires login → CAPTCHA → 2FA before
   any content is visible. Phase 1 opens the sample URL and gets a login page.
   The service doesn't have an "authenticate" phase. An agent would interact with
   the user about credentials.

### Counter (Prior Learnings)

From Iteration 7 (v1): "The runtime's reasoning loop is synchronous — blocks one
LLM slot for 2 hours." The 5-phase model explicitly avoids this. The tradeoff is
intentional: lose adaptivity, gain efficiency and reliability.

From external research: Firecrawl uses the same rigid pipeline (Map → Crawl →
Extract). They handle edge cases by failing fast and telling the user, not by
being adaptive.

**For problem 1 (no sitemap):** Add a fallback in Phase 0. If HTTP discovery
returns < 10 URLs, fall back to browser-based discovery: open the main URL,
extract all links, classify them. This is 1-2 additional LLM calls, not a full
reasoning loop. The service grows a `discoverViaBrowser()` fallback method.

**For problem 2 (wrong sample URL):** Phase 1 should VALIDATE the sample URL.
After opening it, the LLM checks: "Does this page contain the content type the
user described?" If not: "Your sample URL appears to be a listing page, not a
product specs page. Can you provide a sample of an actual product page?" This is
a WS message back to the user — one extra interaction, not a reasoning loop.

**For problem 3 (authentication):** Auth is explicitly Tier 3 (ask-and-wait).
If Phase 1 detects a login page, it sends a WS message: "This site requires
authentication. Please provide credentials or a session cookie." The service
pauses until the user responds. This is the EXISTING QuestionPrompt pattern.

### Net Assessment

**The rigid service model is the RIGHT choice.** The 3 edge cases (no sitemap,
wrong sample, auth) are handled with fallback methods and user interaction, not
with a full agent loop. The 95% case (site has sitemap, sample is valid, no auth)
runs 100× cheaper than an agent. The 5% edge case adds 1-3 more interactions,
still much cheaper than 900 LLM calls.

**Action:** Add `discoverViaBrowser()` fallback in Phase 0. Add sample URL
validation in Phase 1. Add auth detection with user prompt.

## Iteration 2: Does Intent + Sample URL Actually Work for Diverse Sites?

### The Assumption

The user provides: (a) free-text intent like "crawl FAQ content for printers" and
(b) sample URL like `epson.com/faq/SPT_C11CJ67201~faq-00009c6-shared`. The system
extracts a URL pattern, matches it against the sitemap, and filters to relevant pages.
This works beautifully for the Epson scenario. But Epson has clean URL patterns.

### The Problem

1. **Sites with opaque URLs.** Many modern SPAs use hash-based or UUID-based URLs:
   `example.com/app/d7f3a2b1-4e5c-4d2a-9c8b` or `example.com/#/products/12345`.
   There's no human-readable pattern to extract. The sample URL tells you nothing
   about other pages of the same type.

2. **Sites with no sitemap.** ~40% of websites don't have a sitemap.xml. If Phase 0
   finds no sitemap, the "filter 52K URLs to 298" flow collapses. There are no URLs
   to filter.

3. **Dynamic content not in sitemaps.** FAQ pages generated by JS frameworks (React,
   Vue) may not appear in sitemaps. The sitemap might have `/support` but not the
   individual `/support/faq/12345` pages that are client-side routed.

4. **User provides intent but no sample URL.** The sample URL is "optional but helps
   a lot." Without it, Phase 0 has no URL pattern to filter by. It must fall back
   to LLM interpretation of intent alone, which is less precise.

5. **Intent ambiguity.** "Crawl product information" on an e-commerce site. Does
   this mean product descriptions? Prices? Reviews? Specs? All of the above? The
   intent is too vague for automated filtering.

### Counter (Prior Learnings)

From external research: Firecrawl's `map` endpoint also uses HTTP-only discovery
and works fine on ~80% of sites. The 20% that don't work fall through to browser
crawling. This is the same 80/20 split we observed.

From Iteration 8 (v1): "The map phase means the browser path is the FALLBACK,
not the default." This principle still holds. Phase 0 is an optimization that
dramatically helps when it works, not a requirement.

**For problem 1 (opaque URLs):** The sample URL still helps — even with UUIDs,
the service can identify the page TYPE by content. "Pages with FAQ content have
this structure." Then browser-based discovery finds similar pages. Slower but
still targeted by intent.

**For problem 2 (no sitemap):** Fallback to browser-based link discovery. Open
the base URL, extract all internal links, group by URL structure. 1-2 LLM calls
to classify link groups. "Which of these link patterns match 'FAQ content'?"

**For problem 3 (dynamic content):** If content is client-side rendered, Phase 0
won't find it. But Phase 1 WILL — it opens the sample URL in a browser (via MCP).
The browser renders JS. The service sees the rendered content and can build
PageHandlers from it. The gap is only in URL DISCOVERY, not content extraction.

**For problem 4 (no sample URL):** The LLM can interpret intent against the
sitemap URL patterns alone. "User wants 'FAQ content'. Sitemap has `/faq/*`
(3,200 URLs), `/support/*` (500 URLs), `/help/*` (200 URLs). Most likely:
`/faq/*`." Less precise than sample URL, but still filters 52K to 3,900.

**For problem 5 (vague intent):** Phase 3 (Intent Preview) is the safety net.
The service shows: "I think you want 3,200 FAQ pages. Here's a sample of what
I'll extract: [Q&A pairs]. Is this right?" The user can clarify before execution.

### Net Assessment

**Intent + sample URL is an 80/20 optimization, not a hard requirement.**
It dramatically helps for sites with sitemaps and clean URLs (~60-70% of sites).
For the rest, fallbacks exist: browser discovery, LLM intent interpretation,
user clarification in Phase 3. The architecture handles graceful degradation.

**Action:** Document the fallback chain in the HLD: sitemap + sample URL (best)
→ sitemap + intent only (good) → browser discovery + intent (ok) → ask user
for more info (last resort). Ensure Phase 0 has all four paths.

## Iteration 3: Is the Bidirectional WebSocket Extension Realistic?

### The Assumption

The existing progress WebSocket (`/api/admin/progress/subscribe`) is unidirectional
(server → client). We extend it to bidirectional for Crawl Together: the service
sends questions/progress, the user sends approvals/answers.

### The Problem

1. **Different interaction model.** The progress WS sends fire-and-forget events.
   Crawl Together needs request/response: "Approve this plan?" → waits → "Approved."
   This requires message correlation (request ID → response), timeouts, and state
   management that the current WS doesn't have.

2. **Authentication for writes.** The current WS authenticates for READ access.
   If we add WRITE (user sends messages), we need CSRF protection, rate limiting,
   input validation. The WS handler in `progress.ts` has no input processing.

3. **Multiple tabs/devices.** If the user opens two Studio tabs, both connect to
   the WS. When Phase 3 asks for approval, which tab gets it? Both? Race condition
   if user approves in one and edits in the other?

4. **Reconnection state.** User disconnects during Phase 3 (approval pending).
   When they reconnect, the WS must replay the pending question. The current WS
   has no message history — it's purely real-time.

### Counter (Prior Learnings)

From Iteration 7 (v1): "The WebSocket session between Studio and Runtime is
ephemeral — it CANNOT be the source of truth for crawl progress." This led to
Redis-backed state. The same principle applies: WS is a VIEW LAYER. All state
(pending questions, progress, anomalies) lives in Redis/MongoDB. The WS delivers
it, but isn't the source.

**For request/response:** Don't use the WS for request/response. Instead:

- Service persists a `CrawlDecision` to Redis (like the existing
  `pendingDecision` pattern in `crawl.ts` with 1h TTL)
- WS sends a notification: "Approval needed"
- User responds via REST: `POST /api/crawl/together/{jobId}/respond`
- Service polls Redis or gets notified via Redis pub/sub

This is exactly what the existing `POST /api/crawl/batch/confirm` flow does for
QuestionPrompt decisions. Reuse that pattern.

**For reconnection:** When the WS reconnects, it fetches current job state from
Redis. If there's a pending decision, it shows it. No message history needed.

**For multiple tabs:** Only one approval needed. First response wins. Second tab
sees "Already approved" on reconnect.

### Net Assessment

**Don't extend the WS to bidirectional.** Use the existing pattern: WS for
notifications (server → client), REST for responses (client → server), Redis
for state. This is simpler, already battle-tested in the codebase, and avoids
the WS write security issues.

**Action:** Update HLD to clarify: WS is notification-only (like existing progress).
User responses go through REST endpoints. Pending decisions stored in Redis.

## Iteration 4: Does WorkerLLMClient Support Multi-Turn Tool Use?

### The Assumption

`CrawlIntelligenceService` makes LLM calls via `WorkerLLMClient`. In Phase 1-2,
the LLM needs to understand pages by calling MCP tools. But this isn't a simple
prompt→response. It's: prompt → LLM says "call extract_page_structure()" →
service calls the tool → service sends result back to LLM → LLM responds with
understanding. This is a mini tool-use loop.

### The Problem

1. **WorkerLLMClient uses `generateText()` from Vercel AI SDK.** Does this support
   tool calls? Looking at the codebase: `WorkerLLMClient.chat()` calls
   `generateText({ model, messages, tools?, ... })`. The Vercel AI SDK DOES
   support tool calling via the `tools` parameter and `toolResults` in messages.
   So tool use IS possible.

2. **But the service needs to orchestrate the tool loop.** `generateText()` returns
   tool calls. The service must execute them (via MCP), then call `generateText()`
   again with the results. This is a manual loop, not an automatic agent loop.
   Each iteration is a separate `generateText()` call.

3. **Cost tracking per tool call.** Each `generateText()` call returns token usage.
   The service must aggregate these across the tool loop for cost tracking.

### Counter (Prior Learnings)

The Vercel AI SDK's `generateText()` with `maxSteps` parameter automatically
handles multi-step tool calls. Set `maxSteps: 5` and it loops internally:
LLM → tool call → execute → LLM → tool call → execute → final response.
This is exactly what Phases 1-2 need.

Alternatively, the service can do the loop manually for more control:

```
while (hasToolCalls(response)) {
  const results = await executeMcpTools(response.toolCalls);
  response = await llm.chat([...messages, toolCallMsg, toolResultMsg]);
  totalTokens += response.usage;
}
```

Both patterns are established in the codebase (OrgProfileGenerator does sequential
LLM calls; pipeline workers do single-shot calls with structured output).

### Net Assessment

**WorkerLLMClient CAN support tool use.** The Vercel AI SDK's `generateText()`
supports tools natively. The service orchestrates a small tool loop (2-5 steps)
per phase, which is far simpler than a continuous agent reasoning loop. Token
tracking is available per call via `response.usage`.

**Action:** No HLD change needed. Verify in LLD that `WorkerLLMClient.chat()`
passes `tools` parameter correctly and test with MCP tool schemas.

## Iteration 5: Is the Crawl Worker Architecture Sound?

### The Assumption

A new BullMQ worker in search-ai executes PageHandlers mechanically. It receives
a `CrawlJob` with 298 URLs and handlers, opens each in Playwright via MCP, runs
the handler steps, checkpoints to Redis, and submits to the pipeline.

### The Problem

1. **BullMQ job timeout.** BullMQ jobs have a `timeout` setting. A 298-page crawl
   taking 40 minutes will exceed the default job timeout (usually 30s-5min). Need
   to set an appropriate timeout or use the `keepAlive()` pattern.

2. **MCP server connection lifetime.** The worker connects to the MCP server for
   the entire 40-minute crawl. The MCP server's `sessionTimeout` is 30 minutes.
   If the connection drops, the worker loses its browser context. All in-flight
   page state is lost.

3. **Memory pressure.** 298 pages × Playwright browser context = significant memory.
   Even with one page at a time (not parallel), the MCP server's BrowserPool
   accumulates browser contexts. Need explicit context cleanup between pages.

4. **Worker crash recovery.** BullMQ retries failed jobs. But if the worker crashes
   at page 150, BullMQ restarts the ENTIRE job. Without checking the checkpoint,
   it re-crawls pages 1-149. Wasted work + duplicate pipeline submissions.

5. **Concurrent crawl jobs.** If 5 tenants start Crawl Together simultaneously,
   that's 5 workers each running 40-minute jobs. With a single MCP server instance,
   browser pool contention is severe.

### Counter (Prior Learnings)

From Iteration 7 (v1): Checkpoint in Redis was already proposed. This handles
problem 4 — the worker's first action on start/restart is to load the checkpoint
and skip completed URLs.

From Iteration 10 (v1): The execution loop pseudocode already handles this:
`const remainingUrls = job.urls.filter(url => !checkpoint.completedUrlSet.includes(url))`

**For job timeout:** Use BullMQ's `removeOnComplete` + long TTL. Or use the
`updateProgress()` method to keep the job alive. BullMQ also supports
`lockDuration` extension via `job.extendLock()`.

**For MCP session lifetime:** The worker should create a fresh MCP session per
batch of pages (e.g., every 50 pages). Not one session for the entire crawl.
This aligns with the MCP server's `maxPagesPerBrowser=50` setting.

**For memory:** Close browser context after each page. The MCP server's BrowserPool
already supports context lifecycle management.

**For concurrent jobs:** Each job can specify `concurrency: 1` to limit parallel
pages within a job. BullMQ's worker concurrency limits how many jobs run at once.
Set worker concurrency to 3 (= 3 simultaneous crawl jobs max).

### Net Assessment

**The worker architecture is sound with known BullMQ patterns.** All 5 problems
have established solutions: checkpoint resume, lock extension, session batching,
context cleanup, concurrency limits. These are implementation details, not
architectural flaws.

**Action:** Document in LLD: lock extension pattern, 50-page session batching,
per-page context cleanup, worker concurrency = 3.

## Iteration 6: What Happens When the LLM Generates a Bad PageHandler?

### The Assumption

In Phase 2, the LLM navigates sample pages and the system records its actions as
PlaywrightSteps. These compile into a PageHandler that replays mechanically on 298
pages. The handler is validated on 2-3 sample pages before execution.

### The Problem

1. **Sample bias.** 3 sample pages may not represent all 298. If the first 3 pages
   are from one category (e.g., "All-In-Ones"), the handler works for them but fails
   on another category (e.g., "Photo Printers") that has a different page layout.
   Validation passes on the biased sample.

2. **Selectors that work on 95% of pages.** The primary selector `.faq-category-header`
   exists on 283 pages but not on 15 older product pages that use `.accordion-header`.
   The fallback selector `text=FAQs` matches a sidebar link, not the tab. The handler
   extracts wrong content on those 15 pages. No exception thrown.

3. **Timing-sensitive steps.** The handler has `expandAll(.faq-category-header,
{ waitStrategy: 'domStable', delayBetween: 200 })`. This works on fast pages
   but fails on slow pages where AJAX loads take 2 seconds. The expansion completes
   before content loads. Extraction gets empty categories.

4. **The handler captures the WRONG extraction strategy.** Phase 1 identifies inline
   FAQ extraction as optimal. But the LLM, during Phase 2, also clicks individual
   FAQ links (following them to separate pages) for one of the samples because it
   was "curious." The handler includes both strategies. On replay, it does both —
   extracting inline AND visiting individual pages. Double content submission.

### Counter (Prior Learnings)

From Iteration 1 (v1): "Spot-check sampling. After the first 5 models, pause and
show the user a sample of extracted content." This directly addresses sample bias.

From Iteration 5 (v1): "Content fingerprint validation on every Mode 3 replay."
The `expectedOutput` schema catches extraction anomalies: if extracted items < 10
or content length is abnormal, flag the page.

**For sample bias (problem 1):** Pick samples from DIFFERENT URL path segments.
If URLs are `/Support/Printers/All-In-Ones/*/s/SPT_*`, `/Support/Printers/Photo/*/s/SPT_*`,
sample at least one from each distinct path prefix. 3 samples from 3 categories.

**For partial selector failure (problem 2):** The fingerprint validation catches
this. If a page has `.accordion-header` instead of `.faq-category-header`, the
fingerprint's `selectorPresence` check fails. The page is flagged as anomaly.
Phase 5 builds a variant handler.

**For timing (problem 3):** The `waitStrategy: 'networkIdle'` option is better
than `domStable` for AJAX content. The handler should use 'networkIdle' as default
for expansion steps, with a 5-second timeout. If no network activity within 500ms
of the last request, content has loaded.

**For wrong strategy capture (problem 4):** The system should NOT record every
LLM action blindly. It should record only the SUCCESSFUL extraction path. After
Phase 2, the LLM is asked: "Which of these actions are part of the extraction
recipe vs. exploratory?" The system prunes exploratory steps.

### Net Assessment

**Bad PageHandlers are the #1 operational risk.** But the defense-in-depth approach
(diverse sampling, fingerprint validation, output schema validation, anomaly
escalation) means bad handlers are DETECTED quickly, not silently perpetuated.
The cost of a bad handler is a few anomaly pages, not a full-crawl failure.

**Action:** Add to LLD: diverse sample selection algorithm, step pruning in
handler compilation, networkIdle as default waitStrategy, early-stop after
5 anomalies (don't burn through 298 pages if the handler is clearly wrong).

## Iteration 7: Is the Cost Claim of $0.01-0.15 Realistic?

### The Assumption

The revised HLD claims 8-12 LLM calls totaling ~115K tokens for the Epson
scenario. At Claude Haiku pricing: ~$0.03. At GPT-4o: ~$0.50.

### The Problem

1. **Phase 0 intent refinement.** The LLM call includes the full sitemap URL
   list (or at least URL patterns). For a site with 52,847 URLs, even grouped
   patterns could be 5-10K tokens. Add intent + sample URL analysis: ~12K
   input tokens for this single call.

2. **Phase 1-2 page content.** Each sample page's preprocessed HTML is ~3-5K
   tokens. With 3 samples and tool use loops (2-5 tool calls per sample),
   each Phase 2 page is ~15-20K tokens total (input + output across calls).
   3 pages × 20K = 60K tokens for Phase 2 alone.

3. **Phase 5 repair.** Each anomaly page requires loading content (~5K tokens)
   - reasoning about what changed (~2K output). 2-3 anomalies = ~20K tokens.

4. **Revised total:** Phase 0: ~12K. Phase 1: ~15K. Phase 2: ~60K. Phase 5:
   ~20K. Total: ~107K tokens. This is close to the claimed ~115K.

5. **But: retry overhead is not included.** If a tool call fails (MCP timeout,
   browser crash), the service retries. Each retry re-sends context. 20% retry
   rate = ~20K extra tokens. New total: ~127K.

6. **Model choice matters enormously:**
   - Claude Haiku ($0.25/$1.25 per 1M): ~$0.05
   - GPT-4o-mini ($0.15/$0.60 per 1M): ~$0.03
   - Claude Sonnet ($3/$15 per 1M): ~$2.20
   - GPT-4o ($2.50/$10 per 1M): ~$1.50

### Counter (Prior Learnings)

From Iteration 6 (v1): "Token-based cost estimation, not call-based." The revised
HLD already moved to token-based math. The claim of "$0.01-0.15" was for the
cheapest models. The HLD comparison table already shows per-model costs.

The key insight: **cost scales with site COMPLEXITY, not page COUNT.** Whether
it's 298 pages or 2,980 pages, the LLM calls are the same (Phase 0-2 are about
understanding, not per-page processing). Only Phase 5 (repair) scales slightly
with page count (more pages = potentially more anomalies).

For a 10× larger crawl (2,980 pages): Phase 0-2 unchanged (~87K tokens).
Phase 5 maybe 5-10 anomalies instead of 2-3 (~50K tokens). Total: ~137K.
Still under $0.10 with Haiku. The scaling property IS real.

### Net Assessment

**The $0.03-0.50 range (depending on model) is realistic for the Epson scenario.**
The claim of "$0.01" was too aggressive — corrected to $0.03 minimum with the
cheapest models. The key architectural win (cost doesn't scale with page count)
is validated by the math.

**Action:** Update HLD cost table to show honest per-model ranges with retry
overhead. Don't claim $0.01 — claim $0.03-2.20 depending on model.

## Iteration 8: Can This Scale to Millions of Pages? Batch + Fork Analysis

### The Assumption

The current HLD has a single Crawl Worker processing URLs sequentially (one page
at a time, checkpoint each page). For 300 Epson FAQ pages at ~8 sec/page, that's
~40 minutes. Acceptable. But what about enterprise-scale crawls?

### The Problem

1. **1 million pages at 8 sec/page = 92 DAYS.** A single sequential worker is
   completely unusable at enterprise scale. Even a documentation-heavy enterprise
   site (e.g., Microsoft Learn, AWS docs) can have 500K-2M pages.

2. **BullMQ job timeout.** Even with lock extension, a single BullMQ job running
   for 92 days is absurd. Redis memory, lock contention, checkpoint size — all
   break down at this scale.

3. **Single MCP server bottleneck.** One Playwright browser can process one page
   at a time. Even with 50-page session batching, it's still sequential within
   the worker. The MCP server's BrowserPool supports multiple contexts, but our
   worker uses one at a time.

4. **Single-pod failure.** If the pod running the worker crashes at page 500K,
   even with checkpoint resume, the remaining 500K pages are blocked until that
   pod recovers.

5. **Downstream pipeline saturation.** 1M pages × ~5 content items/page = 5M
   items flowing into the BullMQ pipeline (content-processing → docling →
   page-processing → embedding). The pipeline has backpressure limits
   (MAX_QUEUE_DEPTH: 200-1000 per stage). A million-page crawl would flood it.

### The Solution: Batch + Fork Architecture

The key insight: **Phase 0-2 (intelligence) is O(1) — understand the site once.
Phase 4 (execution) is O(N) — process every page. Only Phase 4 needs to scale.**

```
Phase 0-2: INTELLIGENCE (unchanged, 5-10 min)
  Discover 1M URLs, build PageHandlers from 5-10 samples
  Output: PageHandler[] + 1M filtered URLs

Phase 3: APPROVE (unchanged)
  "1M pages, ~500K items, est. 4 hours (8 workers), ~$5-10. Approve?"

Phase 3.5: PARTITION (NEW — zero LLM, pure computation)
  Split 1M URLs into batches of 500-1000 URLs each
  Group by page type if multiple PageHandlers exist
  Each batch → BullMQ child job via FlowProducer

  URL partitioning strategy:
    1. Group URLs by PageHandler type (from Phase 2 page type classification)
    2. Within each type, split into batches of BATCH_SIZE (default: 500)
    3. Assign batch priority: first batches = HIGH (quick feedback),
       later batches = NORMAL (sustained throughput)
    4. Create BullMQ Flow:
       - Parent job: CrawlOrchestrator (tracks aggregate progress)
       - Children: CrawlBatch[] (each processes 500 URLs independently)

  Example for 1M URLs with 2 page types:
    PageHandler A (articles): 800K URLs → 1,600 batch jobs
    PageHandler B (reference): 200K URLs → 400 batch jobs
    Total: 2,000 BullMQ child jobs

Phase 4: EXECUTE (FORKED — multiple workers in parallel)
  Multiple Crawl Workers consume from 'crawl-together-batch' queue
  Each worker processes one batch (500 URLs) independently
  Workers run on different pods → horizontal scaling
  Worker concurrency: 3 per pod (configurable via env)
  With 3 pods × 3 concurrency = 9 parallel batches
  At 500 URLs × 8 sec/page per batch = ~67 min per batch
  9 parallel batches = ~9 × 500 = 4,500 pages per 67-min cycle
  1M pages ÷ 4,500 per cycle = ~222 cycles × 67 min = ~10 days
  With 10 pods × 3 = 30 parallel: 1M pages in ~3.7 days

  BUT: most enterprise crawls are HTTP-based (Mode 1), not Playwright.
  Playwright at 8 sec/page is for COMPLEX pages (JS-rendered, tabs, expand).
  For simple HTTP pages: Go worker at 100 parallel = 10K pages/sec.
  1M simple pages via Go worker: ~100 seconds.

  The batch+fork architecture supports BOTH:
  - Simple pages: fan out to Go worker (HTTP, existing path)
  - Complex pages: fan out to Crawl Worker (Playwright, new path)
  - Mixed: PageHandler classifies each page type, routes accordingly

Phase 4.5: AGGREGATE (NEW — zero LLM)
  Parent CrawlOrchestrator job aggregates:
  - Completed batch count / total batch count
  - Total pages processed / total pages
  - Anomaly pages across all batches
  - Cost tracking (sum of all batch token usage — Phase 5 only)
  Progress → Redis pub/sub → WS notification to Studio

Phase 5: REPAIR (unchanged, but batched)
  Anomaly pages collected across all batches
  If total anomalies > threshold (e.g., 5% of pages):
    → Likely PageHandler issue, not individual page issues
    → Re-sample from anomaly pages, rebuild handler, re-run failed batches
  If anomalies < threshold:
    → Individual page fixes (existing repair flow)
```

### How This Uses Existing Patterns

1. **FlowProducer** — `PipelineFlowBuilder` already creates parent-children
   BullMQ flows. The CrawlOrchestrator uses the exact same pattern: parent job
   with N children. `failParentOnFailure: false` (individual batch failure
   shouldn't kill the entire crawl).

2. **Backpressure** — `PipelineFlowBuilder.checkBackpressure()` already limits
   queue depth. Apply the same to `crawl-together-batch` queue. New constant:
   `MAX_CRAWL_BATCH_QUEUE_DEPTH: 50` (don't enqueue all 2,000 batches at once).

3. **Progressive batch enqueuing** — Don't enqueue all batches upfront. The
   CrawlOrchestrator enqueues N batches at a time (e.g., 20), and adds more
   as batches complete. This prevents Redis memory explosion and enables
   adaptive rate control.

4. **Go worker integration** — For HTTP-only pages (no Playwright needed),
   the partition step routes to the existing `static-crawl` queue consumed
   by the Go worker. The Go worker already handles 1000-URL batches with
   100 parallel HTTP requests. This is 100× faster for simple pages.

### Scaling Numbers

| Scale      | Pages | Batches | Workers (3 pods) | Est. Time (Playwright) | Est. Time (HTTP/Go) |
| ---------- | ----- | ------- | ---------------- | ---------------------- | ------------------- |
| Small      | 300   | 1       | 1                | 40 min                 | 3 sec               |
| Medium     | 10K   | 20      | 9                | 2.5 hours              | 10 sec              |
| Large      | 100K  | 200     | 9                | ~25 hours              | 100 sec             |
| Enterprise | 1M    | 2,000   | 30 (10 pods)     | ~3.7 days              | 17 min              |
| Massive    | 10M   | 20,000  | 30 (10 pods)     | ~37 days               | 2.8 hours           |

**Key insight:** For truly massive crawls (1M+), most pages are HTTP-fetchable.
Only a subset need Playwright (JS-rendered, interactive). The partition step
should classify: HTTP-simple pages → Go worker (fast path), Playwright-required
pages → Crawl Worker (smart path). A typical 1M-page site might be 90% HTTP +
10% Playwright = ~17 min + ~9 hours = ~9.5 hours with 10 pods.

### Checkpoint & Failure Recovery at Scale

**Per-batch checkpoint** (not per-crawl): Each batch job maintains its own
Redis checkpoint. If batch #847 fails, only 500 pages need re-processing.
Other batches continue unaffected.

**Batch retry**: Failed batches retry up to 3 times (standard BullMQ).
After 3 failures, marked as failed. CrawlOrchestrator collects failed
batch URLs for Phase 5 repair or manual review.

**Progressive results**: Each completed batch submits extracted content
to the existing pipeline immediately. The user sees results flowing in
during the crawl, not after. This is critical for UX on long crawls.

**Pause/Resume at batch level**: User can pause the crawl. CrawlOrchestrator
stops enqueuing new batches. In-flight batches complete (max ~67 min wait).
Resume re-enqueues remaining batches.

### Cost at Scale

Phase 0-2 (intelligence): Same ~127K tokens regardless of scale.
Phase 5 (repair): Scales with anomaly count, not page count.
1M pages × 2% anomaly rate = 20K anomaly pages
20K pages ÷ batch of 10 for repair = 2,000 LLM calls × ~5K tokens
= ~10M tokens for repair
Cost (Haiku): ~$3.75 | Cost (GPT-4o): ~$37.50

Total for 1M-page crawl:
Intelligence: $0.03-2.20 (model dependent)
Repair (2%): $3.75-37.50 (model dependent)
Total: $3.78-39.70
Per page: $0.000004-0.00004

Even at scale, the 5-phase model is 100-1000× cheaper than per-page LLM.

### Net Assessment

**The current single-worker design works for the 95% case (< 10K pages).** For
enterprise scale (100K-1M+), batch+fork is required. The architecture supports
it naturally because:

1. Phase 0-2 (intelligence) doesn't change — same cost, same time
2. Phase 4 (execution) is embarrassingly parallel — each batch is independent
3. BullMQ FlowProducer is already in the codebase (PipelineFlowBuilder)
4. Go worker integration provides 100× speed for HTTP-simple pages
5. Progressive batch enqueuing prevents Redis memory explosion

**Action:** Add Phase 3.5 (PARTITION) + Phase 4.5 (AGGREGATE) to the HLD.
Update the 5-phase model to a 7-phase model for enterprise scale. Document the
batch+fork pattern, Go worker routing, and scaling numbers. Add
`CrawlOrchestrator` as a new component alongside `CrawlIntelligenceService`.

## Cross-Cutting Assessment

After 7 iterations on the revised design, the architecture holds up well. The key
risks are operational (bad PageHandlers, timing sensitivity) not architectural
(the 5-phase model, search-ai self-containment, intent + sample URL are sound).

### The 4 Biggest Remaining Risks

1. **PageHandler quality** — Bad handlers = bad data. Mitigation: diverse sampling,
   fingerprint validation, output schema, early-stop on anomaly burst.

2. **Sites without sitemaps** — Phase 0's power comes from sitemap filtering.
   Without it, the system degrades to browser discovery. Mitigation: fallback
   to browser link extraction, still targeted by intent.

3. **MCP server reliability for long crawls** — Long browser sessions are fragile.
   Mitigation: 50-page session batching, per-page context cleanup, checkpoint resume.

4. **Enterprise scale (1M+ pages)** — Single worker is unusable at scale.
   Mitigation: batch+fork via BullMQ FlowProducer, Go worker for HTTP pages,
   progressive batch enqueuing, per-batch checkpointing.

## 1. Overview

### Problem

Workflow connector triggers and actions sometimes produce binary content ŌĆö a Gmail attachment fetched by a "New Email" trigger, a PDF returned from "Find Email", an audio file generated by OpenAI text-to-speech, an image streamed from Slack's `get-file`. The workflow needs to hand a **URL** back to the user so they can open or forward the file, and downstream steps need the same URL so they can pass it to OCR, document parsers, or external systems.

Before this feature, workflow-engine stored attachment bytes in **Redis**, capped at 10 MB, with a 1-hour TTL, and returned a URL of the shape `http://localhost:9080/internal/attachments/<uuid>`. That URL was unreachable from anywhere outside the workflow-engine pod, so connector actions like Gmail "Find Email" looked broken: the `attachments[].data` field came back empty because the file-write step failed silently inside the action runtime.

### Goal

Replace the Redis-backed writer with NFS-backed storage that mirrors search-ai's pattern, give attachment URLs the same shape and routing as other platform-public URLs, and wire the action-side file writer so every connector action that emits files (Gmail, Drive, Outlook, Slack, S3, Azure, Jira, OpenAI TTS) returns a real, openable URL.

### Summary

Workflow-engine connector attachments now flow through a four-layer pipeline:

1. **Storage layer** ŌĆö the same `S3StorageService` from `@agent-platform/shared` that search-ai uses, exposed via a small `FileStorage` interface. With `STORAGE_PROVIDER=local` (the deployed default) bytes land on a `ReadWriteMany` Azure Files Premium share mounted at `/app/uploads/`.
2. **URL signing** ŌĆö HMAC-SHA256 token with a 24-hour TTL, signed with `JWT_SECRET`. The token carries the storage key, tenant ID, and expiry; verification checks signature, expiry, and tenant-key prefix consistency.
3. **Public routing** ŌĆö a dedicated nginx-ingress rule on `/attachments/*` routes directly to workflow-engine. URLs look like `https://agents-<env>.kore.ai/attachments/<uuid>?token=ŌĆ”`. No `workflow-engine` or `internal` segment is exposed to clients.
4. **Cleanup** ŌĆö a periodic sweep (1-hour interval, 25-hour file retention) deletes files older than the token expiry plus a 1-hour grace window. A Redis `SET NX PX` lock dedupes the sweep across replicas.

The `FileStorage` interface (`apps/workflow-engine/src/storage/storage-factory.ts`) is a near-verbatim clone of search-ai's. It supports three providers (`local`, `s3`, `minio`) selected by `STORAGE_PROVIDER`. All deployed environments run `local`; the S3/MinIO branches exist in code but are not used today.

### Token & URL shape

A URL looks like:

```
https://agents-dev.kore.ai/attachments/3f5e8c9d-a1b2-4c3d-9e8f-a1b2c3d4e5f6
  ?token=eyJrIjoiYXR0YWNobWVudHMvdGVuYW50LWFiYy8zŌĆ”&f=invoice.pdf&m=application%2Fpdf
```

| URL segment              | Source                                                                    | Signed?                                         |
| ------------------------ | ------------------------------------------------------------------------- | ----------------------------------------------- |
| Host                     | `WORKFLOW_ENGINE_PUBLIC_URL` (derived from `ingress.host` at deploy time) | ŌĆö                                               |
| Path `/attachments/<id>` | Random UUID generated at write time                                       | ŌĆö (decorative; the real key lives in the token) |
| Query `token`            | HMAC of `{k, t, exp}`                                                     | **Yes**                                         |
| Query `f`                | Original filename (URL-encoded)                                           | No ŌĆö display only                               |
| Query `m`                | Original MIME type                                                        | No ŌĆö but server-side allowlisted on response    |

The token's `k` field is the storage key (`attachments/<tenantId>/<uuid>.<ext>`), `t` is the tenant ID, `exp` is the epoch-ms expiry. On every download, the server:

1. Verifies the HMAC signature with `JWT_SECRET` (timing-safe).
2. Checks `exp` against current time.
3. Asserts `k` starts with `attachments/${t}/` ŌĆö defense against a future signer bug that mismatches key and tenant.

### File layout on disk

```
/app/uploads/
ŌööŌöĆŌöĆ attachments/
    Ōö£ŌöĆŌöĆ tenant-abc/
    Ōöé   Ōö£ŌöĆŌöĆ 862ddefa-895f-4a49-ad10-8fc36336b5f6.pdf
    Ōöé   Ōö£ŌöĆŌöĆ 7a9c3e10-ŌĆ”-ŌĆ”.png
    Ōöé   ŌööŌöĆŌöĆ 4b21f87e-ŌĆ”-ŌĆ”.pdf
    ŌööŌöĆŌöĆ tenant-xyz/
        ŌööŌöĆŌöĆ 91e4d2b1-ŌĆ”-ŌĆ”.docx
```

One folder per tenant. Filenames are random UUIDs + the original extension; the human-readable filename only lives in the URL's `f` query param. Two reasons:

- Easy per-tenant accounting / cleanup (drop the folder if a tenant offboards).
- Avoids cross-tenant filename collisions by construction.

`tenantId`, attachment UUID, and file extension are each sanitized in `buildAttachmentKey()` so a malformed input cannot produce a key with path-traversal characters. The FS-layer traversal guard (`LocalFileStorage.upload`) catches actual attacks; the key-construction sanitization keeps the key well-formed regardless.

### Duplicate handling

Each upload produces a **fresh UUID**. If the same file is attached to three emails, three independent files land on disk. There is no content-hash deduplication. This is deliberate:

- Dedup adds a read+compare on every write.
- Same file across two tenants would either collide names (bad) or violate isolation (worse).
- Disk is cheap; correctness is not.

### TTL & cleanup

Two clocks govern an attachment's life:

| Clock          | Default | Controlled by                                           | Effect                             |
| -------------- | ------- | ------------------------------------------------------- | ---------------------------------- |
| Token TTL      | 24 h    | `ATTACHMENT_TOKEN_TTL_MS` (env, optional override)      | URL returns 403 after expiry       |
| File MAX_AGE   | 25 h    | `ATTACHMENT_FILE_MAX_AGE_MS` (env, optional override)   | File deleted by sweep after expiry |
| Sweep interval | 1 h     | `ATTACHMENT_SWEEP_INTERVAL_MS` (env, optional override) | How often the sweep runs           |

The 1-hour gap between token TTL and file MAX_AGE is intentional. A request opening a URL at minute 23:59 with a still-valid token would otherwise race the deletion at minute 24:00 and corrupt the stream. The extra hour on disk is invisible to users (the URL is already expired by then) and guarantees an in-flight read always finishes cleanly.

The sweep walks `/app/uploads/attachments/<tenant>/`, stats each file, deletes anything older than `ATTACHMENT_FILE_MAX_AGE_MS`. It runs every `ATTACHMENT_SWEEP_INTERVAL_MS` and acquires a Redis lock (`SET NX PX`) so only one replica per cluster sweeps per interval. If Redis is down, the sweep skips itself ŌĆö over-retention is safer than concurrent deletes.

### Tenant isolation

Three independent layers stop one tenant from seeing another's attachments:

1. **Key prefix.** Every write goes under `attachments/<tenantId>/...`. The tenant ID comes from the authenticated session (workflow execution context), never user input.
2. **Token cross-check.** `verifyAttachmentToken` rejects tokens whose `k` does not start with `attachments/${t}/`.
3. **HMAC signature.** The token is cryptographically bound to `JWT_SECRET`. Forging requires the secret.

There is **no per-project scoping** in the key. A URL signed by a workflow in project A is openable from project B within the same tenant. This matches platform-wide convention ("cross-project within a tenant is allowed by design"). If per-project URLs are ever needed, embed `projectId` in the key and add a third assertion in `verifyAttachmentToken`.

### Size limits

| Limit               | Value   | Why                                                                                                                               |
| ------------------- | ------- | --------------------------------------------------------------------------------------------------------------------------------- |
| Per-file writer cap | 50 MiB  | Matches the nginx ingress `proxy-body-size: 50m`. Going above this would still write to disk but fail at the public-URL boundary. |
| Per-PVC capacity    | 10 GiB  | Static PV size; mirrors search-ai default. Sized to hold ~1 day of attachments at moderate traffic.                               |
| Per-share quota     | 100 GiB | Terraform default. Absolute ceiling per env.                                                                                      |

If a connector returns a file larger than the per-file cap, the workflow step fails with `Attachment '<name>' exceeds the 50 MB limit`. Raise the cap by setting `ATTACHMENT_MAX_BYTES` (env override planned; today the value is a source-code constant). Most connectors return files well under this size; the cap exists to stop a runaway connector (e.g. a Gmail attachment of unexpected size) from filling the share between sweep cycles.

### MIME-type safety

Connector outputs include user-supplied MIME types. To prevent an attacker who has obtained a valid URL from flipping `m=application/pdf` to `m=text/html` and triggering HTML rendering, the server allowlists which MIME types it will return as the inline `Content-Type`. Everything outside the list is served as `application/octet-stream`, which makes the browser save the file instead of rendering it.

**Inline-allowed types:** `application/pdf`, `application/json`, `application/zip`, `application/msword`, OOXML formats (Word/Excel/PowerPoint), `text/plain`, `text/csv`, common images (PNG, JPEG, GIF, WebP), common audio (MP3, WAV, OGG), common video (MP4, WebM).

**Anything else** (`text/html`, `image/svg+xml`, `application/xml`, exotic formats) becomes `application/octet-stream` ŌåÆ user sees a download prompt. The file still downloads; only the browser's rendering behavior changes.

## 4. End-to-end flow

A workflow with a Gmail "New Email" trigger that has an attachment:

1. Polling worker fires the trigger. AP runtime adapter pulls the email and decodes the attachment to a `Buffer`.
2. AP piece calls `context.files.write({fileName, data, mimeType})`.
3. Adapter's shim coerces the data to a `Buffer` and calls `ctx.fileWriter(fileName, buf, mimeType)`. The fileWriter has already been bound to the job's tenantId at polling-worker dispatch time.
4. `fileWriterFactory(tenantId)` runs:
   1. Generates a random UUID.
   2. Sanitizes tenantId, UUID, and extension; builds key `attachments/<tenantId>/<uuid>.<ext>`.
   3. Writes the buffer to `/app/uploads/<key>` via the `FileStorage` interface.
   4. Signs an HMAC token over `{k, t, exp}`.
   5. Returns `${WORKFLOW_ENGINE_PUBLIC_URL}/attachments/<uuid>?token=ŌĆ”&f=ŌĆ”&m=ŌĆ”`.
5. The URL ends up in the workflow step output, available to downstream steps and rendered in the workflow UI.
6. User clicks ŌåÆ browser ŌåÆ nginx ingress (matches `/attachments/*` rule) ŌåÆ workflow-engine service.
7. `GET /attachments/:id` verifies the HMAC, asserts key/tenant consistency, reads bytes from `/app/uploads/<key>`, sets allowlisted `Content-Type` and `Content-Disposition`, streams the file.
8. 24 h later: token expires. Any new request gets 403.
9. 25 h later: sweep deletes the file from disk.

The action path is the same except step 1 starts from a workflow action node ("Find Email") and the writer is bound at action-dispatch time (`ConnectorToolExecutor`) instead of at polling-worker dispatch time.

## 5. Failure modes

| Scenario                                 | What happens                                                                                 | Recovery                                                                  |
| ---------------------------------------- | -------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------- |
| NFS share full                           | Write fails with "no space left on device", workflow step errors with clear message          | Bump share quota in infra OR run a sweep cycle manually OR raise PVC size |
| Token expired                            | URL returns 403 with `{success: false, error: 'Invalid or expired token'}`                   | Re-run the workflow to regenerate the URL                                 |
| File deleted, token still valid          | URL returns 404 (race window is < 1h, very rare)                                             | Re-run the workflow                                                       |
| Workflow-engine pod restart mid-download | In-flight HTTP request errors; client retry succeeds. No state lost ŌĆö share is durable.      | Automatic                                                                 |
| Redis down                               | Cleanup sweep skips for the duration. Files over-retain.                                     | Recovers automatically when Redis is back                                 |
| `JWT_SECRET` rotated                     | All existing URLs immediately invalidate (the new signing key produces different signatures) | Coordinated with the platform-wide JWT rotation; users re-fetch URLs      |
| Connector returns >50 MB file            | Workflow step fails with `"Attachment 'name' exceeds the 50 MB limit"`                       | Raise `ATTACHMENT_MAX_BYTES` if the connector genuinely needs it          |

## 7. Security model summary

| Threat                             | Mitigation                                                                                                                                 |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| Forged URL (no valid token)        | HMAC-SHA256 with `JWT_SECRET`; timing-safe comparison                                                                                      |
| Replay after expiry                | `exp` field in payload, checked at verify time                                                                                             |
| Cross-tenant access via token bug  | Verify asserts `payload.k.startsWith('attachments/' + payload.t + '/')`                                                                    |
| Path traversal via malformed input | Three layers: sanitization in `buildAttachmentKey`, traversal guard in `LocalFileStorage.upload`, traversal guard in `readByKey`           |
| MIME confusion / inline XSS        | Allowlist of safe-to-inline types; everything else served as `application/octet-stream`                                                    |
| Disk fill                          | Per-file 50 MB cap + 25 h retention with 1 h sweep                                                                                         |
| Leaked URL ŌåÆ 24 h of bearer access | Documented trade-off; URLs are designed to be shareable. Rotate `JWT_SECRET` if a leak is identified ŌĆö invalidates all URLs platform-wide. |

## 8. Alignment with search-ai

This feature mirrors the storage pattern search-ai already uses, structurally and parametrically:

| Surface                  | Search-AI                                                                   | Workflow-engine                                                         |
| ------------------------ | --------------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| Storage interface        | `FileStorage` (local file: `apps/search-ai/src/storage/storage-factory.ts`) | identical clone (`apps/workflow-engine/src/storage/storage-factory.ts`) |
| Underlying service       | `S3StorageService` from `@agent-platform/shared`                            | same                                                                    |
| `STORAGE_PROVIDER` value | `local` in all envs                                                         | `local` in all envs                                                     |
| Mount path               | `/app/uploads`                                                              | `/app/uploads`                                                          |
| Mount type               | ReadWriteMany Azure Files Premium / EFS                                     | same (sibling share)                                                    |
| Per-tenant prefix        | yes (`documents/<tenantId>/...`)                                            | yes (`attachments/<tenantId>/...`)                                      |
| Token signing            | HMAC-SHA256 with JWT secret                                                 | same                                                                    |
| URL shape                | `{public}/documents/<id>/download?token=ŌĆ”`                                  | `{public}/attachments/<id>?token=ŌĆ”`                                     |
| Cleanup                  | none (manual / no policy)                                                   | hourly sweep with 25 h retention                                        |

The only architectural difference is the cleanup sweep ŌĆö search-ai doesn't have one. If platform-wide retention becomes a requirement, the sweep we wrote here is portable to search-ai.

## 9. Open follow-ups

- **Refactor**: the `files.write` shim is duplicated between `runtime-adapter.ts` (trigger path) and `context-translator.ts` (action path). Extract a shared helper.
- **Per-env tuning**: if a tenant ever needs longer retention or larger files, expose `ATTACHMENT_TOKEN_TTL_MS` / `ATTACHMENT_FILE_MAX_AGE_MS` / `ATTACHMENT_MAX_BYTES` in the deploy configmap (`workflowEngine.configMap` block).
- **S3 flip**: when search-ai flips `STORAGE_PROVIDER` to `s3`, workflow-engine should flip with it ŌĆö the abstraction is in place, only the env vars need to change.
- **Project-scoped URLs**: if cross-project-within-tenant access ever becomes a concern, embed `projectId` in the key and add a third assertion in `verifyAttachmentToken`.
- **Streaming reads**: today the GET handler reads the full file into a Buffer before responding. For large files this could spike memory; switching to `createReadStream().pipe(res)` would amortize.

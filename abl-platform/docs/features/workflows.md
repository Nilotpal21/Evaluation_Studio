# Feature Spec: Workflows & Human Tasks

> **Feature ID**: #48
> **Status**: BETA
> **Owner**: Runtime Team
> **Created**: 2026-03-23
> **Last Updated**: 2026-04-20

---

## 1. Problem Statement

Enterprise agent deployments require orchestrating multi-step processes that span automated actions (HTTP calls, connector operations, agent invocations, tool executions) and human decisions (approvals, data entry, reviews). Without a dedicated workflow engine:

- Agents cannot pause mid-execution to wait for human input and resume deterministically.
- Multi-step automations lack durability -- a pod restart loses in-flight state.
- There is no unified task inbox for human operators to view, claim, and resolve pending tasks.
- SLA tracking, escalation chains, and timeout policies have no system-level support.
- Conditional branching, parallel execution, and loop iteration require ad-hoc orchestration code per deployment.

## 2. Scope

### In Scope

- **Workflow Definition CRUD**: Create, read, update, archive workflow definitions scoped to tenant + project.
- **20 Node Types (canvas-based)**: start, end, condition, loop, delay, text_to_text, text_to_image, audio_to_text, image_to_text, api, function, integration, browser, doc_search, doc_intelligence, human, agentic_app, agent, tool, data_entry.
- **Visual Workflow Builder**: Canvas-based drag-and-drop editor with node/edge graph, auto-save, validation, and execution monitoring panels.
- **Durable Execution**: Restate-backed workflow execution that survives pod restarts and guarantees exactly-once step execution.
- **Human-in-the-Loop (HITL)**: Approval steps, human_task steps with form fields, task assignment/claiming, SLA tracking, escalation chains, timeout policies.
- **Trigger System**: Webhook, polling, and cron-based triggers with connector integration, missed-fire policies, trigger catalog.
- **Expression Resolution**: Template expressions (`{{trigger.payload.orderId}}`, `{{steps.step-1.output.total}}`) resolved against workflow context.
- **Workflow Runtime (compiler)**: Long-running workflow runtime with human task assignment, pause/resume, timeout handling, and ConstructExecutor integration.
- **Studio Integration**: Human task inbox with SWR polling, task detail view, assign/claim/resolve actions, TaskCard response formatting, connector ticket links.
- **Notifications**: Rule-based notifications for step_failed, execution_completed, approval_required, sla_breach, step.waiting_human_task events via email, slack, and webhook channels.
- **Execution Monitoring**: Real-time execution status via Redis Pub/Sub, step-level tracking, execution history, debug flow log with step input/metrics/consoleLogs.
- **Cancellation**: Durable cancellation via Restate promise signal, cascading step status updates and pending human task cleanup.
- **Workflow Versioning**: Version snapshots (nodes, edges, envVars, schemas), 2-state lifecycle (active/inactive) with activate/deactivate, soft-delete, version diffing.
- **Function Node**: V8-sandboxed JavaScript execution with isolated-vm, console log capture, timeout enforcement.
- **Integration Node**: Connector-backed actions with OAuth grant resolution, distributed lock, dynamic action forms.
- **Service JWT for Internal Calls**: Workflow-engine mints short-lived service JWTs for runtime API calls (agent invocation, tool execution).

### Out of Scope

- Cross-tenant workflow sharing or marketplace.
- Custom step type plugin system.
- Workflow-level retry (only step-level retry is supported).
- Blue-green workflow deployment (versioning exists but deployment rollback is not fully specified).

## 3. User Stories

### US-1: Workflow Author

As a workflow author, I want to define multi-step automation workflows with conditional logic, parallel branches, and human checkpoints, so that I can model complex business processes.

**Acceptance Criteria:**

- Can create workflows with all 12 step types via API.
- Workflow definitions are validated against Zod schemas before persistence.
- Workflows are scoped to tenant + project with unique name constraint.
- Can define SLA targets, escalation rules, and notification rules per workflow.

### US-2: Workflow Trigger Manager

As a workflow administrator, I want to configure triggers (webhook, polling, cron) that automatically start workflow executions, so that workflows respond to external events.

**Acceptance Criteria:**

- Can register webhook, polling, and cron triggers via the trigger API.
- Polling triggers use BullMQ scheduler with configurable intervals.
- Cron triggers support standard cron expressions with missed-fire policies (fire_once, fire_all, skip).
- Triggers are deployment-aware (can be pinned to a workflow version).

### US-3: Execution Monitor

As an operations engineer, I want real-time visibility into running workflow executions including step status, timing, and errors, so that I can diagnose issues quickly.

**Acceptance Criteria:**

- Execution records track status per step (pending, running, waiting\_\*, completed, failed, skipped).
- Redis Pub/Sub publishes events for workflow.started, step.started, step.completed, step.failed, workflow.completed, workflow.failed.
- Can query executions by workflow ID, status, and time range.
- Step-level duration and error details are recorded.

### US-3a: Manual Run of Active Version

As a workflow author clicking the Run button in Studio, I want the current **active version** of the workflow to be exercised so that I am testing what users hit in production rather than whatever unsaved edits happen to live on the canvas.

**Acceptance Criteria:**

- `POST /executions/execute` resolves the workflow version with precedence: explicit `workflowVersionId` → active version (`WorkflowVersion.state === 'active'`) → draft (the `Workflow` doc itself).
- Explicit `workflowVersionId` that does not resolve (missing, soft-deleted, or scoped to another workflow/tenant/project) returns 404 `WORKFLOW_VERSION_NOT_FOUND` — never silently falls back to draft.
- Studio does not present a version picker on the Run dialog; the engine resolves active automatically.
- When a version is used, its `definition.nodes`/`edges` feed through `convertCanvasToSteps` (the same path as trigger-engine resolution).
- The resolved `workflowVersionId` and `workflowVersion` are annotated on the `WorkflowExecution` record so the execution monitor shows which version actually ran.
- Workflows that have never been activated still run from draft (first-time development path).

### US-4: Human Task Operator

As a human agent/operator, I want a unified task inbox where I can see all pending tasks assigned to me, claim unassigned tasks, and submit responses with form data, so that I can participate in automated workflows.

**Acceptance Criteria:**

- Pending tasks appear in the Studio inbox with 5-second polling.
- Tasks show type (approval, data_entry, review, decision, escalation), priority, title, description, and due date.
- Can assign tasks to specific users or teams.
- Can claim unassigned tasks.
- Can submit responses with structured form fields, notes, and decisions.
- Task resolution triggers workflow resumption via Restate durable promise.

### US-5: Approval Reviewer

As a project admin or designated reviewer, I want to approve or reject workflow steps that require human authorization, so that sensitive actions have human oversight.

**Acceptance Criteria:**

- Approval steps pause workflow execution and create a human task record.
- Rejection terminates the workflow with APPROVAL_REJECTED status.
- Timeout policies (reject, approve, skip) execute when approval deadline passes.
- Approval decision is recorded in execution history with decidedBy and reason.

### US-6: SLA/Escalation Manager

As a workflow administrator, I want SLA tracking and automatic escalation for human tasks, so that time-sensitive tasks are not forgotten.

**Acceptance Criteria:**

- Tasks have configurable due dates.
- Escalation chains define ordered escalation targets.
- When a task's due date passes, escalation advances to the next level.
- SLA breach timestamp is recorded on the task.
- Notification rules can trigger on sla_breach events.

### US-7: Workflow Cancellation

As an administrator, I want to cancel running workflows, so that I can stop erroneous or unnecessary executions.

**Acceptance Criteria:**

- Cancel signal is delivered via Restate durable promise (sys:cancel).
- Cancellation is checked between non-suspension steps.
- Pending human tasks are cleaned up on cancellation.
- Execution status is updated to 'cancelled' with reason.

## 6. Data Model Summary

### Collections

- **workflows**: Node-based workflow definitions (name, nodes[], edges[], envVars, inputSchema, outputSchema, status, deployment, metadata, triggers[]). Indexes: `{tenantId, projectId, name}` unique, `{tenantId, projectId}`.
- **workflow_versions**: Versioned snapshots of workflow definitions (workflowId, version, definition: { nodes, edges, envVars, inputSchema, outputSchema }, sourceHash, status, changelog). Indexes: `{tenantId, workflowId, version}` unique.
- **workflow_executions**: Execution records (workflowId, workflowVersionId, workflowVersion, status, nodeExecutions[], context, input, output, restateWorkflowId, triggerType, triggerMetadata, webhookMode, webhookDelivery, cancelledAt, nodeExecutions.\*.approval{Decision,DecidedBy,DecidedAt,Reason}, nodeExecutions.\*.mappingErrors[]). Indexes: `{tenantId, workflowId, status}`, `{tenantId, projectId, startedAt}`, `{tenantId, restateWorkflowId}` unique, `{tenantId, status, startedAt}` (stale execution cleanup). `nodeExecutions[]` now always contains both a `start` and `end` boundary step record (first-class lifecycle per FR-41/FR-42); `mappingErrors[]` carries per-field input validation errors on Start (`{name, error}`) and per-output-mapping expression failures on End (`{name, expression, error}`).
- **human_tasks**: Unified human task records (type, status, priority, source discriminated union, fields[], response, escalationChain, dueAt, slaBreachedAt). Indexes: `{tenantId, projectId, status, createdAt}`, `{source.type, source.executionId, source.stepId}`, `{status, dueAt}`.

### Key Relationships

- Workflow 1:N WorkflowVersion (via workflowId)
- Workflow 1:N WorkflowExecution (via workflowId)

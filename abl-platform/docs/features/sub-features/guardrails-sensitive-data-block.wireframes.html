4. Decision matrix modal

Opens from the "?" icon on either screen. Single source of truth for the gate-vs-handle mental model.

Decision matrix — shared component

PII: where do I configure what?

The platform has two screens for PII. Each does a different job. Use this table to decide which one you need.

I want to… Go to
Hide PII from the LLM but still let the agent use it (tool calls work) Settings → PII Protection
Mask PII in agent responses to end users Settings → PII Protection
Reversibly tokenize PII (vault-backed) Settings → PII Protection
Audit every PII detection event Settings → PII Protection
Block messages containing specific PII types outright Guardrails → Sensitive Data Block
Escalate to a human when PII confidence is high Guardrails → Sensitive Data Block
Enforce PII blocking across all projects in a tenant Guardrails (tenant-level policy)
The one-word rule

Settings = "handle" sensitive data so the agent can do its job without exposing it.   Guardrails = "stop" the agent from processing data it must never touch.

Coexistence with PR #989 (Open Item #6)

The new gate prevents the user from flipping the toggle ON. PR #989's banner still fires if a rule was somehow saved enabled-but-incomplete (e.g., older data, schema migration). One mechanism is preventive (UI level); the other is corrective (form level). They cover different failure modes.

Behavior (FR-8.1–8.4):
  • Toggle is visually muted + cursor not-allowed.
  • Tooltip lists missing fields by name (i18n).
  • Compact inline hint "2 fields missing" to the right.
  • Server-side mirror — direct API attempts to enable an incomplete rule return 400 RULE_INCOMPLETE with missingFields[].

12. Auto-deactivation toast + undo

Per FR-7.4, if the last enabled rule in an active policy is disabled, the policy is auto-deactivated. The user sees a toast explaining the change with an Undo CTA (Open Item #9 — confirmed as the proposed behavior).

Auto-deactivation — toast with Undo
ℹ
Policy "abc" was deactivated
The last enabled rule was disabled. No rules are being enforced.
Undo
Why a toast (and not a confirmation modal)?

Modals block the user mid-flow and demand a decision before they understand the context. A toast is non-blocking: it announces the consequence after the fact and offers Undo for 5 seconds — long enough to recover from an accidental click, short enough to disappear when intentional. If the user dismisses the toast or the timeout elapses, the deactivation is final.

Behavior:
  • Toast auto-dismisses after 5 seconds (configurable).
  • "Undo" re-enables the rule AND re-activates the policy in one operation.
  • Telemetry: emit guardrail.auto_deactivation with { policyId, ruleId, undone: boolean }.
  • Empty policy + user toggles activation directly: same path, no toast (handled by FR-7.2 — toggle is disabled at the source).

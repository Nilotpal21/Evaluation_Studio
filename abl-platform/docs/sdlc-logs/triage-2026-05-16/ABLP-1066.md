# ABLP-1066 — LLM Credential Resolution Fails in Evals Pipeline Health

> **Review update — 2026-05-17 (GPT-5.5 high review)** · Verdict: **CONCERN**
>
> - Root cause confirmed at `packages/pipeline-engine/src/pipeline/services/llm-client-factory.ts:117,165,190,220,274` (7+ silent null-return failure modes collapsed into one message).
> - **Repro test can silently return early if `MongoMemoryServer` is unavailable** (`packages/pipeline-engine/src/__tests__/evals-pipeline-health.repro.test.ts:214,215`). Add an explicit `expect()` that the test setup succeeded, or fail loudly if Mongo can't start.
> - **"Valid config should pass" assertion may be testing model/mongoose setup**, not the reported diagnostic collapse. Sharpen the test focus to "the same input that produces 'No LLM model available' today should produce a specific code (`PROVIDER_NOT_CONFIGURED` vs `KEY_INVALID` vs `INFERENCE_DISABLED`)."
> - **Sanitization concern**: resolver currently leaks raw tenant ID into user-visible messages (`llm-client-factory.ts:118`, `eval-preflight.ts:187`). The typed-error contract must include a `userMessage` field that excludes tenant IDs.
> - **Provider/key mismatch is downgraded to warning when credential resolution fails** (`eval-preflight.ts:190`). UI cannot distinguish "not checked" from "checked but risky" — solution needs a structured `{ status: 'pass'|'warn'|'fail'|'not_checked', code, userMessage }` shape, not just typed errors.
> - **Be cautious about a platform-default fallback** in the resolver chain: it can turn a tenant configuration failure into a warning instead of preserving fail-fast preflight behavior (`eval-preflight.ts:183`). Make the fallback opt-in per environment, not default.
>

## Symptom

Evals > Pipeline Health panel shows "Overall: Failed" with two errors:

- "LLM credential resolution failed: No LLM model available for tenant."
- "Could not verify provider/key match (credential resolution failed)."

Tenant has a valid LLM provider/key configured, but the preflight check reports failure.

## Root Cause

**File:** `packages/pipeline-engine/src/pipeline/services/llm-client-factory.ts`

**Line 117-120:** The `resolvePipelineLLM()` function collapses 7+ distinct failure modes into a single generic error message:

```ts
throw new Error(
  `No LLM model available for tenant ${tenantId}. ` +
    `Configure a model in project settings or tenant admin.`,
);
```

The resolution chain (`resolveByModelId` > `resolveByProject` > `resolveByTenant`) silently returns `null` at each level for any of:

| Failure Mode                                             | Location (line) | Returns |
| -------------------------------------------------------- | --------------- | ------- |
| TenantModel not active OR `inferenceEnabled: false`      | 138-165         | `null`  |
| No `ModelConfig({isDefault:true})` for project           | 179             | `null`  |
| TenantModel referenced by ModelConfig inactive           | 183-189         | `null`  |
| TenantModel has no active connection with `credentialId` | 265-275         | `null`  |
| `LLMCredential` not found or missing `encryptedApiKey`   | 279-285         | `null`  |
| Decryption of `encryptedApiKey` fails                    | 287-301         | `null`  |
| Decrypted API key is empty                               | 303-309         | `null`  |
| TenantModel has no `modelId`                             | 313-316         | `null`  |

The most likely real-world cause: tenant has a TenantModel marked `isActive: true` but either `inferenceEnabled: false` (not explicitly set, defaults may vary), OR the model's connection lacks a `credentialId`, OR the credential's `encryptedApiKey` fails decryption (ENCRYPTION_MASTER_KEY mismatch between pipeline-engine and the service that encrypted it).

**Call path:** Studio `POST /api/projects/:id/evals/preflight` (line 34) > Restate `EvalPreflight/check` (`eval-preflight.service.ts:20`) > `runEvalPreflight` (`eval-preflight.ts:417`) > `checkLLMCredentialsAndProviderMatch` (line 166) > `resolvePipelineLLM` (`llm-client-factory.ts:61`).

## Reproduction Test

See `packages/pipeline-engine/src/__tests__/evals-pipeline-health.repro.test.ts`.

Seeds a tenant with an active TenantModel + LLMCredential, then calls `runEvalPreflight(tenantId, projectId)` and asserts the `llm_credentials` check returns `'pass'`. Today this fails because the resolver cannot find or decrypt the credential in the test environment -- reproducing the same "No LLM model available" collapse.

## Future-Ready Solution

### 1. Typed `LlmCredentialResolver` Chain with Explicit Error Codes

Replace silent `null` returns in `resolveCredential()` and each resolution level with a discriminated union result:

```ts
type ResolutionFailure =
  | { code: 'NO_TENANT_MODEL'; tenantId: string }
  | { code: 'MODEL_INACTIVE'; modelId: string }
  | { code: 'INFERENCE_DISABLED'; modelId: string }
  | { code: 'NO_ACTIVE_CONNECTION'; tenantModelId: string }
  | { code: 'CREDENTIAL_NOT_FOUND'; credentialId: string }
  | { code: 'DECRYPTION_FAILED'; credentialId: string; reason: string }
  | { code: 'EMPTY_API_KEY'; credentialId: string }
  | { code: 'NO_MODEL_ID'; tenantModelId: string }
  | { code: 'NO_PROJECT_DEFAULT'; projectId: string }
  | { code: 'NO_TENANT_DEFAULT'; tenantId: string };
```

Each level returns `{ ok: true; data: ResolvedPipelineLLM } | { ok: false; failure: ResolutionFailure }`. The top-level function collects all failures and throws a structured error.

### 2. Pre-flight Diagnostic Categories

The Pipeline Health UI should distinguish three categories:

1. **No provider configured** — no TenantModel with `isActive + inferenceEnabled` exists.
2. **Provider configured, key invalid** — TenantModel found but credential decryption fails or key is empty.
3. **Provider/key mismatch** — credential decrypts successfully but prefix heuristic detects wrong provider.

Each maps to a distinct remediation action displayed in `EvalPreflightPanel.tsx`.

### 3. Project > Tenant > Platform Default Precedence

Add an explicit `platform-default` level (fallback model ID from env `PLATFORM_DEFAULT_MODEL_ID`) so new tenants without any model config still get a diagnostic rather than the generic "No LLM model available" error. The platform default should only produce a `warn` (not fail) since it indicates missing tenant-specific configuration.

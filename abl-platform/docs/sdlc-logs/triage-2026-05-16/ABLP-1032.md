# ABLP-1032 — Duplicate top-level DSL sections silently overwrite

> **Review update — 2026-05-17 (GPT-5.5 high review)** · Verdict: **CONCERN**
>
> - Root cause confirmed at `packages/core/src/parser/agent-based-parser.ts:417,438,463`.
> - **Test does not cover singleton sections** (GOAL, EXECUTION, LANGUAGE) even though the proposed registry depends on singleton-vs-accumulator semantics. Extend the repro at `packages/core/src/__tests__/duplicate-sections.repro.test.ts:16` to assert the singleton path raises a `DuplicateSectionError`.
> - **Test asserts accumulation** for every section but the doc lists "accumulation OR DuplicateSectionError" as the acceptable outcome — pick one per section and align test with the policy.
> - **Naive accumulation would break TOOLS**: duplicate tool names would be preserved into IR compilation (`packages/compiler/src/platform/ir/compiler.ts:1020`) and downstream code expects unique tool keys. The registry MUST specify merge semantics per section (singleton, append, dedupe-by-key).
>
> Full review: [`codex-review.md`](codex-review.md)

## Symptom

When the same top-level section (DELEGATE, HANDOFF, TOOLS, MEMORY, CONSTRAINTS, etc.) appears more than once in an agent file, the parser silently overwrites `doc.<field>` with the output of the last occurrence. No error or warning is raised. The first block's content is irrecoverably lost.

TEMPLATES is the only section that correctly accumulates (spread into existing array).

## Root Cause

**File:** `packages/core/src/parser/agent-based-parser.ts`
**Lines:** 417-462 (dispatch loop inside `parseDocument`)

Every section uses a direct assignment pattern:

```ts
// Line 439
} else if (line === 'DELEGATE:') {
  doc.delegate = parseDelegate(state);   // OVERWRITES previous value
} else if (line === 'HANDOFF:') {
  doc.handoff = parseHandoff(state);     // OVERWRITES previous value
```

The sole exception is TEMPLATES at line 463-465:

```ts
} else if (line === 'TEMPLATES:') {
  const parsed = parseTemplatesBlock(state);
  doc.templates = [...(doc.templates || []), ...parsed]; // ACCUMULATES
```

The dispatch loop never checks whether a section has already been populated before assigning. This is a category error in the parser's section-registration model.

## Reproduction Test

See `packages/core/src/__tests__/duplicate-sections.repro.test.ts`

For each affected section (DELEGATE, HANDOFF, TOOLS, MEMORY), the test:

1. Constructs DSL with two blocks of the same section containing distinct entries
2. Parses via `parseAgentBasedABL`
3. Asserts that both blocks' entries are present in the result (or that a DuplicateSectionError is raised)

All assertions currently fail because only the last block survives.

## Future-Ready Solution

### Design: Schema-Driven Section Registry

Replace the ad-hoc if/else dispatch with a declarative section registry:

```ts
interface SectionDescriptor {
  key: keyof AgentBasedDocument;
  cardinality: 'singleton' | 'accumulator';
  parser: (state: ParserState) => unknown;
}

const SECTION_REGISTRY: Record<string, SectionDescriptor> = {
  'DELEGATE:': { key: 'delegate', cardinality: 'accumulator', parser: parseDelegate },
  'HANDOFF:': { key: 'handoff', cardinality: 'accumulator', parser: parseHandoff },
  'TOOLS:': { key: 'tools', cardinality: 'accumulator', parser: parseTools },
  'MEMORY:': { key: 'memory', cardinality: 'accumulator', parser: parseMemory },
  'CONSTRAINTS:': { key: 'constraints', cardinality: 'accumulator', parser: parseConstraints },
  'TEMPLATES:': { key: 'templates', cardinality: 'accumulator', parser: parseTemplatesBlock },
  // Singleton sections raise on duplicate
  'GOAL:': { key: 'goal', cardinality: 'singleton', parser: parseGoal },
  'EXECUTION:': { key: 'execution', cardinality: 'singleton', parser: parseExecutionConfig },
  'LANGUAGE:': { key: 'language', cardinality: 'singleton', parser: parseLang },
  // ... all other sections
};
```

### Dispatch logic

```ts
const seen = new Map<string, number>(); // section -> first-seen line

for (const [header, descriptor] of Object.entries(SECTION_REGISTRY)) {
  if (line === header) {
    const parsed = descriptor.parser(state);

    if (descriptor.cardinality === 'singleton') {
      if (seen.has(header)) {
        state.errors.push({
          line: state.currentLine,
          column: 0,
          message: `Duplicate section ${header} (first seen at line ${seen.get(header)}). Only one ${header} block is allowed per agent.`,
        });
        break;
      }
      doc[descriptor.key] = parsed;
    } else {
      // Accumulator: concat arrays, merge objects
      const existing = doc[descriptor.key];
      if (Array.isArray(existing) && Array.isArray(parsed)) {
        doc[descriptor.key] = [...existing, ...parsed];
      } else if (isPlainObject(existing) && isPlainObject(parsed)) {
        doc[descriptor.key] = deepMerge(existing, parsed);
      } else {
        doc[descriptor.key] = parsed;
      }
    }

    seen.set(header, state.currentLine);
    break;
  }
}
```

### Benefits

1. Every array-typed section inherits TEMPLATES' accumulator semantics by default.
2. Singleton sections (GOAL, EXECUTION, LANGUAGE, SYSTEM_PROMPT) get a clear duplicate-detection error with line numbers.
3. Adding a new section is a one-line registry entry instead of adding another `else if` branch.
4. The schema enforces the invariant at the type level: `cardinality: 'accumulator'` implies the doc field MUST be `T[]`.
5. Backward compatible: existing single-block agents produce identical output; only multi-block agents gain accumulation.

### Migration

- Phase 1: Add accumulator behavior for array sections. No breaking changes.
- Phase 2: Emit deprecation warnings for singleton duplicates (e.g., two GOAL blocks) for one release cycle.
- Phase 3: Promote warnings to errors.

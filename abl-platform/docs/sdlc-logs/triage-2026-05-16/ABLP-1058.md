# ABLP-1058 — Empty response after `__set_context__` tool call in REASONING step

> **Review update — 2026-05-17 (GPT-5.5 high review)** · Verdict: **CONCERN**
>
> - Root cause is correct on the runtime side (`apps/runtime/src/services/llm/session-llm-client.ts:101,535,947`), but the **compiler executor path also rebuilds assistant content from `result.toolCalls` only** (`packages/compiler/src/platform/constructs/executors/reasoning-executor.ts:343`) — a runtime-only fix will leave the compiler path broken.
> - **Streaming has the same lossy `rawContent` construction** (`session-llm-client.ts:535`); the non-streaming fix must be ported to streaming.
> - Current repro is a **compiler-layer text-preservation proxy**, not a true OpenAI Responses repro. Add a runtime-layer integration test that asserts reasoning items round-trip end-to-end through `session-llm-client`.
>
> Full review: [`codex-review.md`](codex-review.md)

## Symptom

When a REASONING step invokes `__set_context__` (to satisfy an EXIT_WHEN condition), the LLM call immediately following that tool invocation either errors or returns an empty response. The debug trace shows `__set_context__` executing successfully, but the subsequent model call fails with OpenAI's "function_call without reasoning item" error (for o1/o3 reasoning models) or produces no output.

The user sees the generic empty-response message and cannot advance past the step. Retries may succeed if the session state happens to re-enter the loop differently.

## Root Cause

The defect exists in two layers:

### Runtime layer (production code path)

**File:** `apps/runtime/src/services/llm/session-llm-client.ts:105,947`

`rawContent` is typed as `Array<TextContent | ToolUseContent>`. The `toVercelChatResult` method (line 947) and the streaming path (line 535) only emit `text` and `tool_use` blocks — any reasoning/thinking items from the provider response are silently dropped:

```typescript
// session-llm-client.ts:947
const rawContent: Array<TextContent | ToolUseContent> = [];
if (text) {
  rawContent.push({ type: 'text', text });
}
// ... only tool_use blocks added after this
```

**File:** `apps/runtime/src/services/execution/reasoning-executor.ts:2949`

The assistant message pushed into the conversation uses `result.rawContent` (which is missing the reasoning item):

```typescript
messages.push({
  role: 'assistant',
  content: result.rawContent,
});
```

When the next LLM call is made (line 2654), OpenAI's Responses API rejects it because the conversation history contains a `function_call` output without its required preceding `reasoning` item.

### Compiler layer (same architectural defect, not production path)

**File:** `packages/compiler/src/platform/constructs/executors/reasoning-executor.ts:344-352`

The compiler executor builds the assistant message from `result.toolCalls` only — dropping both `text` content and any provider-specific blocks:

```typescript
messages.push({
  role: 'assistant',
  content: result.toolCalls.map((tc) => ({
    type: 'tool_use',
    id: tc.id,
    name: tc.name,
    input: tc.input,
  })),
});
```

## Reproduction Test

**Path:** `packages/compiler/src/platform/constructs/executors/__tests__/reasoning-executor-set-context.repro.test.ts`

**Run:** `pnpm test --filter=@agent-platform/compiler -- reasoning-executor-set-context`

The test provides a fake LLM client that returns both text and tool_use on iteration 1, then asserts the assistant message in the messages array passed to iteration 2 includes the text content from iteration 1. On current code, only tool_use blocks are preserved — the text (proxy for reasoning/thinking) is dropped, causing the assertion to fail.

## Future-Ready Solution

The root issue is that `rawContent` (and the compiler's message construction) uses a closed union (`TextContent | ToolUseContent`) that cannot represent provider-specific items like reasoning/thinking blocks.

### Proposed fix: typed content block union with `ReasoningContent`

1. **Extend `packages/compiler/src/platform/llm/types.ts`:**

```typescript
export interface ReasoningContent {
  type: 'reasoning';
  reasoning: string;
  /** Provider-specific metadata for round-trip */
  providerMetadata?: Record<string, unknown>;
}

export type AssistantContentBlock = TextContent | ToolUseContent | ReasoningContent;
```

2. **Update `ChatResult.rawContent`** to use `AssistantContentBlock[]`.

3. **Update `toVercelChatResult` and streaming path** in `session-llm-client.ts` to capture reasoning items from the Vercel AI SDK result (available via `result.reasoning` or provider metadata).

4. **Both reasoning executors** (runtime at line 2949, compiler at line 344) already push `rawContent` / build content — they would automatically preserve reasoning blocks once `rawContent` includes them.

5. **Provider-specific normalization**: The LLM client layer should normalize reasoning items into the canonical `ReasoningContent` shape regardless of whether the upstream is OpenAI Responses API, Anthropic extended thinking, or another provider. This ensures the reasoning loop is provider-agnostic.

This approach guarantees that ANY content the LLM produces (text, reasoning, tool_use) survives the round-trip through the conversation history, preventing the entire class of "missing item" errors across all providers.

## Problem

The ABL Platform has no parameter transformation layer between LLM tool-call generation and tool execution. Every value a tool needs—including static, session-derived, and config-derived values—must be produced by the LLM as raw JSON. This inflates prompt schemas (~200-300 extra tokens per complex tool), increases hallucination of nested/boilerplate fields, forces more capable (expensive) models for structured output, and raises the overall error surface.

Example: XO 10 webhook requires the LLM to generate a full `nlMeta` object with `botId` (static), `language` (from session), and `channelType` (always `"webhook"`), when only `intent` and `entities` are dynamic.

## Proposed Contract

### (a) `hidden` + `defaultSource` on ToolParameter (Phase 1)

```typescript
// packages/compiler/src/platform/ir/schema.ts — extend ToolParameter
export interface ToolParameter {
  name: string;
  type: string;
  description?: string;
  required: boolean;
  default?: unknown;
  validation?: string;
  enum?: unknown[];
  properties?: ToolParameter[];
  items?: { type: string; enum?: unknown[]; properties?: ToolParameter[] };

  // ─── NEW (Phase 1) ───────────────────────────────────────────────
  /**
   * When true, this parameter is excluded from the LLM-facing JSON schema
   * and resolved at execution time via `defaultSource`.
   */
  hidden?: boolean;

  /**
   * Dot-path expression for runtime resolution of hidden params.
   * Allowed prefixes (allowlisted):
   *   - session.<field>   (e.g. session.botId, session.language)
   *   - config.<field>    (e.g. config.channelType)
   *   - context.<field>   (e.g. context.tenantId)
   *   - literal:<value>   (e.g. literal:webhook)
   */
  defaultSource?: string;
}
```

**Schema filtering** (in `apps/runtime/src/services/execution/prompt-builder.ts`):

```typescript

// Resolve hidden params from defaultSource before session context injection
if (tool?.parameters) {
  const hiddenParams = tool.parameters.filter((p) => p.hidden && p.defaultSource);
  for (const param of hiddenParams) {
    if (!(param.name in params)) {
      params[param.name] = resolveDefaultSource(param.defaultSource!, resolutionContext);
    }
  }
```

### (c) DSL `COMPUTED:` block grammar (Phase 3)

```yaml
TOOLS:
  - name: send_webhook
    description: Send message via XO 10 webhook
    parameters:
      - name: message
        type: string
        required: true
      - name: intent
        type: string
        required: true
    COMPUTED:
      botId: session.botId
      language: session.language
      channelType: literal:webhook
      timezone: config.defaultTimezone
```

Parser semantics:

- `COMPUTED:` block is a sibling of `parameters:` within a tool definition.
- Each key becomes a hidden `ToolParameter` with `hidden: true` and `defaultSource` set to the value expression.
- The `type` is inferred as `string` unless suffixed with `:number` or `:boolean`.
- Computed params do NOT appear in `parameters:` (they are additive).

## Risks & Open Questions

1. **Security: arbitrary session field exfiltration.** The existing `TOOL_SESSION_CONTEXT_PARAM_MAP` is a fixed 3-key allowlist. A generic `defaultSource: 'session.X'` with unrestricted `X` would let tool authors (or malicious DSL) access arbitrary session fields (PII, tokens, internal state). **Mitigation:** Phase 1 MUST enforce an explicit allowlist of resolvable paths. The allowlist should be maintained in `contract-source-data.ts` alongside the existing map. Audit log every resolution attempt against a non-allowlisted path.

2. **Naming collisions.** A `COMPUTED:` key that matches an existing `parameters:` name would cause ambiguity. Phase 3 validator must reject duplicates at compile time. Additionally, hidden param names must not collide with the injected `thought` param (when thinking is enabled).

3. **Ordering w.r.t. existing middleware.** The pre-processor must run BEFORE `validateToolInputs()` (since hidden params won't be in `llmParams`) and BEFORE the existing `TOOL_SESSION_CONTEXT_PARAM_MAP` injection (to avoid double-injection of `session_id` etc.). Proposed order: size check → pre-processor resolution → session context injection → validation → middleware chain → dispatch.

4. **Backward compatibility.** Tools without `hidden` params are unaffected — `hidden` defaults to `undefined`/`false`, and the schema filter is a no-op. No existing tool definitions break. The `COMPUTED:` block is purely additive DSL syntax.

5. **Runtime resolution failures.** What happens when `defaultSource: 'session.language'` resolves to `undefined` at execution time (session field not set)? Options: (a) throw and fail the tool call, (b) pass `undefined` and let validation catch it, (c) skip injection and let the tool handle missing params. Recommend (a) with a clear error message pointing to the tool definition.

6. **LLM caching invalidation.** Since `buildTools()` output is cached via prompt hashing, hiding params changes the schema hash. This is correct behavior (tools with hidden params produce a smaller schema → different cache key). No special handling needed.

7. **Multi-provider schema parity.** `ablTypeToJsonSchema` is called in runtime `prompt-builder.ts`. The Ultravox realtime provider (`packages/compiler/src/platform/llm/realtime/ultravox-realtime.ts:390`) builds schemas independently from `input_schema` — it must also respect `hidden`. Similarly, the KoreVG S2S adapter needs the same filter.

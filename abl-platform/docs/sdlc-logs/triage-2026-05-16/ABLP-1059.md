## Symptom

DSL `ASYNC: true` on a handoff does not reliably trigger async push mode. Streaming mode is never reached even when the remote agent declares `streaming: true`. The platform silently falls back to sync with no error or trace explanation.

## Root Cause

**File:** `apps/runtime/src/services/execution/routing-executor.ts`
**Lines:** 2009-2057

Mode selection is scattered across three sequential `if` blocks with no unified resolver:

1. **Line 2010:** `if (handoffConfig?.async && this.ctx.asyncInfra)` — enters async path. Two bugs:
   - If `this.ctx.asyncInfra` is undefined (e.g., async infra not wired in dev/test), the condition is false and execution silently falls through to sync. There is no error, no trace event — `ASYNC: true` is simply ignored.
   - The remote agent's `capabilities.pushNotifications` is never consulted before entering async mode. If the remote agent does not support push, the call will fail at the remote end, not at the decision point.

2. **Lines 2031-2039:** `if (onChunk) { remoteCard = getRemoteAgentCard(...); if (remoteCard?.capabilities?.streaming) { ... } }` — streaming path is only entered when `onChunk` is provided AND the card advertises streaming. But:
   - If the user is connected (onChunk present) and the DSL says `ASYNC: true`, the async check at line 2010 already short-circuited, so streaming is never even considered as a negotiated fallback.
   - If `onChunk` is absent (REST API call, no WebSocket), streaming is never attempted regardless of card capabilities.

3. **Lines 2057+:** Falls through to plain sync `sendTask()` as the default — no trace of why this mode was chosen.

**Net effect:** There is no single function that takes `(dslAsyncFlag, remoteAgentCard, infraAvailability)` and returns a resolved mode or an error. The truth table from the ticket is not implemented anywhere.

## Reproduction Test

See `packages/a2a/src/__tests__/a2a-handoff-mode.repro.test.ts`.

Tests exercise the 4-row truth table:

- `ASYNC:false + streaming:false` -> sync
- `ASYNC:false + streaming:true` -> streaming
- `ASYNC:true + pushNotifications:true` -> async push
- `ASYNC:true + pushNotifications:false` -> error (fast-fail)

The tests call the extracted current-mode helper and assert the future
`A2AModeResolver` contract. Rows that already match current behavior pass; rows
that require fast-fail capability validation fail.

## Future-Ready Solution

### Introduce `A2AModeResolver`

```typescript
// packages/a2a/src/application/mode-resolver.ts

export type A2AMode = 'sync' | 'streaming' | 'async-push';

export interface ModeResolverInput {
  dslAsync: boolean;
  agentCard: AgentCard | null;
  asyncInfraAvailable: boolean;
  userConnected: boolean; // onChunk present
}

export interface ModeResolverResult {
  mode: A2AMode;
  reason: string; // for trace
}

export class A2AModeResolutionError extends Error {
  constructor(
    public readonly requested: string,
    public readonly unsupported: string,
    message: string,
  ) {
    super(message);
    this.name = 'A2AModeResolutionError';
  }
}

/**
 * Single chokepoint for A2A mode selection.
 * All handoff calls route through this BEFORE dispatch.
 *
 * Truth table:
 *   dslAsync=false, streaming=false  -> sync
 *   dslAsync=false, streaming=true   -> streaming (if userConnected)
 *   dslAsync=true,  push=true        -> async-push
 *   dslAsync=true,  push=false       -> ERROR (fast-fail)
 *   dslAsync=true,  asyncInfra=false -> ERROR (infra not available)
 */
export function resolveA2AMode(input: ModeResolverInput): ModeResolverResult {
  if (input.dslAsync) {
    if (!input.asyncInfraAvailable) {
      throw new A2AModeResolutionError(
        'async-push',
        'asyncInfra',
        'ASYNC:true requested but async infrastructure is not available',
      );
    }
    if (!input.agentCard?.capabilities?.pushNotifications) {
      throw new A2AModeResolutionError(
        'async-push',
        'pushNotifications',
        'ASYNC:true requested but remote agent does not advertise pushNotifications capability',
      );
    }
    return { mode: 'async-push', reason: 'DSL ASYNC:true + remote supports pushNotifications' };
  }

  // Non-async: prefer streaming if available and user connected
  if (input.userConnected && input.agentCard?.capabilities?.streaming) {
    return { mode: 'streaming', reason: 'Remote supports streaming + user connected' };
  }

  return { mode: 'sync', reason: 'Default sync — streaming not available or user not connected' };
}
```

### Integration into `routing-executor.ts`

Replace the three scattered conditionals (lines 2009-2057) with:

```typescript
const remoteCard = await this.getRemoteAgentCard(...);
const modeResult = resolveA2AMode({
  dslAsync: !!handoffConfig?.async,
  agentCard: remoteCard,
  asyncInfraAvailable: !!this.ctx.asyncInfra,
  userConnected: !!onChunk,
});

// Emit mode decision in trace
onTraceEvent?.({
  type: 'a2a_mode_resolved',
  data: { mode: modeResult.mode, reason: modeResult.reason, targetAgent },
});

switch (modeResult.mode) {
  case 'async-push':
    return this.handleAsyncRemoteHandoff(...);
  case 'streaming':
    return this.handleStreamingRemoteHandoff(...);
  case 'sync':
    // fall through to sync sendTask
    break;
}
```

This ensures:

- A single testable pure function owns the decision
- Every decision is traced with its reason
- Mismatches fail fast with descriptive errors
- Adding new modes (e.g., HTTP polling) only requires extending the resolver

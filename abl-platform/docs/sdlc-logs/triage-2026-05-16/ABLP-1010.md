## Symptom

When publishing a module release with HTTP tools that have literal auth values,
the UI shows a single red block of text like:

```
[LITERAL_AUTH_VALUE] tool:list_users: HTTP tool "list_users" has a non-templated auth value. Use auth_profile_ref or {{env. / config. templating instead.. [LITERAL_AUTH_VALUE] tool:get_user: …
```

Users see bracketed error codes, concatenated strings, and no guidance on which
settings page to open. Deployment failures similarly show opaque
`MODULE_BUILD_FAILED` / `AUTH_PROFILE_NOT_FOUND` strings.

## Root Cause

The structured `PublishSafetyIssue[]` (with `severity`, `code`, `source`,
`message`) is collapsed into flat strings at two points:

1. **`packages/project-io/src/module-release/build-module-release.ts:269`**

   ```ts
   errors: blockingIssues.map((i) => `[${i.code}] ${i.source}: ${i.message}`),
   ```

   This flattens typed `{ severity, code, source, message }` objects into
   opaque strings before returning `ModuleReleaseBuildFailure`.

2. **`apps/studio/src/app/api/projects/[id]/module/releases/route.ts:449-455`**

   ```ts
   errors: buildResult.errors.map((msg) => ({ msg, code: ErrorCode.BUILD_ERROR })),
   ```

   The route handler wraps already-flattened strings into generic
   `{ msg, code: 'BUILD_ERROR' }` objects — losing `toolId`, `fieldPath`, and
   the original diagnostic code.

3. **UI path** (`PublishModuleDialog.tsx:153` via `sanitizeError`):
   The `MAX_MESSAGE_LENGTH = 200` threshold in `sanitize-error.ts:60` causes
   multi-tool error strings (which easily exceed 200 chars when concatenated by
   `AppError.messages.join('. ')`) to be replaced by a generic fallback message.
   Even when the string passes through, it renders as a single `<span>` with no
   structure, no per-tool breakdown, no links.

## Reproduction Test

See: `packages/project-io/src/__tests__/module-publish-diagnostics.repro.test.ts`

Calls `buildModuleRelease` with two HTTP tools containing literal auth values
and asserts the response returns structured `diagnostics: PublishDiagnostic[]`
(with `code`, `toolId`, `fieldPath`, `userMessage`). Currently fails because
the response returns flat `errors: string[]`.

## Future-Ready Solution

### 1. Typed `PublishDiagnostic` Schema

```ts
// packages/project-io/src/module-release/publish-diagnostic.ts
export interface PublishDiagnostic {
  severity: 'blocking' | 'warning';
  code: string; // e.g. 'LITERAL_AUTH_VALUE'
  toolId?: string; // e.g. 'list_users'
  fieldPath?: string; // e.g. 'auth' | 'headers.Authorization'
  userMessage: string; // plain-language primary message
  supportMessage?: string; // technical detail for support logs
  actionLink?: string; // e.g. '/studio/tools/list_users#auth'
}
```

### 2. Backend Contract

`ModuleReleaseBuildFailure` returns:

```ts
{
  success: false;
  diagnostics: PublishDiagnostic[];  // structured, typed
  warnings: PublishDiagnostic[];
}
```

The route handler passes `diagnostics[]` through to the JSON response as-is.
Never concatenate or collapse. The existing `errors: string[]` field is kept
temporarily for backward compatibility but deprecated.

### 3. Error-Code Catalog

```ts
// packages/project-io/src/module-release/error-catalog.ts
export const PUBLISH_ERROR_CATALOG: Record<
  string,
  {
    userMessageTemplate: string;
    supportMessage: string;
    actionLinkTemplate?: string;
  }
> = {
  LITERAL_AUTH_VALUE: {
    userMessageTemplate:
      'HTTP tool "{toolId}" uses a hardcoded auth value. Switch to an auth profile or environment variable.',
    supportMessage: 'Non-templated AUTH directive detected in tool DSL.',
    actionLinkTemplate: '/studio/tools/{toolId}#auth',
  },
  AUTH_PROFILE_NOT_FOUND: {
    userMessageTemplate:
      'Auth profile "{fieldPath}" referenced by tool "{toolId}" does not exist in this project.',
    supportMessage:
      'Module references an auth profile from the source project that was not imported.',
    actionLinkTemplate: '/studio/auth-profiles',
  },
  MODULE_BUILD_FAILED: {
    userMessageTemplate: 'Module build failed. Check the diagnostics below for specific issues.',
    supportMessage: 'One or more blocking diagnostics prevented the release build.',
  },
  // ... one entry per known code
};
```

Single source of truth — UI and backend import from the same file.

### 4. UI Rendering — Shared `DiagnosticsList` Component

```tsx
// apps/studio/src/components/modules/DiagnosticsList.tsx
// Used by: PublishModuleDialog, deployment toast, module-build surfaces

<DiagnosticsList diagnostics={diagnostics}>
  {/* Per-diagnostic: */}
  <div>
    <p class="primary">{diagnostic.userMessage}</p>
    <details>
      <summary>Details</summary>
      <code>{diagnostic.code}</code>: {diagnostic.supportMessage}
    </details>
    {diagnostic.actionLink && <a href={actionLink}>Fix this →</a>}
  </div>
</DiagnosticsList>
```

Primary `userMessage` is always visible. `code` + `supportMessage` shown in
expandable "Details". Action links navigate directly to the relevant Studio
surface (Tool Library, Auth Profiles, etc.).

### 5. Consistency

- Publish modal, deployment toasts, and module-build surfaces all use
  `DiagnosticsList` with the same `PublishDiagnostic[]` shape.
- `sanitizeError` is no longer the rendering path for structured diagnostics —
  it remains only for truly unstructured/unexpected errors.

## Verified

The shipped diff introduces:

1. **`buildInlineDebugPayload` helper** — returns `state`, `traceEvents`, `traceContext` only when `includeDebug` is true; otherwise returns `{}`.
2. **`debug` body field** added to `agentChatSchema` (`z.boolean().optional()`).
3. **Query-flag support** — `isTruthyQueryFlag` parses `?debug=true|1` and `?verbose=true|1`.
4. **All response paths gated** — success (line 2979), auth-required (line 2619), queue-full (line 2783), timeout (line 2808), and error (line 3030) all spread `buildInlineDebugPayload` conditionally.
5. **Test coverage** — 57 additional lines in `chat-routes.test.ts` covering query-flag scenarios.

Default API channel responses no longer expose internal state, trace events, or trace context.

## Residual Follow-ups

### 1. UI Curl Example Missing `metadata` / `interactionContext`

**Gap:** `apps/studio/src/components/deployments/channels/tabs/OverviewTab.tsx` lines 163-180 build the `sdkChatExample` curl snippet. The payload only includes `projectId`, `message`, `sessionId`, and optionally `deploymentId`/`environment`. It does NOT include `metadata` or `interactionContext` placeholders.

**Impact:** Users copying the curl from the UI don't know these fields exist; Prathyusha confirmed she missed them because the curl didn't show them.

### 2. Attachments / Document Upload via API

**Gap:** The schema at `apps/runtime/src/routes/chat.ts:1407` accepts `attachmentIds: z.array(z.string().min(1)).optional()` — passing pre-uploaded attachment IDs. However, there is **no multipart/form-data ingress** for direct document upload. The runtime has no file-storage service, no virus-scan step, and no upload endpoint that returns attachment IDs for the chat route to consume.

**Impact:** Users wanting to upload documents via the API channel must first upload via an external mechanism (not documented, not implemented in platform). This is a larger feature spanning storage, security, and API design.

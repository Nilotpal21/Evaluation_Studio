## Description

What's happening

    - The handoff to a remote A2A agent was configured with ASYNC: true in the DSL, but the actual outbound call still went as async: false. The trace confirms a blocking sync call was made, not an async push call.

    - The platform does not consult the remote agent's agent-card.json capabilities (streaming, pushNotifications) when deciding which mode to use. The decision is based purely on the DSL flag and ignores what the remote agent has declared it supports.

    - As a result:

    	- Streaming mode is never reached, even when the remote agent declares streaming: true.

    	- Async push mode is not reliably triggered, even when DSL says ASYNC: true and the remote agent declares pushNotifications: true.

What should happen

The mode should be picked based on both the DSL flag and the remote agent's advertised capabilities:

Additionally:

    - The trace should clearly show which mode was chosen and why.

    - When a mismatch occurs (e.g., ASYNC: true but remote does not support push), the platform should fail fast with a clear error, not silently fall back to sync.

## Description

Summary

When the same top-level DSL section (e.g. DELEGATE:, HANDOFF:, TOOLS:, MEMORY:, CONSTRAINTS:) is written more than once in an agent file, the parser silently overwrites the first occurrence with the last. No error is raised, no warning is emitted. The missing content is only discoverable by inspecting the raw LLM payload in logs — the compiled agent behaves as if the first block never existed.

Steps to Reproduce :

Write two DELEGATE: blocks in a single agent file:

DELEGATE:

- AGENT: SummaryAgent
  WHEN: user asks for a summary
  PURPOSE: "Summarize the conversation"
  INPUT: {}
  RETURNS: {}
  USE_RESULT: "Use the summary in your reply"

DELEGATE:

- AGENT: BillingAgent
  WHEN: user asks about billing
  PURPOSE: "Handle billing queries"
  INPUT: {}
  RETURNS: {}
  USE_RESULT: "Answer based on billing result"

Expected: Both delegate_to_SummaryAgent and delegate_to_BillingAgent tools are available to the LLM.

Actual: Only delegate_to_BillingAgent exists. SummaryAgent is silently dropped. The LLM never sees a tool to reach it.

No compilation error. No warning. The agent appears to compile and deploy normally.

Affected Top-Level Sections

Every direct-assignment section in the dispatch loop is affected. This includes at minimum:

TEMPLATES: is the only exception — it correctly spreads into an accumulator.

Why This Is Hard to Detect

    - Compilation succeeds. The parser produces a valid AST and the IR compiles without error. There is nothing structurally wrong from the compiler's perspective — one valid DELEGATE: block was found.

    - No warning at the discard site. The old value is overwritten by a plain assignment. The parser never checks whether doc.delegate was already populated before writing.

    - The symptom is behavioral, not structural. The agent runs. It just cannot reach the dropped agents. The developer sees no delegate_to_SummaryAgent in the LLM tool list and has no immediate signal as to why.

    - Diagnosis requires log inspection. The only reliable way to discover missing delegates/handoffs today is to enable debug logging and inspect the raw LLM payload to see which delegate_to_ or handoff_to_ tools were injected.

Impact

    - Silent data loss during authoring. A developer splitting DELEGATE: into multiple blocks (e.g. for readability, or after copying a block from another agent) loses all but the last without any indication.

    - Debugging cost. Root-causing requires knowing to look at the LLM payload, not the DSL or the compilation output.

    - Affects coordination correctness. For DELEGATE and HANDOFF specifically, the dropped entries mean entire agent-routing paths are inaccessible at runtime — a functional regression with no observable compile-time signal.

    - Scope. Any of the affected sections above can silently lose content, meaning the issue is not limited to coordination constructs.

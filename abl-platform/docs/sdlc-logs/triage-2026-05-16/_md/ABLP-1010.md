## Description

Problem (user-facing)

Studio surfaces internal, code-like validation text in several module flows. Messages expose raw error codes (e.g. &#91;LITERAL_AUTH_VALUE&#93;) and long concatenated strings instead of clear, human-readable copy that states which tool / which field is wrong and what to change.

A) Publish Module Release

In Publish Module Release (e.g. agents-dev.kore.ai), failed POST …/releases validation is shown in the modal as a single red block of text similar to:

&#91;LITERAL_AUTH_VALUE&#93; tool:list_users: HTTP tool "list_users" has a non-templated auth value. Use auth_profile_ref or {{env. / config. templating instead.. &#91;LITERAL_AUTH_VALUE&#93; tool:get_user: …}}

Issues for users

    - Bracketed codes read like engineer-only diagnostics, not product copy.

    - Repeating the same rule twice (per tool) is noisy; better as one summary + per-tool bullets.

    - Which setting to open (Tool Library → which tool → auth / auth profile) is not spelled out.

    - Field-level mapping (e.g. "Auth on HTTP tool list_users") should be visually primary; machine code can be secondary or in a "Details" expando.

B) Deployment — module dependency resolution

When module dependency resolution fails during deployment, the UI can show a generic toast, e.g. Module dependency resolution failed for this deployment. Fix the reported issues and retry. — without saying whether the blocker is auth profile, module version, entry agent, etc.

Desired outcome / acceptance criteria

    - Primary message in plain language — e.g. "Publishing blocked: HTTP tools must not embed raw auth secrets." Then per affected tool with field context (tool name, "Authentication" / auth_profile_ref).

    - Structured errors for UI — API returns items with toolId or toolName, field / path, userMessage, optional code for support logs — not one concatenated string.

    - Actionable next steps — Link or CTA to Tool Library / Auth profiles as appropriate; no stack traces or secret values.

    - Consistency — Publish modal, deployment toasts, and any diagnostics[] / moduleBuild surfaces use the same vocabulary and formatting rules.

    - Deploy flow — Same bar: name the missing or invalid prerequisite (e.g. auth profile name), not only MODULE_BUILD_FAILED.

Background — deployment repro (unchanged)

Published module imports into another project; deployment fails with MODULE_BUILD_FAILED, diagnostics such as AUTH_PROFILE_NOT_FOUND (source: auth-profile), mountedAgentCount / mountedToolCount 0. Backend may already expose richer diagnostics than the toast; UI must surface them clearly.

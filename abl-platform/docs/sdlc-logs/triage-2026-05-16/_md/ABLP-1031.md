## Description

Summary

HANDOFF and DELEGATE both do the same conceptual thing — transfer work to another agent. Yet their DSL syntax is structurally different enough that:

    - A developer cannot learn one and naturally write the other

    - Migrating a HANDOFF to a DELEGATE (or vice versa) requires a near-complete rewrite of that block

    - The cognitive overhead of keeping two different syntaxes in mind increases the chance of authoring errors

Problem Detail

Both constructs target only agents and both carry a condition (WHEN), a failure strategy (ON_FAILURE), and optional remote configuration (LOCATION, ENDPOINT, PROTOCOL). That is where the similarity ends.

Entry-point keyword is different:

HANDOFF:

- TO: BillingAgent # key is TO

DELEGATE:

- AGENT: BillingAgent # key is AGENT

There is no logical reason for this asymmetry. Both refer to the same kind of entity — an agent name.

Context/input passing uses a completely different model:

HANDOFF uses a named CONTEXT block with its own sub-keys:

HANDOFF:

- TO: BillingAgent
  CONTEXT:
  pass: [userId, orderId]
  summary: "User is asking about their latest invoice"
  history: auto

DELEGATE uses flat key-value maps for both directions:

DELEGATE:

- AGENT: BillingAgent
  INPUT: { child_user: session.userId }
  RETURNS: { session.result: child.summary }
  USE_RESULT: "Use the result to answer the user"

A developer moving a use-case from handoff to delegate has to restructure the entire data-passing block — CONTEXT.pass → INPUT, add RETURNS from scratch, add USE_RESULT. Nothing maps cleanly.

Shared fields: only WHEN, ON_FAILURE, TIMEOUT, and remote config are syntactically identical. Everything else diverges.

Impact

    - Authoring friction: A developer writing their first DELEGATE after knowing HANDOFF (or vice versa) will frequently pick the wrong keyword and structure.

    - Migration cost: Changing a HANDOFF to a DELEGATE is not a keyword swap — it requires restructuring the data-passing block entirely.

    - Documentation burden: Two separate syntax trees must be documented, taught, and remembered independently even though they serve adjacent purposes.

    - Error surface: Divergent entry keys (TO vs AGENT) mean a validator cannot catch "used HANDOFF syntax inside DELEGATE" early — it just silently ignores the unknown key.

Proposed Direction

Align the two constructs on a shared syntactic skeleton. Suggested target shape:

HANDOFF:

- TO: BillingAgent
  WHEN: <condition>
  PASS: [userId, orderId]
  SUMMARY: "User asking about invoice"
  RETURN: true
  ON_FAILURE: continue

DELEGATE:

- TO: BillingAgent # same key as HANDOFF
  WHEN: <condition>
  PASS: { child_user: session.userId } # same keyword, map form for delegate
  SUMMARY: "Summarize order history" # same keyword as PURPOSE
  ON_FAILURE: continue

Specific alignment changes worth scoping:

    - Unify the entry key to TO: for both (parser already accepts \w+ — this is a one-line change per construct)

    - Use PASS: for input data in both — list form for handoff, map form for delegate

    - Rename PURPOSE: → SUMMARY: on delegate to match handoff

This does not require changing runtime semantics — only the parser surface and the IR compilation step need updating. Backward compatibility can be maintained by accepting both the old and new keyword forms during a transition window.

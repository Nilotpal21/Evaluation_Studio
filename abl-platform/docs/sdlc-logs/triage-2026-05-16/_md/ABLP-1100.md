## Description

Problem

The ABL Platform currently has NO PARAMETER TRANSFORMATION LAYER between the LLM generating a tool call and the tool being executed. Every value the tool needs must be produced by the LLM as raw JSON in the tool call response.

This creates significant problems:

    - Larger schemas in the prompt (more tokens describing nested objects)

    - Higher error rates (wrong nesting, hallucinated envelope fields)

    - Costlier models required (simpler models fail on nested structured output)

    - Longer instructions explaining boilerplate instead of domain logic

EXAMPLE: For an XO 10 webhook, the LLM must generate the entire nlMeta object including botId (static), language (from session), and channelType (always webhook). Only intent and entities are truly dynamic.

Current Architecture Gaps

    - NO PARAMETER PRE-PROCESSOR LAYER - Between reasoning-executor.ts:320 and tool-binding-executor.ts:362, parameters pass through unchanged

    - NO COMPUTED/DERIVED PARAMETER DECLARATION - ToolParameter interface has no concept of derived from context/config

    - NO PARAMETER ENRICHMENT MIDDLEWARE - Tool middleware chain only has logging/timing

    - NO PARAMETER SCHEMA FILTERING FOR LLM - All parameters exposed to LLM, even derived ones

    - NO LOOKUP/REGISTRY RESOLUTION - Static values still require LLM generation

Solution (Phased Approach)

PHASE 1 (P1): Hidden params + defaultSource

    - Quick win: Add hidden: true + defaultSource to ToolParameter

    - Filter hidden params from LLM schema

    - Resolve defaults from context at execution time

    - EFFORT: Low (2-3 days)

PHASE 2 (P0): Parameter Pre-Processor Middleware

    - Add ToolParamPreProcessor middleware type

    - Transform LLM output to full tool params

    - Support computed params from session/config/lookup/expression

    - EFFORT: Medium (1 week)

PHASE 3 (P2): DSL COMPUTED block

    - Extend tool DSL with COMPUTED section

    - Best developer experience for tool authors

    - EFFORT: Medium-High (1-2 weeks)

Reference Implementation

Use XO 10 webhook nlMeta construction as validation case:

    - CURRENT: LLM generates ~8 fields, nested 2 levels

    - TARGET: LLM generates ~3 fields, flat (message, intent, entities)

    - AUTO-COMPUTED: botId, language, channelType, timezone

Expected Impact

LLM params to generate: ~8 fields nested → ~3 fields flat

JSON Schema complexity: Nested objects + arrays → Simple flat properties

Prompt instruction tokens: ~200-300 tokens → ~50 tokens

Model tier required: Higher (structured output) → Lower (simple key-value)

Error surface area: High → Low

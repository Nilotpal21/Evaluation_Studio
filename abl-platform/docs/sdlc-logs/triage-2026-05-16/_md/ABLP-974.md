## Description

Issue 1: contextId is client-generated — should be server-generated

What's happening:

The context ID that identifies a multi-turn conversation thread is currently generated by the client. This is a spec violation. If the client omits it or generates a duplicate, multi-turn continuity silently breaks. The server has no control over scoping, uniqueness, or tenant isolation.

What should happen:

On the first turn, the server generates a new contextId and returns it in the response. The client echoes it back on subsequent turns. The server validates and maps it against the session ID.

Issue 2: Task ID not carried forward across turns

What's happening:

On Turn 1, the platform sends no taskId → remote agent generates one and returns it in the Task response. On Turn 2, the platform ignores the returned taskId entirely and sends the next message with no taskId → remote agent creates a brand new task every time. True multi-turn task continuity with the remote agent is broken.

Additionally, once a task reaches a terminal state (completed, failed, canceled), the taskId should be cleared so the remote agent correctly creates a fresh task on the next interaction. This lifecycle is not managed at all.

What should happen:

    - After Turn 1, extract the taskId from the remote agent's response and store it against the session thread

    - On Turn 2+, include that taskId in message.taskId in the outbound request

    - Once the remote task reaches a terminal state, clear the stored taskId

    - Next interaction starts fresh with no taskId and the remote agent issues a new one

Issue 3: Task ID incorrectly logged in traces

What's happening:

The taskId appears correctly in some trace sections (e.g. handoff_progress) but in the a2a_call trace for the agent card discovery step, the taskId is logged as discovery:https://investments-agent-....run.app — the full endpoint URL — instead of a proper task identifier. This is because the discovery call constructs the tracing taskId as {{discovery:$

{endpoint}
}}, which leaks the URL into the trace.

What should happen:

The task ID should be correctly displayed in the trace — the discovery call should not leak the endpoint URL as the task ID.

Issue 4: No way to configure custom metadata for outbound A2A requests

What's happening:

The platform currently sends an auto-constructed metadata block (handoff_from, session values, pass fields, summary, memory grants) with every outbound A2A request. However there is no admin-facing option to define and send custom arbitrary metadata fields to an external A2A agent at the agent config level. This matters when the external agent expects specific metadata fields — e.g. tenant context, auth hints, routing info — that are not part of the standard handoff context.

What should happen:

Allow admins to configure static key-value metadata pairs at the agent config level. These should be merged on top of the existing auto-constructed metadata — not replace it — and sent in every outbound MessageSendParams.metadata call to that agent.

Issue 5: Task state not included in traces

What's happening:

The handoff_progress events use internal runtime phase names (started, suspended, resumed etc.) and do not reflect the actual A2A task states returned by the remote agent. Two states — working and input-required — are silently dropped with no event emitted for them. The actual task state from the remote agent is not surfaced in the trace at all.

What should happen:

Include the actual A2A task state (submitted, working, completed, failed, input-required) in the trace alongside the phase, so the full remote task lifecycle is visible when debugging.

Issue 6: contextId missing from all A2A traces

What's happening:

contextId is never included in a2a_call or handoff_progress trace events. The HandoffProgress interface doesn't define the field. Without contextId in traces, there is no way to correlate all turns of a multi-turn conversation when debugging.

What should happen:

contextId is already available in the routing-executor at the point these events are emitted (session.id). It needs to be passed through and added to the HandoffProgress interface and a2a_call event payload so it appears in every A2A trace entry.

Issue 7 (P2): Protocol version support — only A2A v0.3 JSON-RPC currently supported

What's happening:

The platform currently only supports A2A v0.3 JSON-RPC. External agents may advertise different transports and protocol versions, and there is no way to connect to them.

What should happen:

Add support for:

    - A2A v0.3 HTTP (non JSON-RPC HTTP transport)

    - A2A v1.0 — both JSON-RPC and HTTP methods

Protocol version and transport should be auto-detected from the agent card and configurable as a fallback in the agent config UI.

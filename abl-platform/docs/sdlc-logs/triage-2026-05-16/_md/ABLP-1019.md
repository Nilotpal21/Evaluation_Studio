## Description

Summary

The API channel currently has two significant gaps that limit its usability for real channel integrations:

    - No mechanism to pass or receive metadata from the calling channel or source

    - API response exposes full internal state instead of a clean, channel-friendly payload

Problem 1: No Channel Metadata Support

When invoking an agent via the API channel, there is no way for the caller to attach contextual metadata — such as source system identifiers, user-facing locale, custom headers, or any channel-specific context — to the request. Equally, the response carries no structured metadata back to the caller.

This forces integrators to work around the limitation by encoding metadata into the message payload, which pollutes the conversation context and makes downstream processing fragile.

Expected behavior: The API channel should support a metadata field (or equivalent) on both the request and response, allowing callers to pass and receive arbitrary channel-specific key-value context that is preserved across the interaction without being injected into the agent's reasoning loop.

Problem 2: Overly Verbose, Internally-Exposed Response Payload

The current API response returns the full internal execution state of the agent. This includes:

    - Full session context: tenantId, channelId, contactId, anonymousId, sessionPrincipalId, authScope, identityTier, verificationMethod, etc.

    - Complete agent IR: compiled intermediate representation including system_prompt (full text), all tool definitions with schemas, coordination config, memory config, error handler templates, and 30+ internal message templates

    - Full trace events: every llm_call event with raw messages array (full conversation history), systemPrompt text, tools array with schemas, token counts, latency, model/provider details, and internal toolCalls with thought reasoning

    - Internal context state: raw _raw_input, _clarification_count, _memory_initialized_agent, env vars, and the full gatherProgress object

Example of the problem — a single API response currently returns ~15KB of internal data for a simple credit card balance query, including the complete agent system prompt and all LLM call history.

Expected behavior: The API channel response should return only what a channel integration needs:

{
"sessionId": "...",
"response": "...",
"action": { "type": "delegate", "target": "..." },
"metadata": {}
}

Internal fields (state, traceEvents, agent IR, session auth details, LLM call details) should either be excluded entirely or opt-in via a debug/verbose flag — not returned by default.

Impact

    - Security risk: Internal tenant IDs, session principal IDs, full system prompts, and LLM tool schemas are exposed to any API channel consumer

    - Integration friction: Callers cannot attach source context, requiring workarounds that pollute agent state

    - Payload bloat: A minimal interaction returns megabytes of internal data, increasing latency and bandwidth for channel integrations at scale

Proposed Fix

    - Add a metadata pass-through field to the API channel request/response contract

    - Define a minimal channel-facing response schema (sessionId, response, action, metadata) and strip all internal/execution state from the default response

    - Expose trace events and full state only under an explicit opt-in flag (e.g., ?debug=true or a separate debug endpoint)

For Scenario #1:

What Exists Today

POST /api/v1/chat/agent already supports request-side metadata and structured response payloads.

A client can send:

{
"projectId": "project-1",
"sessionId": "optional-session-id",
"message": "Show me available credit cards",
"metadata": {
"correlationId": "abc-123",
"sourceSystem": "mobile-app",
"caseId": "case-789"
},
"interactionContext": {
"locale": "en-US",
"timezone": "America/New_York"
}
}

Runtime already:

    - accepts metadata

    - validates it as bounded JSON-compatible data

    - passes it into execution as messageMetadata

    - exposes it during the turn as session.messageMetadata / message_metadata

    - keeps it separate from the user’s natural-language message

    - cleans/restores the per-turn metadata after execution

So for caller context, clients should use metadata; for locale/language/timezone, clients should use interactionContext.

How Rich Responses Work

The agent/runtime can already return rich output separately from metadata. These are not supposed to go inside metadata or responseMetadata.

A typical response can include:

{
"sessionId": "session-123",
"response": "Here are the cards you qualify for.",
"action": {
"type": "continue"
},
"responseMetadata": {
"isLlmGenerated": true,
"responseProvenance": {
"schemaVersion": 1,
"kind": "llm",
"disclaimerRequired": false,
"usedLlmInternally": true
}
},
"richContent": {
"cards": [
{
"title": "Platinum Rewards",
"subtitle": "Best for travel",
"description": "3x points on travel purchases"
}
]
},
"actions": {
"elements": [
{
"type": "button",
"id": "apply_platinum",
"label": "Apply"
}
]
},
"contentEnvelope": {
"response": "Here are the cards you qualify for.",
"richContent": {},
"actions": {}
}
}

Functionally:

    - response is the plain text fallback / primary text.

    - richContent carries renderable structured UI payloads.

    - actions carries interactive elements like buttons, choices, forms, etc.

    - contentEnvelope is the normalized combined response envelope.

    - responseMetadata is metadata about the response, such as provenance, LLM usage, disclaimer requirements, and potentially safe correlation fields.

So there is no need to overload metadata for rich UI payloads.

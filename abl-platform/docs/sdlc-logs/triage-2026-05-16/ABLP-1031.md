# ABLP-1031 — HANDOFF vs DELEGATE syntax inconsistency

## Symptom

HANDOFF and DELEGATE are semantically adjacent (both route work to another agent) but use incompatible DSL syntax:

| Concern         | HANDOFF               | DELEGATE          |
| --------------- | --------------------- | ----------------- |
| Entry key       | `- TO:`               | `- AGENT:`        |
| Input data      | `CONTEXT.pass: [...]` | `INPUT: { k: v }` |
| Context summary | `CONTEXT.summary:`    | `PURPOSE:`        |
| History         | `CONTEXT.history:`    | unsupported       |
| Return mapping  | `ON_RETURN.MAP:`      | `RETURNS: {}`     |

A developer cannot learn one syntax and write the other. Migration between the two requires a near-complete rewrite of the block.

## Root Cause

**Parser asymmetry:**

- HANDOFF entry guard (`packages/core/src/parser/agent-based-parser.ts:4450`): matches `- TO: <name>`
  ```ts
  const match = line.match(/^-\s*TO:\s*(\w+)$/); // line 4489
  ```
- DELEGATE entry guard (same file, line 4311): matches `- AGENT: <name>`
  ```ts
  const match = line.match(/^-\s*AGENT:\s*(\w+)$/); // line 4350
  ```

Using `TO:` inside DELEGATE triggers a warning at line 4327-4331 ("Invalid DELEGATE entry ... Expected `- AGENT: AgentName`") and produces an empty delegates array.

**IR asymmetry:**

- `packages/compiler/src/platform/ir/schema.ts:1657` — `DelegateConfig` uses `agent: string`
- `packages/compiler/src/platform/ir/schema.ts:1708` — `HandoffConfig` uses `to: string`
- Different sub-key structures: `input/returns/use_result/purpose` vs `context.pass/context.summary/on_return`

**AST asymmetry:**

- `packages/core/src/types/agent-based.ts:1148` — `DelegateConfig { agent, input, returns, purpose, useResult }`
- `packages/core/src/types/agent-based.ts:1220` — `HandoffConfig { to, context: { pass, summary, history, memoryGrants }, return, onReturn }`

Both produce system tools (`delegate_to_X`, `handoff_to_X`) in the IR compiler at lines 1159-1195 of `packages/compiler/src/platform/ir/compiler.ts`.

## Reproduction Test

See `packages/core/src/__tests__/handoff-delegate-symmetry.repro.test.ts`

The test:

1. Parses DELEGATE with `- TO: BillingAgent` (unified syntax) and asserts it produces a valid delegate entry.
2. Parses HANDOFF with `- TO: SupportAgent PASS: [userId]` and asserts the parser accepts `PASS:` as an alias for `CONTEXT.pass`.
3. Both currently fail because the parser rejects `TO:` in DELEGATE and does not recognize `PASS:` in HANDOFF.

## Future-Ready Solution

### Design: Shared AgentRoutingConstruct Base Type

Introduce a discriminated union base that both HANDOFF and DELEGATE derive from:

```ts
// packages/core/src/types/agent-based.ts

interface AgentRoutingBase {
  to: string; // unified: target agent name
  when: string; // condition
  pass?: string[] | Record<string, string>; // list (handoff) or map (delegate)
  summary?: string; // replaces CONTEXT.summary / PURPOSE
  timeout?: string;
  onFailure?: string | FailureAction;
  remote?: RemoteAgentConfig;
}

interface HandoffRoutingConfig extends AgentRoutingBase {
  mode: 'transfer';
  return: boolean;
  history?: HandoffHistoryConfig;
  memoryGrants?: MemoryGrant[];
  onReturn?: HandoffOnReturnAST;
  async?: boolean;
  asyncTimeout?: number;
}

interface DelegateRoutingConfig extends AgentRoutingBase {
  mode: 'delegate';
  returns?: Record<string, string>;
  useResult?: string;
}

type AgentRoutingConfig = HandoffRoutingConfig | DelegateRoutingConfig;
```

### Parser Changes

1. **Unified entry key:** Accept both `- TO:` and `- AGENT:` for DELEGATE. `- AGENT:` becomes a deprecated alias (emit warning).
2. **Unified data-passing key:** Accept `PASS:` in both sections. For HANDOFF, `PASS: [a, b]` maps to `context.pass`. For DELEGATE, `PASS: { k: v }` maps to `input`.
3. **Unified summary key:** Accept `SUMMARY:` in DELEGATE as alias for `PURPOSE:`.
4. Legacy keys (`AGENT:`, `PURPOSE:`, `CONTEXT:`, `INPUT:`, `RETURNS:`, `USE_RESULT:`) remain accepted during a deprecation window (1 major version).

### IR Compilation Changes

The IR compiler already reads `doc.delegate[].agent` and `doc.handoff[].to`. No IR schema change needed for Phase 1 — the parser normalizes unified syntax into the existing AST shapes:

- `- TO: X` in DELEGATE section sets `config.agent = X`
- `PASS: {...}` in DELEGATE section sets `config.input = {...}`
- `SUMMARY:` in DELEGATE section sets `config.purpose = ...`

### Backward Compatibility Strategy

| Phase | Action                                                                                     | Breaking?   |
| ----- | ------------------------------------------------------------------------------------------ | ----------- |
| 1     | Parser accepts both old and new keywords. New keywords preferred in docs/examples.         | No          |
| 2     | Emit deprecation warnings for legacy keywords (`AGENT:`, `PURPOSE:`, `CONTEXT:`, `INPUT:`) | No          |
| 3     | Remove deprecated keywords (next major version)                                            | Yes (major) |

### Benefits

1. Learn one syntax, write both constructs.
2. Migrating HANDOFF to DELEGATE (or vice versa) is a keyword swap plus adding `RETURNS:`.
3. Validator can detect "HANDOFF keyword used inside DELEGATE" early because both share the same key namespace.
4. Documentation halves: one syntax reference with a `mode` toggle table.

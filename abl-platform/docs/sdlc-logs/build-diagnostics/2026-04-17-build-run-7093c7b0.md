### Why is it "Auto-fixing…" when the YAML looks correct?

**Because the 18 errors are NOT HANDOFF target errors — they're semantic diagnostics.**

The pipeline runs two separate validation layers:

1. **Parse + compile to IR** (`compileABLtoIR`). This succeeded on attempt 3 — `compile_abl resultStatus=pass errorCount=0 warningCount=5`.
2. **Semantic diagnostics** (`runDiagnostics`). This found 18 "actionable" findings at error severity.

When `treatDiagnosticErrorsAsBlocking` is set (it is, for supervisor agents), diagnostic errors route into the fix-loop even though the compiler is happy. The fix-loop LLM tries to resolve them, usually makes marginal progress (18 → 1 or 18 → 2), but can't fully clear.

**The specific diagnostic codes producing the 18 errors were not logged in a structured way** — we only see the aggregate counts (`errorCount=18 actionableCount=10`). Based on the YAML content, likely candidates:

- **H-class HANDOFF CONTEXT rules** — each of 6 HANDOFF rules lacks a `CONTEXT.pass` block. If there's a rule that requires PASS declarations for delegates with RETURN:true, that could be 5 × 1 = 5 findings.
- **Ghost MEMORY fields** — 4 session vars declared (`extracted_order_number`, `extracted_email`, `extracted_product_name`, `confidence_level`) that are never referenced in any HANDOFF, COMPLETE, or FLOW. Each unused field → 1 finding. That's 4 findings (G-09 pattern).
- **Persistent memory namespace** — `persistent: path: user.last_detected_intent` with `scope: user`. If the project's memory config does not declare a `user` namespace, that's 1 finding for the persistent declaration + 1 for the REMEMBER target = 2 findings.
- **Other CONSTRAINTS/LIMITATIONS/TOOLS findings** — the MEMORY section has fields for order number, email, product name, confidence_level which feel like they should drive CONSTRAINTS or tool calls — but don't. Each could be flagged.

Plausible aggregate: 5 (HANDOFF CONTEXT) + 4 (ghost GATHER-like session vars) + 2 (persistent namespace) + 7 (miscellaneous) ≈ 18. Confirmation requires logging `topFindings` list verbatim.

## What the interim patches accomplished

- **Eliminated the structural HANDOFF failure mode**. No more self-referencing catch-alls. No more missing specialists. Triage YAML is now always topology-correct.
- **Forced failures to surface via retry** instead of silently shipping broken supervisors. The user now sees "Auto-fixing..." on the UI across retries, which is more honest than a false `compiled` status hiding a broken HANDOFF.
- **Reduced the Triage final status from error → warning** — the same observable build is no longer blocking.

## What the interim patches do NOT fix

- **The initial generation still produces 18 semantic errors** on every attempt. The structural HANDOFF is right, but the rest of the YAML has issues that semantic diagnostics catch. These are a separate class of failures from HANDOFF_SELF_REFERENCE.
- **Three retry attempts × three fix rounds each = 9 LLM calls** for one supervisor. That's ~37 seconds for the Triage worker (up from ~16 seconds in 3dd30d8e). Cost and latency regressed in exchange for correctness.
- **Prompt size grew to 16K chars on retry** (vs 13.5K on first attempt) because the retry feedback mechanism appends prior errors. Larger prompts make Haiku's attention drift worse.

## Recommendations (prioritized)

1. **Log `topFindings` list at INFO level for every compile_abl diagnostic run.** Without the raw `[CODE] message` strings we cannot tell which rules produce the 18 Triage errors. This was recommendation #6 in 3dd30d8e's report; it remains unactioned and is now blocking further diagnosis.

2. **Upgrade supervisor agents to `claude-sonnet-4-6` in `getModelRecommendation`.** The Triage fails reproducibly across three distinct runs on Haiku (1228a16b, 3dd30d8e, 7093c7b0). Three failure modes — self-reference, HANDOFF deletion, 18 semantic errors — share one common factor: small-model attention degradation on 13–16K char prompts. Sonnet at 200K context with better attention would eliminate most of this. Cost delta is negligible (1 supervisor per build, ~3K tokens output). **This is the single change with the highest expected lift until scaffold+fill lands.**

3. **Compress the retry-feedback section.** When attempt 2/3 runs, the prompt grows to 16K chars because prior errors are appended verbatim. Summarize the retry feedback into ≤10 lines of "the thing to fix" rather than the full error list.

4. **Accelerate scaffold+fill MVP to just HANDOFF + MEMORY.** If the scaffold owns HANDOFF (already proven safe) and MEMORY (the source of many semantic warnings like "ghost fields"), two archetypes' worth of diagnostic errors disappear. This is a smaller slice than the full design proposed in `docs/superpowers/specs/2026-04-17-arch-ai-build-scaffold-fill-design.md` — ship it in 2–3 days instead of 6.

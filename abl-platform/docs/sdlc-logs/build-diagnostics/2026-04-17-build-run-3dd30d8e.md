### Why the fix-loop destroyed the handoffs

`apps/studio/src/lib/arch-ai/helpers/compile-and-fix.ts:201`

```ts
system: `You are an expert ABL developer. Fix the compilation errors in the ABL code below.
Preserve all valid constructs and only change what the compiler feedback requires.
…
Output ONLY the fixed ABL code, no explanations, no markdown fences.

${constructContext}`,
```

The system prompt tells the LLM to "only change what the compiler feedback
requires." Given 12 HANDOFF errors (self-reference + target-not-found), the
cheapest repair for Haiku is **delete the offending lines** — it does not know
that HANDOFF rules are mandatory nor that the topology specifies required
targets. The LLM is **unaware of the topology**: the fix prompt contains only
the failing ABL + error strings + construct syntax reference.

## Patterns across the 5 specialists

### Pattern A — uniform "5 errors" on first compile (OrderSupport, ReturnsRefunds, ProductSupport, ShippingSupport, AccountSupport)

Every specialist reports **exactly 5 errors** on first compile before any
retries. Error count uniformity across 5 independently-generated agents is a
strong signal of a systematic prompt issue, not agent-specific randomness.
Likely candidates (from construct-level examination in the 1228a16b doc):

- `CONSTRAINTS: REQUIRE user_authenticated == true` — 4/6 agents still use
  this undeclared variable (pattern #2 in 1228a16b); G-09 / C-07 validators
  now run but emit as warnings not errors, so there must be another repeat
  error. Need the raw `[CODE] message` list.
- Missing `available_agents` on a specialist that was mis-labelled SUPERVISOR.
- `persistent` memory paths referencing namespaces not declared in project
  memory config.

### Pattern B — retries grow findings (ProductSupport: 10 → 20; ShippingSupport: 10 → 9)

ProductSupport's fix-loop round 2 had **more findings than round 1** (20 vs 10).
This is the same "LLM fix introduces new issues" class as the Triage deletion
issue, just less catastrophic.

### Pattern C — HumanSupport is the only agent that compiles clean from first pass

HumanSupportAgent has 0 outgoing edges in topology → planner emits
`### HANDOFF: None`. With no HANDOFF to get wrong, it passes immediately.
This is consistent: all agents that got HANDOFF wrong first time had
outgoing edges; the one that didn't, didn't.

Commits `8f271fd2f` / `909e8f37c` / `d502bbb4a` shipped the same morning, and
the `.next/BUILD_ID` at 12:03 confirms the bundle this run used includes all
three:

1. **Restored `## Entry-Point Routing — CRITICAL`** — Haiku still generated
   self-references on first pass. The restored section reduces but does not
   eliminate small-model self-reference failure on 13.5K-char prompts.
2. **Plan HANDOFF emits ABL YAML block with `WHEN:`** — format is now correct
   but targets are still wrong.
3. **`enrichedEdges` passes through conditions** — topology context section
   now has WHEN text, but that section is below the plan and CRITICAL sections

The fixes are **necessary but not sufficient**: they clean up the prompt
the initial generator sees, but the **fix-loop** is unchanged and actively
destructive once the initial generation goes wrong.

## Recommendations (priority order)

1. **Fix-loop guardrail (single biggest lever).** In `compile-and-fix.ts:201`:
   - Add to the system prompt: _"You MUST NOT remove any HANDOFF rule. If a
     HANDOFF TO: field references an invalid or self agent, correct it using
     the topology agent list below — never delete. The agent you are fixing
     MUST have a HANDOFF rule for EVERY topology target."_
   - Inject the list of valid topology target names (with conditions) into
     the prompt. The fix LLM currently sees only the broken ABL + error
     strings + construct syntax, and has no way to know what the correct
     TO: target names are.
2. **Make HANDOFF_SELF_REFERENCE blocking at the per-agent level.** The rule
   is semantic and doesn't require other agents to evaluate. Moving it out
   of the `skipCrossAgentValidation: true` path forces the fix-loop to
   actually repair it, or fail noisily, instead of hiding it until
   reconciliation flips the status 80s later.
3. **Structural post-fix check in the worker.** After `compileAndFixWithModel`
   returns success, count HANDOFF rules in the final ABL; if the agent is
   entry/supervisor and `count < plan.handoffs.targets.length`, reject the
   result and retry `generate_agent`. Do not let the worker mark an agent
   "compiled" when it's missing required topology handoffs.
4. **Use Sonnet for supervisor agents in `getModelRecommendation`.** Entry
   agents are the single highest-leverage node: 1-of-N in every build,
   and every specialist's functionality depends on Triage routing correctly.
   Haiku's self-reference failure mode has repeated across three documented
   runs (1228a16b, e490af7e-4a0, 3dd30d8e-93f). Promote supervisors to
   Sonnet or enable extended thinking; the cost is negligible vs. loss of
   a successful build.
5. **Compiler-level keyword enforcement.** When a file declares `SUPERVISOR:`
   but has 0 outgoing handoffs in the topology (or vice-versa), emit a
   blocking parse-level error. This stops the `SUPERVISOR: OrderSupport`
   class of regression from shipping silently.
6. **Log topFindings at INFO level for every per-agent compile.** Currently
   the log records `findingCount=19 errorCount=18 warningCount=0
actionableCount=10` but not the `[CODE] message` strings. Reconstructing
   diagnostic content from the DB post-hoc is possible; logging it inline
   makes live debugging 10× faster and would have shortened this
   investigation from 30 min to 2 min.

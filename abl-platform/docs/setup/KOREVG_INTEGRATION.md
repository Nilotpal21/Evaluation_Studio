# KoreVG Integration

## Overview

KoreVG is a voice channel integration that enables ABL Platform agents to handle phone calls via
WebSocket connections. The integration provides real-time bidirectional communication between
KoreVG (Jambonz) and ABL Platform runtime.

## Architecture

### Full Call Flow

```
Caller
  │
  ▼ dials Twilio number
Twilio
  │ SIP trunk (TWILIO_TRUNK_SID)
  ▼
Jambonz VoIP Carrier (JAMBONZ_VOIP_CARRIER_SID)
  │ dispatches to matching Jambonz Application
  ▼
Jambonz Application
  │ WebSocket: wss://runtime/ws/korevg/{connectionId}?token={inboundAuthToken}
  ▼
KorevgRouter (ABL Runtime)
  │ resolves ChannelConnection → DeploymentResolver → RuntimeExecutor
  ▼
Agent → LLM → Response Verbs → KoreVG → Caller (TTS audio)
```

**Key Characteristics:**

- **Connection-scoped routing**: The `connectionId` in the WebSocket URL is the MongoDB `_id` of
  the `ChannelConnection`. Jambonz embeds it at provisioning time so inbound calls map directly
  to the right project/agent without a phone-number lookup table.
- **Immediate Answer**: Responds within ~1ms to prevent Jambonz timeout during async setup.
- **Event-Driven**: Message-driven architecture with event handlers for session lifecycle.

### WebSocket URL Pattern

```
wss://<domain>/ws/korevg/<connectionId>?token=<inboundAuthToken>
```

**Components:**

- `connectionId`: The MongoDB `_id` of the `ChannelConnection` record. Set automatically by
  provisioning when the Jambonz application is created.
- `token`: 32-byte random hex token, auto-generated at connection creation time, stored in
  `ChannelConnection.config.inboundAuthToken`, and embedded in the Jambonz webhook URL.

## Core Components

### 1. KorevgRouter (`korevg-router.ts`)

**Responsibilities:**

- WebSocket connection handling and upgrade
- Token authentication (constant-time comparison)
- Connection resolution from database by `connectionId`
- Deployment/environment resolution via `DeploymentResolver`
- Session creation and lifecycle management

**Key Features:**

- **Immediate Ack Pattern**: Sends `answer` verb within ~1ms, then sends greeting after async setup completes
- **3-Tier Resolution**: Deployment ID → Environment → Working Copy fallback
- **No Message Buffering**: Early handler only acks `session:new`, no buffering needed

### 2. KorevgSession (`korevg-session.ts`)

**Responsibilities:**

- KoreVG message protocol handling (`session:new`, `verb:hook`, `ack`)
- Runtime executor integration
- Verb generation (gather, say, config)
- WebSocket command API (`redirect` for post-answer greeting)

**Message Types:**

- `session:new`: Initial call setup
- `verb:hook`: User speech transcript, DTMF input
- `ack`: Response with next verb instructions

### 3. VerbBuilder (`verb-builder.ts`)

**Responsibilities:**

- Generates KoreVG verb JSON structures
- Config verb (TTS/STT vendor configuration)
- Gather verb (speech collection with actionHook)
- Say verb (text-to-speech output)

## Authentication Flow

```
1. KoreVG connects: wss://domain/ws/korevg/{connectionId}?token={inboundAuthToken}
2. Runtime extracts token from ?token= query parameter
3. Lookup ChannelConnection by connectionId (_id) via resolveConnectionById()
   (falls back to resolveChannelConnection by externalIdentifier for legacy manual configs)
4. Validate token matches connection.config.inboundAuthToken using timingSafeEqual
5. If valid → proceed, else → close with 1008 (Unauthorized)
```

**Security:**

- Timing-safe token comparison (`tokensMatch`)
- Token stored encrypted in database
- Per-connection tokens (no shared secrets)

## Message Flow

### Call Initiation

```
1. KoreVG → session:new (msgid: "abc123", call metadata)
2. Runtime → [Immediate Ack] { type: 'ack', msgid: 'abc123', data: [{ verb: 'answer' }] }
   ↳ Answered within ~1ms
3. Runtime → [Async Setup] Connection resolution + deployment resolution (~30-52ms)
4. Runtime → [Send Greeting] Command API: { type: 'command', command: 'redirect', data: [verbs] }
   ↳ Config verb + Gather verb with greeting
```

### Conversation Turn

```
1. KoreVG → verb:hook (msgid: "xyz", speech transcript, speech.is_final: true)
2. Runtime → Process message through agent executor
3. Runtime → Generate response (LLM call)
4. Runtime → { type: 'ack', msgid: 'xyz', data: [gather verb with response] }
5. Repeat for each turn
```

### Call End

```
1. User hangs up → KoreVG closes WebSocket
2. Runtime → Cleanup session, remove from sessions Map
```

## Configuration

### ChannelConnection Record

A fully provisioned voice channel connection looks like this in MongoDB:

```json
{
  "channelType": "voice_pipeline",
  "externalIdentifier": "pipeline-voice-prod",
  "projectId": "019c603c-ff4c-7642-8a85-ca3bfea02114",
  "environment": "production",
  "config": {
    "inboundAuthToken": "<auto-generated 32-byte hex, set at creation>",
    "jambonzApplicationSid": "<set at creation by Jambonz provisioning>",
    "jambonzPhoneNumberSid": "<set at creation if JAMBONZ_VOIP_CARRIER_SID configured>",
    "phoneNumber": "+14155551234",
    "phoneNumberSid": "PNxxxxxxxx",
    "asrVendor": "deepgram",
    "asrLanguage": "en-US",
    "ttsVendor": "elevenlabs",
    "ttsVoice": "EXAVITQu4vr4xnSDxMaL"
  }
}
```

**Key config fields:**

| Field                   | Set by                     | Purpose                                              |
| ----------------------- | -------------------------- | ---------------------------------------------------- |
| `inboundAuthToken`      | Auto-generated at CREATE   | Authenticates Jambonz WebSocket connections          |
| `jambonzApplicationSid` | Jambonz provisioning       | Used to update/delete the Jambonz application        |
| `jambonzPhoneNumberSid` | Jambonz provisioning       | Used to unassign the number from Jambonz on deletion |
| `phoneNumber`           | Studio UI (phone purchase) | The Twilio E.164 number assigned to this channel     |
| `phoneNumberSid`        | Studio UI (phone purchase) | Twilio SID — used to release the number on deletion  |

### Deployment Resolution

Routing is configured via the `environment` field on the `ChannelConnection` (set in the Studio
Deployment tab or via `PATCH /api/v1/channel-connections/{id}`).

The `KorevgRouter` passes these to `DeploymentResolver.resolve()` with `allowWorkingCopy: true`:

| Strategy                 | Condition                                 | Behaviour                                                   |
| ------------------------ | ----------------------------------------- | ----------------------------------------------------------- |
| **1. By `deploymentId`** | `connection.deploymentId` is set          | Routes to a specific pinned deployment                      |
| **2. By `environment`**  | `connection.environment` is set           | Routes to the latest active deployment for that environment |
| **3. Working copy**      | Neither is set (`allowWorkingCopy: true`) | Compiles agent DSL on the fly — dev/debug only              |

For production use, always set `environment`. Setting `environment` automatically clears
`deploymentId` and vice versa (backend enforced).

## Key Features

### 1. Immediate Answer Pattern

**Problem**: KoreVG sends `session:new` within ~1ms of connecting, but async setup
(connection + deployment resolution) takes 30-52ms.

**Solution**: Register an early handler that sends `answer` verb immediately, then sends the
greeting via the command API (`redirect`) after async setup completes.

**Benefits:**

- ~51ms faster answer time (1ms vs 52ms)
- Prevents KoreVG timeout
- No message buffering needed

### 2. ActionHook for WebSocket

Even in pure WebSocket mode, KoreVG requires `actionHook` in gather verbs to know where to
send `verb:hook` messages:

```json
{
  "verb": "gather",
  "actionHook": "/ws/korevg/{connectionId}",
  "say": { "text": "Hello! How can I help?" }
}
```

Without `actionHook`, KoreVG would attempt to send results to an HTTP endpoint (not supported).

### 3. BargeIn Support

Users can interrupt agent speech mid-sentence. Handled via:

- `bargeIn: true` in gather verb
- Interrupt detection via `verb:hook` with `speech.is_final: false`
- Session state management to track interrupted responses

## Phone Number Lifecycle

### Purchase (Studio UI)

1. User searches available numbers via `GET /api/voice/twilio/available-numbers`
2. User selects and clicks **Buy Selected Number** → `POST /api/voice/twilio/purchase-number`
3. Twilio purchases the number; if `TWILIO_TRUNK_SID` is set, assigns it to the SIP trunk
4. Studio saves `config.phoneNumber` and `config.phoneNumberSid` to the channel connection

### Release (on channel deletion)

On `DELETE /api/v1/channel-connections/{id}`:

1. If `config.phoneNumberSid` → Twilio releases the number (404 = already gone, swallowed)
2. If `config.jambonzPhoneNumberSid` → phone number unassigned from Jambonz
3. If `config.jambonzApplicationSid` → Jambonz application deleted
4. DB record status set to `inactive`

All three cleanup steps are non-fatal — DB is deactivated regardless of cleanup errors.

## Integration Points

### Database

- `ChannelConnection` model: Stores configuration and credentials
- `connection-resolver.ts`: Resolves connection by `_id` (primary) or `externalIdentifier` (fallback)

### Runtime Executor

- `getRuntimeExecutor()`: Main agent execution engine
- `initializeSession()`: Triggers ON_START and returns greeting
- `processMessage()`: Processes user input through agent flow

### Session Service

- `getSessionService()`: Session persistence and state management
- Session ID generation, lookup, and cleanup

---

## Error Handling

| Error                   | Cause                                   | Response   |
| ----------------------- | --------------------------------------- | ---------- |
| Channel not configured  | No ChannelConnection found for streamId | Close 1008 |
| Authentication required | No token in query param                 | Close 1008 |
| Unauthorized request    | Token mismatch                          | Close 1008 |
| Deployment not found    | Invalid deploymentId/environment        | Close 1011 |

## Performance Characteristics

- **Answer Latency**: ~1ms (immediate ack)
- **Greeting Latency**: ~50-80ms (async setup + LLM call)
- **Turn Latency**: ~500-2000ms (STT → LLM → TTS)

Why a Third-Generation Platform

The enterprise AI agent market has moved through three generations. Each one revealed what the next one needed.

Generation 1: Intent-Based Platforms. Rule-driven dialog trees that matched utterances to intents and followed scripted paths. Deployed at Fortune 500 scale. Powered millions of conversations. But they had a hard ceiling: no reasoning loops, no multi-agent coordination, rigid conversation models. When enterprises needed agents that think — not just match intents — the architecture couldn’t support it.

Generation 2: Flow-Based AI Agents & Multi-Agent Applications. LLM-powered agents with flow-based orchestration, tool execution, supervisors, delegation chains, and multi-agent coordination. Proved that AI agents and multi-agent applications work at enterprise scale — handling real workflows, not just Q&A. But these systems were assembled from disparate components: flow builders, prompt chains, custom glue code, and per-deployment orchestration logic. They lacked a unified programmable foundation, compile-time guarantees, deterministic constraint enforcement, and cross-platform agent federation. Building a 10-agent system still required months of custom integration work, and every team solved the same infrastructure problems (session management, context passing, guardrails) independently.

Generation 3: AI-Native Multi-Agent System. A new AI-native version of the multi-agent system, built on two pillars: ABL (Agent Blueprint Language), a compilable YAML-compliant DSL with compile-time validation and deterministic constraint enforcement; and a purpose-built runtime engine that executes agent definitions as first-class distributed workloads — providing session management, multi-agent orchestration, context passing through delegation chains, tool execution with caller identity, and end-to-end tracing as platform infrastructure, not application code. Together, the language and the engine give AI a structured target it can reason about and generate with high accuracy. Not a pivot from what came before — a progression. The intent platform proved the market. The multi-agent system proved the architecture. The third generation makes it programmable, auditable, and AI-native.

The broader market is experiencing the same evolution. No-code agent builders work for simple FAQ bots, but enterprises quickly discover that real agents require threaded conversations, reasoning loops with guardrails, multi-agent coordination, context passing through delegation chains, and compliance enforcement — none of which fits in a drag-and-drop canvas. Pro-code frameworks offer full flexibility, but every team spends 80% of effort rebuilding the same platform infrastructure — session management, guardrails, context passing, tenant isolation — and 20% on the business logic that actually differentiates their product.

The AI-native multi-agent system exists at the third level: a programmable platform with the right primitives at the right level of abstraction. Developers write 50–200 lines of structured, auditable agent definitions. The platform provides the other 15,000 lines of distributed runtime, orchestration engine, and compliance infrastructure. AI can reason about and generate ABL with high accuracy because the domain is well-structured and compile-time validated — giving a single developer working with AI the output of a team of five.

1. Agent Orchestration & the ABL Language

The ABL Language

ABL is a YAML-compliant domain-specific language purpose-built for defining, orchestrating, and governing AI agents. Agent definitions are plain YAML files — editable in any editor, lintable with standard tools, diffable in pull requests, and version-controlled in Git alongside application code.

ABL uses a Core + Extensions architecture that scales with the problem. A simple chatbot uses only the Core — agent identity, tools, and basic responses (~15 constructs). As requirements grow, teams opt into extensions independently:

  • Expressions
  • Gather
  • Constraints
  • Scripted Flows
  • State Management
  • Coordination
  • Rich Content

Teams never encounter complexity they haven’t opted into. A developer building a simple reasoning chatbot with three tools writes as little ABL as a CrewAI YAML definition. A developer building a 10-agent insurance claims system with identity verification, compliance constraints, and external partner federation uses the full language. The same platform serves both.

Compile-time validation catches broken handoff targets, invalid tool references, unreachable flow steps, malformed constraint expressions, and type mismatches before deployment — never in production.

Python and TypeScript Builder SDKs ship in Release 2, enabling programmatic agent authoring in Jupyter notebooks, CI/CD pipelines, and custom orchestration frameworks.

Execution Modes & Multi-Model Strategy

Three execution modes coexist in the same system, the same session management, and the same orchestration primitives:

Reasoning mode — LLM-driven agent loops that plan, execute tools, observe results, and re-plan. Full tool access and flexible decision-making for complex problems like claims assessment, dispute resolution, and open-ended customer support.

Scripted mode — Deterministic state machines with NER-based entity extraction and explicit step transitions. Sub-500ms latency with no LLM cost per turn. Ideal for voice IVR, identity verification, payment processing, and high-volume transactional flows.

Hybrid — A supervisor in reasoning mode routes to a scripted agent. A scripted agent delegates back to a reasoning agent mid-flow. Teams choose the right execution mode per agent and per step — not per system.

Multi-model routing — The platform natively supports using different models optimized for different types of execution. Agents can declaratively specify model tiers — fast (small, low-latency models for simple classification, entity extraction, and routing), balanced (mid-tier models for standard conversations and moderate reasoning), and reasoning (frontier models for complex decision-making, multi-step problem solving, and nuanced judgment). A supervisor can use a fast model for intent classification while the specialist it delegates to uses a reasoning model for complex claims adjudication. Model selection is configured per agent, per step, or per operation in ABL — not hardcoded globally. Fast models handle the 80% of interactions that are straightforward; reasoning models handle the 20% that require depth.

Multi-Agent Orchestration Patterns

ABL provides five first-class coordination primitives that compose into sophisticated multi-agent systems:

Supervisor Routing classifies intent and routes to specialized child agents with priority rules and conditional logic based on runtime context. A default fallback handles unmatched intents. Supervisors maintain a routing table — not code — making routing changes a configuration update, not an engineering sprint.

Delegation invokes a sub-agent synchronously with explicit typed input/output mapping, configurable timeout, and failure handling strategies (continue, escalate, or respond). Agents can fan out to multiple sub-agents in parallel and synthesize their results into a unified response.

Handoff transfers full conversational control to another agent with typed context passing, configurable conversation history strategies (none, summary only, full transcript, or last N messages), memory grants, and an optional return flow back to the originating agent.

Escalation transfers to a human with priority levels, skill-based routing tags, PII-safe context summaries, and on-complete hooks that define what happens when the human resolves (or doesn’t). The full handoff package — transcript, collected information, sentiment, actions attempted, tools called — travels with the escalation.

A2A Federation enables cross-platform agent communication via the Agent-to-Agent protocol. OAuth2 and webhook-based federation allows agents built on ABL to collaborate with agents running on external partner infrastructure.

Real-World Orchestration Scenarios

Insurance claims spanning multiple departments. A customer files a claim. The Claims Supervisor identifies the claim type and routes to the correct adjuster — auto, property, or medical. The adjuster runs scripted identity verification (policy number + date of birth, 3 attempts max), checks fraud scores, and delegates document verification to a specialist sub-agent. For co-insured claims, the system federates with an external partner’s adjuster via A2A protocol. If the claim exceeds $50K, a constraint fires and escalates to a human supervisor — deterministically, not probabilistically. Context flows through the entire chain: identity tier, policy data, claim history, fraud scores. The developer wrote ~60 lines of ABL. The platform handled session management, context serialization, A2A negotiation, constraint enforcement, and the audit trail.

Financial services with identity-gated access. A banking agent handles account inquiries, but different operations require different verification levels. Checking a balance requires basic verification. Initiating a transfer requires full verification via OTP. Changing account settings requires enhanced verification with manager approval. These constraints are enforced by the engine before tool calls reach the LLM — the LLM never sees balance data if the caller hasn’t verified. The trace shows exactly which constraint fired, when, and what was blocked. An auditor can prove that unauthorized data access is impossible, not just unlikely.

Safety-critical deployments with third-party guardrails integration. For industries with strict safety requirements — healthcare, finance, legal — the platform provides native integration with third-party guardrails models and safety services. Input and output guardrails are declared per agent in ABL and evaluated at the engine level, with configurable actions on violation (block, redact, escalate, respond). The platform connects to external guardrail providers — content safety models, PII detection services, compliance checkers, toxicity classifiers — through standard tool bindings, ensuring that safety enforcement is pluggable, auditable, and not dependent on LLM prompt adherence. Guardrail results flow into the trace system, producing a verifiable audit trail of every check and its outcome.

Voice IVR at scale — 500,000 calls/day. A utility company triage supervisor routes calls to specialized agents. Simple flows (balance lookup, outage reports, service scheduling) run through scripted agents with sub-500ms latency and SSML prosody control — no LLM cost per turn. Complex calls (bill disputes, hazardous situations) route to reasoning agents with full tool access. The supervisor uses a fast model for intent classification; the dispute agent uses a reasoning model for complex adjudication. Both modes share the same session infrastructure, the same tracing, and the same escalation paths. One system replaces two separate codebases.

Retail cross-channel continuity. A customer starts a return request on web chat, provides their order number and uploads photos, then leaves. Hours later, they return on WhatsApp from a personal phone. The platform resolves their identity, rehydrates the session on whatever pod handles the WhatsApp message, and picks up exactly where they left off — gathered fields already populated, conversation context preserved. The developer wrote a linear flow. The platform handled identity resolution, session resume, and channel-appropriate formatting.

Parallel fan-out with synthesis. An agent delegates the same request to three specialists simultaneously — pricing, inventory, and shipping. All three execute in parallel, and the parent agent collects their results and presents a unified answer. Timeout and failure handling are configured per delegation.

Structured Information Gathering

The GATHER construct collects structured data with type safety and validation guarantees. Seven field types (string, number, date, boolean, datetime, array, enum) support validation rules including pattern matching, range constraints, enum enforcement, custom business logic, and LLM-based validation. Users can naturally correct previously provided information (“actually 4 guests, not 3”). Cross-field dependencies, multiple extraction strategies (NER, LLM, hybrid), and activation strategies (required, optional, progressive, data-driven) support complex collection workflows — a claim filing agent gathering 15 fields with date range validation, enum constraints, and required-field enforcement runs on ABL out of the box.

Declarative Constraint Enforcement

Constraints are formal behavioral rules evaluated by the deterministic engine — separate from the LLM. The LLM cannot override them. On violation, the engine takes a configured action: respond with a message, escalate to a human, hand off to another agent, block the operation, or redact sensitive content. Constraints are grouped by phase (pre-search, pre-booking, always) and produce a full audit trail showing which constraint fired, when, and why. This is provable to regulators — not probabilistic.

Session Management, Nested Context & Memory

Production multi-agent systems require sophisticated state management that goes far beyond simple key-value storage. Agent Platform V2 provides this as platform infrastructure — not application code.

Distributed session management. Sessions are durable, distributed, and pod-agnostic. A session created on one pod is fully resumable on any other pod — the platform handles rehydration automatically. Sessions survive pod restarts, rolling deployments, and autoscaling events. Concurrent messages to the same session are serialized safely with optimistic concurrency. Session lifecycle includes creation, active conversation, idle timeout, explicit close, and TTL-based expiration — all configurable per project.

Nested context across agent hierarchies. When agents delegate, hand off, or escalate, context passes through the chain as structured, typed data — not serialized blobs. A supervisor routes to a billing agent, which delegates identity verification to a sub-agent, which escalates to a human — and at every level, the context carries identity tier, tenant ID, conversation history, gathered fields, and caller permissions. Return flows bring results back through the chain with the same structure. Configurable history strategies control what travels with each transition: no history, a summary, the full transcript, or the last N messages. Each agent in the chain operates with its own scoped context while maintaining access to the shared session state it needs.

Persistent memory across sessions. Agents can remember information about users across separate conversations. A customer who provided their account number last week doesn’t need to repeat it. Memory is scoped per user and per agent, with configurable TTLs and explicit store/recall triggers declared in ABL. Auto-store rules fire on conditions — “when the customer provides their email, store it for future sessions.” Memory reads are available at session start, so agents can greet returning users with context from previous interactions.

Typed session variables. SET and CLEAR operations work on typed session variables with path navigation and array access. Variables are scoped to the session and available to all agents in the orchestration chain. Computed values use the expression engine — derived fields update automatically when their dependencies change.

Sliding conversation windows. LLM context doesn’t grow unbounded. Configurable sliding windows cap the conversation history sent to the model, keeping recent context while managing token costs. Different window sizes can be set per agent — a triage supervisor needs minimal history, while a dispute resolution agent needs full context.

2. Building, Testing, Debugging & Managing Agents with AI

The most significant capability shift: AI that can reason about systems and generate code is transforming who can build what and how fast. When AI generates into an unbounded space — raw Python, custom frameworks — its power is diluted by the breadth of decisions. When AI generates into a structured DSL with compile-time validation, named orchestration patterns, and a deterministic runtime — its reasoning concentrates on what the agent should do, not how distributed sessions or context serialization should work. The platform answers the how. AI focuses on the what.

Arch — The AI Solution Architect

Arch is an AI that is contextually aware of every agent, handoff, routing rule, tool binding, constraint, and escalation path in your entire project. It operates across six lifecycle stages:

Ideate — Describe what you need in natural language. Arch interviews you with targeted questions, extracts requirements from uploaded documents (PDFs, API specs, existing YAML configurations), and builds a structured project brief.

Design — Arch proposes a complete agent topology: supervisors, specialists, routing rules, escalation paths, and tool bindings. The topology renders as a live interactive graph. Arch is proactive: “There’s no fallback for unmatched intents — want me to add an escalation path?” Changes happen conversationally: “Make billing a reasoning agent” — Arch updates the topology and explains the trade-offs.

Build — Arch generates syntactically valid, compilable ABL for every agent in the topology. Changes appear as inline diffs. The platform validates in real-time — compilation errors surface immediately and Arch fixes them. Suggestion chips offer next actions: “Add error handling”, “Configure escalation”, “Add constraints for PCI compliance.”

Test — Arch generates test personas (Frustrated Customer, Elderly User, Tech-Savvy Power User) and conversation scenarios. It runs automated test suites with LLM-judge evaluators and surfaces coverage gaps: “The billing agent has no cancellation test — want me to create one?”

Deploy — Pre-flight validation checks compilation, constraint coverage, tool endpoint reachability, evaluation coverage, escalation path completeness, and error handlers. Arch provides commentary on each check.

Improve — Post-deployment, Arch reviews production conversation transcripts and trace data to identify optimization opportunities. It surfaces actionable recommendations tied to real outcomes: “Escalation rate for billing is above threshold — here’s a routing change that could reduce it by 15%.” “The verification agent is using a reasoning model for simple identity checks — switching to a fast model would cut latency by 60% and cost by 80% with no quality loss.” Optimizations are proposed as specific ABL changes with before/after diffs, applied only with human approval — a continuous improvement loop driven by production data, not guesswork.

Arch operates in two modes: Assisted (guided wizard where Arch drives the process with explanations — ideal for domain experts and solution architects) and Pro (direct ABL editing with Arch as a sidebar collaborator — ideal for developers who want speed and control).

4. Analytics, Observability & Enterprise

Analytics & Observability

Operational Observability

End-to-end tracing captures every LLM call, tool execution, agent handoff, constraint check, escalation, and error with session identity, agent name, tenant context, timing, and caller information. Traces propagate context across multi-agent handoffs — a conversation that touches five agents produces one coherent trace tree.

Latency tracking at P50/P95/P99 covers end-to-end response time, time to first token per LLM call, tool execution duration, retrieval latency, and handoff queue time.

Token usage and cost attribution tracks input, output, and reasoning tokens per model. Costs attribute to tenant, project, agent, intent, customer segment, and tool. Budget alerts and spend tracking identify the 5% of requests consuming 50% of token budget.

Real-time dashboards display active conversations, error rates (5-minute rolling), response latency, escalation rates, token spend, top intents, and provider status with live streaming updates.

Agent Performance Analytics

Quantitative metrics per agent: invocation count, step execution count, tool success rate, average steps and turns per conversation, containment and escalation rates, error rate, and average cost per invocation.

100% conversation evaluation uses LLM-as-judge to score every conversation on resolution quality, accuracy, helpfulness, safety, and PII handling. A composite CX score provides an AI-generated customer experience rating per conversation — covering 100% of interactions compared to 5–15% survey response rates.

Core customer service metrics: containment rate, deflection rate, first contact resolution, average handle time, escalation rate, drop-off and abandonment, and resolution accuracy.

Safety monitoring covers hallucination detection, guardrail effectiveness (false positive and false negative rates), prompt injection tracking, and regulatory disclosure verification.

DSL-Native Event Emission

A three-pattern event model unifies all analytics signals. Inline emission (via SDK or ABL declarative syntax) lets developers emit custom business events from tool execution — “Payment Completed”, “Policy Renewed” — with typed properties, version-controlled alongside agent definitions. Synchronous guardrail events emit pass/fail signals from real-time safety checks. Async AI evaluation runs LLM-as-judge scoring in the background with zero latency impact. All three patterns flow into the same analytics system via a unified event model with schema governance.

Tiered evaluation criteria support no-code scorecards, low-code natural language criteria, and pro-code custom scorer functions in TypeScript or Python.

Enterprise Features

Security & Tenant Isolation

Tenant isolation is enforced at every data path — database queries, cache keys, analytics storage, and in-memory structures are all tenant-scoped. Cross-tenant access returns 404 to prevent existence leakage.

Authentication converges three flows (user JWT, SDK session tokens, API keys) into a single identity context via shared middleware. No route or service re-implements authentication. SSO supports SAML (SP-initiated and IdP-initiated) and OIDC. Social login covers Google, LinkedIn, and Microsoft. MFA provides TOTP with recovery codes and account lockout. Device Code Flow enables CLI authentication.

Project-scoped RBAC provides five role levels (Owner, Admin, Operator, Member, Viewer) with granular resource-level permissions. Every route enforces authorization — routes that accept resource IDs verify both permission level and project ownership.

SSRF protection blocks private IP ranges and metadata endpoints on all outbound HTTP from tool execution.

Compliance (PCI, GDPR, SOC 2)

Encryption at rest uses tenant-scoped data encryption keys managed through KMS. Database fields containing sensitive data use field-level encryption. Secrets are never stored in plaintext.

Encryption in transit covers all inter-service communication via TLS and requires authentication before any WebSocket data exchange.

Data minimization enforces TTL-based retention on messages, sessions, and traces with configurable windows. Conversation history uses sliding windows to prevent unbounded growth.

Right to erasure cascades deletion across all associated data — sessions, messages, traces, resolution keys, and cached state. No orphaned PII.

Full audit logging captures authentication events, permission changes, data access, and administrative actions with actor identity, timestamp, and action.

Data sovereignty — the platform deploys on your infrastructure. Your Kubernetes cluster, your VPC, your data center, your sovereign cloud. Customer conversations and PII never leave your boundary. Encryption keys are yours.

Scaling

The runtime is stateless and distributed — any request can land on any pod, and sessions rehydrate automatically across pods. There is no pod-local state as a source of truth.

Rate limiting operates at tenant, API key, and session levels with distributed enforcement. Circuit breakers provide automatic recovery from downstream failures with configurable thresholds per tenant. Distributed locking and optimistic concurrency ensure safe concurrent operations across pods.

Roadmap

Self-Learning Agent Optimization

Arch continuously reviews production conversation transcripts and trace data to identify optimization opportunities — and applies them with human approval:

  • Performance optimization — Identifies agents using reasoning models for simple tasks and recommends switching to fast models, with projected latency and cost savings based on actual production data
  • Routing optimization — Detects misrouted conversations and proposes routing rule adjustments
  • Constraint tuning — Identifies constraints that fire too aggressively or too rarely and suggests threshold adjustments
  • Coverage gaps — Flags conversation topics that no agent currently handles, and proposes new agent designs
  • Gather optimization — Detects collection flows where users consistently struggle and recommends improvements

All recommendations are proposed as specific ABL diffs with before/after comparisons and data-backed rationale. Nothing is applied without human review and approval.

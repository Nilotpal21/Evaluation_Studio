## 1. Feature Inventory — What We're Migrating

## 2. What We're Building — Executive Summary

This design replaces XO Platform's rigid callflow engine with ABL's agent-as-flow architecture — contact center capabilities become **tools** within ABL agents, not a separate flow engine with its own editor, execution context, and step state machine.

### Why We're Changing

XO's callflow system has fundamental limitations that ABL's architecture solves:

| XO Legacy Problem                                                                                                                      | ABL Solution                                                                                                  |
| -------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| **Separate flow editor** — a visual drag-and-drop canvas distinct from bot builder, requiring its own maintenance and feature parity   | **Agent DSL** — contact center logic lives in the same agent definition as everything else                    |
| **Rigid step-by-step execution** — fixed node graph, no reasoning, every branch must be pre-configured                                 | **LLM-powered reasoning** — agent decides routing based on conversation context, handles edge cases naturally |
| **13 custom node types** with their own execution context (`CallflowExecutionContext`, 1,124 lines), state machine, and serialization  | **Standard tool execution** — uses ABL's existing tool pipeline (`HttpToolExecutor`, `ConnectorToolExecutor`) |
| **`botId`/`streamId`/`iId` identity chaos** — three names for the same entity, string concatenation keys (`userId#botId`)              | **Clean identity model** — `agentId`, `contactId`, `tenantId` with proper `Contact` model                     |
| **15+ Redis key patterns** with inconsistent TTLs, no atomic cleanup, orphan risk                                                      | **1 hash per transfer session** — atomic operations, no orphans                                               |
| **SmartAssist concepts leaked as universal** — `check_business_hours`, `set_queue` presented as generic but only work with SmartAssist | **Provider-aware design** — honest about what's SmartAssist-specific vs provider-agnostic                     |
| **Silent error swallowing** — `.catch(() => Promise.resolve())` throughout, no circuit breakers                                        | **ABL resilience patterns** — circuit breaker, retry, rate limiting built into tool executors                 |
| **Skills accumulated implicitly** from ALL prior steps via backward scan — hidden behavior, no developer visibility                    | **Explicit context** — skills passed as tool parameters, visible in agent definition                          |
| **7+ Redis deletes in `finally()` without atomic guarantee** — session cleanup race conditions                                         | **Redis MULTI/EXEC** — atomic session lifecycle                                                               |

## 3. XO Legacy Architecture (What We're Replacing)

### 3.1 How XO Callflows Work

XO uses a **Coordinator/Worker pattern** with RabbitMQ job queues. The callflow engine runs independently from the bot engine:

```
User Message → Bot Engine → Detects agent transfer intent
  → Callflow Engine (separate process)
    → Step 1: CheckBusinessHoursTask → HTTP to SmartAssist
    → Step 2: CheckAgentAvailabilityTask → HTTP to SmartAssist
    → Step 3: SetQueueTask → HTTP to SmartAssist
    → Step 4: AgentTransferTask → Routes to agent desktop
      → Voice: setWaitStatus().onTaskWait() — call held
      → Chat: setCompleteStatus().onCFComplete() — flow ends
```

Each step has its own `TaskDefinition`, `TaskState`, error handling, and branching logic. The entire context is managed by `CallflowExecutionContext` (1,124 lines) with 60+ methods and manual serialization for crash recovery.

### 3.2 What XO's Contact Center Nodes Actually Do

**Critical finding from the code audit:** The three "contact center" nodes (`check_business_hours`, `check_agent_availability`, `set_queue`) are **SmartAssist-specific HTTP calls**. They do NOT work with third-party agent desktops:

```javascript
// BaseTask.invokeAgentDesktopNodes() — ALL three nodes use this
HTTP POST to: {koreAgentUrl}/api/v1/internal/flows/nodes/{nodeType}
// koreAgentUrl = SmartAssist backend (port 5080)
// Auth: internal API key header
```

When using Genesys, Salesforce, or any third-party, these nodes are **irrelevant** — the provider handles its own queue routing, business hours, and agent availability internally. XO's flow designer presents them as universal concepts, but they only work with Kore's native SmartAssist.

### 3.4 XO's Implicit Behaviors (From Code Audit)

These hidden behaviors MUST be preserved for backward compatibility:

1. **Skills accumulate** from ALL prior step results — not just the current node
2. **automationBotId** resolved by scanning ALL previous steps backward
3. **Channel context in Redis** only stored for Kore agent, never third-party
4. **Language codes remapped** before every SmartAssist call (`pt-pt` → `pt_pt` — only Portuguese)
5. **Voice transfers hold the call** (`setWaitStatus`); chat transfers end the flow (`setCompleteStatus`)
6. **Post-agent dialog** only supported for genesys (feature-flagged, `enablePostAgentConversationDialog`)
7. **Queue deadlock prevention** — must call `onTaskWait()` not `Promise.resolve()` in `agentTransferCompleted()`
8. **handleUserRequest re-entry** — during voice hold, user speaking re-triggers the entire transfer
9. **CSAT dialog selection** — filters by `surveyType` (csat vs nps), falls back to first match
10. **Custom flow data** propagated through transfer with `isAgentTransferredTriggered` flag removed

## 4. ABL Architecture (What We're Building)

### 4.1 Agent-as-Flow

Instead of a separate callflow engine, contact center logic lives in an ABL agent:

```yaml

  TOOLS:
    - transfer_to_agent # Core transfer tool
    - ivr_menu # Voice only — DTMF menu
    - ivr_digit_input # Voice only — digit collection

      Transfer to Kore SmartAssist with queue "billing", skills ["billing", "english"]

    For technical issues:
      Transfer to Genesys with queue "tech-support"
      Genesys handles availability and routing internally
```

**Why this is better than XO:**

- The agent reasons about routing instead of following a rigid graph
- Edge cases (business hours closed, no agents, unknown issue) are handled naturally
- No separate editor to maintain — same DSL as all other agents
- Testing via ABL's eval framework, not a separate test harness

**Why it's backward compatible:**

- Same SmartAssist APIs called (just from a tool instead of a callflow node)
- Same agent desktop integrations (same provider configs, same credentials)
- Same `BTAgentIntegration` MongoDB model consumed
- Deterministic flows still possible — write explicit instructions that constrain the agent

## 5. The `transfer_to_agent` Tool

### 5.1 Design Philosophy

**Provider-aware, not provider-agnostic.** XO pretended all providers work the same way (`check_hours → check_availability → set_queue → transfer`). In reality:

- **Kore SmartAssist**: Pre-checks (hours, availability, queue) are separate API calls to SmartAssist. Transfer goes through SmartAssist which routes to the Kore agent desktop.
- **Genesys**: You pass a `queueName` and `deploymentId`. Genesys handles hours, availability, and routing internally.
- **Salesforce**: You pass a `buttonId`. Salesforce handles routing internally.
- **Zendesk**: Single switchboard handoff call. No queue concept exposed.
- **Zoom/Unblu**: Fire-and-forget direct handoff.

The tool accepts common fields AND provider-specific config. No leaky abstraction.

### 5.2 Tool Interface

```typescript
interface TransferToAgentInput {
  // Required
  provider: string; // 'kore' | 'genesys' | 'salesforce' | ...

  // Common (optional, provider interprets what it can)
  skills?: string[]; // Routing skills
  priority?: number; // 0-10, default 5
  message?: string; // Message to user before transfer
  metadata?: Record<string, any>; // Arbitrary context for agent desktop
  postAgentAction?: 'return' | 'end'; // What happens after agent session

  // Kore SmartAssist specific
  kore?: {
    queueId?: string;
    businessHoursId?: string; // Optional: check before transfer
    checkAvailability?: boolean; // Optional: check before transfer
    waitingExperienceId?: string;
    noAgentsFlowId?: string;
    outOfHoursFlowId?: string;
    inQueueFlowId?: string;
  };

  // Genesys specific
  genesys?: {
    queueName?: string; // Genesys's own queue
    organizationId?: string;
    deploymentId?: string;
    apiType?: 'webChatAPI' | 'webMessagingAPI';
    memberInfo?: object; // Custom member info for agent
  };

  // Salesforce specific
  salesforce?: {
    buttonId?: string;
    deploymentId?: string;
    organizationId?: string;
    prechatDetails?: object[];
  };

  // ServiceNow specific
  servicenow?: {
    queueId?: string;
    apiType?: 'queue' | 'vaapi';
    contextVariables?: Record<string, string>;
  };

  // ... other providers follow same pattern
}

interface TransferToAgentOutput {
  success: boolean;
  status: 'transferred' | 'outside_hours' | 'no_agents' | 'queue_invalid' | 'failed' | 'declined';
  conversationId?: string; // Agent desktop conversation ID
  error?: { code: string; message: string };
}
```

### 5.3 Kore SmartAssist Flow (Internal to Adapter)

When `provider: 'kore'`, the adapter internally performs XO's multi-step flow:

```
1. IF businessHoursId provided:
   POST {smartAssistUrl}/api/v1/internal/flows/nodes/businessHours
   Body: { id: businessHoursId, botId: agentId }
   → If !isValid → return { success: false, status: 'outside_hours' }

2. IF checkAvailability:
   POST {smartAssistUrl}/api/v1/internal/flows/nodes/agentsAvailability
   Body: { orgId, accountId, userId, botId, source, queueId, metaInfo: { agentTransferConfig: { skillsIds, ... } } }
   → If !agentAvailability → return { success: false, status: 'no_agents' }

3. Queue validation:
   POST {smartAssistUrl}/api/v1/internal/flows/nodes/queueAvailability
   Body: { queueId, botId }
   → If !isValid → return { success: false, status: 'queue_invalid' }

4. Transfer:
   POST {smartAssistUrl}/liveAgentUrl/{agentId}
   Body: { full transfer payload matching XO's initChat structure }
   → Return { success: true, conversationId }
```

**Backward compatible:** Same SmartAssist APIs, same payload structure, same auth. SmartAssist doesn't know ABL is calling it instead of XO.

### 5.4 Genesys Flow (Internal to Adapter)

When `provider: 'genesys'`, the adapter goes directly to Genesys:

```
1. Resolve bot variables ({{env.ORG_ID}} → actual values)
2. Select API type (webChatAPI or webMessagingAPI)
3. WebChat: POST /api/v2/webchat/guest/conversations → get JWT + WebSocket URL
   WebMessaging: Connect WebSocket with deploymentId
4. Establish bidirectional message channel
5. Return { success: true, conversationId }
```

No SmartAssist pre-checks — Genesys handles routing internally.

### 5.5 Backward Compatibility: Standalone SmartAssist Tools

For customers migrating from XO who want explicit multi-step control:

```yaml
AGENT explicit_flow:
  TOOLS:
    - smartassist_check_hours # SmartAssist-specific
    - smartassist_check_availability
    - smartassist_set_queue
    - transfer_to_agent

  INSTRUCTIONS: |
    Follow this exact sequence:
    1. Check business hours (hours ID: bh-001)
    2. If outside hours, tell user and end
    3. Check agent availability with skills [billing]
    4. If no agents, offer callback
    5. Set queue to "billing" with priority 3
    6. Transfer to kore agent
```

These are thin wrappers around the SmartAssist HTTP client:


### 7.1 XO Legacy: 15+ Redis Key Patterns

XO spreads one agent transfer across many keys:

```
AgentTransfer:{base64}                    # Session marker
userId#botId                              # Kore agent data
conversationId                            # Lookup by conversation
CSAT#userId#botId                         # CSAT tracking
userId#CSAT#botId                         # CSAT variant
SUBFLOWCSAT#userId#botId                  # Subflow CSAT
CF:AGENT_TRANSFER:session:bot:user:ch     # Callflow state
ATCF:session:bot:user:ch                  # Callflow context
POST_AGENT_DIALOG:session:bot:user:ch     # Post-agent trigger
AST:CHANNELCONTEXT:bot:user:ch            # Channel context
agent:genesys:convId                      # Genesys lookup
genesys[host_convId]                      # Genesys recovery
AgentCSAT:bot:user:ch                     # CSAT expiry
AST:ENFORCESINGLE:token                   # Uniqueness check
... and more
```

Cleanup is 7+ `deleteFromRedis()` calls in a `finally()` block without atomic guarantee. Keys can orphan.

### 7.2 ABL: 1 Hash Per Session

```
agent_transfer:{contactId}:{channel} → HASH {
  state:              'pending' | 'active' | 'post_agent' | 'ended'
  provider:           'kore' | 'genesys' | 'salesforce' | ...
  providerSessionId:  'c-xxxx-...'         // Provider's conversation ID
  agentId:            '...'                 // ABL agent ID
  tenantId:           '...'
  projectId:          '...'
  contactId:          '...'                 // ABL Contact model ID
  channel:            'web' | 'voice' | 'email' | ...
  queue:              'billing'
  skills:             '["billing","english"]'  // JSON array
  priority:           5
  metadata:           '{...}'              // JSON — custom data, agent desktop meta
  postAgentConfig:    '{...}'              // JSON — dialog ID, action
  providerData:       '{...}'              // JSON — provider-specific session state
  createdAt:          1709654400
}
TTL: channel-specific (chat: 30min, email: 4hr, voice: session duration, messaging: 30min)
```

**Lookup indexes** (simple string keys pointing to the session key):

```
agent_transfer_by_provider:{provider}:{providerSessionId} → {contactId}:{channel}
TTL: same as session
```

**Cleanup is atomic:**

```typescript
const multi = redis.multi();
multi.del(`agent_transfer:${contactId}:${channel}`);
multi.del(`agent_transfer_by_provider:${provider}:${providerSessionId}`);
await multi.exec();
```

### 8.2 Voice Transfer Behavior

From the XO audit, voice and chat have fundamentally different behavior at transfer:

| Aspect                  | Voice                                               | Chat                                                   |
| ----------------------- | --------------------------------------------------- | ------------------------------------------------------ |
| At transfer             | Runtime holds the call, agent execution suspends    | Agent execution completes (or suspends for post-agent) |
| User during hold        | Speech/DTMF can re-trigger transfer or be forwarded | N/A                                                    |
| Agent response delivery | TTS via voice gateway (AudioCodes/KoreVG)           | SDK message (WebSDK, WhatsApp, Teams, etc.)            |
| Transfer protocol       | SIP INVITE/REFER/BYE                                | HTTP to agent desktop                                  |
| Recording               | Start/stop/pause via Redis pub/sub                  | N/A                                                    |
| Disconnect              | SIP BYE + CDR recording trigger                     | Session end event                                      |

**SIP Transfer abstraction:**

```typescript
interface SIPTransfer {
  type: 'invite' | 'refer' | 'bye';
  target: string; // Phone number or SIP URI
  headers?: SIPHeader[]; // Custom SIP headers (UUI, X-Vendor, etc.)
  callerId?: string; // Override caller ID
}
```

**Target resolution priority** (from XO audit):

1. Agent context `sipTransferURI` / `sipTransferNumber`
2. Bot session metadata `sipTransferURI`
3. Child bot session metadata `sipTransferURI`
4. Provider config `sipTransferId` / `phoneNumber`

ABL simplifies: tool input > agent context > provider config.

### 8.3 Voice Gateway Protocol Support

Two main gateways (from XO audit):

**AudioCodes:** Activity-based protocol

```typescript
// Activities: transfer, hangup, message, config, playUrl, stopRecording
```

**KoreVG (Jambonz):** Command-based protocol


## 10.8 Performance & Scale

### Concurrency Model

| Component                                          | Concurrency Strategy                                                   | Limit                                                      |
| -------------------------------------------------- | ---------------------------------------------------------------------- | ---------------------------------------------------------- |
| **Transfer Session Store**                         | Redis atomic Lua scripts — no pod-level locks                          | Bounded by Redis throughput (~100K ops/sec)                |
| **SmartAssist Client**                             | `undici` connection pool (shared keep-alive agent)                     | `connections: 50` (configurable via `HTTP_TOOL_POOL_SIZE`) |
| **Adapter Registry**                               | Read-only after boot — no lock contention                              | 15 adapters, loaded once                                   |
| **Circuit Breakers**                               | `HybridCircuitBreakerRegistry` — Redis-backed, falls back to in-memory | Per provider: `breaker:agent_transfer:{provider}`          |
| **Genesys WebSocket**                              | One WebSocket per active conversation, Redis PubSub for multi-pod      | Max concurrent per pod configurable                        |
| **Polling adapters** (Salesforce, Nice, Helpshift) | Per-session polling loop with `setInterval` + `unref()`                | Polling interval configurable (default: 3s)                |
| **Event processing**                               | BullMQ worker for inbound agent events (if high volume)                | Concurrency configurable                                   |

### Connection Pooling

```typescript

  keepAliveMaxTimeout: 60_000,

  pipelining: 1, // No HTTP pipelining (SmartAssist may not support)

const providerAgents = new Map<string, undici.Agent>();
function getProviderAgent(provider: string): undici.Agent {
  if (!providerAgents.has(provider)) {
    providerAgents.set(
      provider,
      new undici.Agent({
        keepAliveTimeout: 30_000,
        connections: 20, // Smaller pool per provider
      }),
    );
  }
  return providerAgents.get(provider)!;
}

// Cleanup on graceful shutdown

  smartAssistAgent.destroy();
  for (const agent of providerAgents.values()) agent.destroy();
});
```

### Horizontal Scaling

- **Stateless pods:** All session state in Redis. Any pod can handle any transfer.
- **Genesys WebSocket affinity:** WebSocket connections are pod-specific, but messages route via Redis PubSub. If a pod dies, the recovery mechanism (from XO's `onLoad` pattern) scans Redis for orphaned sessions and reconnects.
- **Polling loops:** Each pod polls only for sessions it owns (tracked in session hash `ownerPod` field). On pod death, cleanup job reassigns.
- **No pod-local state as truth:** All Maps are caches with Redis as source of truth (ABL invariant).

## 12. Key Design Decisions

### 12.1 Why Provider-Aware Instead of Provider-Agnostic

XO's abstraction leaks. `check_business_hours` doesn't work with Genesys. `set_queue` means different things to SmartAssist vs Salesforce. The "universal" interface only works for one provider.

ABL is honest: the `transfer_to_agent` tool has a `provider` field and provider-specific config sections. This is:

- **Clearer** — developers know what works with what
- **More powerful** — provider-specific features are accessible (Genesys metadata, Salesforce prechat, Zendesk tags)
- **Still composable** — common fields (skills, priority, message) work across providers where applicable

### 12.2 Why 1 Tool Instead of 4

XO's 4-step sequence (`check_hours → check_availability → set_queue → transfer`) is really one operation: "route to human." The pre-checks are Kore-specific. Collapsing them into one tool:

- Eliminates state management between steps (XO needed `CallflowExecutionContext` for this)
- Lets the agent handle failures naturally ("no agents available" → agent decides to offer callback)
- Still allows explicit multi-step for backward compat via standalone SmartAssist tools

### 12.3 Why `contactId` Instead of `userId#botId`

XO uses `userId#botId` as a Redis key — a string concatenation that couples identity to a specific bot. ABL's `Contact` model already provides:

- Proper identity management (email, phone, external ID)
- Tenant-scoped isolation
- Merge/dedup support
- Channel history tracking

Using `contactId` as the session anchor gives us proper identity management for free.

### 12.4 Why No Separate Flow Editor

Building a visual flow editor is a multi-quarter effort. XO's flow designer is tightly coupled to its callflow engine. ABL's agent DSL already supports:

- Sequential tool execution
- Conditional logic (LLM reasoning or deterministic instructions)
- Error handling (agent-level)
- Context passing (agent memory/context)
- Composition (sub-agents)
- Testing (eval framework)

The DSL approach is also more powerful — an agent can reason about edge cases that a fixed graph can't anticipate.

## 13. Security Considerations

Agent transfer is a high-trust boundary — it bridges ABL's internal runtime to 15 external agent desktop providers, passes customer PII, and manages credentials for each provider. This section consolidates all security requirements.

### 13.1 Credential Storage & Secrets Management

**Provider credentials** (API keys, OAuth client secrets, JWT signing keys) for all 15 providers are stored in the `BTAgentIntegration` MongoDB collection.

**Requirements:**

- All provider credentials encrypted at rest using ABL's existing `encryptedCredentials` pattern (AES-256-GCM, per-tenant encryption keys) — same pattern as `ChannelConnection.encryptedCredentials` in `packages/database/src/models/channel-connection.model.ts`
- Credentials decrypted only at point of use inside the adapter, never held in plaintext beyond the scope of a single request
- OAuth tokens cached in-memory (`authTokenCache`) with TTLs matching token expiry — cache entries invalidated immediately on HTTP 401 from provider
- No credentials in log output — `createLogger` redaction patterns extended to cover all provider auth field names (`apiKey`, `clientSecret`, `accessToken`, `jwt`, `bearerToken`, `password`, `appSecret`)

```typescript
// Logger redaction (added to shared logger config)
const REDACT_FIELDS = [
  'apiKey',
  'clientSecret',
  'accessToken',
  'refreshToken',
  'jwt',
  'bearerToken',
  'password',
  'appSecret',
  'signingKey',
  'webhookSecret',
  'verifyToken',
];
```

**Token rotation:**

- OAuth2 tokens: automatic refresh via `AuthProvider.refresh()` — on 401, invalidate cache + re-authenticate
- JWT tokens (Genesys, LivePerson): re-sign on expiry, cache with TTL < token lifetime
- Static API keys (Kore, Intercom, Drift): no rotation mechanism in XO — document that rotation requires config update + adapter restart
- Forced rotation support: `AdapterRegistry.invalidateAuth(provider, tenantId)` clears cached tokens for a specific tenant+provider

### 13.2 SSRF Protection

Provider base URLs are configured per-tenant in `BTAgentIntegration`. A compromised or misconfigured tenant could point to internal services.

**Requirements:**

- All outbound HTTP requests from adapters go through ABL's existing SSRF guard (the same `isAllowedUrl()` check used by `HttpToolExecutor`)
- Block RFC 1918 addresses (`10.x`, `172.16-31.x`, `192.168.x`), link-local (`169.254.x`), loopback (`127.x`), and cloud metadata endpoints (`169.254.169.254`)
- Allow SmartAssist internal URL (`koreAgentUrl`) as an explicit exception — it is an internal service by design
- WebSocket URLs (Genesys) validated with the same SSRF check before connection

```typescript
// In every adapter's HTTP call path
import { assertAllowedUrl } from '@agent-platform/compiler/platform';

function validateProviderUrl(url: string, provider: string): void {
  // SmartAssist is the one allowed internal URL
  if (provider === 'kore') return; // Internal service, configured by platform admin
  assertAllowedUrl(url); // Throws on blocked URLs
}
```

### 13.3 Input Validation & Payload Sanitization

The `transfer_to_agent` tool accepts `metadata: Record<string, any>` and provider-specific config objects from agent DSL execution. These payloads flow to external provider APIs.

**Requirements:**

| Field                            | Validation                                         | Max Size                 |
| -------------------------------- | -------------------------------------------------- | ------------------------ |
| `metadata`                       | JSON-serializable, no functions/symbols            | 16 KB                    |
| `providerData` (session hash)    | JSON-serializable                                  | 32 KB                    |
| `skills[]`                       | Array of strings, alphanumeric + hyphen/underscore | 50 items, 100 chars each |
| `message` (pre-transfer message) | String, no script tags                             | 4 KB                     |
| `kore.queueId`                   | String, UUID or alphanumeric                       | 128 chars                |
| `genesys.memberInfo`             | JSON-serializable                                  | 8 KB                     |
| `salesforce.prechatDetails[]`    | Array of objects                                   | 20 items                 |
| Custom SIP headers (voice)       | Allowlisted header names only                      | See 13.6                 |

```typescript
// Payload size guard (applied before adapter dispatch)
const MAX_METADATA_BYTES = 16_384; // 16 KB
const MAX_SKILLS = 50;

function validateTransferInput(input: TransferToAgentInput): void {
  if (input.metadata) {
    const size = Buffer.byteLength(JSON.stringify(input.metadata));
    if (size > MAX_METADATA_BYTES) {
      throw new ToolValidationError(
        'METADATA_TOO_LARGE',
        `metadata exceeds ${MAX_METADATA_BYTES} bytes`,
      );
    }
  }
  if (input.skills && input.skills.length > MAX_SKILLS) {
    throw new ToolValidationError('TOO_MANY_SKILLS', `max ${MAX_SKILLS} skills`);
  }
  // Provider-specific validation delegated to adapter
}
```

### 13.4 Sensitive Data in Redis

The transfer session hash stores `metadata` and `providerData` which may contain customer PII (name, email, account numbers passed as context to agent desktops).

**Requirements:**

- `metadata` and `providerData` fields encrypted before writing to Redis using ABL's existing field-level encryption pattern (same as `redis-session-store.ts` which encrypts `snapshot` fields)
- Encryption key derived from tenant-scoped key (not a global key)
- TTLs enforced on all session keys — no indefinite PII storage in Redis
- Session cleanup (atomic Lua `DEL`) ensures no PII lingers after transfer ends
- Voice session TTL=0 (session duration) must still have a hard upper bound (configurable, default 8 hours) to prevent indefinite PII retention on abandoned calls

```typescript
// Field-level encryption for sensitive session fields
const ENCRYPTED_SESSION_FIELDS = ['metadata', 'providerData', 'skills'] as const;

async function writeSession(
  redis: Redis,
  key: string,
  session: TransferSession,
  encryptionKey: Buffer,
): Promise<void> {
  const fields: Record<string, string> = { ...session };
  for (const field of ENCRYPTED_SESSION_FIELDS) {
    if (fields[field]) {
      fields[field] = encrypt(fields[field], encryptionKey);
    }
  }
  await redis.hset(key, fields);
}
```

### 13.5 Webhook Verification for Inbound Agent Events

Agent desktops send events back to ABL (agent messages, typing, session closed). These inbound webhooks must be authenticated to prevent spoofing.

**Requirements per provider:**

| Provider                                           | Verification Method                  | Implementation                              |
| -------------------------------------------------- | ------------------------------------ | ------------------------------------------- |
| **Kore SmartAssist**                               | Internal API key header              | Validate `x-api-key` matches configured key |
| **Genesys**                                        | WebSocket auth (JWT at connect time) | Token validated at WebSocket handshake      |
| **ServiceNow**                                     | Webhook secret (HMAC-SHA256)         | Verify `x-hook-signature` header            |
| **Intercom**                                       | Webhook secret (HMAC-SHA256)         | Verify `x-hub-signature` header             |
| **Drift**                                          | Webhook token                        | Verify `token` query parameter              |
| **LivePerson**                                     | JWT in webhook payload               | Validate JWT signature                      |

### 13.6 SIP Header Injection Prevention

Custom SIP headers support template variable resolution (`{{context.field}}` → actual value). An attacker controlling context variables could inject malicious SIP headers.

**Requirements:**

- **Allowlisted header names only** — only these custom headers permitted:
  - `User-to-User` (UUI, hex-encoded)
  - `X-KoreReason`
  - `X-NotifyEvent-URL`
  - `X-CALLID`
  - `X-Vendor`
  - Headers explicitly listed in provider config `allowedSipHeaders[]`
- **Value sanitization** — after template resolution:
  - Strip CR/LF characters (prevent header injection)
  - Max 512 bytes per header value
  - UUI header: validate hex encoding
  - URL headers (`X-NotifyEvent-URL`): validate against SSRF guard

```typescript
const ALLOWED_SIP_HEADERS = new Set([
  'user-to-user',
  'x-korereason',
  'x-notifyevent-url',
  'x-callid',
  'x-vendor',
]);

function sanitizeSipHeaders(headers: SIPHeader[], allowedCustom: string[]): SIPHeader[] {
  const allowed = new Set([...ALLOWED_SIP_HEADERS, ...allowedCustom.map((h) => h.toLowerCase())]);
  return headers
    .filter((h) => allowed.has(h.name.toLowerCase()))
    .map((h) => ({
      name: h.name,
      value: h.value
        .replace(/[\r\n]/g, '') // Prevent header injection
        .slice(0, 512), // Max length
    }));
}
```

### 13.7 Cross-Tenant Adapter Isolation

A single `AdapterRegistry` instance serves all tenants. Credentials and config must never leak across tenants.

**Requirements:**

- Adapter `execute()` receives `tenantId` in every call — credentials resolved per-tenant from `BTAgentIntegration` (queried with `{ _id, tenantId }`, never `findById`)
- `authTokenCache` keyed by `{provider}:{tenantId}` — never by provider alone
- Circuit breaker state keyed by `{provider}:{tenantId}` — one tenant's provider outage does not trip the breaker for another tenant
- Provider connection pools (Genesys WebSocket, polling loops) tracked with tenant scope — `activePollingLoops` keyed by `{tenantId}:{contactId}:{channel}`

### 13.8 Rate Limiting

**Requirements:**

- Per-tenant transfer rate limit: configurable, default **100 transfers/minute** per tenant
- Per-contact concurrency limit: `maxConcurrentPerContact: 1` (already in config schema) — prevents duplicate transfers for same customer
- Per-provider rate limit: respect provider-imposed limits (Genesys: 10 req/sec, Salesforce: varies by edition)
- Rate limit enforced via Redis sliding window counter: `at_ratelimit:{tenantId}` with 60-second window

```typescript
const RATE_LIMIT_WINDOW_SEC = 60;
const DEFAULT_MAX_TRANSFERS_PER_WINDOW = 100;

async function checkTransferRateLimit(
  redis: Redis,
  tenantId: string,
  limit?: number,
): Promise<boolean> {
  const key = `at_ratelimit:${tenantId}`;
  const max = limit ?? DEFAULT_MAX_TRANSFERS_PER_WINDOW;
  const current = await redis.incr(key);
  if (current === 1) {
    await redis.expire(key, RATE_LIMIT_WINDOW_SEC);
  }
  return current <= max;
}
```

### 14.1 CRITICAL — Pod-Crash Session Recovery

**XO pattern:** Genesys `onLoad()` scans Redis hash `genesys`, detects dead pods (missing `SERVER_GENESYS_{hostname}` heartbeat key), atomically reclaims sessions via `HDEL + HSET`, reconnects WebSockets with 3-second delay.

**ABL resolution — `SessionRecoveryService`:**

Every pod writes a heartbeat key on startup and refreshes it on an interval:

```typescript
// Heartbeat key: at_pod:{hostname} with 30-second TTL, refreshed every 10 seconds
const POD_HEARTBEAT_TTL = 30;
const POD_HEARTBEAT_INTERVAL = 10_000;

class SessionRecoveryService {
  private heartbeatTimer: NodeJS.Timer;

  async start(redis: Redis, hostname: string): Promise<void> {
    // Write initial heartbeat
    await redis.set(`at_pod:${hostname}`, Date.now().toString(), 'EX', POD_HEARTBEAT_TTL);

    // Refresh on interval
    this.heartbeatTimer = setInterval(async () => {
      await redis.set(`at_pod:${hostname}`, Date.now().toString(), 'EX', POD_HEARTBEAT_TTL);
    }, POD_HEARTBEAT_INTERVAL);

    // On startup: scan for orphaned sessions
    await this.recoverOrphanedSessions(redis, hostname);
  }

  async recoverOrphanedSessions(redis: Redis, hostname: string): Promise<void> {
    // Scan all active transfer sessions
    let cursor = '0';
    do {
      const [next, keys] = await redis.scan(cursor, 'MATCH', 'agent_transfer:*', 'COUNT', 100);
      cursor = next;

      for (const key of keys) {
        const session = await redis.hgetall(key);
        if (!session.ownerPod) continue;

        // Check if owning pod is alive
        const podAlive = await redis.exists(`at_pod:${session.ownerPod}`);
        if (!podAlive) {
          // Claim session atomically
          const claimed = await redis.eval(CLAIM_SESSION_LUA, 1, key, session.ownerPod, hostname);
          if (claimed) {
            await this.reconnectProvider(session, hostname);
          }
        }
      }
    } while (cursor !== '0');
  }

  private async reconnectProvider(session: TransferSessionData, hostname: string): Promise<void> {
    const adapter = adapterRegistry.get(session.provider);
    if (adapter.recoverSession) {
      await adapter.recoverSession(session, hostname);
    }
  }
}
```

**Lua script for atomic claim** — prevents two pods from claiming the same session:

```lua
-- CLAIM_SESSION_LUA
local key = KEYS[1]
local oldPod = ARGV[1]
local newPod = ARGV[2]
local current = redis.call('HGET', key, 'ownerPod')
if current == oldPod then
  redis.call('HSET', key, 'ownerPod', newPod)
  return 1
end
return 0
```

**Adapter recovery interface** — added to `AgentDesktopAdapter`:

```typescript
interface AgentDesktopAdapter {
  // ... existing methods ...
  recoverSession?(session: TransferSessionData, hostname: string): Promise<void>;
}
```

Genesys adapter implements this: re-establishes WebSocket connection (WebChat or WebMessage API), replays any buffered messages. Other adapters with WebSocket transport implement the same. Polling adapters (Salesforce, Nice, Helpshift) restart their polling loop. Direct handoff adapters (Zendesk, Zoom, Unblu) have no recovery needed.

**Session hash extended:**

```
agent_transfer:{tenantId}:{contactId}:{channel} → HASH {
  ... existing fields ...
  ownerPod:       'pod-abc-123'    // Pod currently managing this session
  lastHeartbeat:  1709654500       // Last activity timestamp
}
```

| Test                                           | What It Validates                 |
| ---------------------------------------------- | --------------------------------- |

### 14.3 CRITICAL — CSAT as Session Continuation

**XO pattern:** On `closed` event with `surveyRequired: "YES"`, XO does NOT close the session. Instead it clears agent state, looks up the CSAT dialog by `refId`, replaces session context, and re-injects into the bot message processing pipeline via `botsQ.startJobFlow("process_incoming_messagev1")`. The session stays alive for the survey.

**ABL resolution — CSAT continuation mode:**

The key insight is that CSAT is not a "post-session" action — it's a **session continuation** where the agent transfer ends but the customer session stays alive for survey interaction.

```typescript
// In the agent event handler, when 'closed' event arrives:
async function handleAgentClosed(session: TransferSession, event: AgentClosedEvent): Promise<void> {
  const csatConfig = session.postAgentConfig ? JSON.parse(session.postAgentConfig) : null;

  if (csatConfig?.surveyRequired && !event.isDropOff && !event.isVoiceMail && !event.forceClose) {
    // Transition session to CSAT mode (not end it)
    await sessionStore.updateState(session.key, 'csat_pending');

    // Resolve CSAT dialog/agent
    const csatAgentId = await resolveCsatAgent(session, csatConfig);

    // Emit event to runtime — runtime will invoke the CSAT agent
    // as a continuation of the current session (same contactId, same channel)
    emitSessionEvent({
      type: 'transfer:csat_start',
      tenantId: session.tenantId,
      contactId: session.contactId,
      channel: session.channel,
      csatAgentId,
      csatConfig,
    });

    // Session TTL extended for CSAT duration
    await sessionStore.extendTTL(session.key, csatConfig.csatTimeoutSec ?? 300);
  } else {
    // Normal session end
    await endSession(session, event.reason);
  }
}

async function resolveCsatAgent(session: TransferSession, config: CsatConfig): Promise<string> {
  // XO pattern: filter by surveyType (csat vs nps), fallback to first match
  const surveys = config.surveys ?? [];
  const match = surveys.find((s) => s.type === config.preferredType) ?? surveys[0];
  if (!match) throw new Error('No CSAT survey configured');
  return match.agentId;
}
```

**Runtime integration:** The `transfer:csat_start` event is handled by the runtime's execution pipeline. It invokes the CSAT agent within the same session context — the customer sees a seamless transition from agent to survey. The CSAT agent is a regular ABL agent with instructions to collect satisfaction data.

**CSAT state in session hash:**

```
state: 'csat_pending' | 'csat_active' | 'csat_completed'
csatAgentId: '...'
csatStartedAt: 1709654500
csatType: 'csat' | 'nps'
```

**Why this is better than XO:** XO uses an Agenda MongoDB scheduler as a fallback cleanup job for CSAT keys (4 separate Redis keys per session). ABL uses a single session hash state transition with TTL. If the customer doesn't respond to the survey, the TTL expires and session cleanup runs — no separate scheduler needed.

### 14.4 CRITICAL — Genesys Presigned URL File Upload

**XO pattern:** 5-step async choreography: (1) WS sends presigned URL request → (2) Genesys responds with `PresignedUrlResponse` containing `attachmentId`, `url`, `headers` → (3) XO fetches file from Kore storage via `axios.get(fileUrl)` → (4) XO uploads to Genesys S3 via `axios.put(presignedUrl, buffer, headers)` → (5) Genesys sends `UploadSuccessEvent` confirming the attachment is ready.

**ABL resolution — `GenesysFileUploadHandler` in Genesys adapter:**

```typescript
class GenesysFileUploadHandler {
  private pendingUploads = new Map<
    string,
    { resolve: Function; reject: Function; timeout: NodeJS.Timeout }
  >();
  private readonly MAX_FILE_SIZE = 25 * 1024 * 1024; // 25 MB (Genesys limit)
  private readonly UPLOAD_TIMEOUT = 30_000; // 30 seconds

  async uploadFile(
    ws: WebSocket,
    fileUrl: string,
    fileName: string,
    fileType: string,
  ): Promise<string> {
    // Step 1: Request presigned URL from Genesys via WebSocket
    const requestId = randomUUID();
    const presignedPromise = new Promise<PresignedUrlResponse>((resolve, reject) => {
      const timeout = setTimeout(() => {
        this.pendingUploads.delete(requestId);
        reject(new Error('Presigned URL request timed out'));
      }, this.UPLOAD_TIMEOUT);
      this.pendingUploads.set(requestId, { resolve, reject, timeout });
    });

    ws.send(
      JSON.stringify({
        action: 'onAttachment',
        token: requestId,
        fileName,
        fileType,
        fileSize: 0, // Size not known until download
      }),
    );

    // Step 2: Wait for PresignedUrlResponse
    const presigned = await presignedPromise;

    // Step 3: Fetch file from internal storage
    const fileBuffer = await this.fetchFile(fileUrl);
    if (fileBuffer.byteLength > this.MAX_FILE_SIZE) {
      throw new Error(`File exceeds ${this.MAX_FILE_SIZE} bytes`);
    }

    // Step 4: Upload to Genesys S3 presigned URL
    await this.uploadToPresigned(presigned.url, fileBuffer, presigned.headers, fileType);

    // Step 5: Wait for UploadSuccessEvent (or return attachmentId)
    return presigned.attachmentId;
  }

  // Called by WebSocket message handler when Genesys responds
  handlePresignedResponse(response: PresignedUrlResponse): void {
    const pending = this.pendingUploads.get(response.token);
    if (pending) {
      clearTimeout(pending.timeout);
      this.pendingUploads.delete(response.token);
      pending.resolve(response);
    }
  }

  handleUploadError(error: GenesysUploadError): void {
    // "File type is missing" → send unsupported file type message to user
    if (error.message?.includes('File type is missing')) {
      // Emit event to notify user
    }
  }
}
```

**Memory safety:** `pendingUploads` Map bounded at 100 entries with FIFO eviction (ABL invariant).

```typescript
interface CcaiEventPayload {
  eventType: string; // 'disposition_submitted' | 'agent_accepted' | 'conversation_transfer' | 'conversation_terminated' | 'agent_joined' | 'agent_exited'
  conversationId: string;
  agentId: string;
  tenantId: string;
  contactId: string;
  channel: string;
  sessionId: string;
  timestamp: number;
  metadata: Record<string, unknown>; // Event-specific data (dispositionSummary, transferSummary, etc.)

class SdkEventEmitter {
  constructor(
    private sdkNotificationQueue: Queue,
    private subscriptionStore: SdkSubscriptionStore,
  ) {}

  async emit(event: CcaiEventPayload): Promise<void> {
    // Look up subscriptions for this agent
    const subscriptions = await this.subscriptionStore.getSubscriptions(
      event.tenantId,
      event.agentId,
    );

    for (const sub of subscriptions) {
      if (this.shouldDeliver(sub, event.eventType)) {
        await this.sdkNotificationQueue.add('sdk:event', {
          subscription: sub,
          event,
          callbackUrl: sub.callbackEnabled ? this.buildCallbackUrl(event) : undefined,
        });
      }
    }
  }

  private shouldDeliver(sub: SdkSubscription, eventType: string): boolean {
    return sub.subscribedFor.includes('onAll') || sub.subscribedFor.includes('onEvent');
  }

  private buildCallbackUrl(event: CcaiEventPayload): string {
    return `${config.baseUrl}/api/v1/sdk/callback/${event.tenantId}/${event.sessionId}`;
  }
}
```

### 14.10 HIGH — Disposition & Wrap-Up Flow

**XO pattern:** `disposition_submitted` event arrives after session close. Uses a deferred Redis key (`AST:CCAIMetaInfo-BotKit-Conversation:{conversationId}`) stored during the `closed` event to retrieve context, then forwards to BotKit via `route_event_to_sdk`.

**ABL resolution — deferred event context + SDK notification:**

```typescript
// On session close (closed event), store context for post-close events
async function handleAgentClosed(session: TransferSession, event: AgentClosedEvent): Promise<void> {
  // Store deferred context for disposition_submitted (which arrives after close)
  await redis.set(
    `at_deferred:${session.provider}:${event.conversationId}`,
    JSON.stringify({
      tenantId: session.tenantId,
      contactId: session.contactId,
      agentId: session.agentId,
      channel: session.channel,
      sessionId: session.providerSessionId,
    }),
    'EX',
    3600, // 1 hour TTL — disposition must arrive within 1 hour of session close
  );

  // ... proceed with session end or CSAT continuation
}

// When disposition_submitted arrives (may be minutes after close)
async function handleDispositionSubmitted(event: DispositionEvent): Promise<void> {
  const deferredKey = `at_deferred:${event.provider}:${event.conversationId}`;
  const contextStr = await redis.get(deferredKey);

  if (!contextStr) {
    logger.warn('Disposition received but no deferred context found', {
      conversationId: event.conversationId,
    });
    return;
  }

  const context = JSON.parse(contextStr);
  await redis.del(deferredKey); // One-time use

  // Emit to SDK notification queue
  await sdkEventEmitter.emit({
    eventType: 'disposition_submitted',
    conversationId: event.conversationId,
    agentId: context.agentId,
    tenantId: context.tenantId,
    contactId: context.contactId,
    channel: context.channel,
    sessionId: context.sessionId,
    timestamp: Date.now(),
    metadata: {
      dispositionSummary: event.dispositionSummary,
      dispositionSets: event.dispositionSets,
    },
  });
}
```

### 14.12 MEDIUM — Session Timeout via Redis Keyspace Notification

**XO pattern:** `psubscribe("__keyspace*__:AgentTransfer:*")` detects key expiry. `AST:ENFORCESINGLE` dedup guard prevents double-processing. On expiry: notifies SmartAssist, cleans up Redis keys.

**ABL resolution — BullMQ delayed job instead of keyspace notifications:**

Redis keyspace notifications are unreliable (can be lost under load, require `notify-keyspace-events` config). ABL uses a deterministic approach:

```typescript
// When creating a transfer session, also enqueue a delayed cleanup job
await agentDesktopEventsQueue.add(
  'session:timeout',
  {
    sessionKey: `agent_transfer:${tenantId}:${contactId}:${channel}`,
    tenantId,
    contactId,
    channel,
    provider,
  },
  {
    delay: ttlMs, // Same as session TTL
    jobId: `timeout:${tenantId}:${contactId}:${channel}`, // Dedup by session key
    removeOnComplete: true,
  },
);

// On session TTL extension (every message), reschedule the timeout
await agentDesktopEventsQueue.remove(`timeout:${tenantId}:${contactId}:${channel}`);
await agentDesktopEventsQueue.add(
  'session:timeout',
  {
    /* same */
  },
  {
    delay: newTtlMs,
    jobId: `timeout:${tenantId}:${contactId}:${channel}`,
  },
);

// On normal session end, remove the timeout job
await agentDesktopEventsQueue.remove(`timeout:${tenantId}:${contactId}:${channel}`);
```

**Timeout handler:**

```typescript
case 'session:timeout':
  // Check if session still exists (may have been ended normally)
  const session = await sessionStore.get(job.data.sessionKey);
  if (!session) return; // Already cleaned up

  // Notify provider
  const adapter = adapterRegistry.get(session.provider);
  await adapter.endSession(session.providerSessionId, 'timeout');

  // Clean up
  await sessionStore.end(job.data.sessionKey);
  break;
```

**Why this is better than XO:** Deterministic delivery (BullMQ guarantees), no dependency on Redis `notify-keyspace-events` configuration, built-in dedup via `jobId`, and the job is automatically rescheduled on TTL extension.

### 14.19 MEDIUM — Genesys Cross-Pod WebSocket Routing

**XO pattern:** Redis pub/sub channel `webMessageApiGenesysAgentUserMessage` routes messages between pods — the pod receiving the user message publishes, the pod holding the Genesys WebSocket subscribes and forwards.

**ABL resolution — Redis pub/sub channel per provider:**

```typescript
// Channel name: at_ws:{provider}
const WS_ROUTING_CHANNEL = (provider: string) => `at_ws:${provider}`;

class WebSocketRouter {
  private subscriber: Redis;
  private publisher: Redis;
  private localConnections = new Map<string, WebSocket>(); // sessionKey → ws

  async start(redis: Redis): Promise<void> {
    this.subscriber = redis.duplicate();
    this.publisher = redis.duplicate();

    await this.subscriber.subscribe(WS_ROUTING_CHANNEL('genesys'));
    this.subscriber.on('message', (channel, data) => {
      const msg = JSON.parse(data);
      const ws = this.localConnections.get(msg.sessionKey);
      if (ws && ws.readyState === WebSocket.OPEN) {
        ws.send(msg.payload);
      }
    });
  }

  // Called when a user message needs to go to a provider WebSocket
  async routeToProvider(sessionKey: string, provider: string, payload: string): Promise<void> {
    // Check if we hold the connection locally first
    const ws = this.localConnections.get(sessionKey);
    if (ws && ws.readyState === WebSocket.OPEN) {
      ws.send(payload);
      return;
    }

    // Route via pub/sub to the pod that holds the connection
    await this.publisher.publish(
      WS_ROUTING_CHANNEL(provider),
      JSON.stringify({
        sessionKey,
        payload,
      }),
    );
  }
}
```

### 15.1 Session Recovery — Leader-Elected Scan (Improves 14.1)

**Problem with current design:** Every pod runs `recoverOrphanedSessions()` on startup, executing `SCAN agent_transfer:*` across all keys. With 10,000 concurrent transfers, every pod restart triggers a full scan. If 5 pods restart simultaneously (rolling deploy), 5 parallel SCANs race for the same orphaned sessions.

**Improvement — leader-elected recovery:**

```typescript
class SessionRecoveryService {
  private readonly LEADER_KEY = 'at_recovery_leader';
  private readonly LEADER_TTL = 60; // seconds
  private recoveryTimer: NodeJS.Timer | null = null;

  async start(redis: Redis, hostname: string): Promise<void> {
    // Heartbeat (unchanged)
    await this.startHeartbeat(redis, hostname);

    // Try to become recovery leader (only one pod scans at a time)
    await this.tryBecomeLeader(redis, hostname);

    // Periodic leader election — if leader dies, another pod takes over
    this.recoveryTimer = setInterval(async () => {
      await this.tryBecomeLeader(redis, hostname);
    }, 30_000); // Check every 30 seconds
    this.recoveryTimer.unref();
  }

  private async tryBecomeLeader(redis: Redis, hostname: string): Promise<void> {
    const acquired = await redis.set(this.LEADER_KEY, hostname, 'NX', 'EX', this.LEADER_TTL);
    if (acquired) {
      await this.recoverOrphanedSessions(redis, hostname);
      // Extend leadership TTL after scan completes
      await redis.expire(this.LEADER_KEY, this.LEADER_TTL);
    }
  }

  async stop(): Promise<void> {
    if (this.recoveryTimer) clearInterval(this.recoveryTimer);
  }
}
```

**Additionally — use a Redis SET index instead of SCAN:**

```typescript
// On session create, add to index set
await redis.sadd('at_active_sessions', sessionKey);

// On session end, remove from index set
await redis.srem('at_active_sessions', sessionKey);

// Recovery scans the set (O(N) on active sessions only, not all keys)
const activeKeys = await redis.smembers('at_active_sessions');
for (const key of activeKeys) {
  const session = await redis.hgetall(key);
  if (!session || !session.ownerPod) {
    await redis.srem('at_active_sessions', key); // Stale entry
    continue;
  }
  // ... check ownerPod heartbeat, claim if dead
}
```

**Why better:** O(active sessions) instead of O(all Redis keys). No race between pods. Leader election via `SET NX` is the standard ABL distributed lock pattern.

### 15.3 BullMQ Worker Lifecycle — Graceful Drain (Improves 14.5)

**Problem:** ABL's current shutdown sequence calls `stopChannelQueues()` which closes Queue objects, but BullMQ workers (`inbound-worker`, `delivery-worker`) are not explicitly drained. The new `agent-desktop-events` and `sdk-notifications` workers need proper lifecycle management.

**Improvement — worker drain before queue close:**

```typescript
// In packages/agent-transfer/src/queues/lifecycle.ts

class AgentTransferQueueManager {
  private agentDesktopWorker: Worker;
  private sdkNotificationWorker: Worker;
  private agentDesktopQueue: Queue;
  private sdkNotificationQueue: Queue;

  async start(redis: Redis): Promise<void> {
    this.agentDesktopQueue = new Queue('agent-desktop-events', { connection: redis });
    this.sdkNotificationQueue = new Queue('sdk-notifications', { connection: redis });

    this.agentDesktopWorker = new Worker('agent-desktop-events', processor, {
      connection: redis,
      concurrency: 10,
      stalledInterval: 30_000, // Check for stalled jobs every 30s
      maxStalledCount: 2, // Mark as failed after 2 stalls
    });

    this.sdkNotificationWorker = new Worker('sdk-notifications', sdkProcessor, {
      connection: redis,
      concurrency: 5,
      stalledInterval: 30_000,
    });

    // Wire stalled job detection
    const queueEvents = new QueueEvents('agent-desktop-events', { connection: redis });
    queueEvents.on('stalled', ({ jobId }) => {
      log.warn('Agent desktop event job stalled', { jobId });
      transferErrors.add(1, { errorType: 'stalled' });
    });
    queueEvents.on('failed', ({ jobId, failedReason }) => {
      log.error('Agent desktop event job failed permanently', { jobId, failedReason });
    });
  }

  async shutdown(): Promise<void> {
    // 1. Stop accepting new jobs (close queues)
    // 2. Drain workers — wait for current jobs to finish (5s timeout)
    // 3. Close workers
    // 4. Close queue event listeners

    log.info('Draining agent transfer workers...');

    await Promise.all([
      this.agentDesktopWorker
        .close()
        .catch((err) => log.warn('Agent desktop worker drain timeout', { error: err.message })),
      this.sdkNotificationWorker
        .close()
        .catch((err) => log.warn('SDK notification worker drain timeout', { error: err.message })),
    ]);

    await Promise.all([this.agentDesktopQueue.close(), this.sdkNotificationQueue.close()]);

    log.info('Agent transfer queues shut down');
  }
}
```

### 15.6 Configuration Hot Reload (New)

**Problem:** ABL has a `ConfigWatcher` in `packages/config/src/watcher.ts` but it's not instantiated in the runtime. Provider configs (credentials, endpoints, circuit breaker thresholds) require pod restart to change. In production, rotating a provider's OAuth secret shouldn't require downtime.

**Improvement — Redis pub/sub config notification + selective reload:**

```typescript
// packages/agent-transfer/src/config/hot-reload.ts

class AgentTransferConfigReloader {
  private subscriber: Redis;
  private readonly CONFIG_CHANNEL = 'at_config_reload';

  async start(redis: Redis): Promise<void> {
    this.subscriber = redis.duplicate();
    await this.subscriber.subscribe(this.CONFIG_CHANNEL);

    this.subscriber.on('message', async (channel, message) => {
      if (channel !== this.CONFIG_CHANNEL) return;

      const event = JSON.parse(message) as ConfigReloadEvent;
      log.info('Config reload event received', { type: event.type, target: event.target });

      switch (event.type) {
        case 'provider_credentials':
          // Invalidate auth cache for specific provider+tenant
          await authTokenCache.invalidate(event.target.provider, event.target.tenantId);
          log.info('Auth cache invalidated', event.target);
          break;

        case 'circuit_breaker':
          // Update circuit breaker thresholds without restart
          await circuitBreakerRegistry.updateConfig(
            breakerKey(event.target.provider, event.target.tenantId),
            event.target.config,
          );
          break;

        case 'session_ttl':
          // Update session TTL defaults (takes effect on new sessions)
          sessionConfig.ttl = { ...sessionConfig.ttl, ...event.target.ttls };
          break;

        case 'provider_enabled':
          // Enable/disable a provider for a tenant
          await adapterRegistry.setEnabled(
            event.target.provider,
            event.target.tenantId,
            event.target.enabled,
          );
          break;
      }
    });
  }

  // Called by admin API when config changes
  static async notifyReload(redis: Redis, event: ConfigReloadEvent): Promise<void> {
    await redis.publish('at_config_reload', JSON.stringify(event));
  }
}

interface ConfigReloadEvent {
  type: 'provider_credentials' | 'circuit_breaker' | 'session_ttl' | 'provider_enabled';
  target: {
    provider?: string;
    tenantId?: string;
    config?: Record<string, unknown>;
    ttls?: Record<string, number>;
    enabled?: boolean;
  };
  timestamp: number;
  triggeredBy: string; // Admin user or system
}
```

### 15.7 Cross-Pod WebSocket Routing — Reliable Delivery (Improves 14.19)

**Problem with current design:** Redis pub/sub is fire-and-forget. If the subscribing pod is momentarily disconnected (GC pause, network blip), messages are silently lost. For agent-to-user messages during an active Genesys session, this means customer-facing message loss.

**Improvement — Redis Streams instead of pub/sub:**

```typescript
// Use Redis Streams for cross-pod WebSocket routing
// Streams provide: persistence, consumer groups, acknowledgment, replay

class ReliableWebSocketRouter {
  private readonly STREAM_KEY = (provider: string) => `at_ws_stream:${provider}`;
  private readonly CONSUMER_GROUP = 'ws-router';
  private readonly MAX_STREAM_LEN = 10_000; // Auto-trim

  async start(redis: Redis, hostname: string): Promise<void> {
    const streamKey = this.STREAM_KEY('genesys');

    // Create consumer group (idempotent)
    try {
      await redis.xgroup('CREATE', streamKey, this.CONSUMER_GROUP, '0', 'MKSTREAM');
    } catch (err: unknown) {
      // BUSYGROUP = group already exists, fine
      if (!(err instanceof Error && err.message.includes('BUSYGROUP'))) throw err;
    }

    // Read loop — blocks waiting for new messages
    this.readLoop(redis, streamKey, hostname);
  }

  private async readLoop(redis: Redis, streamKey: string, consumer: string): Promise<void> {
    while (!this.stopping) {
      try {
        const results = await redis.xreadgroup(
          'GROUP',
          this.CONSUMER_GROUP,
          consumer,
          'COUNT',
          10,
          'BLOCK',
          5000, // 5 second block timeout
          'STREAMS',
          streamKey,
          '>',
        );

        if (!results) continue;

        for (const [, messages] of results) {
          for (const [id, fields] of messages) {
            const sessionKey = fields[1]; // field pairs
            const payload = fields[3];

            const ws = this.localConnections.get(sessionKey);
            if (ws && ws.readyState === WebSocket.OPEN) {
              ws.send(payload);
              await redis.xack(streamKey, this.CONSUMER_GROUP, id);
            } else {
              // Not our connection — nack by not acking (another pod will claim it)
              // Or if truly orphaned, it will be reclaimed by pending entry timeout
            }
          }
        }
      } catch (err) {
        log.error('WebSocket router read error', {
          error: err instanceof Error ? err.message : String(err),
        });
        await new Promise((resolve) => setTimeout(resolve, 1000)); // Backoff on error
      }
    }
  }

  async routeToProvider(sessionKey: string, provider: string, payload: string): Promise<void> {
    // Check local first
    const ws = this.localConnections.get(sessionKey);
    if (ws && ws.readyState === WebSocket.OPEN) {
      ws.send(payload);
      return;
    }

    // Publish to stream — guaranteed delivery
    await redis.xadd(
      this.STREAM_KEY(provider),
      'MAXLEN',
      '~',
      this.MAX_STREAM_LEN.toString(),
      '*',
      'sessionKey',
      sessionKey,
      'payload',
      payload,
    );
  }
}
```

**Why better than pub/sub:** Messages survive subscriber disconnections. Consumer group with acknowledgment ensures exactly-once delivery. `XREADGROUP` with `BLOCK` is as efficient as `SUBSCRIBE`. Pending entry list (`XPENDING`) identifies stuck messages for ops review.

### 15.8 Provider Degradation & Fallback (New)

**Problem:** When a provider's circuit breaker opens, the current design returns a hard error. For enterprise customers, a provider outage shouldn't block all agent transfers — there should be a fallback strategy.

```typescript
async function executeWithFallback(
  payload: TransferPayload,
  primaryProvider: string,
  config: ProviderConfig,
): Promise<TransferResult> {
  // Try primary provider
  try {
    const adapter = adapterRegistry.get(primaryProvider);
    const breakerState = await circuitBreakerRegistry.getState(
      breakerKey(primaryProvider, payload.tenantId),
    );

    if (breakerState !== 'open') {
      return await adapter.execute(payload);
    }

    log.warn('Primary provider circuit breaker open', { provider: primaryProvider });
  } catch (err) {
    log.error('Primary provider failed', {
      provider: primaryProvider,
      error: err instanceof Error ? err.message : String(err),
    });
  }

  // Try fallback chain
  if (!config.fallback?.enabled || !config.fallback.providers.length) {
    return {
      success: false,
      status: 'failed',
      error: {
        code: 'PROVIDER_UNAVAILABLE',
        message: `${primaryProvider} is unavailable and no fallback configured`,
      },
    };
  }

  for (const fallbackProvider of config.fallback.providers) {
    try {
      const fallbackAdapter = adapterRegistry.get(fallbackProvider);
      const breakerState = await circuitBreakerRegistry.getState(
        breakerKey(fallbackProvider, payload.tenantId),
      );

      if (breakerState === 'open') continue; // Skip this fallback too

      if (config.fallback.notifyOnFallback) {
        transferErrors.add(1, { provider: primaryProvider, errorType: 'fallback_triggered' });
        log.warn('Falling back to alternate provider', {
          primary: primaryProvider,
          fallback: fallbackProvider,
          tenantId: payload.tenantId,
        });
      }

      const result = await fallbackAdapter.execute({
        ...payload,
        provider: fallbackProvider,
        metadata: {
          ...payload.metadata,
          _fallbackFrom: primaryProvider,
          _fallbackReason: 'circuit_breaker_open',
        },
      });

      return result;
    } catch {
      continue; // Try next fallback
    }
  }

  return {
    success: false,
    status: 'failed',
    error: {
      code: 'ALL_PROVIDERS_UNAVAILABLE',
      message: 'Primary and all fallback providers are unavailable',
    },
  };
}
```

### 15.11 Canary Deployment & Rollback Strategy (New)

**Problem:** The design covers 21 weeks of implementation but has no deployment strategy. Rolling out agent transfer to all tenants simultaneously is risky — a bug in the Kore adapter could break all live agent sessions.

**Improvement — tenant-scoped feature gates with canary rollout:**

```typescript
// Feature gate in tenant config
interface TenantAgentTransferConfig {
  enabled: boolean; // Master switch
  enabledProviders: string[]; // Which providers are active for this tenant
  trafficPercentage: number; // 0-100: percentage of transfers using ABL vs XO legacy
  rollbackOnErrorRate: number; // Auto-rollback if error rate exceeds this (e.g., 0.05 = 5%)
  enabledChannels: ChannelType[]; // Which channels use ABL agent transfer
}
```

**Canary routing in transfer tool:**

```typescript
async function shouldUseAblTransfer(tenantId: string, channel: ChannelType): Promise<boolean> {
  const config = await tenantConfigStore.getAgentTransferConfig(tenantId);

  if (!config.enabled) return false;
  if (!config.enabledChannels.includes(channel)) return false;

  // Traffic percentage — deterministic hash for consistency
  // Same contact always goes to same path (no mid-conversation switching)
  if (config.trafficPercentage < 100) {
    const hash = murmurhash3(tenantId + channel) % 100;
    if (hash >= config.trafficPercentage) return false;
  }

  // Auto-rollback check — if error rate exceeds threshold
  const recentErrorRate = await getRecentErrorRate(tenantId, 300); // Last 5 minutes
  if (recentErrorRate > config.rollbackOnErrorRate) {
    log.error('Auto-rollback triggered for tenant', {
      tenantId,
      errorRate: recentErrorRate,
      threshold: config.rollbackOnErrorRate,
    });
    transferErrors.add(1, { provider: 'abl', errorType: 'auto_rollback' });
    return false;
  }

  return true;
}
```

**Rollout stages:**

| Stage            | Traffic         | Tenants            | Duration | Gate                             |
| ---------------- | --------------- | ------------------ | -------- | -------------------------------- |
| Internal dogfood | 100%            | Kore internal only | 1 week   | Manual                           |
| Early adopters   | 100%            | 5 opt-in tenants   | 1 week   | Manual                           |
| Canary           | 5%              | All tenants        | 1 week   | Auto-rollback at 2% error rate   |
| Gradual          | 25% → 50% → 75% | All tenants        | 3 weeks  | Auto-rollback at 1% error rate   |
| Full             | 100%            | All tenants        | Ongoing  | Auto-rollback at 0.5% error rate |

### 15.13 Adapter Test Harness — Provider Simulator (New)

**Problem:** Integration tests for third-party adapters use `nock` to mock HTTP responses. This works for request/response patterns but cannot simulate WebSocket-based providers (Genesys), long-polling (Salesforce, NiceInContact), or webhook callbacks. Testing these requires a real provider or a proper simulator.

**Improvement — lightweight provider simulator for CI:**

```typescript
// packages/agent-transfer/src/__tests__/simulators/genesys-simulator.ts

import { WebSocketServer } from 'ws';
import express from 'express';

/**
 * Lightweight Genesys simulator for integration tests.
 * Simulates both WebChat API and WebMessaging API.
 *
 * Capabilities:
 * - JWT auth verification
 * - WebSocket lifecycle (connect, message, close)
 * - Presigned URL flow (request → response → upload confirmation)
 * - Configurable latency and error injection
 */
export class GenesysSimulator {
  private httpServer: ReturnType<typeof express>;
  private wsServer: WebSocketServer;
  private connections = new Map<string, WebSocket>();

  // Test controls
  latencyMs = 0;
  shouldRejectAuth = false;
  shouldFailPresignedUrl = false;
  fileUploadShouldFail = false;
  messagesReceived: Array<{ conversationId: string; text: string }> = [];

  start(port: number): void {
    const app = express();
    app.use(express.json());

    // WebChat API: create conversation
    app.post('/api/v2/webchat/guest/conversations', async (req, res) => {
      if (this.shouldRejectAuth) return res.status(401).json({ message: 'Unauthorized' });
      await delay(this.latencyMs);
      const conversationId = `conv-${randomUUID().slice(0, 8)}`;
      res.json({
        id: conversationId,
        jwt: `sim-jwt-${conversationId}`,
        eventStreamUri: `ws://localhost:${port}/ws/${conversationId}`,
        member: { id: `member-${conversationId}` },
      });
    });

    // WebSocket endpoint
    this.wsServer = new WebSocketServer({ noServer: true });
    this.wsServer.on('connection', (ws, conversationId) => {
      this.connections.set(conversationId, ws);
      ws.on('message', (data) => this.handleWsMessage(conversationId, ws, data));
      ws.on('close', () => this.connections.delete(conversationId));
    });

    const server = app.listen(port);
    server.on('upgrade', (req, socket, head) => {
      const conversationId = req.url?.split('/ws/')[1] ?? 'unknown';
      this.wsServer.handleUpgrade(req, socket, head, (ws) => {
        this.wsServer.emit('connection', ws, conversationId);
      });
    });
  }

  // Simulate agent sending a message to user
  sendAgentMessage(conversationId: string, text: string): void {
    const ws = this.connections.get(conversationId);
    if (ws) {
      ws.send(
        JSON.stringify({
          class: 'ChatMessage',
          body: { body: text, bodyType: 'standard' },
          sender: { id: 'agent-123', displayName: 'Test Agent' },
        }),
      );
    }
  }

  // Simulate agent closing the conversation
  closeConversation(conversationId: string): void {
    const ws = this.connections.get(conversationId);
    if (ws) {
      ws.send(JSON.stringify({ class: 'SessionExpiredEvent' }));
      ws.close(1000);
    }
  }
}
```

**Similar simulators for:** Salesforce (HTTP long-polling), ServiceNow (webhook callbacks), NiceInContact (polling with 60s timeout). These run as in-process servers during integration tests — no external dependencies.

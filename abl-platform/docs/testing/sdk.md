## 1. Current State (as of 2026-04-14)

The SDK surface now has strong proof at the browser package, shared-auth, Runtime transport, Runtime timeout-cancellation hardening, Runtime control-plane/authz, Runtime session-route compatibility, Studio API/integration, and isolated browser layers. We have passing coverage for `sdk/init -> sdk_session -> /ws/sdk`, HMAC-verified identity bootstrap, HTTP route enforcement with `X-SDK-Token`, query-string rejection on `/ws/sdk`, project/session/attachment isolation, expired session-list/refresh rejection, refresh behavior including denial when the bound SDK channel is deactivated or rebound away from the originating public key, public-key revocation effects on future bootstrap/refresh requests, route/integration proof for the current preview/share exchange contract and returned SDK claim shape without independent Runtime-vs-Studio signer-boundary proof, fail-closed `endpoint` handling, `connect()` readiness semantics, stale Runtime scope reset on disconnect/auth-close, retry-safe SDK re-bootstrap on the currently tested config fields, consistent pipeline-voice browser capability detection across the package/widget surfaces, code/wiring proof that SDK/browser-facing JIT OAuth callbacks require an explicit public Runtime base, code/wiring proof that Runtime selects distributed OAuth callback state only when Redis is actually ready outside tests, route-module proof that trace/export/generation routes fail closed when project-scoped authorization cannot be verified, RuntimeExecutor-only session-list fallback preserves SDK and non-elevated-user ownership scoping during DB unavailability, explicit denial of runtime-memory fallback for SDK callers and non-elevated user callers on explicit trace/export/generation lookups when persisted ownership cannot be proven, explicit close/delete cleanup using the stored `runtimeSessionId`, explicit SDK `end_session` cleanup even on the default-detach `web_chat` lifecycle, legacy/compatibility alias handling for records that still differ between database id and `runtimeSessionId`, shared-auth ownership middleware pass-through for elevated users on runtime-only `:id` routes, Studio preview/share validation/authz, real active-channel binding for preview/share bootstrap, stale widget-configured channel fallback to the single remaining active SDK channel for preview/share/embed, fail-closed embed generation when no widget capability remains, aligned Studio SDK control-plane route guards, outsider denial on the full Studio Runtime SDK channel proxy surface across list/create/detail/update/delete routes, explicit legacy route removal, isolated browser widget bootstrap, share-link preview bootstrap plus preview mode-toggle continuity without reconnecting the SDK session, authenticated project preview bootstrap/readiness plus local in-browser message-send visibility, focused timeout abort propagation through coordinator/queue/direct execution, suppression of late timeout side effects, and in-process auth-profile proof that session-scoped SDK callers do not reuse tenant tokens during auth-preflight resolution.

The browser lane proves browser-side bootstrap/readiness and local send visibility only: the current specs assert `session_start`, local user-message rendering, and outbound SDK `chat_message` frames. They do not assert a full assistant/model round-trip or deployment-level request acceptance.

Current Runtime timeout behavior is transport fail-closed and now has focused cancellation-hardening proof: `/ws/sdk` sends `Request timed out`, closes the socket, propagates cooperative cancellation through the execution coordinator, LLM queue, and direct executor paths, and suppresses late response/persistence side effects after timeout. This is still focused in-process proof, not deployment-level or multi-pod operational verification.

**Update 2026-04-14 -- WebSocket relocation and SDK heartbeat changes (ABLP-333)**: The Studio WebSocket provider was relocated from app root to chat-tab level (7 commits). The browser SDK `SessionManager` JSON heartbeat timer was removed; connection liveness is now server-side only. The SDK handler added a legacy ping compat shim (`sendLegacyPong`). The internal `/ws` handler keepalive was added then reverted. New unit tests cover CommandPalette HTTP fetching, SDK handler legacy ping, websocket-events ping rejection, SessionManager heartbeat removal guard, and session launcher post-relocation. All planned WS relocation E2E and integration tests remain unwritten. See [WS Relocation test spec](sub-features/ws-relocation.md).

The repo does not yet have full proof for the target security model described in [docs/features/sdk.md](../features/sdk.md). The biggest remaining gaps are:

- broader verified-identity continuity/resume/auth proof beyond the initial bootstrap
- anonymous OAuth/auth-preflight session-scope proof through public APIs
- browser voice E2E
- deployed ingress/log redaction proof
- multi-pod/failover validation
- named `renderables[]` wire-contract coverage across websocket transport and custom Web SDK renderers
- WS relocation E2E proof (no WS on non-chat pages, WS connects on chat tab, session resume after navigate-away-and-back)
- L7 proxy idle timeout survival for internal Studio `/ws` connections (keepalive was reverted)

This guide intentionally separates proof levels so we do not overclaim:

- unit tests
- integration/API-harness tests
- Runtime transport E2E using real HTTP and WebSocket APIs
- real browser E2E
- deployment-level operational verification

## 2. Proof Levels and Status Legend

### Proof Levels

| Level                   | Meaning                                                                                                                                                                                    |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Unit                    | In-process logic validation for one module or component                                                                                                                                    |
| Integration             | Real route/module/API behavior through a local harness, but not a browser, deployment, or hermetic multi-process environment                                                               |
| Runtime transport E2E   | Real HTTP and WebSocket interaction with running Runtime services, usually from a Node client                                                                                              |
| Browser E2E             | Real browser automation against running Studio/Runtime surfaces; proves browser-visible behavior only, not deployment behavior or assistant-response acceptance unless explicitly asserted |
| Deployment verification | Proof collected from deployed ingress, logging, and multi-pod environments                                                                                                                 |

### Status Legend

| Status                 | Meaning                                                                                       |
| ---------------------- | --------------------------------------------------------------------------------------------- |
| PASS                   | Covered by executed tests and aligned with the current contract                               |
| PARTIAL                | Some proof exists, but the target requirement is broader than what is tested today            |
| FAIL / NOT IMPLEMENTED | The required behavior is not implemented or the current implementation is known to violate it |
| NOT TESTED             | No meaningful proof exists yet                                                                |

## 6. Testing Rules for This Feature

These rules apply to future SDK E2E coverage:

- E2E tests must interact only through public HTTP, WebSocket, or browser surfaces.
- E2E tests must not write to the database directly or import Mongoose models for setup/assertions.
- E2E tests must not mock or replace existing platform components.
- External services may be stubbed only when they are true third-party dependencies and the platform interaction remains real.
- Browser tests should use the repo’s browser automation stack against running Studio/Runtime services instead of route-module shortcuts.
- Docs must describe proof honestly. A route-harness HTTP test is integration proof, not browser E2E.
- Browser SDK specs support an attach-to-existing-services mode via `test:e2e:sdk-browser` and `test:e2e:sdk-preview`.
- `pnpm --filter @agent-platform/studio test:e2e:sdk-browser:isolated` is the canonical non-skipping browser proof lane; it self-starts isolated Studio, Runtime, and MongoMemory services.
- Set `SDK_BROWSER_E2E_STRICT=true` to fail instead of skip when Studio/Runtime prerequisites are missing.
- Optional custom self-start is supported by setting `SDK_BROWSER_E2E_RUNTIME_COMMAND` and `SDK_BROWSER_E2E_STUDIO_COMMAND` before running `test:e2e:sdk-browser`.
- Attach-to-existing-services mode requires a reachable Studio login page (`/auth/login`), a healthy Runtime (`/health` returning `ok` or `healthy`), Studio dev-login enabled, and writable Studio APIs used by the browser bootstrap helpers (`/api/projects`, `/api/sdk/keys`, `/api/runtime/sdk-channels`, `/api/sdk/share`, `/api/sdk/embed`).
- Attach-to-existing-services mode is not hermetic: it creates real users/projects/keys/channels/share artifacts in the target environment and should not be treated as proof of persistent-state cleanup or cross-run order-independence.

## E2E Test Scenarios (7 Scenarios)

### E2E-1: SAML SP-Initiated Login End-to-End

**Preconditions:**

- Organization created with verified domain `corp.example.com`
- SAML SSO config active with test IdP (mock SAML IdP running on random port)
- User `alice@corp.example.com` exists as a tenant member

**Steps:**

1. `GET /api/sso/init?email=alice@corp.example.com` -- expect `{ ssoEnabled: true, protocol: 'saml', redirectUrl: '<idp-url>' }`
2. Simulate IdP redirect: `POST /api/sso/saml/callback` with a valid signed SAML assertion (generated by mock IdP) and `RelayState=<orgId>`
3. Follow redirect to `/auth/callback?code=<auth-code>`
4. `POST /api/sso/exchange` with `{ code: '<auth-code>' }` -- expect `{ accessToken, refreshToken, expiresIn }`
5. `GET /api/auth/me` with `Authorization: Bearer <accessToken>` -- expect user profile with tenantId

**Expected Result:** User receives valid JWT tokens with tenant context. Access token contains `tenantId`, `role`, `orgId`.

**Auth Context:** tenant=org's default tenant, user=alice@corp.example.com

**Isolation Check:** `GET /api/sso/init?email=bob@other.com` returns `{ ssoEnabled: false }` (different domain, no SSO config)

### E2E-2: SAML IdP-Initiated Login

**Preconditions:**

- Organization with SAML config where `idpEntityId` is registered
- No RelayState in the SAML response

**Steps:**

1. `POST /api/sso/saml/callback` with a valid signed SAML assertion containing the IdP entity ID, NO RelayState
2. System resolves organization from SAML issuer via `findOrgBySAMLIssuer`
3. Follow redirect to `/auth/callback?code=<auth-code>`
4. `POST /api/sso/exchange` with `{ code: '<auth-code>' }` -- expect tokens

**Expected Result:** Login succeeds despite no RelayState. Organization resolved from SAML issuer element.

**Auth Context:** tenant=resolved from org, user=email from SAML NameID

**Isolation Check:** SAML assertion with unknown issuer returns 400 "Cannot determine organization"

### E2E-3: OIDC Login with Authorization Code Flow

**Preconditions:**

- Organization created with verified domain `oidc-corp.example.com`
- OIDC SSO config active with test OIDC provider (mock running on random port with `/authorize`, `/token`, `/userinfo` endpoints)

**Steps:**

1. `GET /api/sso/init?email=user@oidc-corp.example.com` -- expect `{ ssoEnabled: true, protocol: 'oidc', redirectUrl: '<mock-idp-authorize-url>&state=<state>' }`
2. Extract `state` parameter from redirect URL
3. Simulate authorization callback: `GET /api/sso/oidc/callback?code=mock-auth-code&state=<state>`
4. Follow redirect to `/auth/callback?code=<auth-code>`
5. `POST /api/sso/exchange` with `{ code: '<auth-code>' }` -- expect `{ accessToken, refreshToken, expiresIn }`
6. Decode JWT and verify `tenantId` and `email` claims

**Expected Result:** OIDC flow completes with token exchange and user provisioning.

**Auth Context:** tenant=org's default tenant, user=user@oidc-corp.example.com

**Isolation Check:** `GET /api/sso/oidc/callback?code=mock-code&state=invalid-state` returns 403 "Invalid or expired state parameter"

### E2E-4: MFA Enforcement for Enterprise Plan

**Preconditions:**

- Organization on ENTERPRISE plan with `requireMfa: true`
- User `admin@enterprise.com` has valid credentials but no MFA enabled

**Steps:**

1. `POST /api/auth/login` with email/password -- expect response with `mfaRequired: true` and `partialToken`
2. `POST /api/auth/mfa/setup` with `Authorization: Bearer <partialToken>` -- expect `{ secret, otpauthUrl, recoveryCodes }`
3. Generate TOTP code from `secret` using RFC 6238
4. `POST /api/auth/mfa/confirm` with `{ code: '<totp-code>' }` -- expect `{ success: true }`
5. `POST /api/auth/mfa/verify` with `{ code: '<totp-code>' }` -- expect full access token
6. `GET /api/projects` with the full access token -- expect 200

**Expected Result:** Enterprise users are forced through MFA enrollment before accessing the platform.

**Auth Context:** tenant=enterprise tenant, user=admin@enterprise.com

**Isolation Check:** Non-enterprise plan users can skip MFA setup.

### E2E-5: One-Time Auth Code Exchange (Valid + Expired)

**Preconditions:**

- Valid auth code generated via SSO callback flow

**Steps:**

1. Complete an SSO flow (SAML or OIDC) to generate an auth code
2. `POST /api/sso/exchange` with `{ code: '<valid-code>' }` -- expect 200 with tokens
3. `POST /api/sso/exchange` with the same code again -- expect 401 or 400 (code already consumed)
4. Wait for auth code TTL to expire (60 seconds)
5. Complete another SSO flow to get a new auth code
6. Wait > 60 seconds
7. `POST /api/sso/exchange` with `{ code: '<expired-code>' }` -- expect 401

**Expected Result:** Auth codes are single-use and expire after TTL.

**Auth Context:** N/A (testing auth code lifecycle)

**Isolation Check:** Auth codes from one org's SSO flow cannot be exchanged by a different org's user.

### E2E-6: Force-SSO Blocks Email/Password Login

**Preconditions:**

- Organization with `forceSso: true`, `allowGoogleFallback: false`
- SAML SSO config active
- User `user@forced-sso.com` has email/password and belongs to this org

**Steps:**

1. `POST /api/auth/login` with `{ email: 'user@forced-sso.com', password: '<password>' }` -- expect 403 or redirect to SSO
2. `GET /api/sso/init?email=user@forced-sso.com` -- expect `{ ssoEnabled: true, protocol: 'saml' }`
3. Complete SAML SSO flow -- expect success with tokens
4. Verify `POST /api/auth/google` is also blocked (no Google fallback)

**Expected Result:** Email/password login is blocked for force-SSO orgs. Only SSO login succeeds.

**Auth Context:** tenant=forced-sso org tenant

**Isolation Check:** Users in a different org without force-SSO can still log in with email/password.

### E2E-7: SSO Login Issues JWT with Correct Tenant Context

**Preconditions:**

- Organization with OIDC SSO config
- User `dev@context-test.com` is a member of two tenants: T1 (ADMIN role) and T2 (MEMBER role)
- Default tenant is T1

**Steps:**

1. Complete OIDC SSO login for `dev@context-test.com`
2. `POST /api/sso/exchange` -- get tokens
3. Decode access token JWT -- verify `tenantId=T1`, `role=ADMIN`, `orgId` present
4. `POST /api/auth/switch-tenant` with `{ tenantId: T2 }` -- expect new access token
5. Decode new token -- verify `tenantId=T2`, `role=MEMBER`

**Expected Result:** SSO login resolves the correct default tenant context. Tenant switching works post-SSO.

**Auth Context:** Starts with tenant=T1, switches to T2

**Isolation Check:** Cannot switch to a tenant the user is not a member of (returns 403).

## Integration Test Scenarios (12 Scenarios)

### IT-1: SAML Assertion Signature Verification

**Boundary:** SAML callback route + `@node-saml/node-saml` library + org-repo SSO config lookup

**Setup:** Mock SAML IdP with X.509 certificate. Organization with SAML config containing the IdP certificate.

**Steps:**

1. Generate a properly signed SAML assertion from the mock IdP
2. Call SAML callback with the signed assertion
3. Verify user is created/found and auth code is generated
4. Generate a SAML assertion signed with a DIFFERENT key
5. Call SAML callback with the mismatched-signature assertion
6. Verify 401 "SAML assertion validation failed"

**Expected Result:** Valid signatures accepted, invalid signatures rejected.

**Failure Mode:** If `@node-saml/node-saml` is unavailable, the callback should return 501 or degrade gracefully (not silently accept unsigned assertions).

### IT-2: SAML IdP-Initiated Organization Resolution

**Boundary:** SAML callback route + org-repo `findOrgBySAMLIssuer`

**Setup:** Two organizations with different SAML IdP entity IDs.

**Steps:**

1. POST SAML callback with assertion from IdP-A (no RelayState) -- should resolve to Org-A
2. POST SAML callback with assertion from IdP-B (no RelayState) -- should resolve to Org-B
3. POST SAML callback with assertion from unknown IdP (no RelayState) -- should return 400

**Expected Result:** Organizations correctly resolved by SAML issuer. Unknown issuers rejected.

**Failure Mode:** If org lookup query fails, return 500 (not 400).

### IT-3: OIDC Token Exchange with Mock Provider

**Boundary:** OIDC callback route + external HTTP calls (token endpoint, userinfo) + sso-state-store

**Setup:** Mock OIDC provider with `/token` and `/userinfo` endpoints. OIDC state stored via `storeOIDCState`.

**Steps:**

1. Store OIDC state for a test organization
2. Call OIDC callback with valid code and matching state
3. Verify mock token endpoint was called with correct parameters (grant_type, code, redirect_uri, client_id, client_secret)
4. Verify mock userinfo endpoint was called with access token
5. Verify user was created with correct email and auth code was generated

**Expected Result:** Token exchange and userinfo fetch succeed. User provisioned correctly.

**Failure Mode:** If token exchange returns non-200, callback returns 401.

### IT-4: SSO Config Encrypted Storage Round-Trip

**Boundary:** org-repo `createSSOConfig` / `findSSOConfig` + sso-helpers `decryptSSOConfig`

**Setup:** Encryption service available with master key.

**Steps:**

1. Create an SSO config with SAML data via `createSSOConfig` (encrypted)
2. Read it back via `findSSOConfig`
3. Decrypt via `decryptSSOConfig` and verify all fields match original
4. Update the config via `updateSSOConfig`
5. Read and decrypt again -- verify updated fields

**Expected Result:** SSO config survives encrypt/store/decrypt round-trip without data loss.

**Failure Mode:** If encryption service unavailable, config stores as plaintext JSON (backward-compatible).

### IT-5: Domain Claim and DNS Verification

**Boundary:** domain-service + org-repo domain mapping operations + DNS resolver

**Setup:** Organization A claims `test-domain.com`.

**Steps:**

1. `claimDomain(orgA, 'test-domain.com')` -- expect verification token returned
2. `findDomainMapping('test-domain.com')` -- expect unverified mapping for orgA
3. `verifyDomain(orgA, 'test-domain.com')` with no DNS record -- expect `false`
4. (Mock DNS to return the verification token)
5. `verifyDomain(orgA, 'test-domain.com')` -- expect `true`
6. `findDomainMapping('test-domain.com')` -- expect `verified: true`
7. `claimDomain(orgB, 'test-domain.com')` -- expect error "already claimed"

**Expected Result:** Domain claim, verification, and conflict detection work correctly.

**Failure Mode:** DNS lookup timeout returns `false` (not an error).

### IT-6: MFA Enforcement Check by Plan Tier

**Boundary:** mfa-service `isMFARequired` + workspace-repo + org-repo + compliance-repo

**Setup:** Three organizations: Free plan, Business plan, Enterprise plan with `requireMfa: true`.

**Steps:**

1. `isMFARequired(userId, freeOrgTenantId)` -- expect `false`
2. `isMFARequired(userId, businessOrgTenantId)` -- expect `true`
3. `isMFARequired(userId, enterpriseOrgTenantId)` -- expect `true`
4. Update Business org to `requireMfa: false` -- still `true` (plan-level enforcement)

**Expected Result:** BUSINESS and ENTERPRISE plans enforce MFA regardless of org settings.

**Failure Mode:** If tenant or org not found, returns `false` (fail-open for MFA check).

### IT-7: Recovery Code Single-Use Enforcement

**Boundary:** mfa-service + mfa-repo

**Setup:** User with MFA enabled and 10 recovery codes.

**Steps:**

1. `setupMFA(userId)` -- get recovery codes
2. `confirmMFASetup(userId, validTotpCode)` -- confirm
3. `verifyRecoveryCode(userId, recoveryCodes[0])` -- expect `true`
4. `verifyRecoveryCode(userId, recoveryCodes[0])` again -- expect `false` (already used)
5. `getMFAStatus(userId)` -- expect `recoveryCodesRemaining: 9`
6. `regenerateRecoveryCodes(userId)` -- get 10 new codes
7. `getMFAStatus(userId)` -- expect `recoveryCodesRemaining: 10`

**Expected Result:** Recovery codes are single-use. Regeneration invalidates old codes.

**Failure Mode:** If bcrypt comparison fails, returns `false` (not an error).

### IT-8: SAML Assertion Replay Protection

**Boundary:** sso-state-store + SAML callback

**Setup:** Redis available (or in-memory fallback for test).

**Steps:**

1. Process a SAML callback with assertion ID `_abc123` -- expect success
2. Process the same SAML callback with assertion ID `_abc123` -- expect 400 "SAML assertion already used"
3. Wait for assertion TTL to expire
4. Process the callback with `_abc123` again -- expect success (TTL expired)

**Expected Result:** Replay protection blocks reused assertions within TTL window.

**Failure Mode:** If Redis is down, falls back to in-memory (single-pod limitation).

### IT-9: OIDC CSRF State Validation

**Boundary:** sso-state-store `consumeOIDCState` + OIDC callback

**Setup:** Store an OIDC state parameter.

**Steps:**

1. `setOIDCState('valid-state', { nonce, codeVerifier, orgId })` with 10-second TTL
2. `consumeOIDCState('valid-state')` -- expect data returned
3. `consumeOIDCState('valid-state')` again -- expect `null` (already consumed)
4. `setOIDCState('expiring-state', data, 1)` -- 1 second TTL
5. Wait 2 seconds
6. `consumeOIDCState('expiring-state')` -- expect `null` (expired)
7. OIDC callback with `state=unknown` -- expect 403

**Expected Result:** State is single-use and expires after TTL. Invalid states are rejected.

**Failure Mode:** Redis GETDEL ensures atomicity. In-memory fallback uses delete-after-get.

### IT-10: Auth Code Exchange Lifecycle

**Boundary:** sso-auth-codes store + exchange endpoint

**Setup:** Auth code stored via `storeAuthCode`.

**Steps:**

1. Store auth code with token data (60s TTL)
2. `consumeAuthCode(code)` -- expect token data
3. `consumeAuthCode(code)` again -- expect `null`
4. Store another code with 1s TTL
5. Wait 2 seconds
6. `consumeAuthCode(expiredCode)` -- expect `null`

**Expected Result:** Auth codes are single-use and time-limited.

**Failure Mode:** Concurrent consume requests: only one succeeds (Redis GETDEL atomicity).

### IT-11: SSRF Protection on OIDC Outbound Requests

**Boundary:** OIDC callback `isAllowedExternalUrl` validation

**Setup:** Organization with OIDC config.

**Steps:**

1. Set OIDC `tokenUrl` to `https://valid-idp.example.com/token` -- allow
2. Set OIDC `tokenUrl` to `http://valid-idp.example.com/token` -- reject (non-HTTPS)
3. Set OIDC `tokenUrl` to `https://localhost/token` -- reject
4. Set OIDC `tokenUrl` to `https://127.0.0.1/token` -- reject
5. Set OIDC `tokenUrl` to `https://10.0.0.1/token` -- reject (private)
6. Set OIDC `tokenUrl` to `https://169.254.169.254/token` -- reject (link-local/IMDS)
7. Set OIDC `userInfoUrl` to same patterns -- verify same behavior

**Expected Result:** Only HTTPS URLs to non-private, non-localhost hosts are allowed.

**Failure Mode:** If URL parsing fails, defaults to reject.

### IT-12: JWT Issuance with Tenant Context After SSO

**Boundary:** auth-service `createTokenPair` + `resolveUserContextOrAutoAcceptInvite`

**Setup:** User with membership in a tenant. One pending invitation for new SSO user.

**Steps:**

1. SSO login for existing user with tenant membership -- JWT includes `tenantId`, `role`, `orgId`
2. SSO login for new user with exactly one pending invitation -- auto-accept, JWT includes new tenant context
3. SSO login for new user with two pending invitations -- no auto-accept, `pendingInvitationChoice: true`
4. SSO login for new user with no invitations and no membership -- `needsOnboarding: true`

**Expected Result:** Tenant context correctly resolved for all user states. Auto-accept works for single invitation.

**Failure Mode:** If auto-accept fails (expired invitation), falls through to onboarding flow.

## Unit Test Scenarios

### UT-1: OIDC PKCE Generation and Validation

**Module:** `oidc-service.ts` -- `generatePKCE`, `generateAuthorizationUrl`

**Input:** Random seed

**Expected Output:** `codeVerifier` is 43-128 chars of unreserved characters. `codeChallenge` is base64url-encoded SHA-256 of `codeVerifier`. Authorization URL contains all required parameters.

### UT-2: SSO Config Encryption Round-Trip

**Module:** `sso-helpers.ts` -- `decryptSSOConfig`

**Input:** JSON config encrypted with `EncryptionService.encryptForTenant`

**Expected Output:** Decrypted JSON matches original. Plaintext JSON fallback works for unencrypted data.

### UT-3: Domain Normalization

**Module:** `domain-service.ts` -- domain normalization in `claimDomain`, `verifyDomain`

**Input:** `"Example.COM"`, `" example.com "`, `"EXAMPLE.com"`

**Expected Output:** All normalize to `"example.com"`.

### UT-4: MFA Requirement Check Logic

**Module:** `mfa-service.ts` -- `isMFARequired`

**Input:** Various tenant/org/plan combinations

**Expected Output:** `true` for BUSINESS/ENTERPRISE plans or `requireMfa` org setting. `false` otherwise.

### UT-5: TOTP Code Generation and Verification

**Module:** `mfa-service.ts` -- `generateTOTP`, `verifyTOTPCode`

**Input:** Known secret + known timestamp

**Expected Output:** Code matches RFC 6238 test vectors. Verification succeeds within window, fails outside.

**Module:** `mfa-service.ts` -- `hashRecoveryCode`, `compareRecoveryCode`

**Input:** Generated recovery code

**Expected Output:** Hash is bcrypt. Compare returns `true` for matching code, `false` for wrong code. Case-insensitive (codes uppercased before compare).

## Security & Isolation Tests

- [x] Cross-tenant access: SSO config from Org-A cannot be read by Org-B admin (returns 404)
- [x] Cross-org domain: Domain claimed by Org-A cannot be claimed by Org-B (returns 400)
- [x] Missing auth: All SSO management endpoints require authentication (returns 401)
- [x] Insufficient permissions: Non-admin users cannot modify SSO config (returns 403)
- [x] Input validation: Malformed emails rejected by SSO init, callback, and exchange
- [x] SAML signature verification: Assertions with invalid signatures rejected (401)
- [x] OIDC state validation: Callbacks with invalid/expired state rejected (403)
- [x] Assertion replay: Reused SAML assertions rejected (400)
- [x] Auth code replay: Reused auth codes rejected (401)
- [x] MFA lockout: After N failed TOTP attempts, account locked for configured duration
- [x] SSRF: Outbound OIDC requests to private IPs blocked
- [x] Timing-safe comparison: TOTP verification uses `crypto.timingSafeEqual`

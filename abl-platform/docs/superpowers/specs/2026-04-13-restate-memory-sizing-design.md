# Restate Memory Sizing Runbook — Design

**Date:** 2026-04-13
**Status:** Draft — awaiting user review
**Author:** Rakshak Kundarapu

## 1. Context

`abl-platform-dev-restate-0` entered `CrashLoopBackOff` with 19 restarts in 79 minutes. Kernel OOM log confirmed the cgroup limit (2 Gi) was hit: `anon-rss:2069392kB`. Evidence from pod logs: ~29 Restate partitions opened at boot; three Kafka subscriptions (`abl.session.ended`, `abl.message.agent`, `abl.message.user`) flooded backlog into `service://PipelineTrigger/handleEvent`; RocksDB emitted a `[Stall Detector] ... rocksdb-write-stall-threshold` warning on `local-loglet` within seconds — then the container was OOM-killed.

The rendered `restate.toml` in cluster (`ConfigMap abl-platform-dev-restate-config`) has no memory tuning at all:

```toml
[worker]
internal-queue-length = 1000
[[ingress.kafka-clusters]]
name = "local"
brokers = ["abl-platform-dev-kafka-kafka-bootstrap:9092"]
```

The dev pod is running with 2 Gi limit + default (~29) partitions + no `rocksdb-total-memory-size`. This combination is not in the team's calibration table at `packages/sizing-calculator/src/engine/constants.ts:528` — tier-S pairs 2 Gi with **1 partition**; tier-L pairs ~16 partitions with **8 Gi**. Dev is effectively tier-L partition count on tier-S memory.

## 2. Goal

Produce a single authoritative runbook — `docs/guides/restate-memory-sizing.md` — that:

- Explains _why_ Restate needs the memory it needs (the 5 buckets).
- Gives a _formula_ operators can plug numbers into.
- Cites the team's in-repo calibration table as the source of truth for tier sizing.
- Documents the _empirical verification_ step so a size choice is measured, not guessed.
- Ends with a _worked example_ for the current dev crash, including the exact YAML + TOML snippets a platform engineer applies to `abl-platform-deploy`.

## 3. Non-Goals

- No runtime change in this PR — the live config lives in `bitbucket.org/koreteam1/abl-platform-deploy` (ArgoCD-synced) which is not cloned locally. This PR is documentation only. Applying the change is a separate action.
- No `kubectl patch` of the live StatefulSet (ArgoCD reverts it). Unblocking dev immediately is out of scope for this PR; it can be done by the operator as a one-off if needed.
- No modification to `packages/sizing-calculator` — the tier table there is authoritative and stays as-is. The runbook references it; it doesn't duplicate or override it.

## 4. Deliverable

**One new file:** `docs/guides/restate-memory-sizing.md`

No other files touched. No code changes. No test changes. This is pure operator-facing documentation.

### 4.3 Partition count is bootstrap-immutable — operational caveat

This is a first-class concept in the runbook (§3 surfaces it, §8 applies it) because getting it wrong destroys state.

- **Rule:** Restate pins `partitions` at cluster bootstrap. Keys are hashed via `hash(key) mod N`, so any change to `N` invalidates every existing state lookup. Restate will not silently rehash — depending on version, it either refuses to start or continues with the original `N` and ignores the new setting.
- **Implication for dev (incident 2026-04-13):** the current cluster was bootstrapped with the Restate 1.3 default of 24 partitions. Applying `partitions: 4` against the live PVC is a no-op at best and a corruption risk at worst.
- **Required sequence when `partitions` changes:**
  1. Drain / quiesce Restate producers (pause Kafka subscriptions, stop runtime writes).
  2. Scale the StatefulSet to 0 (`kubectl scale sts abl-platform-dev-restate --replicas=0 -n abl-platform-dev`).
  3. Delete the PVC (`kubectl delete pvc data-abl-platform-dev-restate-0 -n abl-platform-dev`). ArgoCD's `whenDeleted: Retain` policy on the STS preserves the PV object; deleting the PVC is the explicit opt-in to wipe.
  4. Merge the `abl-platform-deploy` PR that sets the new `partitions` value.
  5. Let ArgoCD sync; the STS scales back up with a fresh PVC and bootstraps at the new partition count.
  6. Re-enable producers; run §7 empirical checks.
- **Trade-off called out in the doc:** this is acceptable for **dev** (no durable state worth preserving); it is **not** acceptable for any environment holding in-flight workflow state. For those, the only safe path is to size memory up to fit the existing partition count rather than reducing `N`.
- **Avoidance guidance added to §6 (Tier calibration):** pick the right partition count at first bootstrap. The tier table in `packages/sizing-calculator` is the source of truth for this choice at cluster birth.

### 4.2 Numbers the doc commits to

Locking these into the doc (all Restate 1.3 defaults unless noted):

| Term                                                                   | Value                                                                 |
| ---------------------------------------------------------------------- | --------------------------------------------------------------------- |
| `base` (process overhead)                                              | 400 MiB                                                               |
| Restate 1.3 default `partitions` when unset                            | 24                                                                    |
| `per_partition_overhead`                                               | 80–150 MiB (memtables + per-CF metadata)                              |
| RocksDB default `write_buffer_size` / `max_write_buffer_number` per CF | 64 MiB × 2–3                                                          |
| `ingress_headroom` per Kafka subscription under load                   | 200 MiB                                                               |
| Safety margin                                                          | 20%                                                                   |
| Block-cache share of `rocksdb-total-memory-size`                       | ~50% (rest is memtables + pinned/index/filter blocks)                 |
| Recommended `rocksdb-total-memory-size`                                | 40–50% of container memory limit                                      |
| Dev recommendation                                                     | 4 Gi limit, `partitions: 4`, `rocksdb-total-memory-size = "1500 MiB"` |
| Partition count mutability                                             | **bootstrap-only** — see §4.3                                         |

## 5. Out of scope / deferred

- Multi-node Restate HA (replicated metadata store, `partitions > 4`) — a separate runbook when dev promotes to M+.
- Disk sizing (PVC growth) — 10 Gi is fine for dev today; a future `restate-disk-sizing.md` can mirror the structure of this one.
- Alerting rules — currently Restate is `coroot-tcp` only (per `packages/sizing-calculator/src/engine/coroot-metric-extractors.ts:265`); richer Prometheus scraping is a separate work item.
- Changes to `abl-platform-deploy` — the operator applies the snippets; this PR does not modify that repo.

## 6. Rollout & validation

Because this PR is documentation only:

1. Merge to `develop` (or the target branch) — no runtime impact.
2. Platform engineer opens a follow-up PR in `abl-platform-deploy` against `environments/dev/values.yaml` using the §8 snippets.
3. **Because `partitions` is changing (24 → 4) this is NOT a live roll.** The operator follows the §4.3 re-bootstrap sequence: quiesce producers → scale STS to 0 → delete PVC → merge deploy PR → ArgoCD sync → fresh bootstrap → re-enable producers.
4. Run the §7 empirical checks. Acceptance: RSS steady-state ≤ 75% of 4 Gi; zero `stall` lines in 10-minute window post-drain.

## 7. Risks

- **Risk:** Doc drifts from `sizing-calculator` as tiers evolve. **Mitigation:** §6 explicitly names the constants file as authoritative and instructs readers to update code, not doc, when tiers change.
- **Risk:** Dev backlog is a symptom of a broken `PipelineTrigger` handler, not under-sizing. Bumping memory alone would then mask the real issue. **Mitigation:** §9 (decision tree) forces operators to check consumer liveness before resizing.
- **Risk:** Reader copies the dev snippets into a prod values file. **Mitigation:** the worked example is clearly labelled `dev, 2026-04-13`; §6 tier table is the canonical source for all other environments.
- **Risk:** Reader applies `partitions: 4` to a live non-dev Restate cluster and corrupts or orphans state. **Mitigation:** §3 introduces the bootstrap-immutability rule as a first-class concept; §4.3 gives the mandatory re-bootstrap sequence; §8 worked example explicitly labels this as safe for dev only and warns against partition reduction in any environment with in-flight workflow state.
- **Risk:** Operator runs the deploy PR without first deleting the PVC, chart silently applies `partitions: 4` to an existing 24-partition PVC, Restate ignores it or refuses to start. **Mitigation:** §4.3 and §6 rollout step 3 both make the PVC deletion an explicit ordered step before the deploy PR merges.

Arch AI Blueprint, Eval Pack, and God Mode Quality Loop

A proposal for turning project creation into a governed design-to-quality pipeline: blueprint captures intent, eval pack captures success, project creation persists the implementation, and God Mode continues optimization after the project exists.

Status proposal for review Date 2026-05-12 Audience product, engineering, leadership Recommendation progressive quality flow
00

Strategic Recommendation

My strongest recommendation is not to make evals a single yes/no feature. Treat evals as a progressive quality system with three lanes: fast creation, guided confidence, and autonomous improvement.

What I would pitch

Make guided confidence the default: Arch drafts an eval pack before project creation, creates the project only after health is clean, then offers a post-create baseline run and improvement loop. Keep a fast lane for demos and a power lane for God Mode. This avoids forcing every user into long eval runs while still making quality a first-class part of Arch.

Pitch Product promise Flow Best for Tradeoff My view
1. Fast Create Create a working project with clean syntax and health. Blueprint -> Build -> Health -> Create. Evals are optional later. Demos, prototypes, low-risk internal tools. Fast, but weak behavioral proof. Keep it as a lane, but do not make this the flagship enterprise story.
2. Guided Confidence Create a project plus the tests that prove the design. Blueprint -> Build -> Health -> Draft Eval Pack -> Create -> Optional Baseline. Most onboarding, sales demos, serious first-time project creation. Moderate review step, but much higher trust. Best default. Strongest balance of speed, explainability, and quality.
3. God Mode Launch Create, test, improve, and ship the best version Arch can produce. Guided Confidence -> Baseline -> Failure clusters -> Patch loop -> Regression gate -> Accept. Production-minded customers, complex multi-agent systems, executive demos. Takes time and budget; requires clear stop controls. Make it a premium/power action, not an invisible default.
4. Eval-First Design Define success before architecture; Arch designs to pass tests. Goals + scenarios -> Blueprint -> Build -> Create -> Baseline. Regulated workflows, known SOPs, high-risk support flows. Best rigor, highest upfront effort. Very strong for enterprise templates, but too heavy for generic onboarding.
5. Observe-Then-Evaluate Start with real usage, then generate evals from failures and transcripts. Create -> Pilot sessions -> Mine failures -> Generate evals -> Improve. Ambiguous domains where users do not know scenarios yet. Realistic, but quality comes after exposure. Excellent in-project capability, not enough as the initial quality story.
Default

Guided Confidence

Generate the eval pack before create, run real evals after create. This is the main recommendation.

Fallback

Fast Create

Keep a quick path for simple projects, but label evals as skipped quality coverage.

Power

God Mode Launch

Use budgeted autonomous loops when the user explicitly wants Arch to improve the project after creation.

01

Why This Change

Arch currently proves syntax before it proves behavior. That is not enough for a multi-agent project where routing, handoffs, tool calls, completion, escalation, and business rules can all fail while the UI still looks "created."

Current risk

Blueprint can be too thin

A topology can look correct while generated agent DSL contains dead conditions, invalid calls, wrong return behavior, or hallucinated tool assumptions.

Current risk

Health arrives too late

If critical findings appear after create, users see "project ready" and "project broken" at the same time.

Opportunity

Eval becomes the contract

The blueprint should describe not only what to build, but what must be true for the project to be considered good.

03

Eval Flow Alternatives

We should not assume the first eval idea is the best product shape. The right design depends on friction, runtime truth, and how much autonomous mutation we want Arch to perform during onboarding.

My opinion

The best default is progressive quality: generate an eval pack draft before project creation, create the project once health is clean, then offer a baseline eval and improvement loop immediately after creation. Full God Mode auto-optimization should exist as an explicit power mode with budget, regression checks, and approval controls.

Option Pitch Where evals happen User friction Quality signal Risk Verdict
A. Health gate only Keep onboarding fast. Only compile and run semantic health checks. No eval pack in onboarding; evals only later inside project. Low Medium Misses behavioral failures: wrong answer, weak tone, missing edge cases. Good for demos, too shallow as the main production-ready promise.
B. Blueprint eval intent only Blueprint lists quality dimensions, scenario families, and business rules, but does not generate concrete cases. During blueprint as advisory sections. Low Medium Looks strategic, but does not create runnable proof or coverage pressure. Useful foundation. Not enough by itself.
C. Eval pack draft before create Arch drafts personas, scenarios, expected paths, assertions, and fixtures as an optional artifact. After health gate, before create; runnable after project is created. Medium High Can feel like extra review unless compact and skippable. Recommended default. Best balance of confidence and speed.
D. Required pre-create eval run Project cannot be created until generated agents pass a baseline eval suite. Before create, using a temporary runtime sandbox. High High if real Needs fake project IDs, temporary tool bindings, temporary versioning, and cleanup. This creates lifecycle drift. Avoid for v1. Too much complexity.
E. Post-create baseline plus improve Create project first, then immediately offer "Run baseline and improve" using real runtime, tools, and versions. After create as onboarding continuation. Medium to high Very high Can take time and cost money; needs visible budget and stop controls. Recommended power path. Best place to run real evals.
F. In-project only Keep onboarding about creation. Let users generate and run evals later from the project, agent editor, or Arch overlay. Only after project creation, on demand. Low during onboarding Depends on user follow-through Most users may never run evals, so weak projects ship silently. Good fallback, weaker than guided continuation.
Best default

C + E together

Draft the eval pack before create, then run baseline and improve after create. This keeps artifact quality and runtime truth in the right places.

Fast lane

A + B for simple demos

For a quick prototype, require clean health, capture eval intent in the blueprint, and let the user skip eval-pack generation.

Power lane

C + E with auto-loop

For serious projects, run God Mode with an explicit budget, regression guard, and approval mode. This is Arch's strongest enterprise story.

Eval Pack Design

The eval pack translates blueprint intent into concrete quality checks. It should be generated before create as a draft and made runnable only after create.

Dimension

Routing accuracy

Expected agent path, no dead routes, no duplicate catch-all handoffs.

Dimension

Field discipline

Gather only useful fields, populate completion fields, avoid stale names.

Dimension

Tool correctness

Valid tool references, correct argument shapes, and realistic fixtures.

Dimension

Return lifecycle

Returnable specialists complete; they do not handoff back to parent.

Important constraint

Eval pack generation should be optional and skippable. Project creation should not require eval authoring for simple demos, but production-ready onboarding should strongly recommend it when health or coverage is weak.

06

God Mode Continuation

God Mode is not a separate creation path. It is the quality loop that continues after project creation, using the blueprint and eval pack as source material.

0

Baseline run

Run the accepted eval pack against the created project and store the baseline.

1

Failure clustering

Cluster failures by routing, handoff, gather, tool, business rule, and safety.

2

Patch proposal

Produce a hypothesis, target agent, diff, expected impact, and regression risk.

3

Verification run

Run focused cases plus regression cases before accepting the patch.

4

Versioned accept

Accept only if targets improve and no blocking regression appears.

07

In-Project Arch Capability

Once the project exists, Arch should use the same blueprint and eval artifacts to explain, modify, test, and improve the system from inside Studio.

Ask

"Why is health bad?"

Arch reads health findings, agent DSL, blueprint contracts, and eval failures.

Ask

"Improve routing"

Arch compares expected paths from eval pack against actual traces.

Ask

"Add a workflow"

Arch edits blueprint first, regenerates DSL, and updates eval coverage.

08

Data Flow Audit

Every field that appears in the blueprint or eval pack must either translate deterministically downstream or be marked advisory.

Field family Blueprint Build Create project Eval pack God Mode
Agent topology Source Renders handoffs Persists agents Expected paths Routing regressions
Gather fields Declared GATHER + COMPLETE Agent DSL Required-field assertions Field patch proposals
Tools Tool intent Tool signatures Only real/linkable tools Tool-call expectations Failure diagnosis
Handoffs Conditions + return contract Raw CEL, no quoted expression Agent DSL Expected trajectory Dead route fixes
Success dimensions Targets Advisory, not DSL Project quality config Evaluator dimensions Loop stop criteria

Known anti-pattern to block

A returnable child agent should not end with unconditional HANDOFF WHEN: true RETURN: true back to the parent. The child should complete; runtime returns through the existing handoff stack.

09

Risks and Controls

The main risk is not adding evals. The main risk is adding evals without a strict lifecycle contract, which would create more state drift.

Risk Impact Control
Eval runs before real project exists Fake confidence, no durable version, no real tools Only draft eval packs pre-create; runnable status only after create
Blueprint and generated DSL drift User approves one thing, runtime executes another Blueprint is canonical; DSL is generated and validated from it
God Mode mutates blindly Regressions hidden behind aggregate score improvement Patch proposals require hypothesis, diff, focused run, regression run
Reload resumes wrong phase Duplicate prompts, stale widgets, blocked streaming Persist phase, artifact statuses, active operation locks, and resumable job IDs
10

Rollout Plan

Roll out in small increments so we get value early without a giant all-or-nothing rewrite.

Phase 1

Blueprint eval intent

Add success dimensions, scenario families, and quality gates to blueprint.

Phase 2

Eval pack draft

Generate optional personas, expected paths, assertions, and coverage gaps.

Phase 3

Post-create baseline

Persist eval pack and run baseline evals against the real project runtime.

Phase 4

God Mode loop

Add patch proposals, focused reruns, regressions, budgets, and accept controls.

Recommendation

Run evals only after project creation

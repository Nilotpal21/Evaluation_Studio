Guardrail Summary in Session Header

Small
Spec Section 3: "Session Header shows total guardrail pass/warn/fail counts." Currently the session header shows interactions, duration, tokens, and cost — but no guardrail summary.

Current

ses_a1b2c3d4 Active
Interactions: 4 Duration: 12.3s Tokens: 2,847 Cost: $0.042

Proposed

ses_a1b2c3d4 Active
Interactions: 4 Duration: 12.3s Tokens: 2,847 Cost: $0.042
New
Guardrails ✓ 8 pass ⚠ 2 warn ✗ 1 fail
Data source: Aggregate from all guardrail_* events via extractGuardrailChecks() (already exported as a pure function). Pass = guardrail_check with pass outcome. Warn = guardrail_warning. Fail = guardrail_violation + guardrail_*_blocked.
2

Destructive Action Detection

Small
Spec Section 5: "Destructive tool calls are flagged with an amber warning banner showing the action and scope." Currently tool calls display normally regardless of destructive potential.

Current

TOOL CALL delete_all_records
Method: DELETE
Target: /api/records?scope=all
TOOL CALL search_records
Method: GET

Proposed

TOOL CALL delete_all_records
Method: DELETE
Target: /api/records?scope=all
New
⚠ Destructive action: DELETE on /api/records (scope: all)
TOOL CALL search_records
Method: GET
Detection heuristic: Check tool event data.method for DELETE/PUT or data.destructive === true flag from runtime. The amber/red banner only appears on tool_call steps matching the heuristic — safe calls remain unchanged.

Error-Correlated Memory Diff Highlighting

Small
Spec Section 8: "When an error occurred in the same turn, memory changes get a red tint and an error/retry footer." Currently memory diffs always render with the standard green/amber diff colors regardless of turn outcome.

Current (always same colors)

MEMORY Memory Stored
+ city: "Austin"
~ budget: 1500 → 2000
  bedrooms: 2

Proposed (error turn = red tint)

MEMORY Memory Stored ERROR TURN
New
+ city: "Austin"
~ budget: 1500 → 2000
  bedrooms: 2
⚠ Written during error — may be rolled back on retry
Logic: Pass hasError boolean to MemoryDiff from the parent interaction (already computed in buildSummary). When true: add red border-left to changed lines, show error footer, change step card background to error tint.

Data source: entity_extraction events may carry data.spans: Array<{field, start, end}> or data.sourceText. If spans exist, render highlighted user input above the field table. Color-code each span to match its field row. Fallback: If no span data, hide the source section (current behavior).

Error Flow State / onFailAction Path

Small
Spec Section 11: "When a flow step errors, show the error node and the onFailAction recovery path (retry/skip/escalate) in the breadcrumb." Currently flow breadcrumbs show visited/active/upcoming states but not error or recovery paths.

Node states added: error (red fill + ✗ icon) — step that emitted an error event. error-path (red dashed border + ↺ icon) — the onFailAction recovery node showing action type (retry/skip/escalate) and attempt count.

Data source: flow_step_exit with data.status === 'error' triggers error state. data.onFailAction and data.retryCount populate the recovery node.

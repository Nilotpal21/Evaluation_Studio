## Recommendations for Next Phase (Epic 2)

These are lower-priority items that should be addressed opportunistically during Epic 2 development:

### MEDIUM â€” Mutable Service Factory (schema-discovery-worker.ts:54)

Module-level `let serviceFactory` has no one-time-set guard. If `setDiscoveryServiceFactory` is called while jobs are in-flight, it silently swaps the factory. Low risk today (set once at startup), but a `Object.freeze` or one-time-set pattern would be safer. Address when touching the worker for Epic 2 integration.

### MEDIUM â€” Version Sort on Single-Document Collection (DiscoveredSchemaPersistenceService.ts:76)

The unique compound index `{tenantId, knowledgeBaseId, connectorId}` means only one DiscoveredSchema document exists per connector. The GET endpoint's `.sort({ version: -1 })` is dead code. If version history is needed for Epic 2's mapping engine, the unique index should be changed to `{tenantId, knowledgeBaseId, connectorId, version}`. If not, the sort can be removed.

### LOW â€” SharePoint Discovery: No Retry + Sequential Fetching

- `SharePointSchemaDiscoveryService` fetches lists sequentially (`for...of` loop) while Google Sheets uses `Promise.all` for parallel sheet fetching.
- SharePoint has no `withRetry` wrapper for Graph API rate limits (429/500/503), unlike all other discovery services.
- Address when SharePoint connector work surfaces in later epics.

### LOW â€” Error Response Format Inconsistency

Epic 1 endpoints return `{ success: false, error: { code, message } }` (Architecture Pattern 3). Older endpoints in the same file return `{ error: 'string' }`. Consider standardizing when touching these endpoints for Epic 2.

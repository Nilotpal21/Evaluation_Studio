## What Went Well

### 3-Wave Parallel Sprint Plan

The decision to parallelize stories into waves was the biggest velocity multiplier. Wave 1 ran 3 stories simultaneously (2.1, 2.2, 2.4), Wave 2 ran 3 more (2.3, 2.7, 2.8), and Wave 3 integrated everything. This was possible because:

- Wave 1 stories had no inter-dependencies (rule service, LLM service, CRUD API all independent)
- Wave 2 stories depended on Wave 1 outputs but not on each other
- Wave 3 (orchestration) naturally needed everything from Waves 1+2

### Story Consolidation (2.4+2.5+2.6)

Combining three originally separate stories (CRUD, Bulk Actions, Manual Creation) into a single Story 2.4 was pragmatic. All three modified the same file (`mappings.ts`), so merging eliminated potential conflicts and reduced context-switching overhead.

### Epic 1 Lessons Applied Successfully

- **`getLazyModel` pattern**: Zero violations across all 7 stories. The lesson from Epic 1's retro was explicitly documented in story files and followed perfectly.
- **Code reviews**: Continued on every story. Caught `vi.hoisted()` TDZ issue and `as const` type narrowing requirement in Story 2.9.
- **Story intelligence sections**: Each story referenced previous story patterns, preventing regression.

### Clean Integration (Story 2.9)

The final orchestration worker integrated 6 services on first attempt with 14 passing tests. The adapter patterns (`toMappingSuggestion`, `toConnectorSchemaField`) cleanly bridged interface mismatches between services.

### Graceful Degradation Strategy

The LLM failure handling pattern (established in 2.2, hardened in 2.8, exercised in 2.9) is robust: LLM failure never blocks rule-based results. Circuit breaker upgraded from in-memory to Redis-backed for distributed pod awareness.

## Issues Found

### Pattern: Epic ↔ Model Field Name Mismatch (5/7 stories)

The epic definitions used different field names than the actual MongoDB models:

| Epic Says           | Model Uses          |
| ------------------- | ------------------- |
| `mappingSource`     | `suggestedBy`       |
| `sourceField`       | `sourcePath`        |
| `status: confirmed` | `status: active`    |
| `status: pending`   | `status: suggested` |

Every story had to discover and document this mismatch independently. This created friction in early stories and required careful reading of model source files.

**Action Item:** Update Epic 3+ story definitions to use actual model field names, or create a field name mapping reference document.

### TraceEvent Emission Not Implemented

Multiple stories (2.3, 2.7, 2.8) listed TraceEvent emission in acceptance criteria but deferred implementation as "TBD". The TraceEvent pattern for workers/services isn't well-established yet.

**Action Item:** Establish TraceEvent emission pattern before Epic 3 if observability is needed for UI actions.

## Key Learnings

1. **Parallel wave sprints work well for service-layer epics** — When stories have clear dependency boundaries, running 3 stories simultaneously per wave dramatically accelerates delivery.

2. **Story consolidation prevents merge conflicts** — Combining stories that modify the same files (2.4+2.5+2.6 → single 2.4) is worth the larger story scope.

3. **Epic field names should match model reality** — The persistent mismatch between epic definitions and actual model fields caused unnecessary friction. Future epics should verify field names against source before story creation.

4. **Adapter functions bridge service interfaces cleanly** — The `toMappingSuggestion()` and `toConnectorSchemaField()` pattern in Story 2.9 is the right approach when services have different type expectations.

5. **Graceful degradation should be designed from the start** — Epic 2's LLM failure handling (rule-based continues, circuit breaker protects, worker catches and logs) was planned architecturally and worked perfectly.

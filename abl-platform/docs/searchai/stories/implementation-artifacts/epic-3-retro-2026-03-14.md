## What Went Well

### Brownfield Enhancement Approach

Every story was an enhancement to existing code, not greenfield. The 770-line `FieldsTab.tsx` and 905-line `mappings.ts` were surgically modified rather than rewritten. This preserved all existing functionality while adding new capabilities. The story files explicitly documented existing code line numbers and component APIs, which made surgical edits safe.

### Story Consolidation (3 Consolidations)

Three consolidation decisions eliminated file conflicts and reduced overhead:

- **3.1+3.2** (tab shell + My Fields): Same component, same render tree — no point splitting
- **3.3+3.7** (suggested tab + bulk bar): Bulk bar is part of the suggested tab, not standalone
- **3.5+3.6** (alias edit + enum edit): Combined into single "Edit Field Mapping" dialog with conditional enum section

This pattern was established in Epic 2 (2.4+2.5+2.6) and continued to work well.

### Wave Parallelization for Frontend Epic

The backend-first strategy (Wave 1: Story 3.8) ensured API endpoints existed before frontend stories needed them. Waves 2 and 3 each ran 2 stories in parallel on different tab panels, avoiding merge conflicts since each story touched a distinct section of the component.

### Epic 2 Lessons Applied Successfully

- **`status: 'active'` not `'confirmed'`**: The Epic 2 retro flagged this field name mismatch. Story files explicitly documented it. Despite this, the initial implementation still used `m.status === 'confirmed'` — caught and fixed during code review. The lesson was known but needed vigilance.
- **Route ordering**: Story 3.8 explicitly documented that static routes (GET /tab-stats, GET /review) must register BEFORE parameterized routes (GET /:mappingId). This prevented the same class of bug that plagued earlier Express work.
- **`getLazyModel` mandatory**: Zero violations. Pattern is now muscle memory.

### Existing Component Reuse

Leveraged existing UI components without modification:

- `Tabs` component (already had `count` prop for badges)
- `DataTable` (built-in sorting with `sortable` + `sortValue`)
- `ConfirmDialog` (destructive action confirmation)
- `Badge`, `EmptyState`, `Dialog`, `Select` — all used as-is
- `confidenceVariant()` helper — already mapped to success/warning/error

## Issues Found

### Pattern: Status Filter Mismatch (Critical Bug Caught in Review)

Despite Epic 2's retro explicitly warning about `status: 'active'` vs `status: 'confirmed'`, the My Fields filter was initially written as:

```typescript
const confirmedMappings = mappings.filter((m) => m.status === 'confirmed');
```

This would render an empty My Fields tab. Fixed to `m.status === 'active'`. The mismatch continues to be a trap because "confirmed" is the natural UI language but "active" is the model value.

**Action Item:** Consider adding a TypeScript union type or constant map (`UI_STATUS_TO_MODEL`) to make this impossible to get wrong.

### Express Route Ordering Bug (Critical, Caught in Review)

`GET /:mappingId` was initially placed at line 565, BEFORE `GET /review` (line 868) and `GET /stats/:canonicalSchemaId` (line 990). Express would match "review" and "stats" as mappingId parameters, breaking those endpoints. Fixed by moving GET /:mappingId to the end of the file, after all static routes.

**Root cause:** The story spec said "add BEFORE /:mappingId/confirm" which was correct for the POST routes but the GET was added in the wrong position relative to other GETs.

### Missing DELETE Endpoint

Frontend `deleteMapping()` called `DELETE /api/mappings/:id` but no handler existed in the backend. This was a gap between Stories 3.1 (frontend) and 3.8 (backend) — the delete endpoint wasn't in 3.8's scope but was needed by 3.1's "Remove" action. Added during Wave 1 implementation.

**Action Item:** When stories cross frontend/backend boundaries, explicitly list ALL API operations needed in both story files.

### Dead Code Accumulation During Refactor

After replacing the expandable My Fields section with DataTable, several artifacts remained: `expandedField` state, `confirmedFields` memo, `getMappingsForField` callback, and unused imports (`ChevronRight`, `ChevronDown`, `Card`). These were caught during review and cleaned up.

**Action Item:** After major structural refactors, run a dead code sweep — check for orphaned state variables, unused memos, and stale imports.

### No Story Files for Wave 3 (3.5, 3.6)

Stories 3.5 and 3.6 were implemented without dedicated story files because they were combined into a single dialog during Wave 3. This means there's no implementation record for these stories individually, unlike all other stories which have detailed files.

## Key Learnings

1. **Frontend epics benefit from backend-first waves** — Running Story 3.8 (API routes) in Wave 1 alongside the tab shell ensured all endpoints existed before Wave 2 UI stories needed them. No frontend story was blocked waiting for an API.

2. **Brownfield story specs should include exact line numbers and existing code snippets** — Every Epic 3 story documented the existing code structure with line references. This made surgical modifications safe and prevented accidental rewrites of working code.

3. **Status value mismatches are a persistent trap** — Despite explicit documentation in story files AND the previous retrospective, `'confirmed'` vs `'active'` still caused a bug. The fix is to encode this as a TypeScript constant map, not just documentation.

4. **Express route ordering needs explicit documentation** — Parameterized routes (`:id`) must always come AFTER static routes (`/tab-stats`, `/review`, `/stats/:id`). This should be a documented pattern in the codebase, not tribal knowledge.

5. **Combined stories should still produce story files** — Wave 3's 3.5+3.6 combination was efficient but left no implementation record. Future consolidations should still produce a single combined story file.

# Arch AI Project Feature Capability Audit

**Date:** 2026-05-14
**Status:** WIP audit
**Scope:** Project-level Studio features that Arch AI can see, reason about, modify, or test from ONBOARDING and IN_PROJECT mode.
**Primary evidence:** `apps/studio/src/lib/arch-ai/page-context-registry.ts`, `apps/studio/src/lib/arch-ai/tools/in-project-tools.ts`, `packages/arch-ai/src/types/tools.ts`, `packages/arch-ai/src/knowledge/cards/_mapping.ts`, `docs/superpowers/2026-03-12-abl-platform-feature-map.md`, and existing Arch test guides.

This audit complements `2026-05-05-arch-ai-capabilities-and-gaps-audit.md`. That file is tool/ABL-authoring focused; this file is feature focused: for each project surface, what Arch knows, what it can do, and what is still missing.

## Capability Legend

| Level            | Meaning                                                                                               |
| ---------------- | ----------------------------------------------------------------------------------------------------- |
| **Full**         | Arch has context, knowledge, executable tools, mutation safety, and a verification path.              |
| **Strong**       | Arch can perform most common workflows but has known edge gaps or incomplete coverage.                |
| **Partial**      | Arch can read/analyze or handle a subset of mutations, but important feature workflows remain manual. |
| **Context-only** | Arch receives page context or prompt knowledge, but has no reliable tool surface for the feature.     |
| **Gap**          | Feature is in the project map but Arch lacks meaningful context, knowledge, or tool coverage.         |

## Executive Summary

Arch AI is strongest where feature work maps to project agents, tools, integrations, knowledge bases, model configuration, diagnostics, sessions, and trace analysis. The in-project architect has a broad tool surface: agent CRUD and DSL proposals, project configuration, auth profiles, tools, MCP servers, external agents, variables, integration drafts, connector connections, deployment operations, testing operations, analytics reads, SearchAI operations, memory, diagnostics, simulations, and knowledge-spine lookup.

The biggest gaps are not raw tool count. They are feature completeness, knowledge routing, and production verification:

1. **Channel operations are split and incomplete.** Arch can list channels and configure project-level channel payloads through `deployment_ops`, but it cannot fully create/update/delete channel resources or bind agents to channels through a dedicated `channel_ops` contract.
2. **Workflow authoring is mostly knowledge/context, not executable mutation.** There is a workflow page context and platform cards, but no first-class Arch tool that creates, edits, versions, tests, or deploys workflow definitions.
3. **Evals and experiments are shallow.** `testing_ops` can run a runtime test, list evals, and create an eval shell, but scenario persistence and full eval lifecycle are deferred.
4. **Analytics is read-only and narrow.** `analytics_ops` currently supports `metrics` and `anomalies`; intent and quality-score pipelines are explicitly unavailable in executor code.
5. **Governance, guardrails config, alerts, customer insights, quality monitor, and agent performance are not yet full operational copilots.** They have page context and/or general diagnostics, but not feature-specific mutation and verification loops.
6. **Knowledge coverage is good for ABL constructs and integrations, but uneven for feature-specific operations.** Cards exist for channels, deployments, auth profiles, connections, KB administration, workflows, testing/evals, API management, and external agents. Missing or weak areas include experiments, billing, governance, alerts, inbox approvals, project import/export, and advanced analytics.

## Cross-Cutting Gaps

1. **Feature inventory is not machine-enforced.** The feature map, page context registry, Arch tools, knowledge cards, and test guides can drift independently.
2. **Capability naming is not normalized.** Page context capabilities such as `workflow_design`, tool names like `testing_ops`, and knowledge card topics are related but not joined by a single taxonomy.
3. **Read/context coverage is often mistaken for action coverage.** Workflows, analytics surfaces, governance, and experiments demonstrate this most clearly.
4. **Mutation safety is uneven across feature domains.** Agent DSL changes have proposals, diffs, compile checks, simulation, and apply. Tools/auth/variables/KB mutations have narrower dependency and rollback visibility.
5. **Verification loops are not uniform.** Some actions naturally end in test/simulation/health checks; others return success without prompting Arch to prove production wiring.
6. **Knowledge cards can teach nonexistent tools.** The clearest current example is `channel_ops` in channel/external-agent guidance while executable channel support lives under `deployment_ops` and `platform_context`.
7. **Tool registries can drift despite the dual-alignment contract.** `ToolName`, `TOOL_CLASSIFICATION`, and actual executable Studio tools are not currently identical (`create_project`, `kb_crawl`, `kb_schema`).

## Proposed Audit Test Shape

1. Parse `PAGE_CONTEXT_REGISTRY` and extract feature keys and capability tags.
2. Parse `ToolName`, `IN_PROJECT_SPECIALIST_TOOL_MAP`, and `buildInProjectTools` registered tool names.
3. Parse `CARD_FILE_COVERAGE` and card-router registrations.
4. Build a matrix:
   - feature route/page
   - page-context capabilities
   - relevant tools/actions
   - relevant knowledge cards
   - docs/test guide paths
   - classification: Full, Strong, Partial, Context-only, Gap
5. Fail if a feature has neither tool coverage nor knowledge coverage and is not explicitly marked planned/context-only.
6. Warn if knowledge cards mention tool names that are absent from `ToolName` or absent from in-project registration.

## Immediate Human-Readable Gap List

- **Channels:** missing `channel_ops`; current support cannot complete full channel CRUD and agent binding.
- **Workflows:** missing `workflow_ops`.
- **Evals:** missing scenario persistence and full lifecycle.
- **Analytics:** `intents` and `quality_scores` are executor stubs; page-level optimization surfaces need deeper tools.
- **Experiments:** no executable or knowledge coverage.
- **Inbox approvals:** no page context or tool coverage.
- **Alerts:** no page context, knowledge, or tool coverage.
- **Governance:** no first-class registry/compliance workflow coverage.
- **Guardrails config:** relies on ABL edits, not policy CRUD.
- **Project import/export/modules:** mature package exists, but Arch does not expose it as a feature capability.
- **Advanced ABL authoring:** scaffold/prompt gaps remain for NLU, entities, lookup tables, templates, hooks, attachments, destinations, behavior profiles, and advanced execution constructs.
- **Tool registry drift:** `ToolName`, tool classification, and executable tool registration are not yet generated from one source.

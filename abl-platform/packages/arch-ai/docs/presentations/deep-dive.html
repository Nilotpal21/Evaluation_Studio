How it works

The split: coordinator decides, specialist speaks.

Generative models are good at language and bad at lifecycle. Arch AI keeps the two responsibilities separate. The deterministic coordinator owns the rules of the game; the right specialist plays each round.

Coordinator — the rules

Pure functions over typed session metadata. Cannot be talked out of a rule.

  • Validates phase transitions against a static edge table
  • Evaluates exit criteria as predicates, not vibes
  • Classifies user intent as small or large scope
  • Picks the next BUILD gate from a derived queue
  • Enforces guards: max turns, stalls, loops, timeouts

Specialist — the conversation

Focused system prompt + curated tool set. Speaks only when invoked by the coordinator.

  • Writes the conversation, asks questions, proposes designs
  • Calls server-side tools to fetch facts and validate
  • Calls client-side tools (ask_user, collect_file) to hand control to the user
  • Requests proceed_to_next_phase only when ready
  • Cannot advance phases or mutate state directly
The four phases

Onboarding is a deterministic pipeline.

Each phase has exactly one specialist, one exit predicate, and one forward edge. The only legal backward edge is BUILD → BLUEPRINT for large-scope mutations — used when the user changes their mind about agent topology mid-build.

01

Interview

Onboarding Specialist · clipboard icon

Captures the foundational specification — what the agent does, who talks to it, what tone it uses, what entities it collects, and what constraints apply. Conversational; never a form.

Inputs gathered

  • Project name + one-line summary
  • Channels (web, voice, SMS, email)
  • Tone & voice persona
  • Primary intents & success criteria
  • Entities to collect, validation rules
  • Hand-off / escalation thresholds
  • Optional file uploads (existing docs, transcripts)

Tools available

ask_user collect_file platform_context update_specification add_conversation_note proceed_to_next_phase

Exit criterion

canExitInterview(specification) — required spec fields populated (name, summary, channels, intents). Coordinator-evaluated, not LLM-decided.

02

Blueprint

Multi-Agent Architect · network icon

Designs the multi-agent topology. Picks a pattern (single agent, supervisor + workers, pipeline, fan-out), defines hand-offs, and computes a deterministic build order via Kahn topological sort. The user approves the topology before proceeding.

What the architect decides

  • Topology pattern from the decision-tree vocabulary
  • Agent count, names, and roles
  • Hand-off edges + their guard conditions
  • Build order (topological sort)
  • Tool ownership — which agent owns which integration

Tools available

propose_topology refine_topology classify_pattern ask_user approve_topology proceed_to_next_phase

Exit criterion

metadata.topologyApproved === true. Until approval is captured in session metadata, the phase machine refuses to advance.

03

Build

ABL Construct Expert · code icon

Generates compilable ABL agent code, one agent at a time, in topological build order. Each agent goes through skeleton → validate → auto-fix → compile. Reviewed sequentially via the build-gate queue, with progress streamed live.

Per-agent pipeline

  • buildSkeleton — structural scaffold
  • validatePreCompile — Zod + grammar checks
  • autoFixABL — repairs common issues
  • Compile via @abl/compiler
  • Status: compiled or warning
  • Mock-server scaffold per external tool

Tools available

generate_agent validate_agent apply_fix extract_tools agent_review approve_agent

Exit criterion

Every agent in the topology has status compiled or warning. Build progress is streamed via build_* SSE events.

04

Create

Coordinator-driven · no LLM call

Assembles the project summary from session metadata and persists the project. No LLM round-trip — purely coordinator + service work. The user clicks Create; Arch persists the spec doc, topology, agents, and a fresh decision journal.

What gets persisted

  • Spec document (markdown SSOT)
  • Approved topology (with build order)
  • All compiled agent files
  • Mock-server artifacts
  • Decision journal (append-only)
  • Project memory snapshot

Triggers

create_project

Exit criterion

metadata.projectId set. Session transitions to COMPLETE; the user is moved to the project workspace.

The cast

11 specialists, two pools.

Click any card to see capabilities, tools, knowledge focus, and when the coordinator routes to it. Onboarding pool drives the four-phase flow; in-project pool runs after the agent ships.

Gating & guardrails

Five mechanisms keep the loop sane.

A general-purpose LLM left to its own devices will drift, loop, or skip steps. Arch AI layers five deterministic gates on top so the conversation cannot escape the rails without the coordinator noticing.

1 · Phase machine

A static edge table validates every transition. INTERVIEW → BLUEPRINT, BLUEPRINT → BUILD, BUILD → CREATE, plus the one legal backward edge BUILD → BLUEPRINT. Forward edges check the source phase's exit predicate; the backtrack skips it. Anything else throws InvalidTransitionError.

2 · Session state machine

Gate-free design. Sessions alternate between IDLE and ACTIVE; pending widget interactions persist as pendingInteraction on the session, not as a distinct state. Atomic transitions via findOneAndUpdate with the precondition in the filter — no read-then-write race.

3 · Scope classifier

Categorizes user intent as SMALL (typo, copy, single-field edit) or LARGE (add/remove agent, redesign topology, change channel set). LARGE during BUILD triggers the backtrack; diffTopologyAgainstBuildState preserves work for unchanged agents.

4 · Build-gate queue

Pure function over (topology, files, approvedAgents). Returns one of three decisions: next (review this agent), needs_generation (call LLM to generate more files), all_done (advance phase). No persistent queue collection to drift.

5 · Loop detection

Per-turn SHA-256 fingerprint of LLM output. If the model emits the same response twice in a row, the executor trips a guard and returns guard_tripped rather than burning tokens in a tight loop.

6 · Executor guards

Per-turn counters: max reinvocations, stall windows, cumulative time. Checked before each LLM call, not after — a runaway is caught before the next round-trip. Retriable errors (rate limit, transient 5xx) get up to 2 retries with 500 ms backoff.

Knowledge layer

35+ cards. Selected per turn, within budget.

The LLM never has to invent ABL syntax, platform limits, or workflow patterns. Every relevant rule is encoded as a small card and injected into the system prompt — only when relevant to the current turn.

How selection works

  1. Phase + specialist define the eligible card pool
  2. selectKnowledgeCards(query, capabilities) ranks by relevance
  3. The selector adds cards in priority order until the prompt budget is hit
  4. PLATFORM_LIMITS_CARD is always included — limits never get skipped

What a card contains

  • Construct grammar (gather, handoff, flow…)
  • Real ABL examples that compile
  • Common mistakes and the right pattern
  • When NOT to use it
  • Cross-references to related cards

Why budget matters

Big prompts dilute attention and explode cost. Selecting the smallest relevant card set keeps the model focused and the latency tight. Cards load into the same prompt section every time so the model knows where to look.

Tools & execution

Two flavors. Validated. Time-boxed.

Every tool the LLM can call is declared as a JSON Schema. Inputs are validated before execution; outputs are typed; both kinds — server-side and client-side — cleanly compose with the multi-turn loop.

Server-side tools

Coordinator runs the executor, feeds the result back to the LLM in the same stream. The model can call multiple in a turn.

  • JSON Schema input validation pre-execution
  • 30 s timeout per call (toolTimeoutMs)
  • Errors classified as retriable / fatal
  • Up to 2 retries with 500 ms backoff for transient errors
  • Examples: generate_agent, validate_agent, extract_tools

Client-side tools

Executor emits tool_call + done and stops the stream. UI renders a widget; user answers; coordinator calls resume() with the result.

  • ask_user — choice, free text, slider, file
  • collect_file — text or image, with size limits
  • agent_review — approve/edit a generated agent
  • Pending interaction persists on the session — survives page reloads
Streaming protocol

12 typed Server-Sent Events.

Every event has a Zod schema. The Studio client parses the same schemas the runtime emits — no string-fiddling, no implicit contracts. Activity events give the UI breadcrumbs between large gaps.

specialist
First event of every turn — banner with name + icon.
text_delta
Streaming token chunk from the LLM. Append-only.
tool_call
Tool invocation with id, name, and validated input.
tool_result
Server-side tool result. isError if applicable.
activity
UI breadcrumb — "Thinking…", "Running search_docs…", "Response ready".
file_processed
File ingestion succeeded — extracted text / image refs ready.
file_error
File ingestion failed with a typed reason code.
file_context_change
Files added/removed from the active context.
suggestion
Inline chip suggestions categorized for UX.
quality_floor
Spec/topology dropped below quality threshold — surfaced as actionable issues.
build_*
Per-agent BUILD progress: start · stage · compiled · enriched · error · diagnostics.
done
Turn ended. UI clears typing indicators, enables input.
Persistence

One source of truth, append-only history.

Five collections, all scoped by tenant. Nothing in process memory is treated as authoritative — every decision is journaled, every transition is atomic.

arch_sessions

Live session state, sliding-window messages[] capped at 200, pending interaction, build progress. Atomic transitions via findOneAndUpdate with precondition in filter.

arch_journal

Append-only typed events. Five record types: decision, consultation, mutation, validation, analysis. Status transitions only — content never mutates.

arch_spec_documents

Per-project markdown specification — the SSOT that survives long after the session is archived. Editable paths are explicitly enumerated.

arch_files (GridFS)

User-uploaded files referenced by content blocks. Server-only access; never bundled into client code.

arch_audit_log

Operational/security entries from AuditLogEmitter. Separate sink from the journal — security concerns isolated.

arch_project_memory

Cross-session facts about the project. In-project specialists pull from here. Tenant + project scoped.

FAQ

Questions that come up.

Does the LLM ever decide to advance phases?

No. The LLM can request advancement by calling the proceed_to_next_phase tool, but the coordinator validates the request against the source phase's exit criteria. If the criteria fail, the request is rejected and the LLM gets a tool result telling it what's still missing.

What happens if the user changes their mind mid-build?

The scope classifier categorizes the request. SMALL changes (rewording, field tweaks) stay in BUILD. LARGE changes (add/remove agent, redesign topology) trigger the only legal backward edge — BUILD → BLUEPRINT. diffTopologyAgainstBuildState preserves files and approvals for unchanged agents so the user doesn't redo work.

How does Arch avoid hallucinated ABL syntax?

Three layers. (1) The abl-construct-expert system prompt embeds the canonical ABL grammar reference. (2) The knowledge layer injects relevant construct cards per turn (gather, handoff, flow, hooks). (3) Generated ABL is validated and auto-fixed before the user sees it — the model gets a tool result with errors and re-tries.

What if a tool call hangs?

Every server-side tool runs with a 30-second timeout (configurable via toolTimeoutMs). On timeout, the executor emits an isError: true tool result and the multi-turn loop continues. Retriable errors (rate limit, transient 5xx) get up to 2 retries with 500 ms backoff before being surfaced as fatal.

Can two clients race a phase transition?

Transitions are atomic. SessionService.transition uses findOneAndUpdate with the precondition in the filter, not a read-then-write pattern. Two clients see one success and one well-typed InvalidTransitionError the API can return as a 409.

What survives a session resume after days?

Everything material. buildResumeSnapshot reconstructs the last specialist, pending interaction, last decision, and partial topology/build state from the session, journal, and spec document. The UI renders a "where we left off" banner — no transcript replay required.

How is tenant isolation enforced?

Every Mongoose model uses the shared tenant-isolation plugin from @agent-platform/database. Every query scopes by tenantId. Cross-tenant access returns 404 (not 403) by design — never leak the existence of resources you can't see.

Does Arch AI work without an LLM provider?

The package consumes an LLMStreamClient interface — it doesn't depend on any specific provider SDK. The Runtime injects the implementation. CREATE phase runs entirely without an LLM. Diagnostics is a pure rules engine.

Where is the source of truth — chat or spec?

The spec document. Chat is the input modality; the spec doc is the durable artifact the project lives by. Approved fields propagate from the session into the spec doc on every turn. The journal records every mutation with from/to + rationale so you can reconstruct the timeline.

Can I extend Arch with a new specialist?

Yes. Add a system prompt under prompts/specialists/, register the ID in SPECIALIST_IDS or IN_PROJECT_SPECIALIST_IDS, declare tools in PHASE_TOOL_MAP or IN_PROJECT_TOOLS, and add a display banner in the executor. The architecture is designed for additive extension.
